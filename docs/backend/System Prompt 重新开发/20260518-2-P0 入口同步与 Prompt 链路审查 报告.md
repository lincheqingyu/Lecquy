# P0 入口同步与 Prompt 链路审查

> 更新日期：2026-05-18
> 类型：审查报告
> 关联：[System Prompt 重新开发指导](./20260518-1-System%20Prompt%20重新开发指导%20技术规范.md)

## 1. P0 结论

P0 已完成入口规则同步和 prompt 构建链路审查。

当前后端代码事实是：`LAYERED_PROMPT=true` 时，运行时已经走“分层 system prompt”链路；但它仍然每轮重新构建 system，并且时间、runtime、mode、model、active skill 等动态字段仍可能进入 `system` 字段。P1 不能继续在现有 builder 内打补丁，应该在运行时外侧增加 `FrozenSystemSnapshot`，把现有 builder 包成一次性 snapshot 生成器，再由会话复用 snapshot 的 `systemText`。

## 2. 已同步入口

已同步：

- 根目录 `CLAUDE.md`
- 根目录 `AGENTS.md`
- `docs/README.md`
- `docs/backend/README.md`
- `docs/backend/System Prompt 重新开发/README.md`

入口规则已从旧的“稳定前缀 + 末位 `<env>` 动态 system”改为：

- 会话级 `FrozenSystemSnapshot`
- 同一 snapshot 生命周期内 API `system` 字段字节级稳定
- 可变内容走用户问题前 `<system_prompt_update>`
- update 必须是 `cumulative since snapshot`
- MemoryRecall 不进入 system
- Anthropic `cache_control` 只停在 Anthropic adapter

## 4. 当前主调用链

运行时入口在 `SessionRuntimeService`：

```text
executeRun
  -> manager.appendMessage(userMessage)              # 用户可见消息先入会话事件树
  -> buildContextMessages(...)
  -> executeSimple / executePlan
     -> buildRunSystemPrompt(...)
     -> runSimpleAgent / runManagerAgent / runWorkerAgent
```

`buildContextMessages` 当前有两条路径：

- `LAYERED_PROMPT=true`：调用 `buildMemoryRecallMessages`，MemoryRecall 以 `role=user` 的 synthetic message 进入上下文。
- 非 layered：调用 `buildMemoryRecallBlockLegacy`，再交给 `buildAugmentedContext` 兼容旧注入方式。

`buildRunSystemPrompt` 当前也有两条路径：

- 非 layered：调用 `buildSystemPromptLegacy`。
- `LAYERED_PROMPT=true`：调用 `loadStartupSlices`，再调用 `buildLayeredSystemPrompt`。

## 5. 当前 layered system 序列

`LAYERED_PROMPT=true` 时，当前 system prompt 是一条字符串，内部由 serializer 包 `LAYER` 标签：

```text
system:
<LAYER:system>
identity
tooling
tool-call-style
safety
skills index
workspace
documentation
time
managed AGENTS/TOOLS
runtime
extra-instructions
</LAYER>

<LAYER:mode name="simple|plan">
role directive
</LAYER>

<LAYER:startup>
<CAPABILITY>
SOUL
IDENTITY
USER PROFILE
MEMORY SUMMARY
</LAYER>

<LAYER:skill id="...">
active skill content
</LAYER>

<LAYER:user_preference>
USER.md preference
</LAYER>
```

该序列由下面的函数组合：

```text
prompt-serializer.buildLayeredSystemPrompt
  -> system-prompts.buildSystemLayerSlice
  -> system-prompts.buildModeLayerSlice
  -> system-prompts.buildStartupLayerSlice
  -> system-prompts.buildSkillLayerSlice
  -> system-prompts.buildUserPreferenceSlice
  -> prompt-serializer.serializeSystemPrompt
```

serializer 已经做了两个对 P1 有价值的约束：

- 按 `PromptLayer` 数值排序，不依赖调用方数组顺序。
- 拒绝 `MemoryRecall` / `LiveTurn` 动态层进入 `serializeSystemPrompt`。

## 6. 当前 legacy system 序列

`LAYERED_PROMPT` 未开启时，旧入口直接拼接为普通文本：

```text
identity
role directive
tooling
tool-call-style
safety
skills index
workspace
documentation
time
project context
runtime
extra-instructions
```

legacy 入口仍由 simple / manager / worker 的兼容 API 暴露：

- `buildSimpleSystemPrompt`
- `buildManagerPrompt`
- `buildWorkerPrompt`

这些 API 当前仍被 runner fallback 和测试引用，P1 不应删除。

## 7. 上下文文件读取事实

当前 `context-files.ts` 的关键事实：

- `.lecquy/SOUL.md`、`.lecquy/IDENTITY.md`、`.lecquy/USER.md`、`.lecquy/MEMORY.md` 是用户可编辑上下文。
- `.lecquy/AGENTS.md`、`.lecquy/TOOLS.md` 在当前代码中不会读取磁盘正文，而是由 `buildManagedAgentsContent` / `buildManagedToolsContent` 生成托管内容。
- `loadStartupSlices` 当前读取 `SOUL.md`、`IDENTITY.md`、`USER.md`，并通过 `loadMemorySummary` 读取 memory summary。
- worker 角色会清空 `SOUL` / `IDENTITY` / `USER`，只保留托管系统内容。
- `USER.md` 解析器只允许 profile / preference 双切片，preference 不得包含越权确认、绕过权限等行为指令。

这意味着 P1 的 `sourceHashes` 不能只记录文件 hash，还要记录“托管生成内容 hash”和“运行时 inventory hash”。

## 8. 当前 messages 序列样例

当前 layered 路径下，模型请求的语义顺序可以概括为：

```text
system: <layered system prompt string>

user: <LAYER:memory_recall>
...
</LAYER>

user: <augmented context from buildAugmentedContext>

user: <kira current message>
```

与最终目标的差距：

- MemoryRecall 当前标签是 `<LAYER:memory_recall>`，目标是 `<retrieved_memory priority="low">`。
- 当前还没有 `<system_prompt_update>`。
- 当前 API replay / visible transcript / augmentation 的分层语义尚未成为显式数据契约。

## 9. 与目标架构的主要差距

| 差距 | 当前表现 | P1 / 后续处理 |
|---|---|---|
| system 每轮可变 | `buildRunSystemPrompt` 每轮调用 builder，time/runtime/model/mode 可改变 system 字节 | P1 增加 snapshot 存储和复用 |
| 动态字段仍在 system | time、runtime、active skill 等仍在 system layer | P1 先冻结创建时状态；P2 再走 update |
| 没有 snapshot 类型 | 只有 `LayeredPromptResult.systemPrompt` 和 slice hash | P1 新增 `FrozenSystemSnapshot` 数据契约 |
| 没有 snapshot 持久化 | 会话事件树可存 custom entry，但尚未用于 system snapshot | P1 使用 `custom:system_prompt_snapshot` 或等价持久化 |
| MemoryRecall tag 未对齐 | 当前输出 `<LAYER:memory_recall>` | P3 调整为 `<retrieved_memory priority="low">` |
| `.lecquy/system-prompt/` 仍在用 | `prompt-module-files.ts` 仍优先读取旧模板目录 | P1 保持兼容，不删除；后续迁移到 BASE/TOOLS |
| AGENTS/TOOLS 磁盘文件未读 | 当前由 managed 内容生成 | P1 source hash 记录 generated managed content |

## 10. P1 影响

P1 的正确切入点不是重写 `system-prompts.ts`，而是：

1. 在 prompt core 内新增 snapshot 类型和 builder。
2. snapshot builder 第一版包住现有 `loadStartupSlices + buildLayeredSystemPrompt`。
3. 在 `SessionRuntimeService.buildRunSystemPrompt` 外侧增加 `ensureFrozenSystemSnapshot`。
4. 同一 `sessionId` + `snapshotId` 生命周期内直接返回 snapshot 的 `systemText`。
5. 把变更检测和 `<system_prompt_update>` 留给 P2。

验收口径：在不删除旧 builder、不重写 `.lecquy/system-prompt/` 的前提下，做到同一会话多轮请求的 `system` 字节不变。
