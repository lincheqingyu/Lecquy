## 0 结论

Lecquy 继续使用现有 `/api/v1/chat/ws` WebSocket 作为传输层，审批协议向 Codex App Server 学习，抽象成三事件请求/响应/回执：

1. **服务端发起请求**：`server_request`
2. **客户端回传决策**：`server_request_response`
3. **服务端确认请求已解决**：`server_request_resolved`
4. **工具项继续走现有生命周期**：`tool_call_start` / `tool_call_end`

核心原则：**WS 是 transport，审批是 request/response protocol**。本协议对齐 Claude Code / Codex 的「单进程单会话」心智，为后续桌面客户端迁移（Electron/Tauri）预留最小迁移成本 —— 整套协议在桌面端用 IPC 替换 WS 即可复用。

## 1 为什么不是继续用 `confirm_required`

`20260423-2` 提出的 `confirm_request` / `confirm_response` 相对现状前进一步，但仍偏「事件通知」，缺陷集中在：

- 名字绑定权限确认，难以复用到文件 diff、命令编辑、计划确认等后续场景。
- 缺少 `requestId / itemId / turnId` 生命周期关联。
- 没有显式的「请求已被服务端消费并解决」的回执事件，前端按钮易出现悬空态。
- hard deny 与 ask 共用同一事件通道（见 `20260423-1 §2.3` 遗留问题）。

Codex App Server 的启发：审批不是普通 stream delta，而是 **server-initiated request**。客户端必须明确返回一个 decision，服务端再用 resolved 事件关闭 UI 状态。

## 2 与 Codex App Server 的概念映射

| Codex 概念 | Lecquy 对应 | 说明 |
| --- | --- | --- |
| `threadId` | `sessionKey` / `sessionId` | Lecquy 已有会话标识，前端主要使用 `sessionKey` |
| `turnId` | `runId` | 一次用户请求对应一次运行 |
| `itemId` | `toolCallId` | 绑定到时间线上的一个工具项 |
| `serverRequest` | `server_request` | 服务端向客户端发起待决请求 |
| approval response | `server_request_response` | 客户端批准、拒绝或取消 |
| request resolved | `server_request_resolved` | 服务端确认该 request 已被消费 |
| command/file completed | `tool_call_end` | 实际工具执行结果仍走现有工具生命周期 |

Lecquy 不在 M1 改成完整 JSON-RPC 2.0 envelope，因为当前共享协议已经是 `{ event, payload }`。但新增事件的语义要接近 JSON-RPC：**每个 request 有唯一 id、只能被解决一次、响应必须关联原 request**。

### 3.7 会话管理事件：`session_subscribe`

```ts
export interface SessionSubscribePayload {
  readonly sessionKey: string
}

export const sessionSubscribeSchema = z.object({
  sessionKey: z.string().min(1),
})
```

**为什么需要单列一个事件**：

- WS 连接建立后默认没有会话上下文。`run_start` / `run_resume` 都是会触发 runtime 写动作的事件（启动或续跑 agent），不适合作为"纯重连恢复 UI"的入口。
- 原规范依赖 `run_start` 路径内嵌的 `session_restored` 做恢复，前提是客户端会再发一次 run。但"刷新页面 / 断线重连回到进行中会话"的场景不应触发新的 run，需要一个**只绑定、不写入**的专用入口。
- `session_subscribe` 由此独立出来：只做 projection 查询 + notifier 绑定 + pending `server_request` 重放，**不**产生任何 runtime 副作用。

**服务端处理合同**（见 §4.8）：

1. 按 `sessionKey` 在 `SessionRuntimeService.getProjection` 查询投影。
2. 会话不存在 → 返回 `error { code: 'SESSION_NOT_FOUND' }`，前端应清除本地 `sessionKey` 并跳转空态。
3. 会话存在 → `bindNotifier` 把该 WS 加入 `notifiers.get(sessionKey)` 的 Set。
4. 发 `session_bound` + `session_restored`（沿用现有生命周期事件）。
5. 调 `getPendingServerRequests(sessionKey)` 重放所有未结算的 `server_request`。

**安全边界**：`session_subscribe` 是 read-only 事件。它不能启动 run、不能取消 run、不能解决 pending request，后端实现必须严格保证不触发任何写动作。

## 4 工具审批生命周期

### 4.1 ask / Confirm

```text
agent loop
  → tool permission check returns Confirm
  → backend broker.create() 生成 requestId/itemId
  → WS server_request(kind=tool_approval) 广播给该 sessionKey 所有活跃 WS
  → agent loop await broker promise（挂起）
  → frontend 输入区替换为审批卡
  → frontend 发 server_request_response
  → backend 校验 sessionKey + runId + requestId + itemId
  → WS server_request_resolved（广播给所有 WS）
  → broker resolve/reject agent loop 的 promise
  → agent loop 继续：执行工具 or 注入拒绝观察
  → WS tool_call_end
```

### 4.2 accept

`decision = accept`：不新增规则；只恢复当前 pending tool call；工具执行完成后走 `tool_call_end(status=success | error)`。

### 4.3 accept_for_session（M2）

- 使用服务端生成的 `ruleSuggestion`（见 §9 M2）。
- 调用 `PermissionManager.applyUpdate` 注入 `source: 'session'` 规则。
- 恢复当前工具调用。后续同规则命中不再弹卡。

### 4.4 accept_for_project（M3）

- 调用 `applyUpdate` 后持久化到项目配置（`.claude/settings.json`）。
- 必须写审计日志。

### 4.5 decline

- 不执行工具。
- 发 `server_request_resolved(status=declined)`。
- 后端向 agent loop 注入拒绝观察文本（见 §4.7）。

### 4.6 hard deny（与 ask 协议分流）

hard deny 不发 `server_request`。处理方式：

- 直接发 `tool_call_end(status=error, errorMessage, errorDetail)`。
- `errorDetail.code = 'permission_denied'`。
- `errorDetail.ruleContent` 携带命中规则内容供前端展示。
- 前端渲染为只读拦截卡，**无任何审批按钮、无例外申请入口**。
- 向 agent loop 注入：`工具 {toolName} 已被安全策略阻止，无法执行。命中规则：{rule.content}。请换一种不触发该规则的方案。`

### 4.7 三类终态的 Agent 观察文案模板

保持简洁，不做 prompt engineering 过度优化。M1 固定中文模板：

| 终态 | 注入文案 |
|---|---|
| `decline` | `用户拒绝执行该操作：{toolName}。请询问用户下一步。` |
| `expired` | `权限审批超时，未执行 {toolName}。` |
| `cancelled`（run_cancel） | `用户已取消本次运行，未执行 {toolName}。` |
| hard deny | `工具 {toolName} 已被安全策略阻止。命中规则：{ruleContent}。请换一种不触发该规则的方案。` |

M4 再考虑 i18n 和按场景优化。

### 4.8 WS 重连与 pure-reconnect 流程

前端重连分两类场景，对应不同入口事件：

**场景 A：刷新页面 / 断线重连、不启动新 run**

```text
浏览器恢复连接
  → WS 重新握手 /api/v1/chat/ws
  → 客户端本地仍持有 sessionKey
  → 客户端 emit session_subscribe { sessionKey }
  → 后端 getProjection(sessionKey)
      ├─ 不存在 → error { code: 'SESSION_NOT_FOUND' } → 客户端清 sessionKey
      └─ 存在
          → bindNotifier(ws, sessionKey)
          → emit session_bound { sessionKey, sessionId, kind, channel, created: false }
          → emit session_restored { sessionKey, sessionId, status, runId, messageCount }
          → replayPendingServerRequests(sessionKey)  // 重发所有未结算 server_request
  → 前端按 requestId 去重、按最新 expiresAt 更新 pendingRequest
```

**场景 B：用户继续输入新一轮 / 启动新 run**

```text
客户端 emit run_start { route, sessionKey? }
  → 后端 resolveSession(route, sessionKey)
  → bindNotifier（复用与 A 相同的绑定逻辑）
  → emit session_bound + session_restored（如果 bound.restored=true）
  → replayPendingServerRequests（如果 restored）
  → runtime.startRun(...)  // 继续启动新 run
```

**不变式**：

- `session_subscribe` 永不触发 `startRun` / `resumeRun` / `cancelRun` / `resolveServerRequest`。
- `session_bound` / `session_restored` / pending `server_request` 三者的发射顺序在两条路径上一致。
- notifier 绑定使用扁平 `Set<NotifierFn>`（§5.3），多 Tab 并存时均能收到广播。
- pending request 的 `requestId` 是幂等键，前端收到重复 `server_request` 必须按 id 去重而不是叠加。

**为什么不复用 `run_resume`**：`run_resume` 会触发 runtime 的恢复执行逻辑（对已暂停 pause_required 状态的 run 下发恢复信号）。"纯刷新页面看既有会话"不应该重启任何 agent loop，`session_subscribe` 因此必须独立存在。

## 5 后端实现边界

### 5.1 新增 `confirmation-broker.ts`

新增文件：`backend/src/runtime/confirmation-broker.ts`

职责：

- 创建 pending request，生成 `requestId`（uuid v4）、`itemId`。
- 保存 `{ resolve, reject, abortController, expiresAt, sessionKey, runId }`。
- 按 `sessionKey + runId + requestId + itemId` 四元组校验响应。
- TTL **默认 10 分钟**，`setTimeout` per request，到期发 resolved(status='expired', source='timeout')。
- `cancelByRun(sessionKey, runId, source='run_cancel')` 取消该 run 下所有 pending。
- 为重连客户端重放 pending request（见 §5.4）。

**存储与调度**：

- **M1 存储**：进程内 `Map<requestId, PendingEntry>`，扁平结构。
- **进程重启**：所有 pending 全部丢失，不做持久化、不做重启广播。前端重连后 `session_restored` 里不包含已失效的请求，前端本地按 TTL 自然过期。
- **多进程**：M1 不支持（Lecquy 当前单进程）。
- **持久化**：M1 不承诺，M2+ 按实际需求评估。

### 5.2 `tool-permission.ts` 改造

当前 `PermissionTier.Confirm` 分支：

```ts
// 当前代码：
options.onEvent?.({ type: 'confirm_required', ..., description })
throw new Error(description)
```

改为：

```ts
// 目标代码：
const outcome = await broker.create({
  sessionKey, runId, toolName, args, displayCommand,
  matchedRule, mode,
})
if (outcome.status === 'accepted') {
  // 继续执行（return 后工具调用继续）
} else {
  // decline / cancelled / expired：抛可识别错误，由上层转为观察文本
  throw new PermissionDeclinedError(outcome.status, outcome.message)
}
```

**hard deny 分支同步迁移**：

```ts
// 当前（有问题）：
options.onEvent?.({ type: 'confirm_required', ... })
throw new Error(description)

// 目标：
throw new HardDenyError({ code: 'permission_denied', ruleContent })
```

上层 `session-runtime-service.ts` 捕获 `HardDenyError` 后转成 `tool_call_end(status=error, errorDetail={ code:'permission_denied', ruleContent })`。**彻底废除** hardDeny 走 `confirm_required` 的旧路径。

### 5.3 `session-runtime-service.ts`

新增 API：

```ts
resolveServerRequest(payload: ServerRequestResponsePayload): {
  ok: boolean
  reason?: 'not_found' | 'already_resolved' | 'sessionKey_mismatch'
         | 'runId_mismatch' | 'itemId_mismatch'
}

getPendingServerRequests(sessionKey: string): ServerRequestPayload[]
```

**notifiers 结构调整**：

```ts
// 当前：
private readonly notifiers = new Map<string, (event, payload) => void>()

// 目标：
private readonly notifiers = new Map<string, Set<NotifierFn>>()
```

扁平 Set，无 peerId，无 ownership。所有注册到该 sessionKey 的 WS 都会收到 `server_request`（广播）；首个有效应答获胜，其余返回 `already_resolved`。

**cancelRun 联动（时序关键）**：

```ts
cancelRun(sessionKey, runId?) {
  // 1. 先级联取消 broker
  this.broker.cancelByRun(sessionKey, runId ?? active.runId, 'run_cancel')
  // 2. 再 abort
  active.abortController.abort()
}
```

顺序不能反 —— 先 cancel broker 让 pending 条目干净地发出 resolved(status='cancelled')，再 abort 让 agent loop 的 `await broker` 被 AbortSignal 唤醒。

### 5.4 `ws/chat-ws.ts`

新增客户端事件路由：

```ts
if (event === 'server_request_response') {
  const parsed = serverRequestResponseSchema.safeParse(payload ?? {})
  if (!parsed.success) return sendError('invalid_payload')
  const result = runtime.resolveServerRequest(parsed.data)
  // result.ok=false 时不回包（前端依赖 resolved 广播判断状态）
}
```

**重连重放**：`session_restored` 发送完毕后，调 `getPendingServerRequests(sessionKey)` 把仍 pending 的请求重新广播给该 WS。前端按 `requestId` 去重。

### 5.5 审计日志（JSONL）

新增 `backend/src/runtime/approval-audit.ts`，写入路径 `<workspace>/.claude/audit.jsonl`，每行一条 JSON：

```ts
interface ApprovalAuditEntry {
  ts: number          // Unix ms
  runId: string
  itemId: string
  toolName: string
  displayCommand?: string
  decision: 'accept' | 'accept_for_session' | 'accept_for_project'
          | 'decline' | 'cancel' | 'expired' | 'run_cancel' | 'hard_deny'
  ruleContent?: string
}
```

**M1 范围**：

- 只记录终态，不记录「审批发起」事件。
- 无 UI 查看器（M3 做）。
- 无 OTel 集成（M4 再评估）。
- 文件直接 append，无 rotation。

## 6 前端实现边界

### 6.1 `useChat` 状态

加单一字段：

```ts
const [pendingRequest, setPendingRequest] = useState<ServerRequestPayload | null>(null)
```

处理规则：

- 收 `server_request` → `setPendingRequest(payload)`
- 收 `server_request_resolved` → `setPendingRequest(null)`
- 收 `tool_call_end` → 走现有 tool item 状态更新路径（不需改动）

重连处理：收到第二份相同 `requestId` 的 `server_request`，若本地已有则忽略，只用最新 `expiresAt` 更新。

### 6.2 审批卡 UI 定位（输入框替换）

**M1 只做一种视觉方案**：

- 当 `pendingRequest !== null` 时，**底部消息输入区**整体替换为 `<ApprovalCard>`。
- `<ApprovalCard>` 展示：
  - 标题：`需要批准：{toolName}`
  - 命令预览：`{displayCommand}`（代码块）
  - 权限模式：`{mode}`
  - 两个按钮：`批准此次` / `拒绝`
- **时间线不单独渲染审批状态条**。`tool_call_start` 的 description 字段承担"等待审批"的视觉表达，`tool_call_end` 承担结果表达。

**M1 按钮**：仅 `accept` 与 `decline`。

**M2 按钮扩展**：增加 `accept_for_session`（仅 UI 按钮，规则由服务端生成；见 §9 M2）。

**M3 按钮扩展**：增加 `accept_for_project`。

### 6.3 Hard deny 只读卡

Hard deny 走 `tool_call_end(status=error, errorDetail.code='permission_denied')`，前端：

- 渲染样式与普通 tool error 区分：🛡 盾牌图标 + 标题「已被安全策略阻止」
- 显示 `errorDetail.ruleContent`
- 无任何按钮（不提供"申请例外"/"重试"入口）
- 底部固定文案：`如需放行，请修改 .claude/settings.json 中的权限规则。`

### 6.4 并发约束

由于后端 `activeRuns` 本就禁止同 sessionKey 并发 run，同一时刻同一会话最多 1 个 pending `server_request`，前端**不需要**审批队列 UI。

## 7 安全规则

1. **requestId 仅服务端生成**（uuid v4），前端不得伪造。
2. **四元组校验**：`server_request_response` 必须携带 `sessionKey + runId + requestId + itemId`，broker 四项任一不匹配即拒绝并审计。
3. **幂等**：已 resolved 的 request 再收响应，返回 `ok:false, reason='already_resolved'`，不重复执行工具。
4. **规则内容服务端生成**（M2）：`accept_for_session` / `accept_for_project` 的 `rule.content` 必须与服务端 `ruleSuggestion.content` 完全相等；不等直接拒绝。前端**不得**在 UI 里让用户编辑 rule content（详见 §9 M2）。
5. **TTL 到期自动 expired**（不是 declined），source='timeout'，发 resolved 并向 agent 注入超时观察。
6. **run_cancel 级联**：`cancelRun` 先取消 broker pending（source='run_cancel'），再 abort agent loop。
7. **hard deny 不可恢复**：永远不允许通过 `server_request_response` 放行 hard deny。hard deny 不走 server_request 通道。

## 8 与现有协议的兼容性

当前 `shared/src/ws-events.ts` 已有事件：`run_start` / `run_resume` / `run_cancel` / `run_state` / `tool_call_start` / `tool_call_end` / `pause_requested`。

本规范在审批生命周期新增三类事件（§3.1 / §3.2），并新增一个会话管理事件 `session_subscribe`（§3.7），不修改已有生命周期。

**为什么不复用 `pause_requested`**：

- `pause_requested` 是 **run 级**挂起等待单一字符串输入（人工输入 / Plan Resume），生命周期绑 `pauseId`。
- `server_request` 是 **item 级**多选决策，生命周期绑 `requestId + itemId`，需要支持规则注入、幂等响应、TTL 自动撤回。
- 强合会让 `PausePacket` 类型膨胀且 `resolvedAt` 字段语义重载。

两者并存：`pause_requested` 保留给人工输入 / Plan Resume，`server_request` 服务于工具审批。

## 11 最终决策表（2026-04-24）

本次设计经过「第一轮 Codex 初稿 → Claude 审查 → 8 项决策征询 → 过度设计复盘 → 简化锁定」四轮迭代，最终锁定如下决策。本表供后续 review 追溯。

| ID | 主题 | 最终决策 | 废弃方案 |
|---|---|---|---|
| D1 | 多客户端所有权 | 扁平广播 + 首个应答获胜（无 peerId，无 ownership） | 严格所有者（peerId 绑定 WS 连接）—— 单用户场景过度设计 |
| D2 | TTL + 持久化 | 进程内 Map + setTimeout(10min) + 重启全清、不广播 | 60min TTL / M2 承诺持久化 —— YAGNI |
| D3 | 审计日志 | 6 字段 JSONL（`.claude/audit.jsonl`） | 13 字段 + OTel 预留 —— 过度占坑 |
| D4 | hard deny UI | 只读拦截卡 + 显示规则 + 无任何按钮 | 提供「申请例外」入口 —— 与硬拦截语义冲突 |
| D5 | Agent 观察文案 | 三类简洁中文模板（decline / expired / cancelled） | 复杂 prompt engineering —— M1 先观察再优化 |
| D6 | `rule?` 字段 M1 态度 | 类型保留、运行期**直接忽略**、不审计不警告 | 运行期警告 —— 为假想攻击面做防御 |
| D7 | 审批卡 UI | **仅输入区替换卡**，时间线不插状态条 | 输入区 + 时间线双状态 —— 双状态同步复杂度 |
| D8 | M2 规则建议生成 | displayCommand 确定性模板化（抄 Claude Code） | LLM 生成 + 哈希校验 + source 扩展字段 —— M2 重大过度设计 |
| D9 | WS 重连恢复入口 | 独立 `session_subscribe` 事件（只读绑定，零副作用） | 复用 `run_start`/`run_resume` —— 绑定语义与 run 启动语义耦合，纯刷新场景被迫触发 runtime 写动作 |

协议演进原则：**真正破坏性变更**走新 event name（如 `server_request_v2`），**不**预留 `SERVER_REQUEST_PROTOCOL_VERSION` 之类字段。

心智对齐：**单进程单会话**（对齐 Codex CLI / Claude Code CLI）。桌面客户端迁移路径上，本协议 IPC 化替换 WS 即可复用，无迁移成本。
