## 1 设计哲学

### 1.1 问题域

在任何让 AI 代理执行真实世界动作的系统中，必须回答三个问题：

1. **什么可以做？** — 规则体系：来源、优先级、匹配策略
2. **谁来决定？** — 决策引擎：规则优先，模式兜底，分类器托底
3. **出了事怎么追溯？** — 审计：完整记录每一次决策

Claude Code 用 9400+ 行代码回答了这三个问题，核心洞见有三条：

- **声明式规则 + 命令式模式** —— 规则是静态配置，模式是运行时开关，二者正交
- **多源合流 + 稳定排序** —— 不同来源的规则用固定优先级合并，保证可预测
- **深度防御** —— 即便配置全空，内置规则 + 分类器 + 文件保护仍能挡住大多数攻击

### 1.2 Lecquy 的定位调整

Lecquy 的执行环境与 Claude Code 并不完全相同：

| 维度 | Claude Code | Lecquy |
| --- | --- | --- |
| 运行形态 | 终端 CLI | Cowork / API 驱动的后端 |
| 工具集合 | 20+ 种原生工具 | 6 种核心工具（bash / edit / write / read / delete / search） |
| 用户交互 | 命令行同步确认 | 事件订阅 + 前端 UI 异步确认 |
| 配置渠道 | CLI flag + settings | CLI flag + session + 项目 + 用户 + 内置 |
| 现有层次 | 无历史包袱 | 已有 `PermissionTier`（Auto / Preamble / Confirm）|

因此 Lecquy 的实现目标不是 1:1 复刻，而是：

1. **保留 Claude Code 的决策骨架**（五种模式、五级来源、三种行为）
2. **削减复杂度**（目标 3000 行以内，实际 2757 行）
3. **与 `PermissionTier` 共存**（新引擎 → `tier-bridge.ts` → 旧接口）
4. **为 2.0 AI 分类器预留接口**（`CommandClassifier` 接口 + 规则驱动兜底）

## 2 核心模块深度拆解

### 2.1 `permissions.ts`（1486 行）

Claude Code 的决策引擎心脏。核心逻辑可抽象为伪代码：

```
function checkPermission(tool, args, rules, mode, preflights):
    // A. 工具特定前置
    pre = runPreflights(tool, args)
    if pre.deny: return deny

    // B. 规则匹配（按来源优先级 + 精确 > 通配符排序）
    matched = findFirstMatch(rules)
    if matched.deny: return deny

    // C. mode 决策矩阵
    if mode == bypass: return allow
    if mode == plan:   return plan
    if needsAsk(pre, matched):
        if mode == acceptEdits and isEditTool(tool): return allow
        if mode == dontAsk:                         return deny
        return ask
    return allow
```

**Opus 的核心观察**：

1. **`preflight` 与规则是叠加的** —— 即使规则 `allow`，如果 `preflight` 说 `deny`，最终仍 `deny`；Lecquy 保留这一点（见 `checker.ts` 的 `runPreCheck`）。
2. **三种行为的不对称性** —— `allow` 只能被 `preflight=deny` 否决，`deny` 不可被覆盖，`ask` 则由 mode 决定落点。
3. **"第一条命中即返回"依赖排序稳定性** —— 这要求 `sortRulesByPriority` 必须是稳定排序，且同来源内按"精确 → 通配符"排序。

**Lecquy 的简化**：

- 规则扁平化为 `{ source, behavior, toolName, content? }` 四元组，砍掉 Claude Code 的 `ruleValue / applicableTools / nestedRules` 等装饰字段。
- 规则排序只做两层：来源优先级（`RULE_SOURCE_PRIORITY`）+ 精确/通配符（`toolName === '*'` 排后）。
- `preflight` 只保留 bash 和文件类，其他工具走规则 + mode。

### 2.2 `filesystem.ts`（1777 行）

Claude Code 的文件保护模块。最核心的是四个概念：

1. **危险文件黑名单**（shell rc、ssh key、.env、credentials...）
2. **危险目录黑名单**（.git、.ssh、node_modules...）
3. **系统路径前缀**（/etc、/usr、C:/Windows...）
4. **符号链接逃逸**（lstat + realpath + 工作区边界检查）

Claude Code 还做了**路径遍历**（`..`）、**UNC 攻击**（`\\?\`、`\\.\`）、**百分号编码**（`%2e%2e`）的检测。

**Opus 的核心观察**：

- 这些检查不是互斥的，必须**组合**才能防住真实攻击。
- Windows 路径与 POSIX 路径混用会出惊喜（`C:\Users\u\.env` 上 `path.basename` 在 POSIX 下返回全串）。

**Lecquy 的实现**：

文件保护被拆成两个模块，职责单一：

- `dangerous-paths.ts`（172 行）—— 常量清单 + `isDangerousFile / isInDangerousDirectory / isProtectedSystemPath / isDangerousPath`
- `path-validation.ts`（199 行）—— 跨平台归一化 + `containsPathTraversal / containsVulnerableUncPath / resolveWithinWorkspace / matchGlob`

两者叠加在 `file-operations.ts`（231 行）的 `canReadFile / canEditFile / canDeleteFile` 中：

```
baseValidate(filePath, workspaceDir):
    1. 路径遍历？→ deny
    2. UNC 攻击？→ deny
    3. 系统前缀？→ deny
    4. 危险目录？→ deny
    5. 在工作区外？→ deny
    6. 符号链接逃逸？→ deny
    7. 返回 allow / ask（危险文件 ask）
```

### 2.3 `yoloClassifier.ts`（1495 行）

Claude Code 里复杂度最高的模块，负责给任意 bash 命令打"安全/可疑/危险"三级标签。

**真实情况**：泄露源码里 `yoloClassifier` 大量调用一个叫 `classifierDecision` 的内部 AI 分类器（LLM-based）。对 Lecquy 而言，AI 分类不现实（成本、延迟、离线），所以必须回退到规则驱动。

**Opus 的核心观察**：

- **能力评估**：规则能覆盖约 70-80% 的真实危险命令；剩下的是语义级危险（比如 `cat /etc/shadow > /tmp/x` vs `cat > /tmp/x`），要靠 AI 或 sandbox。
- **规则坑点**：
  - `\brm\b` 会把 `myremove.sh` 当成 rm —— 必须用词边界
  - `curl ... | bash` 的 `|` 会被"拆复合命令"破坏 —— 必须**先在完整命令上扫 DENY，再拆**
  - `rm -rf /` 末尾是 `/` 不是单词字符 —— `\b` 在这里不成立，要用 `\s*$` 兜底
- **组合策略**：DENY 优先，ASK 兜底，allow 是默认。这与 `yoloClassifier` 中的 "block > warn > pass" 一致。

**Lecquy 的实现**（`bash-classifier.ts` 271 行）：

```
classifySync(command):
    if not command.trim():
        return allow

    // 先全量扫 DENY（跨子命令的危险模式）
    for rule in DENY_PATTERNS:
        if rule.pattern.test(command):
            return deny

    // 拆子命令
    subs = splitCompoundCommand(command)

    // 子命令再扫 DENY（如 "echo hi && rm -rf /"）
    for sub in subs:
        for rule in DENY_PATTERNS:
            if rule.pattern.test(sub):
                return deny

    // 子命令扫 ASK
    for sub in subs:
        for rule in ASK_PATTERNS:
            if rule.pattern.test(sub):
                return ask

    return allow
```

**规则内容**（节选）：

- **DENY_PATTERNS**（10 条）：`rm_root`、`rm_rf_wildcard_home`、`dd_to_device`、`mkfs`、`fdisk_destructive`、`fork_bomb_bash`、`fork_bomb_python`、`cat_random_to_device`、`curl_pipe_shell`、`rm_etc`
- **ASK_PATTERNS**（25+ 条）：`sudo`、`su`、`rm`、`rmdir`、`find_delete`、`curl`、`wget`、`ssh`、`scp`、`rsync_remote`、`pkg_install`、`pip_install`、`npm_global`、`npm_install`、`systemctl`、`chmod`、`chown`、`kill -9`、`git_push_force`、`git_reset_hard`、`sql_drop`、`sql_truncate`、`sql_delete_from`、`crontab`、`iptables`...

### 2.4 `classifierDecision.ts`

Claude Code 里把 yolo 分类器结果翻译为最终 permission 决策的小模块。真实逻辑非常薄：

```
decide(classifierResult, mode, rules):
    if classifierResult.level == 'deny': return deny
    if rules.allow matches: return allow
    if classifierResult.level == 'ask': return ask
    return allow
```

Lecquy 把这一层折叠进了 `checker.ts` 的 `resolveByMode`，没有单独抽模块 —— 保持最少层级。

## 3 Lecquy 精简策略

| Claude Code | Lecquy | 理由 |
| --- | --- | --- |
| 9400+ 行 | 2757 行 | 删冗余、合并同类、工具更少 |
| 10+ 模式 | 5 模式 | 内部 `auto/bubble` 外部不可见，合并掉 |
| 规则字段 20+ | 5 字段 | `ruleValue/applicableTools/nestedRules` 实战用不到 |
| AI yolo 分类器 | 规则驱动 + 可插拔 | MVP 不上 AI，接口留好 |
| `denialTracking` 独立模块 | 并入审计 | 拒绝只是决策之一，无需特殊化 |
| 规则合并算法复杂 | 扁平数组 + 稳定排序 | 足以表达所有用例 |

**总体原则**：保留决策骨架和核心数据结构，砍掉所有为 CLI 交互设计的冗余字段和 AI 专用模块。

### 5.2 决策矩阵（完整版）

| preCheck ＼ mode | default | dontAsk | plan | acceptEdits | bypassPermissions |
| --- | --- | --- | --- | --- | --- |
| `deny` | **deny** | **deny** | plan（不执行） | **deny** | **deny** |
| `ask` | ask | **deny** | plan | 编辑/读取→allow，其他 ask | **allow** |
| `allow` / `null` + 规则 ask | ask | **deny** | plan | 编辑/读取→allow，其他 ask | **allow** |
| `allow` / `null` + 无规则 | allow | allow | plan | allow | allow |
| 命中 `allow` 规则 | allow（除非 preCheck=deny） | allow | plan | allow | allow |
| 命中 `deny` 规则 | **deny** | **deny** | plan | **deny** | **deny** |

> 关键约束：
> 1. `preCheck=deny` 任何 mode 下都 deny（不可覆盖）
> 2. `bypassPermissions` 仍然 respect `deny`（规则 deny 和分类器 deny）——这是安全优先的刻意选择
> 3. `plan` 模式总是返回 `plan`，不执行任何操作

### 5.3 规则优先级

```
命中优先级（同一工具下）：
  cliArg(40) ─ session(30) ─ projectSettings(20) ─ userSettings(10) ─ builtin(0)

同来源内：
  toolName 精确 ─ toolName = '*'
```

排序由 `sortRulesByPriority` 完成（稳定排序），匹配时取第一条。

## 6 关键实现细节

### 6.1 路径遍历的"假阳性 + 假阴性"

**假阳性**：`a..b.txt` 含有 `..`，但不是遍历。
**假阴性**：`foo/./../bar` 经过规范化才露馅。

`path-validation.ts` 的 `containsPathTraversal` 做三步：

1. 先统一斜杠，解 URL 编码（`%2e%2e` → `..`）
2. 按 `/` 分段，只有**整段**等于 `..` 才算
3. 再用 `path.normalize` 看是否跳出了工作区

### 6.2 符号链接逃逸

`file-operations.ts` 的 `checkSymlinkEscape` 逻辑：

```
if !lstat(path).isSymbolicLink(): return safe
target = realpath(path)
if !target.startsWith(workspaceDir): return deny
```

降级处理：如果 `lstat` 报 ENOENT（尚未创建的文件），视为安全（因为没有链接可言）。如果 `realpath` 抛错（断链），视为 ask。

### 6.3 沙箱环境变量白名单

`sandbox-adapter.ts` 默认只透传：
`PATH / HOME / USER / USERNAME / SHELL / TERM / LANG / LC_ALL / LC_CTYPE / TMPDIR / TEMP / TMP / TZ`

**不**透传：
`DB_PASSWORD / API_KEY / GITHUB_TOKEN / AWS_* / ANTHROPIC_API_KEY / OPENAI_API_KEY ...`

调用方可通过 `options.env` 显式注入特定变量。

### 6.4 审计的非阻塞写入

`manager.check` 调用审计时用 `void this.sink.write(record).catch(() => {})`：

- 审计不拖慢主路径
- 审计失败不反推给调用方
- 用于合规审计场景时建议叠加 `JsonFileAuditSink`（磁盘）+ 告警

### 6.5 规则更新的不可变性

`updater.ts` 的 `applyUpdate` 是**纯函数**：输入旧数组，输出新数组，不修改原数组。这让 `manager` 可以安全地在订阅回调里 diff：

```
manager.subscribe((ev) => {
    if (ev.type == 'rulesChanged') {
        const diff = computeDiff(prev, ev.rules)
        ui.render(diff)
    }
})
```

## 8 与现有系统的兼容桥

Lecquy 现有 `backend/src/agent/tool-permission.ts` 使用 `PermissionTier` 三档枚举包装工具。Opus 采取**非侵入集成**：

```
新引擎 (PermissionDecision)                旧接口 (PermissionTier)
────────────────────────                  ──────────────────────
allow                          ─────▶      Auto
ask                            ─────▶      Confirm
deny                           ─────▶      Confirm + hardDeny 标记
plan                           ─────▶      Confirm  (仅预览)
acceptEdits 模式的编辑/读取     ─────▶      Preamble
```

翻译器：`tier-bridge.ts` 的 `bridgeResult(result, { mode, toolName })`。

集成建议见 `20260420-5-最终集成和运维指南.md`。关键步骤：

1. `createPermissionAwareTools` 内部注入 `PermissionManager.check`
2. 根据 `bridgeResult` 决定 Auto/Preamble/Confirm
3. Confirm + hardDeny 时直接拒绝，不弹确认

## 9 后续演进路线

### 9.1 2.0 — AI 分类器

现有 `RuleBasedBashClassifier` 走规则匹配。接入 AI 分类器只需实现 `CommandClassifier` 接口：

```ts
class AIBasedBashClassifier implements CommandClassifier {
  readonly name = 'ai-classifier'
  async classify(input) {
    const response = await this.llm.send(PROMPT + input.command)
    return parseResponse(response)
  }
}

manager.setClassifier(new AIBasedBashClassifier())
```

建议：规则 → AI 组合（`RuleFirstThenAI`），先用规则过滤明显的 deny，再交给 AI 处理 ambiguous 区间。

### 9.2 2.1 — MCP 工具动态注册

现有 `toolName` 列表是硬编码的。当 MCP 生态接入后，可用如下策略：

- 新增字段 `PermissionRule.mcpServer?: string` 指定哪个 MCP 服务的工具
- `preflight` 根据 `mcpServer.metadata.capability` 自动推导 `destructive=true/false`

### 9.3 2.2 — 分布式审计

`JsonFileAuditSink` 适合单机。上云后可：

- 新增 `HttpAuditSink` 推 OTel/Grafana Loki
- `CompositeAuditSink` 串联 InMemory + Http
- 保证非阻塞：失败不回传

### 9.4 2.3 — 策略即代码

当前规则是 JSON 配置。可演进到：

- OPA / Rego 策略（运行时 eval）
- TypeScript 谓词（编译时类型保证）
- 混合：简单规则走 JSON，复杂条件走代码
