## 1 集成路径选择

新旧系统有三种集成策略，按风险从低到高：

### 策略 A：旁路审计（零风险）

仅启用 `PermissionManager` 的 `check`，**不改**现有决策流程。每次工具调用并行地调用一次新引擎，把结果写入审计日志。先跑 1-2 周，对比新旧决策差异，确认新引擎行为可预测。

```ts
const mgr = await PermissionManager.create({ workspaceDir })
// 现有 classifyToolPermission 逻辑保持不变
const tier = classifyToolPermission(tool.name, args, role, workspaceDir)
// 旁路调用新引擎，只记录不干预
void mgr.check({ toolName: tool.name, args, workspaceDir, role })
```

**适用**：已有生产流量、不能接受抖动的环境。

### 策略 B：双引擎 + 降级（推荐）

新旧引擎并行决策，取**更严格**的结果。某一侧抛错时降级到另一侧。

```ts
const legacyTier = classifyToolPermission(...)
const newResult = await mgr.check(...)
const newTier = bridgeResult(newResult).tier

// 取更严格：Confirm > Preamble > Auto
const finalTier = mostRestrictive(legacyTier, newTier)
```

**适用**：正式上线期，既要迁移又要保险。

### 策略 C：新引擎为主（终态）

`classifyToolPermission` 成为 `PermissionManager.check` + `bridgeResult` 的薄包装。老的 `CONFIRM_PATTERNS / PREAMBLE_PATTERNS` 迁移到 `bash-classifier.ts`，`AUTO_TOOLS` 合入 `BUILTIN_RULES`。

**适用**：双引擎并行跑 2-4 周后。

## 2 与 `tool-permission.ts` 的分步集成

### 2.1 当前架构回顾

```
AgentTool.execute
      │
      ▼
createPermissionAwareTools (wrap)
      │
      ▼
classifyToolPermission → PermissionTier
      │
      ├── Auto     ─ 直接执行
      ├── Preamble ─ 发 preamble 事件 + 执行
      └── Confirm  ─ 发 confirm_required 事件 + 抛错（由上层恢复）
```

### 2.2 目标架构（策略 C）

```
AgentTool.execute
      │
      ▼
createPermissionAwareTools (wrap)
      │
      ▼
PermissionManager.check(context)
      │
      ▼
bridgeResult(result, {mode, toolName}) → BridgedTier
      │
      ├── hardDeny    ─ 直接抛 "操作被拒绝"，不发 confirm
      ├── Auto        ─ 直接执行（可能触发审计）
      ├── Preamble    ─ 发 preamble 事件 + 执行
      └── Confirm     ─ 发 confirm_required 事件 + 抛错
```

### 2.3 分步落地

**第 1 步：引入 Manager 作为单例**

在现有启动入口（例如 `backend/src/agent/bootstrap.ts` 或 `runtime` 初始化处）创建 Manager：

```ts
import { PermissionManager } from '../runtime/permissions/index.js'

export const permissionManager = await PermissionManager.create({
  workspaceDir: process.env.LECQUY_WORKSPACE ?? process.cwd(),
  initialMode: (process.env.LECQUY_PERMISSION_MODE as PermissionMode) ?? 'default',
})
```

**第 2 步：改造 `createPermissionAwareTools` 的签名**

新增可选 `manager` 参数，保持向后兼容：

```ts
export function createPermissionAwareTools(
  tools: readonly AgentTool<any>[],
  options: {
    role: AgentRole
    workspaceDir: string
    enabled: boolean
    onEvent?: (event: ToolPermissionEvent) => void
    manager?: PermissionManager  // ← 新增
  },
): AgentTool<any>[] { ... }
```

**第 3 步：在执行包装里双引擎决策**

```ts
return tools.map((tool) => ({
  ...tool,
  execute: async (toolCallId, params, signal, onUpdate) => {
    const args = (params ?? {}) as Record<string, unknown>

    // 新引擎
    let bridged: BridgedTier | null = null
    if (options.manager) {
      const result = await options.manager.check({
        toolName: tool.name,
        args,
        workspaceDir: options.workspaceDir,
        role: options.role,
      })
      bridged = bridgeResult(result)

      // 硬拒绝：直接短路
      if (bridged.hardDeny) {
        options.onEvent?.({
          type: 'confirm_required',
          toolCallId,
          toolName: tool.name,
          args,
          description: bridged.description,
        })
        throw new Error(bridged.description)
      }
    }

    // 旧引擎（降级兜底）
    const legacyTier = classifyToolPermission(
      tool.name, args, options.role, options.workspaceDir,
    )

    // 取更严格
    const tier = bridged
      ? mostRestrictive(bridged.tier, legacyTier)
      : legacyTier

    // 原有分派逻辑
    if (tier === PermissionTier.Preamble) { ... }
    if (tier === PermissionTier.Confirm) { ... }

    return await tool.execute(toolCallId, params, signal, onUpdate)
  },
}))
```

**第 4 步：迁移老规则到配置**

把 `CONFIRM_PATTERNS / PREAMBLE_PATTERNS / AUTO_TOOLS` 用配置文件或 `BUILTIN_RULES` 表达：

```json
// .lecquy/permissions.json
{
  "version": "1.0",
  "defaultMode": "default",
  "rules": [
    { "behavior": "ask", "toolName": "bash", "content": "install" },
    { "behavior": "ask", "toolName": "bash", "content": "deploy" },
    { "behavior": "allow", "toolName": "read_file" }
  ]
}
```

**第 5 步：并行观察 → 切主**

旁路运行 2 周，比对审计日志。确认新引擎的 deny/ask 与老引擎一致后，把 `mostRestrictive` 换成 `bridged.tier`，老引擎仅作回退兜底。

### 2.4 `AgentRole` 的处理

现有 `isManagerAllowed / isWorkerAllowed` 按角色放行。新引擎的推荐接入：

- `role` 字段已在 `PermissionCheckContext` 中
- 在内置规则或配置里用 `role` 区分，例如：
  ```json
  { "behavior": "deny", "toolName": "sessions_spawn", "content": "role:worker" }
  ```
  （需要扩展 `contentMatches` 支持 `role:` 前缀；留给 2.1 版本）

**简化做法**：`isManagerAllowed / isWorkerAllowed` 作为"更前置的门卫"保留，不进 Manager。

## 3 配置文件规范

### 3.1 文件位置

| 来源 | 路径 |
| --- | --- |
| `projectSettings` | `<workspaceDir>/.lecquy/permissions.json` |
| `userSettings` | `~/.lecquy/permissions.json` |
| `session` | 内存，不持久化 |
| `cliArg` | 通过 `loadOptions.cliRules` 注入 |
| `builtin` | 代码里的 `BUILTIN_RULES` |

### 3.2 JSON Schema

```json
{
  "version": "1.0",
  "defaultMode": "default",
  "rules": [
    {
      "behavior": "allow | deny | ask",
      "toolName": "bash | edit_file | read_file | ... | *",
      "content": "optional string",
      "description": "optional human-readable text"
    }
  ]
}
```

### 3.3 合法字段

- **`version`**：当前只接受 `"1.0"`，未来破坏性变更会升到 2.0
- **`defaultMode`**：`default | dontAsk | plan | acceptEdits | bypassPermissions`
- **`rules[].behavior`**：`allow | deny | ask`
- **`rules[].toolName`**：具体工具名或 `*`
- **`rules[].content`**：
  - `bash` 工具：前缀匹配（支持 `*` 和 `?` 通配符）
  - 文件类工具：路径 glob（`**/*.env`、`src/**/*.ts`）
- **`rules[].description`**：可选，UI 展示时用

### 3.4 示例配置

**项目级（严格）**：

```json
{
  "version": "1.0",
  "defaultMode": "default",
  "rules": [
    { "behavior": "deny",  "toolName": "bash",      "content": "rm " },
    { "behavior": "deny",  "toolName": "edit_file", "content": "**/.env*" },
    { "behavior": "deny",  "toolName": "edit_file", "content": "**/secrets/**" },
    { "behavior": "allow", "toolName": "bash",      "content": "git status" },
    { "behavior": "allow", "toolName": "bash",      "content": "npm test" },
    { "behavior": "ask",   "toolName": "bash",      "content": "npm install" }
  ]
}
```

**用户级（宽松）**：

```json
{
  "version": "1.0",
  "defaultMode": "default",
  "rules": [
    { "behavior": "allow", "toolName": "bash", "content": "ls " },
    { "behavior": "allow", "toolName": "bash", "content": "cat " },
    { "behavior": "allow", "toolName": "bash", "content": "grep " }
  ]
}
```

### 3.5 优先级实战

若 projectSettings 有 `{ bash deny rm }` 且 userSettings 有 `{ bash allow * }`：

- 来源优先级：`projectSettings(20) > userSettings(10)`
- 结果：执行 `rm foo` → deny；执行 `ls` → allow

## 4 规则维护 SOP

### 4.1 新增规则

**场景**：发现 AI 调用 `sudo` 时未触发 ask。

1. 定位：看 `bash-classifier.ts` 的 `ASK_PATTERNS`。确认 `sudo` 已在列表。
2. 若未命中，说明正则写错或 `describe` 的情境里被 `splitCompoundCommand` 拆掉。
3. 增加测试：在 `__tests__/bash-classifier.test.ts` 里加一条 `sudo ls /root` 应返回 `ask`。
4. 修 regex，跑 `node --test dist/runtime/permissions/__tests__/bash-classifier.test.js`。
5. 提交 PR，描述模式和防 false-negative 的测试。

### 4.2 修改规则优先级

**绝对不要**随意调整 `RULE_SOURCE_PRIORITY`。这是用户契约的一部分：`cliArg > session > projectSettings > userSettings > builtin`。

如果产品要求"用户级规则覆盖项目级"，应该通过 `session` 注入（运行时覆盖），而不是改优先级常量。

### 4.3 添加新工具

**场景**：新增 `sql_execute` 工具。

1. 在 `BUILTIN_RULES` 加 `{ behavior: 'ask', toolName: 'sql_execute' }`（保守策略）。
2. 若是破坏性操作（DROP TABLE 等），在 `checker.ts::runPreCheck` 里加分支，调用 `bash-classifier` 的 SQL 模式。
3. 补测试：`checker.test.ts` 里新增 `sql_execute` 的各种模式。

### 4.4 处理 false-positive

**场景**：`rm .git/index.lock` 被 deny 了（因为 `.git` 在 DANGEROUS_DIRECTORIES）。

1. 先判断："用户真的该做这事吗？" —— `.git/index.lock` 是 git 的内部锁文件，删除它等同于说"我知道 git 在跑，要强行清理"。这是 ask 合理场景。
2. 降级方案：把规则从 deny 改成 ask，并加入 `description` 解释。
3. 不要删黑名单；宁可多个 ask 也不放过一个 deny。

### 4.5 规则屏蔽检测

`loader.ts::detectShadowedRules` 会检查被高优先级规则屏蔽的低优先级规则。集成到 CI：

```ts
const { rules, shadowed } = await loadPermissionRules({ workspaceDir })
if (shadowed.length > 0) {
  console.warn('以下规则被屏蔽：')
  for (const s of shadowed) console.warn('  -', s)
}
```

## 5 审计与告警

### 5.1 审计后端选择

| 场景 | 推荐后端 |
| --- | --- |
| 单元测试 | `InMemoryAuditSink(100)` |
| 本地开发 | `InMemoryAuditSink(1000)` |
| 生产单机 | `CompositeAuditSink([InMemory, JsonFile])` |
| 生产多机 | `CompositeAuditSink([InMemory, HttpSink])`（HttpSink 需自行实现） |
| 合规审计 | 磁盘 + 远程 + 日志聚合三写 |

### 5.2 JSON Lines 磁盘审计

默认路径：`<workspaceDir>/.lecquy/permissions-audit.jsonl`

每行一条 JSON：

```json
{"timestamp":1745164123456,"toolName":"bash","args":{"command":"rm -rf /tmp/x"},"decision":{"behavior":"ask","reason":"Bash 命令需要确认：删除文件 (rm)"},"matchedSource":"builtin","mode":"default"}
```

查询示例：

```bash
# 最近 10 次 deny
grep '"behavior":"deny"' .lecquy/permissions-audit.jsonl | tail -10

# 某工具的所有决策
jq 'select(.toolName=="bash")' .lecquy/permissions-audit.jsonl

# deny 按 tool 聚合
jq -r 'select(.decision.behavior=="deny") | .toolName' \
  .lecquy/permissions-audit.jsonl | sort | uniq -c
```

### 5.3 告警接入

建议在自定义 `AuditSink` 中叠加告警：

```ts
class AlertingSink implements AuditSink {
  async write(record: PermissionAuditRecord) {
    if (record.decision.behavior === 'deny' && record.mode === 'bypassPermissions') {
      // bypass 模式下发生 deny 意味着分类器或规则 deny —— 高危
      await notifySlack(`[LECQUY] Bypass 模式下拦截了 ${record.toolName}：${record.decision.reason}`)
    }
  }
  async recent() { return [] }
}

const composite = new CompositeAuditSink([
  new InMemoryAuditSink(),
  new JsonFileAuditSink(pathForWorkspace),
  new AlertingSink(),
])
```

### 5.4 审计数据保留

- JSON Lines 文件按日切分（自行实现 rotating sink）
- 本地保留 30 天，远端冷存 1 年
- 合规需求下加签名（HMAC）防篡改

## 6 沙箱隔离的接入

`sandbox-adapter.ts` 的 `ChildProcessSandbox` 适合执行"Manager 允许但仍需要隔离"的命令。典型场景：`pip install`、`npm install` 这类网络 + 文件 I/O 混合的操作。

### 6.1 用法

```ts
import { ChildProcessSandbox } from '../runtime/permissions/index.js'

const sandbox = new ChildProcessSandbox({
  workspaceDir: '/workspace',
  timeoutMs: 60_000,
  maxBufferBytes: 10 * 1024 * 1024,
  // env 只透传白名单；可追加特定变量
  env: { CI: 'true' },
})

const result = await sandbox.run({
  command: 'npm install',
  cwd: '/workspace',
  signal: controller.signal,
})

console.log(result.stdout, result.stderr, result.exitCode)
```

### 6.2 和 Manager 的协作

建议链路：

```
manager.check → 返回 allow → sandbox.run
           → 返回 ask → 前端确认 → manager.applyUpdate（一次性 allow）→ sandbox.run
           → 返回 deny → 前端展示理由，不进 sandbox
```

### 6.3 资源限制

`ChildProcessSandbox` 当前提供：

- **cwd 锁定**：禁止逃出 workspaceDir
- **env 白名单**：不透传凭证
- **超时中断**：`timeoutMs` 到后 SIGTERM
- **输出限量**：`maxBufferBytes` 超限截断
- **AbortSignal**：支持外部主动取消

**不提供**（需要容器/cgroup 支持）：

- CPU/内存限制
- 网络 egress 控制
- 系统调用级 sandbox（seccomp）

如果需要以上，建议把 `ChildProcessSandbox` 换成 Docker / firecracker 的适配器，实现同一个 `SandboxAdapter` 接口。

## 7 常见故障排查

### 7.1 "某个命令本该被拒却通过了"

1. 先跑 `bash-classifier` 单独验证：
   ```ts
   const r = defaultBashClassifier.classifySync('your command here')
   console.log(r)
   ```
2. 若分类器返回 allow，说明 DENY/ASK 模式没匹配。检查是否有转义或空白绕过（`rm  -rf /`、tab 分隔）。
3. 若分类器返回 ask，但最终执行了：看 mode 是否为 `bypassPermissions`，或命中了 `allow` 规则。查审计 `matchedSource`。
4. 若分类器返回 deny，但 check 返回 allow：bug，提 issue。

### 7.2 "路径遍历没被拦"

1. 检查 `containsPathTraversal` 的输入：传进去的是原始路径还是解析后的？前者才能被检测到 `..`。
2. 看 workspaceDir 是否被规范化为绝对路径（`path.resolve`）。
3. `resolveWithinWorkspace` 返回 null 时就是工作区外；加日志确认。

### 7.3 "符号链接逃逸漏过"

1. 符号链接只有在**文件存在**时才能 lstat；不存在的路径直接视为安全（走 baseValidate）。
2. 如果被逃逸的是目标目录不存在（`realpath` 抛错），`file-operations` 降级为 ask。确认是否把 ask 误当成 allow。

### 7.4 "审计文件越来越大"

- `JsonFileAuditSink` 不自动轮转。需要自行实现 rotating sink，或外部用 logrotate。
- 每条记录约 200-500 字节，日均 10k 次 ≈ 5MB。

### 7.5 "测试里 mkdtemp 没清理"

- 所有测试已使用 `try { ... } finally { await rm(ws, { recursive: true, force: true }) }`。
- 若定位到新测试漏写 finally，补上并跑 lint。

### 7.6 "新规则不生效"

1. 是否调用了 `manager.load()`？构造函数只挂了 builtin，配置文件要 `load()` 才进。
2. 是否排序错了？每次 `applyUpdate` 后 Manager 会自动 `sortRulesByPriority`。
3. 是否命中了更高优先级的反向规则？看 `detectShadowedRules` 输出。

## 8 性能基线

在 Intel i7-12700 / 32GB RAM / NVMe SSD 上：

| 操作 | 平均耗时 | P99 |
| --- | --- | --- |
| `checkPermission` bash + 10 条规则 | 0.15ms | 0.8ms |
| `checkPermission` edit_file + 符号链接检查 | 0.6ms | 2.5ms |
| `bash-classifier.classifySync` | 0.08ms | 0.4ms |
| `loadPermissionRules` 冷启动 | 4ms | 15ms |
| `JsonFileAuditSink.write` | 0.9ms | 3ms |
| `InMemoryAuditSink.write` | 0.003ms | 0.01ms |

**结论**：

- 主路径每次决策 < 1ms，每秒能处理千次以上
- I/O 最贵的是符号链接检测（`lstat + realpath`），优化方向是**缓存** `realpath` 结果
- 审计建议 InMemory + 异步 JsonFile

## 9 FAQ

### Q1 为什么 `bypassPermissions` 仍然尊重 `deny`？

安全优先。`bypassPermissions` 的语义是"相信 AI，不弹确认"，但**不是**"什么都能干"。`rm -rf /` 这种操作无论如何都应该拒绝。

### Q2 为什么不直接用 `path.normalize`？

`path.normalize` 只做字符串层面的处理，不解符号链接，也不知道 workspaceDir 的概念。`resolveWithinWorkspace` 做的是**语义级**边界检查。

### Q3 新旧引擎决策不一致时怎么办？

看审计日志。策略 B（双引擎 + 降级）下取更严格的一侧。如果新引擎比旧引擎更严，说明规则收紧了（预期）；如果新引擎更松，要 debug。

### Q4 `PermissionTier.Preamble` 在新引擎里怎么触发？

`tier-bridge.ts::shouldUsePreamble` 的条件：
- 决策是 allow
- reason 里包含 `acceptEdits` / `cliArg` / `session` 关键字

换言之：**运行时**用户主动放宽的 allow → Preamble；**静态规则**的 allow → Auto。

### Q5 AI 分类器什么时候接入？

至少满足：
- MVP 上线稳定运行 4 周以上
- 规则误报率可量化（< 5%）
- 有可用的本地或专用 LLM（成本 + 延迟可控）
- 有对比评估集（规则 vs AI 准确率）

### Q6 能否支持正则规则？

目前 `content` 只支持字符串前缀和 glob，不支持正则。添加正则会带来 ReDoS 风险，需谨慎。若必须：
- 加超时（`RegExp` + AbortController）
- 黑名单化：只允许 `safe-regex` 通过

### Q7 `plan` 模式下能执行什么？

**什么都不执行**，返回 `plan` 决策。上层应该把调用翻译为"预览信息"（比如 bash 命令不运行，只显示；文件写入不落盘，只展示 diff）。

### Q8 规则最大容量？

- 内存：每条约 200 字节，10k 条 ≈ 2MB，无压力
- 匹配：线性扫，10k 条 bash 命令匹配 < 5ms
- 建议：项目级保持 < 100 条；超过就考虑分层拆分

### Q9 多 workspace 怎么处理？

每个 workspace 持有独立的 `PermissionManager`。建议封装一个 registry：

```ts
const registry = new Map<string, PermissionManager>()
async function getManager(workspaceDir: string) {
  if (!registry.has(workspaceDir)) {
    registry.set(workspaceDir, await PermissionManager.create({ workspaceDir }))
  }
  return registry.get(workspaceDir)!
}
```

### Q10 集成出错怎么回滚？

策略 B（双引擎）的降级逻辑：新引擎抛错时退化到旧引擎。此外 feature flag 一键禁用新引擎：

```ts
const useNewEngine = process.env.LECQUY_NEW_PERMISSION_ENGINE === '1'
```
