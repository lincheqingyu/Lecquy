## 一、排查结论摘要

**根因确认**：`sessions_history` 工具的输出未经 `TOOL_OUTPUT_LIMIT`（50K 字符）截断，且底层 `history()` 返回的 `SessionMessageRecord[]` 可以包含 base64 图片数据（实测单条消息 5.3MB）。agent 调用此工具时，该图片内容被 `JSON.stringify(rows, null, 2)` 完整输出，直接撑爆上下文。

**爆炸半径**：4 个 session-tools（`sessions_history` / `sessions_list` / `sessions_send` / `sessions_spawn`）全部没有 `TOOL_OUTPUT_LIMIT` 截断。其余 7 个 agent 工具（bash / read-file / write-file / edit-file / todo-write / skill / request-user-input）均有输出保护或输出天然可控。

## 三、爆炸数据溯源

### 3.1 真实会话文件

```
.lecquy/sessions/v3/sessions/sess_1776231258671_vh518p5c.jsonl    5.1 MB  (65 行)
.lecquy/sessions/v3/sessions/sess_1776306100773_ywwd6epm.jsonl     837 KB
.lecquy/sessions/v3/sessions/sess_1776336719285_pqwlsgoo.jsonl     197 KB
.lecquy/sessions/v3/sessions/sess_1776357499023_wevl2nv7.jsonl     176 KB
```

### 3.2 5.1MB 文件的内部结构

65 行中，第 15 行单行占 **5,361,967 字符**（5.1MB），是一条 `type: "message"` 的 entry：

```json
{
  "type": "message",
  "message": {
    "role": "user",
    "content": [
      { "type": "text", "text": "识别图片内容" },
      { "type": "image", "data": "iVBORw0KGgo...（5.3MB base64）" }
    ]
  }
}
```

### 3.3 数据流追踪

```
agent 调 sessions_history({ sessionKey: "sess_1776231258671_...", limit: 50 })
  → createSessionsHistoryTool.execute()
  → service.history(sessionKey, limit)       // 默认 limit=50，返回最近 50 条
  → rebuildSessionProjection()                // 遍历所有 entry，筛出 message 类型
  → messages.slice(-limit)                    // 取最后 50 条
  → 如果最后 50 条包含该图片消息
  → JSON.stringify(rows, null, 2)             // ❌ 无截断，直接输出 5MB+
  → 上下文爆炸
```

即使用默认的 `limit=50`，只要最近 50 条消息里包含那张图片，输出就是 5MB+。

## 四、逐个工具缺陷详析

### 4.1 `sessions-history.ts`（核心问题）

| 维度 | 现状 | 问题 |
|---|---|---|
| limit 默认值 | 无默认，`Type.Optional(Number({ minimum: 1, maximum: 500 }))` | agent 不传则为 undefined，虽 service 有 fallback 50，但 max=500 太宽 |
| 输出截断 | `JSON.stringify(rows, null, 2)` 直出 | 无 TOOL_OUTPUT_LIMIT 截断 |
| 'current' 关键字 | 不支持 | 直接传给 `service.history('current')`，service 抛错"会话不存在" |
| 工具描述 | "读取指定会话的历史消息。" | 未说明 limit 默认值、截断行为、不要传 'current' |

### 4.2 `sessions-list.ts`

| 维度 | 现状 | 问题 |
|---|---|---|
| limit 默认值 | 无默认，`Type.Optional(Number({ minimum: 1, maximum: 200 }))` | agent 不传则为 undefined |
| messageLimit | `Type.Optional(Number({ minimum: 0, maximum: 50 }))` | 若 agent 传 messageLimit>0，recentMessages 可能包含图片 base64 |
| 输出截断 | `JSON.stringify(rows, null, 2)` 直出 | 无 TOOL_OUTPUT_LIMIT 截断 |
| 工具描述 | "列出最近会话，可选附带最近消息。" | 未风险提示 |

### 4.3 `sessions-send.ts`

| 维度 | 现状 | 问题 |
|---|---|---|
| 输出截断 | `JSON.stringify({ runId, status, reply })` 直出 | `reply` 是目标 session 的 agent 完整回复，可能包含大量内容块 |
| timeoutSeconds | 默认 30s，max 600s | 合理 |

风险等级标注为"中"是因为 `sessions_send` 的输出通常是一条回复而不是多条历史消息，但回复本身也可能是工具调用堆，上限不可控。

### 4.4 `sessions-spawn.ts`

| 维度 | 现状 | 问题 |
|---|---|---|
| 输出截断 | `JSON.stringify(result)` 直出 | result 来自 `service.spawnTask()`，返回类型 `SpawnTaskResult`，包含 sessionKey/runId 等控制字段，长度可控 |

风险等级"中"是因为 `SpawnTaskResult` 结构较小，但无截断保护仍是缺口。

## 六、修复方案（对照原指令的修正）

### 6.1 `sessions-history.ts` — 需改 4 处

1. **加 `TOOL_OUTPUT_LIMIT` 截断**：`JSON.stringify` 后 `slice(0, TOOL_OUTPUT_LIMIT)`，并附加截断提示
2. **收紧 limit**：default=20, max=200（原来无 default, max=500）
3. **支持 `current` 关键字**：解析为 `getCurrentToolSessionKey()`，解析失败时返回友好 error 而非抛异常
4. **更新工具描述**：说明 limit 默认值、截断行为、不要传 'current'

### 6.2 `sessions-list.ts` — 需改 2 处

1. **加 `TOOL_OUTPUT_LIMIT` 截断**
2. **加 default limit**：默认 20

### 6.3 `sessions-send.ts` — 需改 1 处

1. **加 `TOOL_OUTPUT_LIMIT` 截断**：对 reply 做截断保护

### 6.4 `sessions-spawn.ts` — 需改 1 处

1. **加 `TOOL_OUTPUT_LIMIT` 截断**

### 6.5 `runtime.ts` / `index.ts` — 无需修改

`getCurrentToolSessionKey` 已存在，`sessions-spawn.ts` 已在使用。`sessions-history.ts` 直接 import 即可。
