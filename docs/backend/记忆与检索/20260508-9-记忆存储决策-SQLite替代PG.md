# 记忆存储决策：SQLite + FTS5 替代 PostgreSQL + pgvector

更新日期：2026-05-08

## 1. 决策

> **Lecquy 的记忆底座采用 SQLite + FTS5，现阶段不要引入向量检索。**

## 2. 上下文

- 之前基线记录在 `20260408-3-Runtime Memory Compact 决策沉淀.md`，当时选型为 `PostgreSQL + pgvector + pg_trgm`。
- 实际开发中发现单机场景下引入 PostgreSQL + pgvector 的运维成本（依赖安装、数据迁移、环境配置）远高于 SQLite。
- FTS5 是 SQLite 内置全文搜索引擎，支持 BM25 排序，配合 `better-sqlite3` 在 Node.js 中零依赖启动。

## 3. 选型依据

| 维度 | PostgreSQL + pgvector | SQLite + FTS5 |
|------|----------------------|---------------|
| 运维成本 | 需安装 PG、插件、管理连接 | `better-sqlite3` 单文件，无守护进程 |
| 启动速度 | 需等待数据库连接池就绪 | 毫秒级打开文件 |
| 记忆场景匹配度 | 强在向量相似度搜索，但一期无此需求 | FTS5 的 BM25 原生支持全文排序，契合当前"关键词 + 重要性 + 时间衰减"的召回模型 |
| 开发体验 | 类型转换、连接管理、Docker 依赖 | 同步 API、类型友好、无外部依赖 |
| 扩展路径 | 向量检索可直接接入，不需改造 | 后续需要向量检索时可在 SQLite 上叠加 `sqlite-vec` 插件，或换回 PG |

## 4. 已落地工作

- `第二周第1件`：`better-sqlite3` + FTS5 替代 PG，extraction-runner 同步改造，tag 落地
- `第二周第2件`：prompt-injector 切到 SQLite 召回，复合排序公式（BM25 + importance + 时间衰减 + 项目软优先）
- `第1件冒烟修复`：watermark 表 + 提取水位机制 + dedupe 清理

## 5. 约束

- 除非记忆召回出现明确的语义相似度瓶颈（如"找一段忘记关键词但记得意思的内容"），否则不引入向量检索。
- 后续如果决定引入向量检索，优先评估 `sqlite-vec` 插件，其次再考虑换回 pgvector，避免引入外部依赖。
