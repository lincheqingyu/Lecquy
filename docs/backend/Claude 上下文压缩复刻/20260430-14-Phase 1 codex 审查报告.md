## 3. Findings

### 3.1 Blocker

### F-001 [Blocker]

**位置：** `backend/src/memory/compact.ts:123-129`

**当前状态：**

```ts
const argsStr = JSON.stringify(part.arguments).slice(0, 200)
// 工具状态作为 output 摘要（block 上不存储完整输出）
const statusStr = part.status === 'error'
  ? `[error] ${truncateText(part.errorMessage ?? '未知错误', COMPACTION_TOOL_OUTPUT_MAX_CHARS)}`
  : '[已完成]'
lines.push(`  ↳ ${part.name}(${argsStr}) → ${statusStr}`)
```

**期望状态：**
`formatHistoryForCompaction` 应把工具调用序列化为 `↳ tool(args) → output`，其中 args 截断到 200 字符，tool output 文本截断到 2000 字符；成功工具调用不能只输出 `[已完成]`，否则 compaction LLM 无法看到被压缩历史中的工具结果事实。

**依据：**
文档 7 第 4.1 节、文档 9 修订版第 4.2 / 4.5 节、文档 8 第 4.4 / 7 节。

**修复建议：**
补齐工具结果来源，把 tool output 文本纳入 `SessionToolCallContentBlock` 或在 compact 序列化阶段可读取的历史 entry 中关联读取，再统一用 `truncateText(output, COMPACTION_TOOL_OUTPUT_MAX_CHARS)` 写入 `↳` 行。

### 3.2 Major

### F-003 [Major]

**位置：** `backend/src/runtime/session-runtime-service.ts:1522-1528`

**当前状态：**

```ts
if (await applyCompactionIfNeeded(manager, {
  model: model.id,
  apiKey,
  timeoutMs: this.cfg.COMPACTION_TIMEOUT_MS,
})) {
  await this.refreshProjection(sessionKey)
}
```

**期望状态：**
`session-runtime-service.ts` 调用 `applyCompactionIfNeeded` 时应有独立 try/catch，压缩流程任意异常都只记录日志并继续 finally 清理；当前 `writeMemorySummary` 或 `refreshProjection` 失败会让 finally 抛错，可能把已完成的对话 run 改写成失败感知。

**依据：**
文档 8 第 2.2 节、文档 13 第 2.1 节对 `session-runtime-service.ts` 的必查重点。

**修复建议：**
用 `try { ... } catch (error) { logger.warn('[compact] 自动压缩失败，跳过本轮压缩', { ... }) }` 包住 compaction 调用和随后的 refresh。
