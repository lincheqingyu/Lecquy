## 5. codex 行为约束

- **严格按依据审查，不引入新设计判断。** 如果发现"看起来可以更好但文档没要求"的写法，标 Nit 或不标，**不**升级为 Major
- **不**建议 Phase 2 / Phase 3 应该怎么做。即使发现了 token 估算的明显缺陷，也只能标记为"已知遗留项（Phase 2 处理）"，不写改动建议
- **不**重新讨论文档 7 已经收敛过的问题。例如：codex 可能本能地觉得"compaction LLM 失败应该抛错让用户感知"，但文档 7 4.2 已判为降级。这种 finding 一律不出
- **不**评估测试是否能跑通（这是 Sonnet 自检的责任），只看测试设计是否覆盖文档 7 5.7 列出的场景
- **不**做风格类批量整改建议（如 import 排序、空行规范）。这些由项目 lint 工具负责
- 每条 finding 必须可执行（指出在哪里改、改成什么样），**不**写"建议优化"或"考虑改进"这种含糊措辞
- 报告语言中文，文件路径 / 类型名 / 字段名保留英文（遵循 `.claude/rules/language.md`）

## 6. T7.5 Opus 分诊环节（强制）

codex 报告完成后，**必须**经 Opus 分诊后再交给 Sonnet 修复。这一环节不可跳过。

### 6.1 为什么需要分诊

codex 是外部 reviewer，不持有项目设计史，会出现两类 false positive：

- **设计选择被误判为缺陷**：例如标 Major"降级路径用 logger.warn 不抛错"，实际是文档 7 4.2 的明确决策
- **过度审查**：例如建议为单一调用路径补充防御性 try/catch，实际上调用栈上层已有保护

让 Sonnet 直接吃下未过滤的报告，会被 false positive 拖偏方向。

### 6.2 Opus 的分诊产出

在文档 14 末尾追加一节"## 7. Opus 分诊结论"，按 finding 编号逐条标注：

| 标记 | 含义 | 处置 |
| --- | --- | --- |
| `accept` | 真实问题，按 finding 描述修复 | 进入 T8 由 Sonnet 修 |
| `reject` | False positive，附文档 7 / 8 章节引用作为依据 | 不修 |
| `discuss` | 触及文档 7 已收敛之外的边界，需要重新设计判断 | 升级到 Opus 单独决策，决策结论写回文档 7 第 4 / 5 节并版本号 +1 |

格式示例：

```
| Finding | Severity | 分诊 | 依据/说明 |
| --- | --- | --- | --- |
| F-001 | Blocker | accept | （无） |
| F-002 | Major | reject | 文档 7 4.2 明确降级不抛错 |
| F-003 | Major | discuss | 触及 previousSummary 在 LLM 输出格式不合规时的二次行为，文档 7 未覆盖 |
```

### 6.3 分诊行为约束

- Opus 不修改 codex 的原始 finding 描述，只在分诊节追加判断
- `reject` 必须附依据章节引用，不能只写"不修"
- `discuss` 标记后必须在同一份报告中给出当前轮的临时处置建议（防止阻塞 T8）；最终决策写回文档 7 走完整迭代
