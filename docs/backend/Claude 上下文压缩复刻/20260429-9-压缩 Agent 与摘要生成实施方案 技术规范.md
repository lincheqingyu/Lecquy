## 1. 本文档的定位

## 2. 为什么当前模板摘要不够

当前摘要输出示例：

```
此前摘要：已实现 SessionManager 的 buildSessionContext，修复了 firstKe...
已压缩 42 条较早消息，保留最近 10 条原文。
- 用户: 帮我看下 compact.ts 里的 resolveCompactSource 为什么会在 startIndex...
- 助手: 这里的问题是 findIndex 在找不到 entry 时返回 -1，然后 slice(-1) 会...
- 用户: 好，那现在帮我改一下让它在 startIndex 为负时直接 return null
- 助手: 我来修改 resolveCompactSource() 函数，在 startIndex < 0 时直接返回...
```

**这个摘要对 LLM 的价值极低：**

- 没有"当前任务目标是什么"
- 没有"已经完成了什么"
- 没有"下一步计划做什么"
- 没有"哪些文件是关键文件"
- 没有"有哪些重要技术约束"

LLM 下一轮收到这个摘要，几乎无法从中恢复会话上下文，只能依赖 recent tail 的 10 条原文。

**LLM 摘要对应的输出示例（预期效果）：**

```markdown
## 目标
- 修复 Lecquy 上下文压缩中 resolveCompactSource 在 firstKeptEntryId 找不到时的边界问题

## 约束与偏好
- 保持 append-only session tree 不变
- 不改变 compact entry 的持久化结构

## 进度
### 已完成
- 定位 bug：findIndex 返回 -1 时 slice(-1) 行为异常
- 修复 resolveCompactSource()：startIndex < 0 时 return null

### 进行中
- 添加对应单元测试

### 受阻
- （无）

## 关键决策
- 选择 return null 而非 fallback 到全量消息，避免重复压缩

## 下一步
- 完成 compact.test.ts 的边界测试
- 确认 refreshProjection() 在 compact 后立即调用

## 关键上下文
- getDurableMessageEntries() 只计 user/assistant 消息，compaction entry 不计入 50 条阈值
- firstKeptEntryId 必须在 messageEntries 中存在，否则 resolveCompactSource 返回 null

## 相关文件
- backend/src/memory/compact.ts：压缩逻辑主文件，resolveCompactSource() 在此
- backend/src/runtime/pi-session-core/session-manager.ts：buildSessionContext() 读取 compaction entry
```

这个摘要下一轮 LLM 能立即理解当前状态，不需要重新从 10 条消息里推断。

## 4. 接口设计

### 4.1 新的 `buildCompactSummary` 签名

不再暴露 `llmClient` 参数。由函数内部直接使用 `completeSimple` 发起 LLM 调用，model 与 apiKey 从调用方传入。

```ts
// compact.ts

interface CompactSummaryInput {
  readonly source: CompactSource
  readonly model: string          // model ID，如 "claude-haiku-4-5"；来源：bound.projection.model（见文档 11 事实 2）
  readonly apiKey: string         // 来源：cfg.LLM_API_KEY（见文档 11 事实 3）
  readonly timeoutMs: number      // 来源：cfg.COMPACTION_TIMEOUT_MS（见第 9 节）
}

interface CompactSummaryResult {
  readonly summary: string
  readonly method: 'llm' | 'template'   // 记录实际使用的生成方式
  readonly llmError?: { name: string; message: string }  // 仅降级时存在
}

async function buildCompactSummary(
  input: CompactSummaryInput
): Promise<CompactSummaryResult>
```

### 4.2 LLM 调用的 message 组织

历史记录序列化为单一文本块，作为 `role: 'user'` 消息送入 compaction LLM。**不复刻** OpenCode 的 model-messages 重放方式（详见文档 7 第 4.1 节）。

工具调用必须嵌入所属"助手段落"内部，**不**单独成行。格式：

```
[用户] 帮我看下 compact.ts 里 resolveCompactSource 的 startIndex 边界

[助手] 我来读取这个文件。
  ↳ read_file({"path":"backend/src/memory/compact.ts"}) → import { promises as fs } from 'node:fs'\n... (前 2000 字符)
  ↳ findIndex 在 entry 不存在时返回 -1，slice(-1) 会取最后一条而不是空数组，这就是 bug 根因。

[用户] 那帮我改一下让 startIndex 为负时直接 return null

[助手] 我来修改 resolveCompactSource()。
  ↳ edit_file({"path":"backend/src/memory/compact.ts","old":"...","new":"..."}) → 编辑成功
  ↳ 已加上 if (startIndex < 0) return null 的早返回。
```

序列化约束：

- 不送入 `thinking` 内容（过程性产物，OpenCode 亦丢弃）
- 不送入 model / provider / timestamp 字段
- args 摘要：JSON 序列化后保留前 200 字符
- output 摘要：纯文本截断到 `COMPACTION_TOOL_OUTPUT_MAX_CHARS`（默认 2000 字符，见第 9 节）
- 工具调用以 `  ↳ tool_name(args 摘要) → output 摘要` 形式嵌入助手段落；多次工具调用串成多个 `↳` 行
- 助手文本与工具调用之间不插额外空行，保留段落整体感
- 不同角色的相邻消息之间用单个空行分隔
- `extractSessionText` 对 `toolCall` 类型返回空字符串（见文档 11 事实 4），因此工具调用需手动格式化，不能依赖 `extractSessionText`

**SUMMARY_INSTRUCTIONS（system prompt，每次不变）：**

### 4.4 完整 prompt 构建

`buildCompactionPrompt` 返回 `{ system: string; user: string }`：

```ts
function buildCompactionPrompt(input: {
  history: string
  previousSummary?: string
}): { system: string; user: string } {
  const userParts: string[] = []

  userParts.push('以下是需要压缩的对话历史：')
  userParts.push('<conversation-history>')
  userParts.push(input.history)
  userParts.push('</conversation-history>')

  if (input.previousSummary) {
    userParts.push('')
    userParts.push('请在以下锚定摘要的基础上更新：')
    userParts.push('- 把上方对话历史中的新事实合并进来')
    userParts.push('- 删除已被新事实超越或不再成立的内容')
    userParts.push('- 保留仍然成立的细节')
    userParts.push('- 输出仍然按 system prompt 中 SUMMARY_TEMPLATE 的章节结构')
    userParts.push('')
    userParts.push('<previous-summary>')
    userParts.push(input.previousSummary)  // 完整传入，不截断（见文档 7 第 4.3 节）
    userParts.push('</previous-summary>')
  } else {
    userParts.push('')
    userParts.push('请根据以上对话历史生成一份新的锚定摘要。')
  }

  return {
    system: SUMMARY_INSTRUCTIONS,
    user: userParts.join('\n'),
  }
}
```

`previousSummary` 完整传入，**不做任何字符截断**（见文档 7 第 4.3 节收敛结论）。

### 4.5 history 序列化

```ts
// 常量定义见第 9 节"配置项与默认值"
const COMPACTION_MAX_OUTPUT_TOKENS = 4_096
const COMPACTION_TOOL_OUTPUT_MAX_CHARS = 2_000

function formatHistoryForCompaction(entries: SessionEventEntry[]): string {
  const lines: string[] = []

  for (const entry of entries) {
    if (entry.type !== 'message') continue

    const role = entry.message.role === 'user' ? '用户' : '助手'
    const parts = entry.message.content

    if (typeof parts === 'string') {
      if (parts.trim()) lines.push(`[${role}] ${parts}`)
      continue
    }

    if (!Array.isArray(parts)) continue

    // 助手消息：把文本与工具调用组合为一个段落
    const paragraphLines: string[] = []
    for (const part of parts) {
      if (part.type === 'text' && part.text?.trim()) {
        paragraphLines.push(part.text.trim())
      } else if (part.type === 'toolCall') {
        // 工具调用以 ↳ 行嵌入助手段落（见 4.2 格式约束）
        const argsStr = JSON.stringify(part.args ?? {}).slice(0, 200)
        const outputStr = truncate(extractToolOutput(part), COMPACTION_TOOL_OUTPUT_MAX_CHARS)
        paragraphLines.push(`  ↳ ${part.name}(${argsStr}) → ${outputStr}`)
      }
      // thinking / image / file 类型跳过
    }

    if (paragraphLines.length > 0) {
      lines.push(`[${role}] ${paragraphLines.join('\n')}`)
    }
  }

  return lines.join('\n\n')
}

function truncate(text: string, maxChars: number): string {
  if (text.length <= maxChars) return text
  return text.slice(0, maxChars) + `... (截断，原长 ${text.length} 字符)`
}
```

## 5. 降级处理

LLM 摘要失败时，不 crash，回退到当前模板方式。必须留下可观测痕迹（文档 7 第 4.2 节）：

```ts
async function buildCompactSummary(input: CompactSummaryInput): Promise<CompactSummaryResult> {
  try {
    const summary = await callCompactionLLM(input)
    return { summary, method: 'llm' }
  } catch (error) {
    logger.warn('[compact] LLM 摘要失败，降级到模板方式', {
      reason: error instanceof Error ? error.name : 'unknown',
      message: error instanceof Error ? error.message : String(error),
    })
    const summary = formatCompactSummaryFallback({
      previousSummary: input.source.previousSummary,
      compactedMessages: input.source.compactedMessages,
      recentTailCount: COMPACT_RECENT_TAIL,
    })
    return {
      summary,
      method: 'template',
      llmError: {
        name: error instanceof Error ? error.name : 'unknown',
        message: error instanceof Error ? error.message : String(error),
      },
    }
  }
}
```

降级原因记录到 compaction entry 的 `details` 字段（新增字段见第 6 节）：

```ts
manager.appendCompaction(
  result.summary,
  source.firstKeptEntryId,
  estimateTokensBefore(source),
  {
    trigger: 'message_threshold',
    compaction_method: result.method,           // 新增：'llm' | 'template'
    llm_error: result.llmError,                 // 新增：仅降级时存在
    kept_message_count: COMPACT_RECENT_TAIL,
    compacted_message_count: source.compactedMessages.length,
    compacted_through_entry_id: /* ... */,
  },
)
```

**异常分类（全部走降级）：**

- 超时（AbortError）/ 网络错误 / 429 / 5xx / 抽象层异常 → 降级 + `logger.warn`
- 调用前置准备失败（model 为空、apiKey 缺失）→ `logger.error` + 降级
- LLM 返回非法格式（缺章节）→ 软校验，只记 `logger.warn`，不重试，不 throw（见文档 7 第 5.7 节）
- **不**使用 `console.warn` / `console.error`，统一改用 `logger`

## 6. `applyCompactionIfNeeded` 的签名变化

`CompactionOptions` 必须设计为**开放结构**，方便 Phase 2 不改 schema 直接扩展（文档 7 第 5.5 节）。不暴露 `llmClient`：

```ts
export interface CompactionOptions {
  readonly model: string          // model ID（来源：bound.projection.model，见文档 11 事实 2）
  readonly apiKey: string         // 来源：cfg.LLM_API_KEY（见文档 11 事实 3）
  // Phase 2 预留字段：
  readonly estimatedTokens?: number
  readonly modelContextWindow?: number
}

export async function applyCompactionIfNeeded(
  manager: SessionManager,
  options: CompactionOptions,
): Promise<boolean>
```

`compaction entry.details` 同步预留 Phase 2 字段：

```ts
{
  trigger: 'message_threshold' | 'token_overflow',  // Phase 2 会扩展
  compaction_method: 'llm' | 'template',
  llm_error?: { name: string; message: string },
  kept_message_count: number,
  compacted_message_count: number,
  compacted_through_entry_id: string,
  // Phase 2 预留：
  estimated_tokens_before?: number,
  model_context_window?: number,
  threshold_used?: number,

```

// apiKey 来源：cfg.LLM_API_KEY（见文档 11 事实 3，cfg 无 ANTHROPIC_API_KEY 字段）

## 7. model 选择策略

复用 `bound.projection.model`（与主对话同源，文档 7 第 5.3 节）：

```ts
const compactionModel = bound.projection.model ?? this.cfg.LLM_MODEL
```

**降级触发条件（直接走模板降级，记 warn）：**

- model 为空（`bound.projection.model` 为 undefined，且 `cfg.LLM_MODEL` 也未配置）
- apiKey 缺失（`cfg.LLM_API_KEY` 未配置）
- `completeSimple` 抛出"unsupported model"异常（如非 Anthropic / 非 OpenAI 兼容的本地 vLLM）→ 在 `callCompactionLLM` 的 try/catch 中捕获，走降级

## 8. 超时与上限

| 配置项 | 起步值 | 层级 | 说明 |
| --- | --- | --- | --- |
| `COMPACTION_TIMEOUT_MS`（env） | 60_000 ms | `cfg.COMPACTION_TIMEOUT_MS` | 硬超时，起步值，上线后根据 p99 耗时调整 |
| `COMPACTION_MAX_OUTPUT_TOKENS`（常量） | 4_096 | `compact.ts` 常量 | SUMMARY_TEMPLATE 典型输出 800–1500 tokens，预留余量 |
| `COMPACTION_TOOL_OUTPUT_MAX_CHARS`（常量） | 2_000 | `compact.ts` 常量 | 与 OpenCode 一致，在 `formatHistoryForCompaction` 序列化时执行 |

`completeSimple` 调用示例（文档 7 第 5.1 节）：

```ts
import { completeSimple, type UserMessage } from '@mariozechner/pi-ai'

// completeSimple 完整签名：
// completeSimple<TApi>(model: Model<TApi>, context: Context, options?: SimpleStreamOptions)
// 见文档 11 事实 1：支持 signal（AbortSignal）和 maxTokens 字段

const prompt = buildCompactionPrompt({
  history: formatHistoryForCompaction(source.compactedMessages),
  previousSummary: source.previousSummary,
})

const response = await completeSimple(
  resolvedModel,   // Model<TApi> 对象，由 model ID 在调用层解析
  {
    systemPrompt: prompt.system,
    messages: [{
      role: 'user',
      content: prompt.user,
      timestamp: Date.now(),
    } satisfies UserMessage],
  },
  {
    apiKey: input.apiKey,
    signal: AbortSignal.timeout(input.timeoutMs),
    maxTokens: COMPACTION_MAX_OUTPUT_TOKENS,
  },
)
```

**关于硬超时：** 60 秒是起步值，**不是**最终决策。Phase 1 上线后必须采集实际耗时分布（p50 / p95 / p99），根据 p99 反推合理上限。超时后走模板降级（与其他失败路径一致）。如 p99 超时降级率 > 5%，优先调高超时值而不是改设计。

## 9. 配置项与默认值

本节定义 T3 的三个配置字段落点，代码改动在 T6 中执行。

### 9.1 字段命名

参照 `env.ts` 现有风格（全大写 SCREAMING_SNAKE_CASE，顶层平铺，不嵌套）：

| 字段 | 命名 | 层级 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| 压缩超时 | `COMPACTION_TIMEOUT_MS` | `cfg` 顶层（env 暴露） | `60_000` | `z.coerce.number().int().default(60_000)` |
| 最大输出 token | `COMPACTION_MAX_OUTPUT_TOKENS` | `compact.ts` 常量（不入 env） | `4_096` | service 默认值，改动需改代码 |
| 工具输出截断 | `COMPACTION_TOOL_OUTPUT_MAX_CHARS` | `compact.ts` 常量（不入 env） | `2_000` | service 默认值，改动需改代码 |

### 9.2 env.ts 新增（仅 `COMPACTION_TIMEOUT_MS` 入 env）

```ts
// backend/src/config/env.ts，在 LLM_* 字段组之后追加

/** 压缩摘要 LLM 调用硬超时（毫秒） */
COMPACTION_TIMEOUT_MS: z.coerce.number().int().default(60_000),
```

### 9.3 compact.ts 常量

```ts
// backend/src/memory/compact.ts

/** 压缩摘要 LLM 最大输出 token 数 */
const COMPACTION_MAX_OUTPUT_TOKENS = 4_096

/** 序列化历史时工具输出的最大字符数 */
const COMPACTION_TOOL_OUTPUT_MAX_CHARS = 2_000
```

### 9.4 可调整范围

- `COMPACTION_TIMEOUT_MS`：env 暴露，运维可通过环境变量调整；建议范围 30_000–120_000，低于 30s 在高负载下降级率显著
- `COMPACTION_MAX_OUTPUT_TOKENS`：代码常量，更改需重新部署；SUMMARY_TEMPLATE 典型输出 1–2k tokens，4k 已充裕
- `COMPACTION_TOOL_OUTPUT_MAX_CHARS`：代码常量，与 OpenCode 对齐；降低会损失工具调用细节，升高会增加 compaction LLM 输入 token

## 10. `writeMemorySummary` 同源

`writeMemorySummary()` 写入 `.lecquy/memory-summary.md` 的内容必须与 `manager.appendCompaction(summary, ...)` 中的 `summary` **完全一致**（文档 7 第 5.6 节）：

```ts
// applyCompactionIfNeeded 内部
const result = await buildCompactSummary({
  source,
  model: options.model,
  apiKey: options.apiKey,
  timeoutMs: cfg.COMPACTION_TIMEOUT_MS,
})

// appendCompaction 与 writeMemorySummary 用同一个 result.summary
manager.appendCompaction(
  result.summary,
  source.firstKeptEntryId,
  estimateTokensBefore(source),
  {
    trigger: 'message_threshold',
    compaction_method: result.method,
    llm_error: result.llmError,
    kept_message_count: COMPACT_RECENT_TAIL,
    compacted_message_count: source.compactedMessages.length,
    compacted_through_entry_id: /* ... */,
  },
)
await writeMemorySummary(manager.getCwd(), result.summary)
```

禁止出现"compaction entry 是 LLM 摘要、文件是模板摘要"或反之的不一致。

## 13. 后续迭代方向

Phase 1 完成后，可以进一步做：

- **compaction model 可配置**：在 `.lecquy/config.json` 支持 `compaction.model` 字段，单独配置
- **摘要 token 预算可配置**：支持 `COMPACTION_MAX_OUTPUT_TOKENS` 入 env，控制摘要长度上限
- **摘要语言可配置**：中文 / 英文 SUMMARY_INSTRUCTIONS 可通过 env 切换
- **compaction 摘要可查看**：前端展示历史 compaction entry 的摘要内容（审计用）
- **触发策略升级为 token-aware**（Phase 2）：引入 token 估算，替换 50 条消息阈值
- **tool result prune 机制**（Phase 3）：历史工具输出截断，减少近端 token 浪费
