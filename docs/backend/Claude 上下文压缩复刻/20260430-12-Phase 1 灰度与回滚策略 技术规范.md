# Phase 1 灰度与回滚策略

更新日期：2026-04-30

> Phase 1 是触达每一次 run 末尾的改动 —— compaction 摘要从纯字符串模板升级为 LLM 调用，错误传播路径与可观测性都发生了变化。本文档给出上线节奏、feature flag 设计、可观测性指标、回滚预案分级与摘要质量抽样审计五项策略，并对工作规划文档（文档 10）第 7.1 节列出的三个开放问题给出明确判断。
>
> 上游依赖：文档 6（事实对比）、文档 7（收敛后的设计）、文档 8（审查清单）。下游使用方：T6 编码阶段在落地接口时按本文档预留 flag 与日志字段。

---

## 1. 三个开放问题的收敛

### 1.1 是否加 feature flag → 加，但只做 1-bit 全局开关

**判断：** 在 cfg 中加一个布尔 flag（命名待 T3 与其他 compaction 配置项一起定，建议 `compaction.llmSummaryEnabled`），不做用户级 / session 级 / 百分比灰度。

**决策依据：**

1. **compaction 是 run 末尾的后台优化，不是用户主诉路径。** 文档 7 第 4.4 节已明确摘要继续以合成 user 消息插入，对话本身不会被 compaction 失败中断。在这个前提下，"按用户灰度"或"按 session 灰度"的复杂度收益极低。
2. **文档 7 第 4.2 节内置了模板降级。** 单次 compaction LLM 调用失败已自动回退到 `formatCompactSummaryFallback()`，相当于"自带运行时熔断"。flag 的真正价值是在"系统性问题（prompt 模板写错、token 估算偏差、模型层接口变化）"导致降级率系统性升高时一键回退到老路径，而不是替代单次降级。
3. **1-bit 开关与现有 cfg 风格一致，长期维护成本最低。** Lecquy 现有 cfg 没有百分比灰度基础设施，强行引入会越界到 Phase 1 范围之外。

### 1.2 回滚优先用 flag 还是 revert → 优先改 flag

**判断：** 默认通过关闭 flag 回滚（L1），revert PR 仅作为 L1/L2 不收敛时的兜底（L3）。

**决策依据：**

1. **flag 关闭即刻回到模板路径，不走部署流程。** 关 flag 的 MTTR 远小于 revert PR + redeploy。
2. **revert 会丢失代码。** 重新开启 LLM 摘要需要 cherry-pick 或重新合入，反而增加变更量。flag 让代码留在 main 分支，回滚是配置变更而非代码变更。
3. **revert 的真正适用场景是"flag 关闭后仍异常"。** 例如新代码引入了影响模板降级路径的公共逻辑 bug —— 这种情况说明改动溢出了 flag 边界，必须 revert。

### 1.3 是否在 compaction entry 多存 history 指纹 → 不存

**判断：** 不引入 hash / 指纹字段。已有的 `firstKeptEntryId` 与 `compacted_through_entry_id` 已经足够支撑事后审计。

**决策依据：**

1. **session tree 是 append-only。** 被压缩的 messageEntries 不会被改写或删除，凭 entry ID 范围即可从 session tree 还原原文，不存在"原文已丢失需要 hash 校验"的场景。
2. **指纹的成本不止字段本身。** 还要锁定 hash 算法、序列化字段顺序、跨版本兼容性，并补不匹配场景的测试。这些约束与"防被外部篡改"的极小收益不成比例。
3. **真正缺的是质量观测，不是完整性校验。** 摘要质量回退要靠"采样审计 + 降级率指标"发现（见第 6 节），而不是 hash。

## 2. 上线节奏

| 阶段 | 环境 | flag 默认值 | 主要动作 | 进入下一阶段的指标 | 观察天数 |
| --- | --- | --- | --- | --- | --- |
| 0 | 开发 + CI | `false` | 代码合入 main，flag 默认 false。**生产行为零变化。** 跑全量单测 / 集成测试，覆盖文档 7 第 5.7 节列出的边界 | 单测全绿 + 文档 8 清单逐条勾选 | 1 天 |
| 1 | 内部 dogfooding | `true`（仅内部环境覆盖） | 内部账号触发 ≥ 5 次真实 compaction，人工对比摘要质量 | 0 次未捕获异常 + 摘要章节结构正确率 100% | 2 天 |
| 2 | 生产首发 | `true`（生产环境覆盖） | 全量开启。盯日志：降级率、耗时分布、摘要长度分布、结构错误率 | 7 日内降级率 < 5% 且无 L3 / L4 级事件 | 7 天 |
| 3 | 稳定期 | `true` | 持续观察聚合指标，每周复盘一次抽样审计结果 | 连续 2 周无关键事实缺失 | 14 天 |
| 4 | 默认开启 | 把 cfg 默认值改为 `true`（下次 release） | flag 仍保留作为长期回滚开关，**不**做"清理 flag" PR | —— | 长期 |

阶段 1 → 2 的开关动作以**生产环境配置**完成，不需要改代码，符合 1.2 节的回滚链路。

## 3. feature flag 设计

**字段位置：** 与 T3 的三个 compaction 配置项同层。当前候选命名：

```ts
// 仅占位，最终命名以 T2/T3 修订后的文档 9 为准
cfg.compaction.llmSummaryEnabled: boolean
```

**env 变量：** `COMPACTION_LLM_SUMMARY_ENABLED`（与 `compactionTimeoutMs` 一致地暴露 env，方便运维即时切换，无需走 PR）。

**默认值：** 阶段 0-1 默认 `false`，阶段 2 起通过 env 覆盖为 `true`，阶段 4 进入下次 release 时把 cfg 默认值切到 `true`，env 仍保留作为最终回滚开关。

**实施位置：** 在 `buildCompactSummary()` 的入口判断 —— flag 关闭时**不**进入 `try { callCompactionLLM() } catch { 模板 }` 分支，直接走 `formatCompactSummaryFallback()` 并把 `compaction_method` 标记为 `'template'`，`details` 中**不**写 `llm_error`（因为根本未尝试调用）。

```ts
async function buildCompactSummary(input): Promise<CompactSummaryResult> {
  if (!cfg.compaction.llmSummaryEnabled) {
    return { summary: formatCompactSummaryFallback(input), method: 'template' }
  }
  // 文档 7 第 4.2 节定义的 try/catch 降级路径
  try { ... } catch { ... }
}
```

这样 flag 关闭时两条产出（`compaction entry.summary` 与 `writeMemorySummary` 写入文件）天然继续走文档 7 第 5.6 节要求的同源路径，不会破坏审查清单第 5.3 节的一致性约束。

## 4. 可观测性指标

### 4.1 单次 compaction 的结构化日志

每次 `applyCompactionIfNeeded()` 完成时输出一条 `logger.info`（降级走 `logger.warn`，按文档 7 第 4.2 节），关键字段：

| 字段 | 来源 | 用途 |
| --- | --- | --- |
| `sessionKey` | manager | 关联具体会话 |
| `compaction_method` | result.method | 区分 LLM / 模板 |
| `duration_ms` | 调用前后时间差 | 耗时分布 |
| `summary_chars` | summary.length | 摘要长度分布 |
| `previous_summary_present` | input.previousSummary != null | 迭代摘要占比 |
| `compacted_message_count` | source 计算 | 触发规模 |
| `model_id` | bound.projection.model | 降级与模型相关性分析 |
| `llm_error` | catch 分支 | 仅降级时存在 |

这些字段全部走 Lecquy 现有 logger，**不引入新基础设施**（满足文档 10 第 7.5 节验收标准）。

### 4.2 聚合指标（按日，可由现有日志导出）

- 降级率：`count(compaction_method='template' & flag_enabled=true) / count(*)`，目标 < 5%；> 5% 触发 L1 评估
- LLM 摘要耗时分布：p50 / p95 / p99，p99 决定下一轮 `compactionTimeoutMs` 是否需要调整（文档 7 第 5.4 节预留的可调项）
- 摘要长度分布：发现"摘要 < 200 字符"的异常压缩（多为 LLM 输出格式破损）
- 多轮 compaction 内的事实迭代正确率：从抽样审计获得（见第 6 节）

### 4.3 报警阈值

- 降级率单日 > 10%：立刻评估，1 小时内决定是否 L1 回滚
- 降级率单日 > 5%：进入观察名单，连续 2 天则评估
- p99 耗时超过 `compactionTimeoutMs * 0.8`：调整超时配置（L2）

## 5. 回滚预案分级

| Level | 典型触发场景 | 动作 | 生效时间 | 数据影响 |
| --- | --- | --- | --- | --- |
| L1 | 降级率突增、prompt 输出结构性错乱、模型层接口异常 | 关闭 `COMPACTION_LLM_SUMMARY_ENABLED` env | 配置热加载（取决于 cfg 重载机制） | 无；新 entry 走模板，已写入 entry 保留 |
| L2 | 超时频发但摘要质量正常 / `maxTokens` 不足 / tool output 截断过紧 | 调整对应 cfg 参数（`compactionTimeoutMs` / `compactionMaxOutputTokens` / `compactionToolOutputMaxChars`） | 同 L1 | 无 |
| L3 | L1 关闭后仍有异常（说明影响范围超出 LLM 路径） | revert Phase 1 PR | 走部署流程 | 无；已写入 LLM 摘要 entry 保留 |
| L4 | 已写入的摘要被发现严重错误（敏感信息泄露 / 严重事实错误） | 追加修复 entry 覆盖问题 entry，**不**做物理删除 | 人工介入 | session tree append-only，仅以追加形式修正 |

L1 是默认手段；L2 用于参数级问题；L3 用于代码级问题；L4 是脏数据应急。预期 99% 事件停在 L1。

## 6. 摘要质量审计抽样

上线初期 2 周（覆盖阶段 2 + 阶段 3 前半段）执行人工抽样审计：

1. **抽样：** 每天从 LLM 摘要中随机抽 5 条，按 `sessionKey + compactionEntryId` 锁定
2. **重建原文：** 通过 `firstKeptEntryId` 与 `compacted_through_entry_id` 从 session tree 拉对应 messageEntries 区间
3. **对照评估：** 摘要与原文逐章节核对，按以下三类标记问题：
   - 关键事实缺失：原文中明确出现的目标 / 文件路径 / 关键决策在摘要中漏写
   - 错误事实：摘要中的事实与原文冲突
   - 结构破坏：缺章节、章节顺序错、未保留 SUMMARY_TEMPLATE 中要求的"（无）"占位
4. **记录与复盘：** 写入审计 csv（字段：日期、sessionKey、entryId、问题类别、严重度、原文摘录），每周复盘一次

**停止抽样的条件：** 连续 2 周（每天 5 条 = 70 条样本）无"关键事实缺失"或"错误事实"。此时进入阶段 3 后半段，转为被动审计 —— 仅在用户反馈"AI 不记得之前说过的事"时按 sessionKey 调出最近 compaction entry 做事后比对。

抽样规模可视降级率与人力调整，但**不**改抽样的来源（永远从生产 LLM 摘要采样，不从内部环境采样）。

## 8. Phase 1 显式不做的事（灰度维度）

为避免范围漂移：

- 不做用户级 / session 级 / 百分比灰度（见 1.1）
- 不引入 history 指纹 / hash 字段（见 1.3）
- 不引入新的可观测基础设施（指标全部基于现有 logger）
- 不改变 compaction entry 的 schema，仅扩展 details 中的字段（与文档 7 第 5.5 节一致）
- 不做"清理 feature flag" PR（见第 2 节阶段 4）
