# 上下文压缩关键点审查清单

更新日期：2026-04-30（首版 2026-04-29）

---

## 1. 本文档的用途

这份文档是一份**开发过程中的检查表**，不是架构分析，也不是实施计划。

目的是：每次修改上下文压缩相关代码时，对照这份清单检查是否遗漏了关键约束。

适用场景：

- 修改 `compact.ts` 中的触发逻辑
- 修改 `compact-summary.template.ts` 或升级为 LLM 摘要
- 修改 `augmented-context-builder.ts` 中的消息顺序
- 修改 `session-manager.ts` 中的 `buildSessionContext()`
- 修改 `session-runtime-service.ts` 中的 `applyCompactionIfNeeded()` 调用点

## 2. 触发策略相关

### 2.1 触发阈值

- [ ] 触发阈值是否仍然是固定消息条数（`COMPACT_TRIGGER_MESSAGE_EVENTS = 50`）？
  - 如果是：记录这是 prototype 级策略，不适合生产使用
  - 如果已改为 token-aware：确认 token 估算公式是否合理（字符数/4 是粗估，可能有 2x 误差）

- [ ] 是否考虑了 model context window 的差异？
  - 不同 model 的 context window 差距可达 10x（32k vs 200k），固定阈值对它们都是错的

- [ ] 触发时机是否在 run 结束后？
  - 如果是：本轮 LLM 调用如果本身已撑爆 context，这次 run 会失败，压缩救不了
  - 这是当前架构限制，记录为已知问题

### 2.2 触发入口

- [ ] 所有路径（simple / plan manager / plan final-answer / run_resume）是否都会触发 `applyCompactionIfNeeded()`？
  - 确认调用点在 `session-runtime-service.ts`，不是分散在各路径里

- [ ] `applyCompactionIfNeeded()` 失败时是否有 try/catch？
  - 压缩失败不应该中断整个 run

## 3. 摘要生成相关

### 3.1 摘要内容质量（当前模板方式）

- [ ] 摘要中是否有语义内容（目标、进度、关键决策）？
  - 当前模板方式只有"截断采样"，不包含任何语义信息
  - **这是最大的已知缺陷**，等待 Phase 1 升级为 LLM 摘要

- [ ] `COMPACT_MAX_SUMMARY_CHARS = 1200` 是否足够容纳关键信息？
  - 对于长会话，1200 字可能极度不够
  - 升级为 LLM 摘要后，摘要长度应由 LLM 自行决定（设置合理的 max_tokens 上限）

### 3.2 摘要迭代

- [ ] 有 `previousSummary` 时，是否正确读取并传入？
  - 当前：从 `latestCompaction.summary` 读取，截取 280 字传入（不完整）
  - 升级后：应完整传入，让 LLM 基于完整旧摘要迭代

- [ ] `previousSummary` 是否来自最近一次压缩？
  - 确认 `findLatestCompaction()` 返回的是 entries 中最新的 compaction entry

- [ ] 多轮压缩后，摘要是否能正确累积？
  - 验证：压缩 3 次后，第 3 次的摘要是否包含第 1、2 次压缩的关键信息

### 3.3 摘要生成方法标记与降级监控

- [ ] compaction entry 的 `compaction_method` 字段是否正确写入（`'llm' | 'template'`）？
  - 用于区分本次摘要是 LLM 生成还是降级到模板方式（依据：文档 7 4.2）

- [ ] LLM 摘要生成失败时是否走 `logger.warn` 而非 `console.warn`？
  - 必须通过结构化日志可观测降级事件（依据：文档 7 4.2）

- [ ] LLM 失败时 `details.llm_error` 是否被记录？
  - `llm_error` 应包含 `{ name: string; message: string }`，仅在降级时存在（依据：文档 7 4.2）

### 3.4 摘要写入

- [ ] `writeMemorySummary()` 写入的文件路径是否正确？
  - 当前写入 `.lecquy/memory-summary.md`，确认路径与 `buildContextMessages()` 读取的一致

- [ ] LLM 摘要生成失败时是否有降级处理？
  - 升级为 LLM 摘要后，必须实现降级：LLM 失败 → 使用模板摘要

## 4. 裁剪边界相关

### 4.1 tail 保留量

- [ ] `COMPACT_RECENT_TAIL = 10` 是否合理？
  - 10 条消息可能是 1k tokens（短消息），也可能是 100k tokens（文件读取结果）
  - 固定条数对 token 量没有保证

- [ ] recent tail 里是否可能有超大 tool output？
  - 如果有：进入下一轮 LLM 调用时，这些大 tool output 会占满剩余 context
  - 需要 tool output 截断（Phase 3）

### 4.2 firstKeptEntryId 的正确性

- [ ] `firstKeptEntryId` 是否指向正确的消息？
  - 确认它指向 `candidateMessages[candidateMessages.length - COMPACT_RECENT_TAIL]`
  - 确认这个 entry 的 id 在 session tree 中存在且唯一

- [ ] `buildSessionContext()` 是否正确从 `firstKeptEntryId` 开始重放？
  - 检查 `session-manager.ts` 中的相关逻辑：找到 compaction entry → 找到 firstKeptEntryId → 从此处开始

### 4.3 compacted 历史的隐藏

- [ ] `buildSessionContext()` 是否完全跳过 `firstKeptEntryId` 之前的历史消息？
  - 这些消息不应出现在送给 LLM 的 contextMessages 中
  - 它们在 append-only session tree 中仍然保留（用于审计/回放），但不应进入上下文

### 4.4 历史序列化的 tool_use / tool_result 断裂处理

- [ ] `formatHistoryForCompaction` 序列化是否正确处理了 tool_use / tool_result 切口断裂场景？
  - compactedMessages 被时间轴切割，可能在工具调用与其结果之间断裂（不成对）
  - 必须序列化为 `↳ tool_name(args) → output` 单行格式，避免 model messages 的配对校验报错
  - tool output 应截断到 2000 字符（依据：文档 7 4.1）

## 5. 上下文组装顺序相关

### 5.1 compact 场景的顺序稳定性

- [ ] compact 场景下的上下文顺序是否固定？

  当前 `augmented-context-builder.ts` 的顺序：
  ```
  [compact summary message]
  [recent tail messages]
  [memory recall block]  ← 这个位置在 compact 和非 compact 场景下不同
  [当前用户输入]
  ```

  - 确认 memory recall block 在 compact 场景下的插入位置是否与非 compact 场景一致
  - 不一致会导致共享前缀不稳定（影响 prompt caching）

- [ ] compact summary message 是否始终是第一条消息？
  - 确认 `sessionContext.compaction.summaryMessageIndex === 0` 的判断逻辑

### 5.2 非 compact 场景的顺序

- [ ] 非 compact 场景下是否有多余的 compact-related 逻辑？
  - 确认 `buildAugmentedContext()` 在 `sessionContext.compaction === null` 时直接推送全量 `sessionContextMessages`

### 5.3 摘要与文件存储的同源性

- [ ] `writeMemorySummary()` 写入文件的内容是否与 `compaction entry.summary` 字段完全一致？
  - 无论是 LLM 生成还是模板降级，两处必须同源，不能出现"entry 是 LLM 摘要、文件是模板"的不一致（依据：文档 7 5.6）
  - 确认 `writeMemorySummary` 接收的是 `buildCompactSummary` 返回的 `result.summary`

## 9. 容易犯的错误（已知坑）

### 9.1 `getDurableMessageEntries()` 只计 user/assistant 消息

`getDurableMessageEntries()` 过滤掉了 type 不是 `"message"` 的 entries（如 `compaction`、`run_started` 等）。

**坑：** 如果把 compaction entry 或 run 事件误计入 50 条，触发时机会比预期更早。

### 9.2 `candidateMessages` 从上次压缩后的 `firstKeptEntryId` 开始

```ts
if (latestCompaction?.type === 'compaction') {
  const startIndex = messageEntries.findIndex(
    (entry) => entry.id === latestCompaction.firstKeptEntryId
  )
  if (startIndex >= 0) {
    candidateMessages = messageEntries.slice(startIndex)
  }
}
```

**坑：** 如果 `firstKeptEntryId` 在 messageEntries 中找不到（数据不一致），`startIndex` 为 -1，`candidateMessages` 退化为全量，可能导致重复压缩。

需要确认：`resolveCompactSource()` 在 `startIndex === -1` 时的行为是否安全。

### 9.3 `buildCompactSummary` 的总长截断在 `formatCompactSummary` 的最外层

```ts
return truncate(lines.join('\n'), input.maxSummaryChars ?? COMPACT_MAX_SUMMARY_CHARS)
```

**坑：** 截断发生在最终 join 之后，这意味着 `previousSummary` 可能已经占用了大量字符，导致后续采样消息被完全截掉。

升级为 LLM 摘要后，这个问题自动消失，但如果保留模板方式作为降级，需要修复这个截断顺序。

### 9.4 `writeMemorySummary` 写的是上一次摘要 + 最近消息采样，不是语义化摘要

当前 `writeMemorySummary()` 写入文件系统的内容，就是 `buildCompactSummary()` 的输出。如果这个文件被 `buildMemoryRecallBlockLegacy()` 读取并注入上下文，模型实际收到的是截断字符串，不是有意义的摘要。

**升级为 LLM 摘要后，必须同步更新 `writeMemorySummary()`，确保写入文件的是 LLM 生成的结构化摘要。**

### 9.5 `bound.projection.model` 非 Anthropic provider 时 `completeSimple` 不可用

`bound.projection.model` 可能是 OpenAI、本地 vLLM 等非 Anthropic 兼容的 model。当选用这类 model 时，`completeSimple` 调用会失败（无 Anthropic API 兼容层）。

**处理方式：** 在 compaction LLM 调用点用 try/catch 包住，若捕获"unsupported model"异常，立即降级到模板方式记日志，不向用户暴露错误（依据：文档 7 5.3）。

## 10. 快速参考：当前关键常量

| 常量 | 当前值 | 位置 | 含义 |
| --- | --- | --- | --- |
| `COMPACT_TRIGGER_MESSAGE_EVENTS` | 50 | `compact.ts` | 触发压缩的消息条数阈值 |
| `COMPACT_RECENT_TAIL` | 10 | `compact.ts` | 保留最近 N 条消息 |
| `COMPACT_MAX_SUMMARY_CHARS` | 1200 | `compact-summary.template.ts` | 摘要总长上限 |
| `COMPACT_PREVIOUS_SUMMARY_CHARS` | 280 | `compact-summary.template.ts` | 上次摘要截取长度 |
| `COMPACT_SAMPLE_MESSAGE_CHARS` | 140 | `compact-summary.template.ts` | 单条消息采样长度 |
| `COMPACT_SAMPLE_MESSAGE_LIMIT` | 8 | `compact-summary.template.ts` | 采样消息条数上限 |

升级为 LLM 摘要后，后 5 个常量可以废弃（仅保留为降级路径使用）。
