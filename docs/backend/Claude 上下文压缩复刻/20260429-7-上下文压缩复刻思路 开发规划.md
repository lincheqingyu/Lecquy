# 上下文压缩复刻思路

更新日期：2026-04-30（首版 2026-04-29）

## 1. 复刻的核心前提

### 1.1 Lecquy 与 OpenCode 的架构差异不应强行抹平

OpenCode 是单体 Go-style Effect 服务架构，session 是 message stream，compaction 任务本身是一条 user 消息。

Lecquy 是 append-only session tree + projection 体系，有 WebSocket 协议、plan/manager/worker 工作流分离。

**复刻的目标是对齐"效果"，而不是复刻"形式"。** 以下几点是有意不改的：

- 不把 Lecquy 的 session tree 改造为 message stream
- 不把 append compaction entry 改为"真实 user 消息"
- 不把 plan/worker 工作流改为 OpenCode 的单体 runLoop
- 不复刻 Effect/Layer 的服务依赖注入体系

### 1.2 最值得复刻的三件事

按价值排序：

1. **摘要生成从模板升级为 LLM agent**（语义质量差距最大）
2. **触发策略从消息条数升级为 token-aware**（时机合理性差距最大）
3. **tool result prune 机制**（token 浪费最直接）

tail 保留量（固定 10 条 → token budget）是锦上添花，可以在前三项稳定后再推进。

### 1.3 不宜过早收敛的问题

在动手之前，以下几个问题需要先想清楚，而不是直接照搬 OpenCode 的答案：

- Lecquy 的 compaction agent 应该用哪个 model？是跟主对话共用，还是单独配？
- 摘要应该用中文还是英文？（OpenCode 是英文，但 Lecquy 是中文产品）
- `previousSummary` 的迭代方式：是真正合并（OpenCode 做法），还是只拼接旧摘要？
- token 估算用什么方式？（OpenCode 用 `Token.estimate(JSON.stringify(msgs))`，Lecquy 目前没有 token 估算工具）

## 2. 当前基线的有利条件

Lecquy 已经具备这些基础，不需要从零开始：

- `applyCompactionIfNeeded()` 是统一入口，改触发策略不需要改调用点
- `appendCompaction()` 与 `buildSessionContext()` 已解耦，改摘要内容不影响持久化结构
- `compact-summary.template.ts` 是单一定义，升级为 LLM 调用只需改这一个地方
- `augmented-context-builder.ts` 已经把 compact summary 和 recent tail 的组装逻辑分开
- `session-runtime-service.ts` 已有 `buildContextMessages()` 统一上下文入口

## 3. 分阶段复刻路线

### Phase 1：摘要生成升级为 LLM agent（最高优先级）

**目标：** 让压缩摘要从"截断字符串"变为"LLM 推理的语义浓缩"。

**核心改动点：**

`compact.ts` 的 `buildCompactSummary()` 当前是纯函数，改为异步 LLM 调用：

```
buildCompactSummary(source)
→ 构造 prompt（历史消息 + SUMMARY_TEMPLATE）
→ 调用 LLM（单次 completion，不需要 tool）
→ 返回结构化摘要文本
```

**需要决策的问题：**

1. **用哪个 model？**
   - 选项 A：与主对话共用同一个 model（简单，但摘要质量取决于用户选的 model）
   - 选项 B：单独配置 compaction model（复杂，但更灵活；OpenCode 的做法）
   - 选项 C：写死一个轻量 model（claude-haiku 之类），独立于主对话
   - 推荐：先选 A（最小改动），后续再支持 B

2. **SUMMARY_TEMPLATE 用中文还是英文？**
   - Lecquy 是中文产品，用户对话也是中文
   - OpenCode 的模板是英文
   - 推荐：将 OpenCode 的 7 个字段翻译为中文，但字段结构保持一致

3. **送给 compaction LLM 的 messages 如何组织？**
   - OpenCode 的做法：`MessageV2.toModelMessagesEffect(head, model, { stripMedia: true, toolOutputMaxChars: 2000 })`
   - Lecquy 对应的是：把 `compactedMessages`（被裁剪的部分）转换为 `AgentMessage[]` 送入 LLM
   - tool output 需要截断（OpenCode 截到 2000 字），避免摘要 LLM 本身被 tool output 撑爆

4. **`applyCompactionIfNeeded` 的签名需要变化**
   - 当前是同步/异步函数，返回 `boolean`
   - 升级后需要接收 model 和 api key（或从 session context 读取）
   - 调用点在 `session-runtime-service.ts`，需要把 model 信息传进来

**摘要模板草案（中文版）：**

```
按照下方 <template> 内的 Markdown 结构输出，保持章节顺序，不要输出 <template> 标签本身。

<template>
## 目标
- [单句总结用户的核心任务]

## 约束与偏好
- [用户明确的约束条件、偏好或规范，无则写"（无）"]

## 进度
### 已完成
- [已完成的工作，无则写"（无）"]
### 进行中
- [当前进行中的工作，无则写"（无）"]
### 受阻
- [当前阻塞点，无则写"（无）"]

## 关键决策
- [已做的重要决策及原因，无则写"（无）"]

## 下一步
- [按优先级排列的后续行动，无则写"（无）"]

## 关键上下文
- [重要技术事实、错误信息、开放问题，无则写"（无）"]

## 相关文件
- [文件或目录路径：与任务的关联原因，无则写"（无）"]
</template>

规则：
- 所有章节必须保留，即使内容为空也写"（无）"。
- 使用简洁的要点，不要写成段落式散文。
- 保留精确的文件路径、命令、错误字符串和标识符。
- 不要提及摘要过程本身，也不要说"上下文已被压缩"之类的话。
```

**迭代摘要（有 previousSummary 时）：**

```
（在 prompt 末尾追加）
请在以下锚定摘要的基础上进行更新：合并上方对话历史中的新事实，
删除已过期的内容，保留仍然成立的细节。

<previous-summary>
{{ previousSummary }}
</previous-summary>
```

**验收标准：**

- 压缩后，下一轮对话中 LLM 能正确引用压缩前的目标和关键文件
- 多轮压缩后，摘要内容能正确迭代（旧事实被保留，新事实被融入）
- 压缩 LLM 调用失败时，回退到当前模板方式（降级处理，不 crash）

### Phase 2：触发策略升级为 token-aware

**目标：** 不再用固定的 50 条消息作为触发阈值，而是根据当前 session 的 token 估算量决定是否压缩。

**核心改动点：**

`resolveCompactSource()` 当前：

```ts
if (messageEntries.length < COMPACT_TRIGGER_MESSAGE_EVENTS) return null
```

改为：

```ts
const estimatedTokens = estimateSessionTokens(messageEntries)
const threshold = getCompactionThreshold(modelContextWindow, outputReserved)
if (estimatedTokens < threshold) return null
```

**需要解决的子问题：**

1. **token 估算**：Lecquy 目前没有 token 计数工具。
   - 选项 A：字符数 / 4 的粗估（OpenCode 也用了这个 `Token.estimate = charCount / 4`，只是 fallback）
   - 选项 B：接入 Anthropic 的 `countTokens` API（精确，但有额外 API 调用开销）
   - 推荐：先用选项 A（简单，足够触发判断），后续支持 B
   - 粗估公式：`Math.ceil((text.length) / 4)`，对所有消息的文本内容求和

2. **threshold 计算**：
   - 需要知道当前 model 的 context window 大小
   - Lecquy 当前在 `session-runtime-service.ts` 里已有 model 信息
   - 可以从 model spec 读取 `contextWindow`，再减去 `min(20000, maxOutputTokens)` 作为 threshold
   - 推荐预留 80% 触发（即估算 token 达到 `threshold * 0.8` 时触发），给 LLM 回复留空间

3. **触发时机**：
   - Lecquy 当前在 run 结束后触发，OpenCode 在 streaming 中途也能触发
   - **Phase 2 不改触发时机**，只改触发策略（条数 → token 估算）
   - 在 run 结束后用 token 估算做判断，已经是比当前条数阈值大得多的改进

4. **保留 tail 的 token 感知**：
   - 同步把 `COMPACT_RECENT_TAIL = 10` 改为 token budget（usable 的 20%，上限 8k tokens）
   - 从最近消息倒序累计，直到 budget 耗尽

**验收标准：**

- 短消息长对话（每条 20 字）不再在 50 条时就压缩
- 长消息短对话（每条 10k tokens）能在 3-5 条时就触发压缩
- 压缩后 recent tail 的 token 量在合理范围内（不超过 8k tokens）

---

### Phase 3：tool result prune 机制

**目标：** 避免历史 tool output 占用大量上下文 token，尤其是文件读取、搜索结果等大体积 output。

**核心改动点：**

在压缩完成后，或 run 结束后，扫描 recent tail 之外的历史消息，对 tool output 做截断标记。

```
prune(manager):
  entries = manager.getEntries()
  从最新消息倒序扫描 tool result entries：
    - 跳过最近 2 个 user turn（保护近端）
    - 累计 tool output token：
      - total <= PRUNE_PROTECT (40k) → 保护，跳过
      - total > PRUNE_PROTECT → 加入待 prune 列表
    - if (pruned_total > PRUNE_MINIMUM (20k)):
      → 对这些 entries 打上 pruned = true 标记
      → 下次 buildSessionContext() 时，这些 tool output 替换为 "[output pruned]"
```

**与当前架构的适配：**

Lecquy 的 session tree 是 append-only，不能直接修改历史 entry。可以：

- 选项 A：在 compaction entry 里记录被 prune 的 entry ID 列表，`buildSessionContext()` 跳过它们
- 选项 B：在 `buildAugmentedContext()` 里做 tool output 截断（不改 entry，只改送给 LLM 的消息）
- 推荐：选项 B（最小改动，不改持久化结构）

**验收标准：**

- 长时间对话后，historical tool output 不再占满 recent tail 之外的上下文
- prune 后下一轮 LLM 调用的 token 估算明显下降

---

### Phase 4：streaming 中途溢出检测（中期目标）

这是 OpenCode 与 Lecquy 差距最根本的一点，但也是改动最大的。

**前提条件：** Phase 1-3 完成，且 Lecquy 已有 token-aware 触发策略。

**核心思路：**

在 `executeSimple()` / `executeManager()` 等 LLM 调用点，监听每次 step 结束后的 token 用量。如果检测到溢出：

1. 中断当前 LLM 调用
2. 触发压缩
3. 用新的上下文重新发起 LLM 调用

这需要 LLM 调用层支持"中途返回 compact 信号"，与当前 Lecquy 的 runSimpleAgent 接口有较大差距，需要单独设计。

**Phase 4 不在本轮规划内，作为中期路线记录。**

## 4. 设计问题收敛结论

本节是对原"留待 Opus 推敲"的四个问题的收敛回答。每个问题给出选定方案、决策依据，以及在 Lecquy 当前架构下的具体落地路径。

### 4.1 compaction LLM 的 message 组织方式 → 选 B（文本块）

**结论：** 把被压缩的历史序列化为单一文本块，作为一条 `role: "user"` 消息送入 compaction LLM，**不**复刻 OpenCode 的 model-messages 重放方式。

**决策依据：**

1. **任务性质决定形态。** Phase 1 的目标是"语义浓缩"，compaction LLM 不需要在 message-tree 语境下扮演角色，保留 message 边界元数据（role / id / parts / model 信息）的价值远低于实现成本。
2. **避开 tool_use / tool_result 配对约束。** Anthropic API 对 `tool_use` 与 `tool_result` 必须成对的校验非常严格。`compactedMessages` 是按时间切的中间段，前后切口可能落在 tool_use 与其 tool_result 之间，直接以 model messages 重放会触发 4xx。文本块方案在序列化时把工具调用降级为单行 `[工具:name] args 摘要 → output 截断`，绕过这个边界。
3. **方便统一截断 tool output。** OpenCode 在 `toModelMessagesEffect` 里通过 `toolOutputMaxChars: 2000` 控制 tool output；Lecquy 用文本块只需在序列化时调用一次 `truncate(toolOutput, 2000)`，比在 message 转换层做更可控。
4. **未来可平滑升级到方案 A。** 如果实验证明保留 message 结构能提升摘要质量，只需替换 `formatHistoryForCompaction()` 实现，外层接口不变。

**Lecquy 落地形式：**

工具调用必须嵌套在所属"助手段落"内部，**不要**让 `[工具:name]` 单独成行成为独立角色。否则 LLM 容易把工具结果当成第三方发言，进而在"## 进度"里写出"工具说……"这种不通顺的摘要。

推荐格式：

```
[用户] 帮我看下 compact.ts 里 resolveCompactSource 的 startIndex 边界

[助手] 我来读取这个文件。
  ↳ read_file({"path":"backend/src/memory/compact.ts"}) → import { promises as fs } from 'node:fs'\n... (前 2000 字符)
  ↳ findIndex 在 entry 不存在时返回 -1，slice(-1) 会取最后一条而不是空数组，这就是 bug 根因。

[用户] 那帮我改一下让 startIndex 为负时直接 return null

[助手] 我来修改 resolveCompactSource()。
  ↳ edit_file({"path":"backend/src/memory/compact.ts","old":"...","new":"..."}) → 编辑成功
  ↳ 已加上 if (startIndex < 0) return null 的早返回。
```

约束：
- 不送入 `thinking` 内容（过程性产物，对最终摘要无增量信息，OpenCode 也丢弃）
- 不送入 model / provider / timestamp 字段
- 工具调用以 `  ↳ tool_name(args 摘要) → output 摘要` 形式嵌入助手段落；多次工具调用串成多个 `↳` 行
- args 摘要：JSON 序列化后保留前 200 字符
- output 摘要：纯文本截断到 2000 字符（OpenCode 一致）
- 助手文本与工具调用之间不插额外空行，保留段落整体感
- 不同角色的相邻消息之间用单个空行分隔

### 4.2 compaction 失败时的降级策略 → 选 A（模板降级 + 可观测）

**结论：** compaction LLM 调用失败一律降级到 `formatCompactSummaryFallback()`（即原模板方式），不向用户暴露错误，但必须留下可观测痕迹。

**决策依据：**

1. **OpenCode 的"硬失败"是其架构特性，不应照搬。** OpenCode 在 streaming 中途触发压缩，如果 compaction 也失败，下一轮 LLM 调用必然撑爆 — 没有继续路径，只能终止。Lecquy Phase 1 在 run 结束后触发，不存在"下一轮一定会爆"的紧迫性，没有理由学这个严格策略。
2. **降级到模板方式不会比升级前更差。** 最坏情况仍然是 Phase 0 的字符截断摘要，与升级前一致；不会出现"压缩越升级、稳定性越差"的回归。
3. **压缩是后台优化，不是用户主诉。** 把内部 LLM 调用失败抛成 run-level 错误，违背用户对"对话能继续"的核心心智。
4. **必须留观测痕迹。** 用户无感 ≠ 团队无感。失败原因必须能被审计。

**Lecquy 落地形式：**

```ts
async function buildCompactSummary(input: CompactSummaryInput): Promise<CompactSummaryResult> {
  try {
    const summary = await callCompactionLLM(input)  // 内含可配置硬超时，见 5.4
    return { summary, method: 'llm' }
  } catch (error) {
    logger.warn('[compact] LLM 摘要失败，降级到模板方式', {
      reason: error instanceof Error ? error.name : 'unknown',
      message: error instanceof Error ? error.message : String(error),
    })
    const summary = formatCompactSummaryFallback({ ... })
    return { summary, method: 'template', llmError: error }
  }
}
```

`compaction entry.details` 字段必须新增：
- `compaction_method: 'llm' | 'template'`
- `llm_error?: { name: string; message: string }`（仅降级时存在）

**异常分类：**
- 超时 / 网络错误 / 429 / 5xx / 抽象层异常 → 都走降级
- 调用前置准备失败（apiKey 缺失、model 配置异常）→ logger.error + 降级，不抛错
- **不在 Phase 1 处理"压缩 LLM 自身溢出"。** 当前触发条件下，序列化 history 体量约 50-60k tokens，远低于 200k context window。极端长会话的兜底由 Phase 2（更早触发）和 Phase 3（tool result prune）联合解决。

### 4.3 previousSummary 的迭代边界 → 完整传入 + 显式合并指令

**结论：** previousSummary 完整传入，**不做任何字符截断**，且必须在 prompt 中显式要求 LLM "合并新事实、删除已过期内容、保留仍然成立的细节"。

**决策依据：**

1. **结构完整性 >> token 节省。** 一份 SUMMARY_TEMPLATE 输出大约 1.5-2k tokens，相对 200k context window 占比约 1%。从中省下 1k tokens 的代价是把"## 关键上下文""## 相关文件"等关键章节直接砍掉，后续摘要无法继承这些事实。
2. **OpenCode 的合并语义是必要功能。** 如果只拼接旧摘要、不让 LLM 主动合并，"## 已完成"会随轮数无限增长，违背"锚定摘要"设计意图。
3. **previousSummary 与 history 必须在 prompt 里语义分离。** 如果把 previousSummary 当作 history 的一部分塞进 conversation block，LLM 会把它当成对话历史中的一条消息处理；必须独立放入 `<previous-summary>` 块，并通过指令告诉 LLM 这是"上次的锚定摘要"。

**Lecquy 落地形式：**

prompt 末尾追加（仅当存在 previousSummary）：

```
请在以下锚定摘要的基础上更新：
- 把上方对话历史中的新事实合并进来
- 删除已被新事实超越或不再成立的内容
- 保留仍然成立的细节
- 输出仍然按下方 SUMMARY_TEMPLATE 的章节结构

<previous-summary>
{完整的 latestCompaction.summary，不截断}
</previous-summary>
```

**与现有 candidateMessages 逻辑配合：**
- `compact.ts:55-57` 已经做了"自上次 compaction 起切片"，所以 history 自然只含本次新增；previousSummary 与 history **不会**重复同一批事实
- 多次 compaction 时 `findLatestCompaction()` 返回最新一条，即可拿到最近一次的完整摘要

**降级路径同步处理：** `formatCompactSummaryFallback()` 中保留 `COMPACT_PREVIOUS_SUMMARY_CHARS = 280` 截断逻辑（降级路径下 previousSummary 是结构化 markdown，截断仍会破坏结构 — 但降级本身已是退化模式，可以接受）。

### 4.4 摘要应该作为 system / user / assistant 消息？ → Phase 1 维持 user 消息现状

**结论：** Phase 1 **不动**外层组装顺序，摘要继续以合成 user 消息（`createSyntheticUserContextMessage`）的方式插入；assistant / system 形式作为后续单独的优化项，不在本轮范围内。

**决策依据：**

1. **任务边界控制。** Phase 1 的核心改动是 `buildCompactSummary()` 内部从"模板"换成"LLM"。同时改 `augmented-context-builder.ts` 的消息形态会让 bug 定位成本爆炸。
2. **拒绝 OpenCode 的 assistant 消息方案。** OpenCode 把摘要做成带 `summary: true` 的 assistant 消息，是因为它需要与 compaction user 消息配对（OpenCode 的 session 是 message stream）。Lecquy 是 append-only event tree，没有这个配对约束。伪造 assistant 消息的实际代价是**污染审计**：assistant 消息通常带 model / provider 字段，伪造一条带不真实 model 字段的 assistant 消息会让历史回放误判"这条回复来自哪个模型"，破坏 session tree 作为审计源的语义。
3. **"system prompt 注入块"是更合理的长期方向，但需要 system prompt 改造为 base + dynamic 双层结构。** 这是独立优化，不应耦合到 Phase 1。
4. **augmented-context-builder.ts 的"memory recall 在 compact 与非 compact 场景下插入位置不同"问题（详见审查清单 8 节 5.1）独立处理。** Phase 1 不处理这个一致性问题，作为已知遗留项继续标注。

> 关于 prompt caching 的说明：当前合成 user 消息的方式本身就不是 caching-friendly 的（compact 与非 compact 场景的 prefix 已经不同，审查清单 5.1 已标注）。因此"改成 assistant 消息会破坏 caching"不构成反对理由 — 现状对 caching 也不友好。caching 是预先存在的独立问题，应在统一 augmented-context-builder 组装顺序时一起处理，而不是用来论证 4.4 这个决策。

**后续可选优化（不在本轮）：** 把摘要从 user 消息升级为 system prompt 的 dynamic 块，与基础 system prompt 一起进入 prompt caching 静态部分，同时统一 memory recall / compact summary 的插入顺序。

## 5. Phase 1 前置约束（实施前必须明确）

本节是从上面四个收敛结论中沉淀出的工程边界。在 Phase 1 编码开始前需要逐项确认到位，否则实施过程中会反复返工。

### 5.1 LLM 调用层用 `completeSimple`，不用 `runSimpleAgent`

`@mariozechner/pi-ai` 暴露的 `completeSimple` 已被 `session-v2/session-service.ts` 用于"自动标题生成"（参考其 `createSimpleTitle()` 实现）。compaction 摘要的需求与之完全同构：单轮、无 tool、纯文本输出。

```ts
import { completeSimple, type UserMessage } from '@mariozechner/pi-ai'

const response = await completeSimple(model, {
  systemPrompt: SUMMARY_INSTRUCTIONS,   // 见 5.2
  messages: [{
    role: 'user',
    content: buildCompactionPrompt({ history, previousSummary }),
    timestamp: Date.now(),
  } satisfies UserMessage],
}, {
  apiKey,
  signal: AbortSignal.timeout(cfg.compactionTimeoutMs),  // 见 5.4，起步 60_000，可配置
  maxTokens: 4_096,
})
```

**不用 `runSimpleAgent`**：它会强制装载 simple tools、permission manager、memory flush 等 side effect，全是本场景不需要的。

### 5.2 system prompt 与 user prompt 的边界

文档 9 把 SUMMARY_TEMPLATE 全部放在 user 消息里。更合理的拆分：

- **system prompt（每次相同）：** SUMMARY_TEMPLATE 主体（章节结构、输出规则、不要提及压缩过程等约束）
- **user prompt（每次变化）：** "请压缩以下历史"指令 + history 文本块 + 可选的 SUMMARY_UPDATE_ANCHOR + previousSummary

这样切分后：
- 指令与数据分层，符合 Anthropic 推荐做法
- system prompt 命中 prompt caching 的概率更高（虽然 user 部分每次不同会让缓存大部分失效，但 system 仍能受益）
- 调试时容易区分"是模板写错了"还是"是数据序列化错了"

### 5.3 model 选择策略

复用 `bound.projection.model`（与主对话同源）：

```ts
const compactionModel = bound.projection.model ?? this.cfg.DEFAULT_MODEL
```

**降级触发条件：**
- model 不存在 / 无 apiKey → 直接走模板降级，记 warn
- 非 Anthropic / 非 OpenAI 兼容 model（如本地 vLLM）：在 `callCompactionLLM` 用 try/catch 包住，若 `completeSimple` 抛"unsupported model"，走模板降级

后续单独配置 `compaction.model` 不在 Phase 1 范围。

### 5.4 token 上限与超时

| 维度 | 起步值 | 说明 |
| --- | --- | --- |
| `maxTokens`（输出） | 4_096 | SUMMARY_TEMPLATE 输出典型 800-1500 tokens，预留 4k 给 LLM 余地 |
| 硬超时 | 60 秒（**起步值，需实测调整**） | 见下方说明 |
| tool output 截断 | 2000 字符 | 与 OpenCode 一致，在 `formatHistoryForCompaction` 序列化时执行 |

**关于硬超时：** 60 秒是起步值，**不是** Phase 1 的最终决策。compactedMessages 在触发时通常 40-50 条（含 tool output），序列化后 30-60k 字符，加上 SUMMARY_TEMPLATE 与 previousSummary，输入侧典型 50-60k tokens。摘要 LLM 生成 1-2k tokens 输出的实际耗时取决于 model、provider、网络与峰值排队，波动很大。设计阶段无法预判合理上限。

实施约束：
- 把超时值做成**配置项**而非常量，初始默认 60s
- Phase 1 上线后必须采集实际耗时分布（p50 / p95 / p99），根据 p99 反推合理上限
- 超时后走模板降级（与其他失败路径一致），降级率作为可观测指标长期跟踪
- 如果发现 60s 在峰值时降级率显著（>5%），优先调高超时值而不是改设计

输入侧不需要主动限制（在 200k context model 上 50-60k tokens 远低于上限，且 Phase 2 会从源头改进触发判断）。

### 5.5 `applyCompactionIfNeeded` 接口的可扩展性

`CompactionOptions` 必须设计为开放结构（open shape），方便 Phase 2 不改 schema 直接扩展：

```ts
export interface CompactionOptions {
  readonly llmClient?: never  // 不直接暴露 client，由内部根据 model + apiKey 调 completeSimple
  readonly model: Model<'openai-completions'>
  readonly apiKey: string
  // Phase 2 新增字段（保留扩展位）：
  readonly estimatedTokens?: number
  readonly modelContextWindow?: number
}
```

compaction entry 的 `details` 字段也要预留扩展：

```ts
{
  trigger: 'message_threshold' | 'token_overflow',  // Phase 2 会扩展
  compaction_method: 'llm' | 'template',
  llm_error?: { name: string; message: string },
  kept_message_count: number,
  compacted_message_count: number,
  compacted_through_entry_id: string,
  // Phase 2 预留：
  estimated_tokens_before?: number,
  model_context_window?: number,
  threshold_used?: number,
}
```

### 5.6 writeMemorySummary 与 compaction entry summary 必须同源

`writeMemorySummary()` 写入 `.lecquy/memory-summary.md` 的内容必须与 `manager.appendCompaction(summary, ...)` 中的 `summary` 完全一致 — 不论是 LLM 输出还是模板降级输出。

```ts
const result = await buildCompactSummary(input)
manager.appendCompaction(result.summary, ..., { compaction_method: result.method, ... })
await writeMemorySummary(manager.getCwd(), result.summary)
```

避免出现"compaction entry 是 LLM 摘要、文件是模板摘要"或反之的不一致。

### 5.7 测试边界（在文档 9 第 8 节基础上补充）

文档 9 已覆盖单元测试与集成测试，本节补：

- **mocked LLM 返回非法格式（缺章节）的处理**：Phase 1 用"软校验" — 只记 warning，不重新调用 LLM。理由：避免在出错路径再生成新的 LLM 调用，放大失败成本。
- **超时单测**：把 `compactionTimeoutMs` 设成测试值（如 100ms），mock 一个挂起更久的 client，验证 `AbortSignal.timeout` 起作用、走降级路径、`compaction_method` 为 `'template'`、`llm_error.name` 为 `AbortError` 或等价。
- **previousSummary 完整传入的单测**：构造 previousSummary > 1k 字的场景，验证不被截断（断言 prompt 中包含 previousSummary 的尾段）。
- **多轮 compaction 的事实迭代**：覆盖审查清单 3.2 节的"压缩 3 次后摘要含第 1、2 次的关键事实"。
- **tool_use / tool_result 配对断裂场景**：构造一个被切口分裂的工具调用，验证 `formatHistoryForCompaction` 序列化为 `[工具:...]` 单行而不是抛错。

### 5.8 Phase 1 显式不做的事

为防止范围漂移，明确以下事项不在 Phase 1 范围：

- 修改 `augmented-context-builder.ts` 的消息组装顺序
- 修改 `appendCompaction` 的存储 schema（仅扩展 details 字段）
- 改变摘要在主对话上下文中的角色（保持合成 user 消息）
- 引入 token 估算工具（Phase 2）
- 实现 tool output 历史 prune（Phase 3）
- 对 compaction LLM 输出做强校验或重试
