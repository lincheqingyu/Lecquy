# Memory 路径收敛 技术规范

更新日期：2026-04-10

所属开发包：包 3（Memory 路径收敛）
前置依赖：包 0 + 包 1 + 包 2
预估工期：1-2 天

## 1. 目标

把当前并存的两条 memory 注入路径（legacy 文件系统 + PG DB recall）收敛为 v2 的「startup summary + query recall」双通道。确保 memory recall 不出现在 `role: system` 中，只落在 `messages[]` 的 user 轮次内。

## 2. 当前状态

### 2.1 Legacy 路径（文件系统）

- 文件：`memory/store.ts` → `loadMemoryInjectionText()`
- 读取：主记忆 `MEMORY.md` + 今日日记 `memory-{today}.md` + 昨日日记 `memory-{yesterday}.md`
- 格式化为 `## Memory Recall` 章节
- 注入位置：system prompt 尾部（通过 `buildProjectContextSection` 中的 MEMORY.md）

### 2.2 DB 路径

- 文件：`memory/prompt-injector.ts` → `buildMemoryRecallBlock()`
- 条件：`PG_ENABLED=true` 且 `userQuery` 有效
- 查询：`searchEventMemories(query, top_k=5)`
- 注入位置：作为 `contextMessages` 注入

### 2.3 问题

1. 两条路径可能同时生效，导致 memory 内容重复注入
2. MEMORY.md 完整常驻在 system prompt 中，污染共享前缀
3. memory recall 进了 system 字段，每回合变化会击穿 prefix cache

## 3. 文件变更清单

| 文件 | 操作 | 说明 |
|---|---|---|
| `memory/prompt-injector.ts` | 改造 | 统一为 `buildMemoryRecallMessages()` |
| `memory/store.ts` | 改造 | 新增 `loadMemorySummary()`，MEMORY.md 不再被 prompt 引用 |
| `memory/compact.ts` | 改造 | compact 产出写到 MEMORY.summary.md |
| `session-runtime-service.ts` | 改造 | 串接新路径 |

## 4. 改造 `memory/store.ts`

### 4.1 新增函数

```typescript
/**
 * 读取 .lecquy/MEMORY.summary.md
 * 这是 startup context 层的唯一 memory 来源
 *
 * @returns 文件内容；不存在则返回空字符串
 */
export async function loadMemorySummary(workspaceDir: string): Promise<string>
```

实现要点：

- 路径固定：`${workspaceDir}/.lecquy/MEMORY.summary.md`
- 文件不存在 → 返回 `''`，不报错
- 读取后按 `STARTUP_BUDGETS.memorySummary` (400 tokens) 截断
- 不做任何格式化加工，原文返回

### 4.2 原有函数保留

`loadMemoryInjectionText()` 保持不变，作为 DB 不可用时的 fallback 数据源（但不再注入 system prompt，改为注入 messages）。

## 5. 改造 `memory/prompt-injector.ts`

### 5.1 新函数签名

```typescript
import { AgentMessage } from '../agent/types'

/**
 * 构建 memory recall 消息
 * 产物落在 messages[] 的 user 轮次中，不进 system prompt
 *
 * @returns AgentMessage[] — 0 或 1 条消息
 */
export async function buildMemoryRecallMessages(options: {
  /** 当前用户提问（用于 recall 查询） */
  userQuery: string
  /** 会话 ID */
  sessionId: string
  /** 是否启用 PG */
  pgEnabled: boolean
  /** 工作区路径（用于 fallback 读取日记文件） */
  workspaceDir: string
}): Promise<AgentMessage[]>
```

### 5.2 实现逻辑

```
1. 如果 pgEnabled
   a. searchEventMemories(userQuery, top_k=5)
   b. 有结果 → 格式化为 recall 文本
   c. 无结果 → 走 fallback

2. fallback（PG 不可用或无结果）
   a. loadMemoryInjectionText(workspaceDir)
   b. 有内容 → 使用该内容
   c. 无内容 → 返回空数组

3. 如果有 recall 内容
   → 返回 [{
     role: 'user',
     content: '<LAYER:memory_recall>\n' + recallText + '\n</LAYER>'
   }]

4. 如果无 recall 内容
   → 返回 []
```

### 5.3 保留旧函数

```typescript
/**
 * @deprecated 使用 buildMemoryRecallMessages 替代
 */
export async function buildMemoryRecallBlock(...): Promise<string | null>
```

## 6. 改造 `memory/compact.ts`

### 6.1 当前状态

compact 流程产出写到 `MEMORY.md`。

### 6.2 改造

compact 产出改为写到 `MEMORY.summary.md`：

```typescript
// 旧路径
const memoryPath = path.join(workspaceDir, '.lecquy', 'MEMORY.md')

// 新路径
const summaryPath = path.join(workspaceDir, '.lecquy', 'MEMORY.summary.md')
```

`MEMORY.md` 保留为完整原始记忆的存档文件，不再被 prompt 系统引用。它的作用变为：

- recall 源（PG 不可用时作为 fallback 的数据源）
- 人工审查的完整记忆档案

### 6.3 写入纪律

- `MEMORY.summary.md` 只能由 compact pipeline 写入
- memory recall 查询结果不得回写 `MEMORY.summary.md`
- flush 流程仍写到日记文件 `memory-{date}.md`（不变）

## 7. 改造 `session-runtime-service.ts`

### 7.1 关键改动点

在 `sendRun` / `buildAugmentedContext` 方法中：

```typescript
// ---- 旧路径 ----
// memory 拼进 contextMessages 或 system prompt 尾部
// MEMORY.md 作为 context file 被 buildProjectContextSection 读取

// ---- 新路径（LAYERED_PROMPT=true）----

// 1. loadMemorySummary() 的产物已通过包 2 的 loadStartupSlices 进入 startup 层
//    这里不需要额外处理

// 2. buildMemoryRecallMessages(userQuery) 作为 messages 注入
const recallMessages = await buildMemoryRecallMessages({
  userQuery: lastUserMessage.content,
  sessionId,
  pgEnabled: config.pgEnabled,
  workspaceDir,
})

// 3. 构建最终 messages 数组
const messages = [
  ...recallMessages,          // memory recall（0 或 1 条）
  ...conversationMessages,    // 历史对话
  lastUserMessage,            // 当前用户消息
]

// 4. memory recall 不再出现在 systemPrompt 中
```

### 7.2 与包 2 的配合

`loadStartupSlices` 内部已经调用 `loadMemorySummary()` 并将结果拼入 startup slice。包 3 不需要重复调用，只需要确保：

- `MEMORY.summary.md` 的写入（compact）和读取（startup）指向同一文件
- startup 层读到的是 compact 产出，不是 recall 查询结果

## 8. 验收标准

### 8.1 协议测试

| # | 用例 | 预期 |
|---|---|---|
| 1 | `LAYERED_PROMPT=true` 发送到 LLM 的 messages | messages 中有 `<LAYER:memory_recall>` 标签 |
| 2 | `LAYERED_PROMPT=true` 的 system prompt | 不含 `<LAYER:memory_recall>` |
| 3 | `LAYERED_PROMPT=true` 的 system prompt | 不含 `## .lecquy/MEMORY.md` 段 |
| 4 | MEMORY.summary.md 不存在 | startup slice 中 memory summary 为空，不报错 |
| 5 | compact 触发后 | 写入 MEMORY.summary.md，不写入 MEMORY.md |
| 6 | PG 可用 + 有 recall 结果 | messages[0] 包含 DB 检索内容 |
| 7 | PG 不可用 | fallback 到日记文件内容 |
| 8 | PG 可用但无结果 | fallback 到日记文件内容 |
| 9 | 无任何 recall 内容 | messages 中无 memory_recall 消息 |

### 8.2 序列化稳定性测试

| # | 用例 | 预期 |
|---|---|---|
| 1 | recall 内容每回合变化 | system prompt 字节不变 |
| 2 | MEMORY.summary.md 内容不变 | startup slice 字节不变 |

### 8.3 回归测试

| # | 用例 | 预期 |
|---|---|---|
| 1 | `LAYERED_PROMPT=true` 端到端 simple 模式 | 对话正常，memory 可用 |
| 2 | `LAYERED_PROMPT=false` | 旧路径不受影响 |
| 3 | `PG_ENABLED=false` + `LAYERED_PROMPT=true` | 文件系统 fallback 正常工作 |
