# 隐藏 Prompt v2 重构 开发规划

更新日期：2026-04-10

## 1. 背景

本文档是 [`20260409-1-Lecquy 隐藏 Prompt 与模式边界 技术规范.md`](./20260409-1-Lecquy%20隐藏%20Prompt%20与模式边界%20技术规范.md) 的 v2 实施总领纲要。

v1 文档经过架构评审后，识别出 3 个高风险问题（分层表述不统一、startup context 与 system 重叠、manager/worker 授权协议缺失）和 8 个中低风险问题。本规划的目标是把评审结论和后续讨论中确定的修正方案（Qwen3.5/OpenAI 兼容请求结构、USER.md 双切片协议、capability block 归类、权限三档、skill 常驻规则）落成可执行的开发任务。

## 2. 总原则

1. 先改结构，不先改文案。
2. 先让 `systemPrompt` 的分层和字节稳定可测，再去接 `USER.md / MEMORY.summary / skill`。
3. 不要一上来同时改 `system-prompts.ts`、`context-files.ts`、`skill-loader.ts` 的所有逻辑，风险太高。
4. 每个开发包完成后做一次 `session-runtime-service.ts` 接入 + 端到端回归测试，确保主路径可用再推进下一包。

## 3. 唯一分层真源

按优先级从高到低，注入顺序从前到后：

| # | 层 | 载体 | 优先级 | 稳定性 | 语言 | prefix cache |
|---|---|---|---|---|---|---|
| 1 | system | 运行时硬编码 | 最高 | 常驻不变 | 英文 | 是 |
| 2 | mode | `simple.md` / `plan.md` | 仅 system 可覆盖 | 会话级不变 | 英文 | 是 |
| 3 | startup context | SOUL / IDENTITY / USER.profile / MEMORY.summary / CAPABILITY | 不可覆盖 mode | 会话级稳定 | 中文 | 是 |
| 4 | skill runtime | 命中后注入的单个 skill 正文 | 不可覆盖 1–3 | 命中后至模式切换前保持 | 英文规则 + 中文示例 | 是 |
| 5 | user preference | USER.md preference 切片 | 仅影响风格/粒度 | 会话级稳定 | 中文 | 是 |
| 6 | memory recall | query 检索结果 | 仅事实，不构成规则 | 每回合动态 | 原文语言 | 否 |
| 7 | live turn | 当前用户消息 + 工具调用 | —— | 每回合变化 | 用户语言 | 否 |

硬规则：

- 注入顺序 = 优先级顺序，1–5 位于 prefix cache 段，6–7 位于动态段。
- 下层永远不能改写上层规则，只能在既有边界内补充事实或风格。
- 通用「结果 / 验证 / 下一步」答复结构归 system 层，mode 只描述差异。
- 语言策略：结构英文，示例中文，示例不构成规则。

## 4. 物理请求格式（Qwen3.5 / OpenAI 兼容）

面向 OpenAI Chat Completions 协议，`messages[]` 中**有且仅有一条 `role: system`**，内部按 `<LAYER:system> / <LAYER:mode> / <LAYER:startup> / <LAYER:skill> / <LAYER:user_preference>` 固定顺序拼接。

```jsonc
{
  "model": "qwen3.5-max",
  "messages": [
    {
      "role": "system",
      "content": "<LAYER:system>\n...\n</LAYER>\n\n<LAYER:mode name=\"simple\">\n...\n</LAYER>\n\n<LAYER:startup>\n...\n</LAYER>\n\n<LAYER:skill id=\"pdf\">\n...\n</LAYER>\n\n<LAYER:user_preference>\n...\n</LAYER>"
    },
    {
      "role": "user",
      "content": "<LAYER:memory_recall>\n...\n</LAYER>\n\n<LAYER:live_turn>\n用户当前消息\n</LAYER>"
    },
    { "role": "assistant", "content": "..." },
    { "role": "user", "content": "..." }
  ]
}
```

硬规则：

- 标签名、顺序、层间换行数量均为字节级契约，任何调整需走文档变更流程。
- `memory_recall` 与 `live_turn` 禁止写入 `role: system`，必须落在 `messages[1..]` 的 user 轮次内。
- 实现侧提供单元测试校验序列化字节稳定性。

## 5. 开发包总览

按「先序列化层 → 上下文层 → 权限层 → skill 层」推进：

```
包 0 (类型基座)           ← 0.5 天，零风险
  ↓
包 1 (Prompt Serializer)  ← 2-3 天，核心
  ↓
包 2 (Startup Context)    ← 1-2 天
  ↓
包 3 (Memory 收敛)        ← 1-2 天，可与包 2 尾部并行
  ↓
包 4 (Mode / Permission)  ← 2-3 天，最复杂
  ↓
包 5 (Skill Runtime)      ← 1-2 天
```

## 7. 集成策略

`session-runtime-service.ts` 是把 system prompt、memory、tool set、agent runner 串起来的中枢。不作为单独开发包，而是每个开发包完成后做一次接入：

| 开发包完成后 | 在 session-runtime-service 中的改动 |
|---|---|
| 包 0 | 无 |
| 包 1 | `sendRun` 中把 `buildSystemPrompt` 替换为 `buildLayeredSystemPrompt`，旧路径 fallback |
| 包 2 | startup 数据源从 `readPromptContextFiles` 切到 `loadStartupSlices` |
| 包 3 | memory 注入从 system prompt 尾部移到 `messages[0]` |
| 包 4 | tool 创建时注入 permission 中间层，worker 启动参数改为 `WorkerRunOptions` |
| 包 5 | skill 工具调用触发 `SkillSession.loadAndFreeze()`，重建 system prompt |

## 8. 当前代码基线关键事实

以下信息来自对当前仓库的实际读取，是各开发包的实现前提：

### system prompt 拼装链路

- 入口：`system-prompts.ts` → `buildSystemPrompt(options)`
- 按顺序拼接 12 个章节：identity → role → tooling → tool-call-style → safety → skills → workspace → documentation → time → project context → runtime → extra instructions
- 模板系统：`prompt-module-files.ts`，14 个 handlebars 模板，支持 `.lecquy/system-prompt/` 覆盖

### context files 加载

- 入口：`context-files.ts` → `readPromptContextFiles()`
- 6 类文件：SOUL / IDENTITY / AGENTS(虚拟) / USER / TOOLS(虚拟) / MEMORY
- `AGENTS.md` 由 `buildManagedAgentsContent()` 动态生成
- `TOOLS.md` 由 `buildManagedToolsContent()` 动态生成
- worker 角色仅加载 AGENTS + TOOLS

### skill 加载

- 入口：`skill-loader.ts`
- 三层优先级：bundle > workspace > runtime
- frontmatter 仅有 name / description / direct_return
- 无 manifest 校验、无 specificity、无常驻管理

### tool 注册

- simple：8 个基础 + extension tools
- manager：read_file + skill + todo_write + request_user_input + session tools
- worker：与 simple 完全一致（含 bash/edit_file/write_file）
- 无权限分级中间层

### memory 注入

- legacy 路径：`store.ts` → `loadMemoryInjectionText()` → 主记忆 + 今日/昨日日记
- DB 路径：`prompt-injector.ts` → `buildMemoryRecallBlock()` → PG 向量检索 top 5
- 两条路径并存，均可能注入 system prompt 尾部

## 9. 验收总原则

每个开发包都要有 3 类测试：

1. **序列化稳定性测试**：同一输入连续序列化字节完全一致；某层变化不影响其他层字节
2. **协议/权限测试**：各层边界、权限分档、授权载体的正确性
3. **回归测试**：新路径覆盖旧路径的信息范围，端到端主路径可用

## 10. 非目标

本轮重构不包括：

- 重写 prompt 文案内容（只改结构和注入方式）
- 实现完整第三方 skill 市场
- 自动从 simple 跳转到 plan
- 前端 UI 改造
- RAG 接入 runtime 主链路
