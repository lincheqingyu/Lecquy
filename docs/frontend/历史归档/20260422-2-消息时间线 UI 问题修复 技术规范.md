# 消息时间线 UI 问题修复 技术规范

## 前置阅读

本文档是 [20260422-1-消息时间线与工具动作呈现 技术规范](./20260422-1-消息时间线与工具动作呈现%20技术规范.md) 的后续。实现层在符合该规范定义的渲染架构下，存在三处视觉与交互缺陷，本文档定义修复方案。

## 问题一：write_file 流式草稿默认折叠，写入过程不可见

### 现象

调用 `write_file` 工具时，用户看不到内容逐步写入的过程——文件卡片（`OperationRow`）默认处于折叠状态，流式草稿视图从未呈现在屏幕上。

### 架构背景

按 20260422-1 规范，`write_file` 成功时不渲染原始 tool 卡，改为在对应位置渲染文件动作节点（`ArtifactOperationCard`）。写入过程中 artifact 状态为 `draft`，`ArtifactTrace.tsx` 的 `OperationRow` 负责渲染此阶段，已内置 `PreviewContent` 流式动画（每 16ms 推进一个 chunk）。

### 根本原因

`ArtifactTrace.tsx` 第 134 行：

```tsx
// OperationRow 内
const [isExpanded, setIsExpanded] = useState(false)  // ← draft 状态下也默认折叠
```

流式动画逻辑已完整实现，但卡片永远处于折叠状态，用户无法看到写入过程。

### 修复方案

**将 draft 阶段的卡片默认展开。**

```tsx
const isDraft = entry.artifact?.status === 'draft'
const [isExpanded, setIsExpanded] = useState(isDraft)
```

附加规则：
- 写入完成（`draft → ready`）后，卡片切换为 `ReadyOperationRow`，默认折叠，保持现有行为
- 写入过程中用户若手动折叠，尊重用户操作，不强制重新展开
- `PreviewContent` 及其 `isStreaming` 逻辑无需修改

### 关键文件

- `frontend/src/components/artifacts/ArtifactTrace.tsx`，第 131–171 行（`OperationRow`）

## 问题二：思考模块图标与展开竖线不对齐，字号不一致

### 现象

1. 思考触发按钮中的 `Sparkles` 图标中心与展开内容左侧竖线（`border-l-2`）不在同一垂直轴上
2. 触发按钮 label（`思考了 x.xs`）字号为 `text-[13px]`，展开内容字号为 `text-[14px]`，两者不一致

### 根本原因

`MessageItem.tsx` 第 629–679 行，`renderThinkingGroup` 函数：

```tsx
// 触发按钮：Sparkles 图标 size-3.5（14px），图标中心约在 x = 7px
<button className="inline-flex items-center gap-1.5 ... text-[13px]">
  <Sparkles className="size-3.5" />
  <span>{label}</span>
</button>

// 展开内容：border-l-2 竖线在 x = 0，与图标中心偏差约 7px
<div className="mt-1.5 border-l-2 border-border pl-3 text-[14px] ...">
  {content}
</div>
```

偏差来源：竖线无左偏移（`ml-0`），而图标中心在 `x ≈ 7px`；字号来源于两处独立的硬编码值。

### 修复方案

```tsx
// 修改前
<div className="mt-1.5 border-l-2 border-border pl-3 text-[14px] leading-[1.55] text-text-secondary select-text">

// 修改后（对齐图标中心 + 统一字号）
<div className="mt-1.5 ml-[7px] border-l-2 border-border pl-3 text-[13px] leading-[1.55] text-text-secondary select-text">
```

**实施前请确认：**
- `ml-[7px]` 为 Tailwind CSS 4 arbitrary value，请在浏览器实测是否与图标中心精确对齐
- 若按钮存在隐式 padding 导致图标实际偏移量不是 7px，应以实测值为准（可选 `ml-1.5` 即 6px 作为备选）

### 关键文件

- `frontend/src/components/chat/MessageItem.tsx`，第 629–679 行（`renderThinkingGroup`）

## 问题三：文件动作卡片视觉层级过重

### 现象

"Created a file" 卡片包含：图标、粗体标题、灰色文件名、状态 badge、展开/折叠 chevron，完成后还有一层嵌套的文件 pill。与 Claude.ai 等同类产品的极简单行风格相差较大，视觉噪声偏高。

### 架构背景

按 20260422-1 规范，文件动作节点的核心语义是"在执行位置标记发生了什么"，标签（`Created a file`、`Updated a file`）本身是正确的动作语义输出，不需要修改。需要调整的是视觉呈现的繁简程度。

### 当前结构（`ArtifactTrace.tsx`）

**`OperationRow`（draft 阶段）**：
```
[FileText] Created a file（粗体）  filename.md（灰色）  [生成中 badge]  [ChevronDown]
└─ 展开：21rem 高度内联代码预览区
```

**`ReadyOperationRow`（完成阶段）**：
```
[FileText] Created a file（粗体）  [ChevronDown]
└─ 展开：文件 pill（再一次点击才能打开文件）
```

### 修复方案

目标：单行极简，减少交互层级，与时间线上其他 tool 动作行（`Viewed`、`Ran`、`Loaded`）视觉权重保持一致。

**Draft 阶段**（写入中）：

```
[LoaderCircle 转圈] 正在写入 filename.md
```

- 无展开/折叠
- 写入完成后自动切换为 Ready 样式
- 内联流式预览区可根据方案一的修复结果决定保留或移除（若问题一的默认展开方案用户体验足够，可考虑移除 21rem 预览区，简化为纯状态行）

**Ready 阶段**（完成后）：

```
[FileText] Created filename.md  （文件名可点击，直接打开右侧面板，无需再展开一层）
```

- 去掉外层展开/折叠交互
- 文件名直接作为点击目标，调用 `onOpenArtifact`
- 去掉嵌套 pill，减少一层交互

**参考实现骨架：**

```tsx
// Ready 阶段（简化后）
function ReadyOperationRow({ entry, onOpenArtifact }) {
  const label = entry.artifact?.name ?? resolveOperationDetail(entry)
  const isClickable = Boolean(entry.artifact && onOpenArtifact)

  return (
    <div className="flex items-center gap-2 px-1 py-0.5 text-[13px] text-text-muted">
      <FileText className="size-3.5 shrink-0" />
      <span className="shrink-0 text-text-secondary">
        {entry.trace?.subtitle ?? 'Created a file'}
      </span>
      {isClickable ? (
        <button
          type="button"
          onClick={() => onOpenArtifact(entry.artifact)}
          className="truncate text-text-muted transition-colors hover:text-text-secondary"
        >
          {label}
        </button>
      ) : (
        <span className="truncate">{label}</span>
      )}
    </div>
  )
}
```

**实施前请确认：**

- `DocumentPanel.tsx`（右侧文档预览面板）是否在所有路由/场景下均可用；若存在不可用路径，需保留 `ReadyArtifactLink` 的降级逻辑
- 移除 21rem 内联预览区后，draft 阶段写入过程是否仍有合适的视觉反馈（问题一修复后的展开卡片是否足够）

### 关键文件

- `frontend/src/components/artifacts/ArtifactTrace.tsx`，第 131–236 行（`OperationRow`、`ReadyOperationRow`、`ArtifactOperationCard`）
- `frontend/src/app/home/components/DocumentPanel.tsx`（确认右侧面板可用性）

## 验收标准

- `write_file` 写入过程中，用户无需点击即可看到文件内容逐步出现（问题一）
- 写入完成后卡片自动恢复为折叠/单行状态（问题一）
- 思考展开内容的左侧竖线与 `Sparkles` 图标中心在同一垂直轴上（问题二）
- `思考了 x.xs` 标签字号与展开内容字号一致（问题二）
- 文件动作节点呈现为单行极简样式，视觉权重与其他 tool 动作行一致（问题三）
- 文件名可直接点击打开右侧面板，无多余展开层级（问题三）
- 上述三处修复不破坏 20260422-1 规范定义的时间线顺序和语义映射

---

## 实施优先级建议

三个问题可独立实施，建议按以下顺序：

1. **问题二**（思考对齐）：改动最小，风险最低，单处 className 修改
2. **问题一**（流式默认展开）：改动集中，一行 `useState` 初始值变更
3. **问题三**（文件卡片重构）：改动范围最大，依赖前两项稳定后再推进
