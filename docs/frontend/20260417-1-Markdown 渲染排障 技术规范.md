# Markdown 渲染排障 技术规范

## 文档目的

这份文档是 Lecquy 前端 markdown 渲染链路的排障入口。

以后只要出现下面这类问题，先看这份文档，再决定是否改代码：

- 标准 markdown 渲染异常
- Mermaid 图表不显示、显示成源码、主题错误
- ASCII 树图 / 框图渲染异常
- 复制按钮位置异常
- 流式输出时闪烁、分段错乱、样式丢失

目标只有一个：

- 优先保证标准 markdown 正确渲染
- ASCII 图表只做“纯图表块”增强，不污染标准 markdown 主路径

## 当前正式渲染架构

2026-04-17 起，前端 markdown 渲染已从 `MessageItem.tsx` 内的自研解析逻辑迁移到 `streamdown` 体系。

当前主链路如下：

```text
消息文本
  ↓
StreamdownMarkdown.tsx
  ↓
streamdown
  ├── @streamdown/code
  ├── @streamdown/mermaid
  └── @streamdown/cjk
  ↓
AsciiAwareBlock
  ├── 纯 ASCII 图表块 → AsciiDiagramBlock
  └── 其他内容       → streamdown 默认 Block
```

## 关键文件

- `frontend/src/components/chat/StreamdownMarkdown.tsx`
  统一 markdown 封装。插件、Mermaid 配置、流式/静态模式切换都在这里。
- `frontend/src/components/chat/AsciiDiagramBlock.tsx`
  纯 ASCII 图表卡片渲染。复制按钮也在这里。
- `frontend/src/components/chat/ascii-diagram.ts`
  ASCII 图表识别、树图/框图解析规则。
- `frontend/src/components/chat/MessageItem.tsx`
  消息 UI 容器。这里只能调用 `StreamdownMarkdown`，不应该再自己解析 markdown。
- `frontend/src/index.css`
  `streamdown` 所需的 `@source`、全局 token、局部样式覆盖都在这里。
- `frontend/src/components/artifacts/ArtifactPanel.tsx`
  Artifact markdown 预览入口。
- `frontend/src/app/home/components/DocumentPanel.tsx`
  附件 markdown 预览入口。

## 必须遵守的边界

### 1. 标准 markdown 是主路径

除“纯 ASCII 图表块”外，其他内容都必须交给 `streamdown`。

这意味着下面这些内容不应该走自定义 ASCII 渲染：

- 含 `###` 标题的教程型 markdown
- 含 `---` 分隔线的文档
- 含列表、引用、表格的普通 markdown
- 含嵌套代码围栏的 markdown 示例
- 普通 ` ```markdown ` 或 ` ```text ` 代码示例

### 2. ASCII 图表只在纯图表块触发

当前判断入口是：

- `isStandaloneAsciiDiagramBlock(text)`

它的设计原则：

- 需要存在足够多的 Box Drawing 字符
- 不能命中常规 markdown 结构
- 不能包含代码围栏

如果未来某个教程文档被错误识别成 ASCII 图表，优先改这个判定函数，不要去改 `MessageItem.tsx` 做补丁。

### 3. Mermaid 只能走 streamdown 插件

当前 Mermaid 正式路径是：

- `@streamdown/mermaid`

不要重新引入以下旧路径：

- `MermaidBlock.tsx`
- 在 `MessageItem.tsx` 内手动 `import mermaid`
- 针对 ```` ```mermaid ```` 再写一套分支 parser

### 4. MessageItem 不再负责 markdown 解析

以下旧思路不要再回退：

- `splitMarkdownSegments`
- `renderTextBlock`
- `renderInlineMarkdown`
- 在 `MessageItem.tsx` 内手写 fenced code / list / table / heading parser

`MessageItem.tsx` 的职责现在只有：

- 决定哪里渲染 markdown
- 决定是否处于 streaming 状态
- 管理消息级 UI 状态

## 当前实现约定

### Streamdown 入口

`StreamdownMarkdown.tsx` 当前负责：

- 接入 `code / mermaid / cjk` 插件
- 通过 `isAnimating` 在 `static / streaming` 之间切换
- 跟随页面深浅色模式给 Mermaid 传 theme
- 用 `AsciiAwareBlock` 拦截纯 ASCII 图表块

### ASCII 图表增强

`AsciiDiagramBlock.tsx` 当前负责：

- 纯树图 / 框图 / 原始 ASCII 图回退渲染
- 图表卡片容器
- 图表复制按钮，位置固定在卡片内部右上角
- 树图导线和框图拆箱逻辑

### 样式入口

`index.css` 当前负责：

- `@import "streamdown/styles.css";`
- `@source "../node_modules/streamdown/dist/*.js";`
- `@source "../node_modules/@streamdown/code/dist/*.js";`
- `@source "../node_modules/@streamdown/mermaid/dist/*.js";`
- `@source "../node_modules/@streamdown/cjk/dist/*.js";`
- 提供 `streamdown` 依赖的颜色 token
- 对消息区 markdown 做少量样式覆盖

如果 `streamdown` 的 UI 突然变成裸样式，先检查这里，而不是先怀疑 React 组件。

## 排障顺序

以后出现渲染 bug，按这个顺序查。

### 第 1 步：先拿到原始文本

不要只看截图。

必须先确认模型实际输出的是哪一种：

- 普通 markdown
- ` ```mermaid ` 围栏
- 纯 ASCII 图
- 教程型 markdown 里夹了一段 ASCII 图
- 流式未闭合文本

没有原始文本时，几乎所有修复都会误判。

### 第 2 步：判断 bug 属于哪一层

按下面四类分：

1. `streamdown` 标准 markdown 问题
2. Mermaid 插件问题
3. ASCII 纯图表识别问题
4. CSS / 主题 / 按钮定位问题

不要一上来就改解析逻辑。

### 第 3 步：先验证是否误触发 ASCII 图表分支

重点看：

- `isStandaloneAsciiDiagramBlock(text)`

如果内容本来应该按标准 markdown 渲染，却进了 `AsciiDiagramBlock`，问题通常在这里。

常见症状：

- 标题、分隔线、列表被吃掉
- 教程文档整体被包进图表卡片
- 代码示例被当作树图

### 第 4 步：如果是 Mermaid，先查插件链路

确认三件事：

- 文本是否真的是 ```` ```mermaid ```` 围栏
- `StreamdownMarkdown.tsx` 是否仍然注入了 `@streamdown/mermaid`
- `index.css` 是否仍然包含 `streamdown` 与 mermaid 插件的 `@source`

常见症状：

- Mermaid 直接显示源码
- Mermaid 渲染失败
- 深色模式颜色错误
- Mermaid 控件样式丢失

### 第 5 步：如果是样式错乱，先查 CSS，不先改 TSX

先检查：

- `streamdown/styles.css` 是否仍被导入
- `@source` 路径是否还有效
- `--color-background`、`--color-card`、`--color-border`、`--color-muted`、`--color-primary` 等 token 是否还在
- 本地覆盖类名是否和 `data-streamdown` 标记一致

常见症状：

- 代码块按钮没样式
- 表格边框消失
- Mermaid 卡片背景不对
- 整体 spacing 失真

## 常见问题与建议修法

### 问题 1：教程型 markdown 被错误渲染成 ASCII 图表

优先检查：

- `ascii-diagram.ts` 里的 `MARKDOWN_STRUCTURE_REGEX`
- `isStandaloneAsciiDiagramBlock(text)`

建议修法：

- 扩大“命中 markdown 结构则放弃 ASCII 图表”的范围
- 不要在 `MessageItem.tsx` 里加特殊 case

### 问题 2：纯 ASCII 树图没有被增强，直接按普通 markdown 显示

优先检查：

- 原始文本是否包含足够多 Box Drawing 字符
- 文本里是否混入了标题、列表、代码围栏

建议修法：

- 调整 `isAsciiDiagramContent(text)` 的阈值
- 调整 `isStandaloneAsciiDiagramBlock(text)` 的拦截条件

### 问题 3：Mermaid 显示成源码

优先检查：

- 围栏语言是不是 `mermaid`
- `plugins={{ mermaid: mermaidPlugin }}` 是否仍存在
- 是否误把 `controls`、`plugins`、`BlockComponent` 改坏

### 问题 4：复制按钮位置不对

优先检查：

- ASCII 图表：`AsciiDiagramBlock.tsx` 内 `DiagramCard`
- 代码块 / 表格 / Mermaid：`streamdown` 默认 controls 样式 + `index.css` 覆盖

设计约束：

- ASCII 图表复制按钮必须在图表卡片内部右上角

### 问题 5：流式输出时闪烁或者半截 markdown 错乱

优先检查：

- `StreamdownMarkdown.tsx` 的 `mode`
- `isAnimating` 传入逻辑
- 调用方是不是把本来静态内容错误当成 streaming

当前 `MessageItem.tsx` 里 streaming 依据是：

- `message.stepStatus === 'started'`
- 或 `thoughtTiming?.status === 'running'`

## 不允许的回退方案

以后修 bug 时，不要做这些事：

- 不要把 markdown 主路径改回 `MessageItem.tsx` 手写 parser
- 不要为 Mermaid 重新写独立渲染主路径
- 不要用“某个截图特例”去硬编码内容匹配
- 不要为了修 ASCII 图表去破坏标准 markdown
- 不要只改样式掩盖结构判断错误

如果一个修复需要重新引入 `splitMarkdownSegments` 这一类逻辑，基本方向就是错的。

## 验证命令

每次改完渲染逻辑后，至少跑这两条：

```bash
pnpm --dir frontend exec tsc -b
pnpm --dir frontend build
```

如果只看页面截图，不跑这两条，不算完成。

## 推荐回归用例

至少手测下面几类文本：

1. 普通 markdown
   - 标题、列表、引用、表格、链接、行内代码
2. Mermaid
   - 单个 ```` ```mermaid ```` 代码块
3. 纯 ASCII 树图
   - 只包含树图和少量节点文本
4. 纯 ASCII 框图
   - 只包含 Box Drawing 框图
5. 教程型 markdown
   - `###` + 正文 + ```` ``` ```` 示例 + `---`
6. 混合内容
   - 正文段落 + Mermaid + 普通代码块 + 表格
