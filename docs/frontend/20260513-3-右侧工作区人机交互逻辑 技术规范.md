## 1. 结论

Lecquy 的右侧交互应以 Claude Code 的“伴随式工作区”为主参考：右侧不是设置抽屉，也不是弹窗，而是在用户做真实任务时持续陪伴主时间线的 **context inspector / task companion**。

人机交互逻辑的核心不是“有几个面板”，而是解决真实使用中的 5 个动作：

1. 用户要知道 agent 正在做什么。
2. 用户要知道 agent 为什么这么做。
3. 用户要快速查看 agent 产出的文件或引用的文件。
4. 用户要在高风险动作出现时批准、拒绝或暂停。
5. 用户要在必要时调整 runtime，但不让设置打断主任务。

因此右侧应采用以下原则：

- **主时间线负责叙事**：中央展示发生顺序，保持可读。
- **右侧负责检查和介入**：右侧展示当前对象、进度、上下文、审批和 runtime。
- **用户显式操作优先**：用户点开的内容不能被普通自动事件抢走。
- **阻塞事件最高优先**：审批、权限、危险命令必须置顶。
- **自动打开必须克制**：只在用户没有明确关注对象时弱自动打开。
- **关闭不等于遗忘**：关闭右侧只是释放屏幕，不清空任务状态。

## 2. 设计探索

### 2.1 使用者

真实使用者是 kira，一个人在本机长期开发 Lecquy。打开界面时通常处于以下状态：

- 刚从 Claude Code / Codex 的工作流迁移回来，希望 Lecquy 接住连续任务。
- 在读文档、改前端、看 agent 输出，频繁在“对话、文件、进度、审批”之间切换。
- 不希望 UI 像后台管理系统一样展示全部配置。
- 更需要“当前任务能不能继续、有没有风险、产物在哪里”，而不是“全部功能入口”。

### 2.2 产品领域概念

本交互设计属于个人强 Agent 的本地工作台，关键概念是：

- 任务现场：当前 session、当前目标、当前上下文。
- 事件时间线：用户输入、agent 思考、tool 动作、文件产物按顺序发生。
- 伴随检查区：右侧随任务变化，展示当前最需要检查的对象。
- 权限闸门：高风险操作必须停下来让用户决定。
- 记忆召回：agent 为什么知道这件事，来源于哪些偏好、决策、错误或承诺。
- 运行时：模型、采样、tools、reasoning、kernel、memory runtime 是当前 agent 的工作状态。
- 产物链路：从 tool 动作到 artifact，再到预览、下载、打开目录。

### 2.3 色彩和气质

交互逻辑本身不定义视觉 palette，但右侧的操作感应来自一个“安静本地工作台”：

- 白色工作纸面：内容模块应干净，不像控制台告警墙。
- 浅灰对话底色：模块之间露出主背景，形成 Claude Code 式轻分层。
- 淡边线：区域可分辨，但不抢阅读焦点。
- 少量蓝色或深色：只给当前选择和主要动作。
- 琥珀/红色：只用于审批风险、警告和错误。
- 单色图标：图标只解释动作，不装饰配置项。

### 2.4 签名交互

Lecquy 的签名交互应是 **右侧随任务切换，但不抢用户焦点**。

具体表现：

- 用户点文件名，右侧切到文件预览。
- agent 长任务运行时，右侧在未 pinned 时切到 Progress。
- 权限审批出现时，右侧强制切到 Approval。
- 用户正在看 Runtime 或某个 Artifact 时，普通 progress / draft 不抢占，只在时间线给出可点击目标。
- 审批完成后，右侧回到审批前的非阻塞 mode。

### 2.5 需要避免的默认设计

| 默认做法 | 问题 | Lecquy 应替换为 |
|---|---|---|
| 把右侧做成设置页 | 使用者真实工作时大多数时间不是调参数 | 任务伴随工作区 |
| 每种内容一个抽屉 | 状态互相打架，用户不知道关哪个 | 单一 RightRail + mode |
| 自动弹出所有新信息 | 打断阅读和检查文件 | 弱自动打开 + pinned 保护 |
| 把审批塞在输入框上方 | 阻塞动作不够明显，也挤压输入区 | Approval mode 置顶 |
| 展示完整 prompt / 全量 memory | 信息过载，变成后台 | 摘要 + 可追溯入口 |

## 3. Claude Code 参考点

### 3.1 应该模仿的部分

Claude Code 的右侧体验值得模仿的是“伴随而非模态”：

- 右侧卡片和主对话同屏存在，不阻断阅读。
- Progress / Context / 文件列表是任务态信息，不是后台配置。
- 用户可以一边读对话，一边看右侧引用文件或任务进度。
- 右侧关闭后，主时间线仍然完整可用。
- 右侧内容视觉权重低，主要用于检查，而不是吸引注意。

### 3.2 不能照搬的部分

Lecquy 不是 Claude Code 的复刻，不能只做 Progress / Context / 文件列表：

- Lecquy 有长期记忆，需要解释 memory recall。
- Lecquy 有 `.lecquy/` prompt 文件，需要 Runtime / Kernel 检查入口。
- Lecquy 近期要自建 harness，必须承载 sandbox / permission / approval。
- Lecquy 是唯一入口，右侧要支持跨任务连续性，而不是一次性编码 session。

### 3.3 取舍

主交互模仿 Claude Code，信息内容贴合 Lecquy：

```text
Claude Code:  Progress / Project files / Context / Preview
Lecquy:       Progress / Context / Artifact / Approval / Runtime / Memory
```

## 4. 总体人机模型

页面应形成稳定的空间语义：

```text
左侧：去哪里      会话、项目、历史、任务入口
中间：发生什么    用户输入、agent 回复、思考、tool、文件动作时间线
右侧：看什么      当前对象、进度、上下文、审批、runtime、记忆
顶部：当前身份    会话标题、主题、右侧入口、少量全局动作
```

用户不需要学习复杂布局，只需要形成一个直觉：

> 我在中间和 Lecquy 对话；如果要看细节、文件、进度、上下文或批准操作，就去右侧。

## 5. 真实工作场景推演

### 5.1 冷启动继续一个任务

背景：kira 打开 Lecquy，继续昨天的前端重构。

期望交互：

1. 左侧选中上次会话。
2. 中央恢复消息时间线和输入框。
3. 宽屏下右侧默认显示 `Context`，展示当前 session、项目规则、最近相关文件、已召回 memory 摘要。
4. 如果用户关闭右侧，右侧保持关闭，不因普通消息恢复自动打开。

设计理由：

- 刚进入任务时，用户最需要“我现在在哪、上下文是什么”。
- 右侧默认 Context 参考 Claude Code 的 Context 卡，但 Lecquy 应额外展示记忆和 `.lecquy/` 约束摘要。

### 5.2 用户发起一个长任务

背景：用户说“按规范实现 RightRail 第一版”。

期望交互：

1. 用户发送消息后，中央时间线开始流式展示 agent 回复。
2. 若右侧未打开或处于 `Context`，自动切到 `Progress`。
3. `Progress` 展示当前目标、todo、正在执行项、最近 tool 动作。
4. 用户点中央某个 tool / 文件事件时，右侧切到对应对象。
5. 如果用户已经显式打开 `Runtime` 或某个 `Artifact`，长任务 progress 不抢占，只在右侧 header 或时间线中显示轻提示。

设计理由：

- 长任务时用户最常见的心理动作是“它现在卡在哪”。
- 但用户显式打开的内容通常意味着正在检查，不能被自动进度抢走。

### 5.3 agent 生成文件

背景：agent 调用 `write_file` 生成 markdown 或 html。

期望交互：

1. 中央时间线在真实执行位置显示 `正在写入 filename`。
2. 如果右侧关闭或处于非 pinned `Context/Progress`，右侧弱自动切到 `Artifact` 并展示草稿。
3. 如果用户正在看另一个文件或 Runtime，不自动切换。
4. 文件 ready 后，右侧继续显示正式内容，header 给出 copy path / show in folder / refresh。
5. 中央文件名始终可点击，点击后显式切到该 artifact 并 pin。

设计理由：

- Claude Code 里文件/上下文是右侧伴随对象，Lecquy 应保持同屏可查。
- 自动打开只解决“用户没在看别的东西”的场景，不做强打断。

### 5.4 用户审批高风险操作

背景：agent 要执行删除文件、写入敏感路径、联网、git reset、沙箱升级。

期望交互：

1. 右侧强制切到 `Approval`。
2. 中央时间线保留请求出现的上下文，但不把主要按钮塞在输入区。
3. `Approval` 显示动作、风险、影响范围、命令或 diff preview。
4. 用户可选择允许一次、允许本 session、允许本项目、拒绝、取消任务。
5. 审批完成后右侧回到 `lastNonBlockingMode`。
6. 审批未完成前，普通 artifact/runtime/progress 不能覆盖 Approval。

设计理由：

- 审批是阻塞态，不是普通通知。
- 放右侧比弹窗更贴近 Claude Code 的伴随工作台，也保留中央上下文。

### 5.5 用户检查 agent 为什么这样回答

背景：agent 引用了一个历史偏好或项目规则，用户想知道来源。

期望交互：

1. 中央回复里可出现轻量 memory/context chip，或右侧 Context 中列出来源。
2. 用户点击后，右侧切到 `Memory` 或 `Context` 的展开项。
3. 只展示本轮相关记忆，不展示全量数据库。
4. 可提供设计占位动作：过时、记错、不要再用、提升为稳定偏好。
5. 当前阶段这些治理动作可以只做 UI 占位或后续任务，不强行落地。

设计理由：

- Lecquy 的护城河是“懂 kira”，但可解释性不能变成后台表格。
- 用户真实需要的是纠错和追溯，不是管理 memory.db。

### 5.6 用户临时调整 runtime

背景：模型失败、输出太短、tools 关闭、需要切 reasoning。

期望交互：

1. 用户点顶栏设置按钮。
2. 右侧切到 `Runtime`，并设置 `pinnedMode='runtime'`。
3. Runtime 修改自动保存，不需要额外弹窗。
4. 普通 Progress / Artifact draft 不抢占 Runtime。
5. 如果出现 Approval，Approval 临时置顶；完成后回 Runtime。
6. 用户按 Esc 或点击关闭后清除 Runtime pin。

设计理由：

- Runtime 是当前 agent 的工作状态，不是全局后台设置。
- 用户调 runtime 时通常正在排查问题，不能被普通事件打断。

### 5.7 用户切换会话

背景：kira 从一个前端任务切到另一个后端任务。

期望交互：

1. 中央切换到新会话。
2. artifactRef、approvalRef、memoryRef 清空或按 session 恢复。
3. 如果新会话有历史 task context，右侧显示该 session 的 Context。
4. Runtime 不随 session 自动打开。
5. 旧 session 的 pinned artifact 不泄漏到新 session。

设计理由：

- 右侧是当前任务的检查区，不能跨 session 残留旧对象。

### 5.8 窄屏或窗口被压缩

背景：用户在小窗口中查看 Lecquy。

期望交互：

1. 小于 900px 时 RightRail 变 overlay，不挤压中央时间线。
2. 小于 640px 时 RightRail 变全屏 sheet。
3. 输入框聚焦时，非 Approval 的 RightRail 可以自动收起或保持背后。
4. Approval 不因输入框聚焦自动消失。

设计理由：

- 主时间线和输入框是核心任务路径，不能被右侧挤坏。
- 审批是唯一可覆盖输入区优先级的状态。

## 6. RightRail 状态模型

### 6.1 状态定义

```ts
type RightRailMode =
  | 'context'
  | 'progress'
  | 'artifact'
  | 'memory'
  | 'runtime'
  | 'approval'

type RightRailOpenReason =
  | 'user'
  | 'system'
  | 'restore'
  | 'approval'

interface RightRailState {
  isOpen: boolean
  mode: RightRailMode
  openReason: RightRailOpenReason
  pinnedMode: RightRailMode | null
  lastNonBlockingMode: RightRailMode
  artifactRef: ArtifactRef | null
  memoryRef: MemoryRef | null
  approvalRef: ApprovalRef | null
  progressRef: ProgressRef | null
}
```

### 6.2 mode 分组

| 分组 | mode | 说明 |
|---|---|---|
| 阻塞态 | `approval` | 必须先处理，不允许普通关闭 |
| 用户检查态 | `artifact` / `memory` / `runtime` | 用户显式点开后进入 pinned |
| 系统伴随态 | `progress` / `context` | 可以被用户检查态覆盖 |
| 默认态 | `context` | 没有明确对象时展示 |

### 6.3 状态转移

| 事件 | 当前状态 | 新状态 | 说明 |
|---|---|---|---|
| 用户点设置 | 任意非 approval | `runtime` open + pinned | 显式操作 |
| 用户点文件 | 任意非 approval | `artifact` open + pinned | 显式操作 |
| 用户点 memory 来源 | 任意非 approval | `memory` open + pinned | 显式操作 |
| 长任务开始 | closed/context/progress 且未 pinned | `progress` open | 系统伴随 |
| 长任务开始 | artifact/runtime/memory pinned | 不切换 | 尊重用户焦点 |
| draft artifact 出现 | closed/context/progress 且未 pinned | `artifact` open，openReason=system | 弱自动打开 |
| draft artifact 出现 | artifact/runtime/memory pinned | 不切换 | 只在时间线提示 |
| approval 出现 | 任意状态 | `approval` open，保存 lastNonBlockingMode | 阻塞置顶 |
| approval 完成 | approval | 回 `lastNonBlockingMode` | 恢复现场 |
| 用户 Esc | runtime/context/progress | close | 释放屏幕 |
| 用户 Esc | artifact/memory | 回 context 或 close | 取决于是否还有默认内容 |
| 用户 Esc | approval | 不关闭 | 必须处理 |
| 切换 session | 任意非 runtime | reset refs，按新 session context | 防止旧对象泄漏 |

## 7. 打断规则

### 7.1 用户显式优先

用户显式打开某个 mode 后，系统自动事件不得覆盖它，除非是 `Approval`。

显式操作包括：

- 点击文件名或附件。
- 点击设置按钮。
- 点击 memory/context 来源。
- 手动切换右侧 tab/mode。
- 拖拽调整 artifact 宽度。

### 7.2 系统弱自动打开

系统可以自动打开右侧，但必须满足：

- 当前没有 `pinnedMode`。
- 当前不是 `approval`。
- 当前没有用户正在查看的 artifact/runtime/memory。
- 窗口宽度允许右侧存在。

弱自动打开适用：

- 长任务进入 progress。
- 首个 draft artifact 出现。
- 从空会话进入已有任务，展示 context。

### 7.3 阻塞优先

Approval 是唯一可以覆盖用户显式 mode 的状态。

要求：

- Approval 出现时保存 `lastNonBlockingMode`。
- Approval 完成后恢复。
- Approval 不响应普通 Esc 关闭。
- 如果用户拒绝或取消任务，右侧回到 Progress 或 Context，并在中央时间线保留结果。

### 7.4 关闭规则

关闭右侧只影响屏幕占用，不应清空底层数据。

| 内容 | 关闭后保留什么 |
|---|---|
| Artifact | 保留 artifactRef，可从时间线再次点开 |
| Progress | 保留任务状态，任务继续运行 |
| Runtime | 保留配置草稿和自动保存结果 |
| Memory | 保留本轮 recall 结果 |
| Approval | 不允许普通关闭 |

## 8. 中央时间线与右侧协作

### 8.1 中央时间线的职责

中央只回答：

- 用户说了什么。
- agent 想了什么。
- agent 做了什么。
- agent 最终给了什么。

中央不承担：

- 长文件完整预览。
- 全量 tool 输出。
- 完整 diff。
- 运行时设置。
- 记忆数据库管理。
- 审批详情的主要操作区。

### 8.2 右侧的职责

右侧回答：

- 这个事件的对象是什么。
- 当前任务运行到哪。
- 这个文件内容是什么。
- 这条记忆来源是什么。
- 这个命令风险是什么。
- 当前 agent runtime 是什么。

### 8.3 点击路径

| 中央对象 | 点击结果 |
|---|---|
| 文件名 / artifact | 打开 `Artifact` mode |
| tool 错误摘要 | 打开 `Progress` 或错误详情 |
| memory chip | 打开 `Memory` mode |
| context chip | 打开 `Context` mode |
| permission requested | 打开 `Approval` mode |
| 模型名 / runtime hint | 打开 `Runtime` mode |

## 9. mode 级交互规范

### 9.1 Context

默认展示：

1. 当前任务标题和 mode。
2. 当前 session / route。
3. Context Stack 摘要：StartupContext、UserPreference、MemoryRecall、LiveTurn。
4. 本轮相关文件。
5. 当前约束：CLAUDE.md / AGENTS.md / 命中 skill 的关键约束。
6. 风险提示：上下文过大、记忆低置信、工具失败、等待用户。

交互：

- 点击文件进入 Artifact。
- 点击 memory 摘要进入 Memory。
- 点击 kernel 文件进入 Runtime > Kernel。
- Context 不展示完整 prompt。

### 9.2 Progress

默认展示：

1. 当前 goal。
2. Todo 状态。
3. 正在执行 step。
4. 最近 tool 动作。
5. 等待用户或阻塞原因。

交互：

- 运行中自动刷新。
- 用户可折叠已完成项。
- 点击 tool 相关文件进入 Artifact。
- 点击审批提示进入 Approval。

Claude Code 模仿点：

- 像 Claude Code 的 Progress 卡一样，偏阶段和状态，不做百分比进度条。
- 长任务只展示“当前在做什么”和“下一步”，不要变成日志流。

### 9.3 Artifact

默认展示：

- 文件标题、类型、大小或更新时间。
- preview/source 切换。
- copy / show in folder / refresh / download。
- draft 状态和流式锁底。

交互：

- 用户手动滚动上方时解除锁底。
- 用户滚回底部重新锁底。
- 新 draft 不覆盖 pinned artifact。
- 同一文件多次生成时，点击时间线旧事件默认打开最新版本；历史版本另立版本管理任务。

### 9.4 Memory

默认展示：

- 本轮召回的 decisions、mistakes、preferences、project facts、commitments。
- 每条展示 summary、kind、confidence、source、status。

交互：

- 点击 source 回到相关 session 或 context 来源，当前阶段可占位。
- “过时 / 记错 / 不要再用 / 提升为稳定偏好”先作为后续动作设计，不在本轮强实现。
- 不展示全量 memory.db。

### 9.5 Runtime

默认展示：

```text
AGENT
SAMPLING
TOOLS
REASONING
ROLE CONTEXT
KERNEL
MEMORY RUNTIME
```

交互：

- 点击设置按钮进入 Runtime 并 pin。
- 所有可编辑项自动保存。
- Runtime 不被普通 progress / artifact draft 覆盖。
- Approval 可临时覆盖，完成后返回 Runtime。

### 9.6 Approval

默认展示：

1. 请求动作。
2. 风险说明。
3. 影响范围。
4. 命令、文件路径或 diff preview。
5. 可选决策：允许一次、允许本 session、允许本项目、拒绝、取消。
6. 规则建议。

交互：

- 不允许普通关闭。
- 操作后立即显示提交中状态。
- 完成后恢复 `lastNonBlockingMode`。
- 拒绝后中央时间线显示拒绝结果，任务进入 paused/failed 的实际状态。

## 10. 顶栏入口逻辑

顶栏不应堆满 mode 按钮。参考 Claude Code，顶部只放少量稳定入口：

- 主题切换。
- RightRail toggle。
- Runtime 入口可以继续使用齿轮，但语义应从“Settings”转为“Agent Runtime”。

RightRail 内部可以有 mode header，但不建议把 6 个 mode 都做成常驻 tab。更好的方式：

- 默认 header 显示当前 mode 名和对象。
- Context / Progress / Artifact / Memory / Runtime / Approval 通过事件和点击进入。
- 需要手动切换时，放在 header 的轻量下拉或 icon menu 中。

原因：

- 用户真实工作时不是“选 tab”，而是“点一个对象查看”。
- Claude Code 的右侧也是任务对象驱动，不是后台 tab 页。

## 11. 键盘和焦点

### 11.1 Esc

| 状态 | Esc 行为 |
|---|---|
| dropdown/menu 打开 | 先关闭 dropdown/menu |
| runtime | 关闭 RightRail，清 pinned |
| artifact | 回 context 或关闭 RightRail |
| memory | 回 context |
| progress | 关闭 RightRail 或回 context |
| context | 关闭 RightRail |
| approval | 不关闭，聚焦拒绝/取消或保持当前焦点 |

### 11.2 焦点管理

- 打开 Approval 后，焦点进入 Approval 的主操作区。
- 打开 Artifact 后，焦点不强行抢到预览区，避免打断中央阅读。
- 打开 Runtime 后，焦点不自动进入第一个输入框，避免误输入。
- 关闭 RightRail 后，焦点回到触发按钮或中央输入框。

### 11.3 输入区优先

- 用户正在输入时，非 Approval 的自动右侧切换应延后。
- Approval 可以打断输入，但需要保留输入草稿。

## 12. 响应式交互

| 宽度 | 行为 |
|---|---|
| >= 1200px | RightRail 可常驻，默认 Context 或 Progress |
| 900-1199px | RightRail 可打开但更克制，artifact 可覆盖式显示 |
| 640-899px | RightRail overlay，不挤压中央 |
| < 640px | 全屏 sheet，左上返回，底部保留主要动作 |

移动端原则：

- 不因为右侧而压缩输入框到不可用。
- Approval 优先级仍最高。
- Runtime 长表单在移动端全屏滚动。
- Artifact 宽内容优先 source/preview 内部横向滚动。

## 13. 会话级持久化

建议持久化：

- 每个 session 的 `lastMode`。
- 每个 session 最近打开的 artifactRef。
- RightRail 是否被用户关闭。

不建议持久化：

- `approval`。
- `pinnedMode` 跨冷启动恢复。
- Runtime 自动打开状态。
- 跨 session 的 artifactRef。

规则：

- session 切换时，优先恢复该 session 的 Context。
- 如果该 session 有正在运行的任务，优先 Progress。
- 如果该 session 有 pending approval，优先 Approval。

## 14. Claude Code UI 模仿硬规则

本节是给后续 Codex / Claude Code 编码阶段看的强约束。目的不是再描述交互，而是防止实现者读到当前 Lecquy 旧文件后，被 `SettingsDrawer.tsx`、现有卡片样式、旧颜色 token 和历史补丁带偏。

### 14.1 基准

右侧工作区的 UI 基准是用户提供截图里的 Claude Code 右侧栏，而不是当前 Lecquy 的 `SettingsDrawer`。

实现时必须优先对照 Claude Code 的这些特征：

- 右侧栏是**主工作区旁的伴随层**，不是从屏幕边缘滑出的设置抽屉。
- 右侧栏背景和页面背景接近，整体轻、白、安静，没有强边框包围感。
- 内容由几个独立小卡片组成，每张卡片占满右侧栏可用宽度。
- 卡片之间用留白分隔，不用粗线分隔。
- 卡片圆角偏小到中等，接近 `10-14px` 的克制圆角；禁止大面积 `rounded-2xl/rounded-3xl` 气泡感。
- 卡片边框极淡，阴影极轻，视觉上像“放在纸面上的轻卡片”，不是浮窗、modal 或 dashboard widget。
- 卡片标题短，通常是 `Progress`、项目名、`Context` 这类对象名；标题区右侧只放折叠 chevron 或少量动作按钮。
- 文案灰度层级清楚：标题深、正文中灰、说明浅灰；不要用大量彩色状态标签。
- 图标小而少，服务结构和动作，不装饰每一行信息。
- 列表项行高舒适但不松散，适合长期扫读。
- 右侧卡片不展示“全功能入口”，只展示当前任务会用到的对象。

### 14.2 禁止在当前 UI 上反复大补丁

后续实现 RightRail 时，禁止把当前 `SettingsDrawer` 作为视觉母版继续打补丁。

本任务的性质应表述为：**结构重构为主，业务逻辑迁移为辅**。

术语边界：

| 术语 | 在本任务中的含义 | 允许继承什么 | 不允许继承什么 |
|---|---|---|---|
| 结构重构 | 重新设计右侧布局、状态模型、组件边界和视觉原语 | 需求、数据流、保存行为 | 旧抽屉结构、旧卡片层级、旧 className |
| 业务逻辑迁移 | 把旧组件中仍有价值的 model/context/memory 保存逻辑搬到新组件或 hook | API 调用、防抖保存、preset 数据结构、错误处理 | 旧 UI 容器、旧 Section 样式、旧 dropdown 外观 |
| 视觉迁移 | 把旧 SettingsDrawer 的视觉样式搬到 RightRail | 不允许 | 当前任务禁止视觉迁移 |
| 兼容补丁 | 在旧 UI 上加条件分支模拟新设计 | 不允许 | 当前任务禁止兼容补丁路线 |

因此，后续实现时不要说“迁移 SettingsDrawer UI”。正确说法是：

- **重构**右侧 shell、状态机和视觉原语。
- **迁移**模型 preset、context file、memory runtime 的保存逻辑。
- **重写**Runtime / Artifact / Progress / Context 的右侧呈现方式。
- **删除或废弃**旧抽屉外壳和旧样式，而不是把它们塞进新 RightRail。

明确禁止：

- 在旧 `SettingsDrawer` 外再套一层 RightRail。
- 保留旧 `isSettingsOpen` / `openDocument` 两套互斥逻辑，再用条件判断修补新行为。
- 继续复制旧的 `Section`、`SettingRow`、`InlineDropdown` 样式到所有 mode。
- 在旧卡片基础上继续加 `border`、`shadow`、`rounded-*` 来“调得更像 Claude”。
- 为了少改文件而保留两个右侧布局体系。
- 用当前 UI 的色彩、间距、圆角作为默认值，再局部覆盖。
- 出现“当前设置抽屉样式 + Claude Code 卡片样式”混杂的过渡态。

允许并推荐：

- 从头新建 `RightRailShell`、`RightRailCard`、`RightRailHeader`、`RightRailListItem` 等小原语。
- 把旧 `SettingsDrawer` 拆为纯数据/逻辑 hook 与 `RuntimePanel` 内容组件。
- 直接删除旧右侧布局分支，而不是维护兼容层。
- 先重构结构，再迁移内容。
- 为了代码质量进行必要的文件拆分。
- 在改动面更大但结构更清楚时，优先选择重构，不选择脆弱补丁。

判断标准：

> 如果实现方案需要在旧组件里继续叠 5 个以上视觉条件分支，说明应该停止补丁，改为抽新组件。

### 14.3 代码质量优先级

RightRail 属于布局和交互主干，代码质量优先于“少改几行”。

实现必须满足：

- 单一状态模型：只能有一个 RightRail 状态机主导右侧，不允许 settings/document/approval 各自维护独立抽屉状态。
- 内容与容器分离：`RightRailShell` 只管宽度、开合、mode、响应式；各 mode 只管内容。
- 视觉原语复用：卡片、标题、列表项、空态、折叠区、动作按钮必须用统一原语。
- 旧逻辑迁移时先识别职责：model preset 保存、context file 保存、memory runtime 保存属于逻辑；旧 `Section` 样式不属于可继承资产。
- CSS class 不做长链补丁：超过一屏的 className 或大量条件拼接，应抽组件或 helper。
- mode 内容不互相 import 具体实现细节；共享类型和回调由上层传入。
- 自动切换逻辑集中在 reducer / state helper，不散落在多个 `useEffect` 里互相抢状态。
- 文件预览、runtime、approval 都通过同一个 shell 呈现，不再各自写 `<aside>` 外壳。

### 14.4 Claude Code 视觉规则

后续编码时，右侧栏应按下面的视觉规则实现：

| 项目 | 规则 |
|---|---|
| 右侧栏宽度 | 默认约 `360-420px`，常规 Context/Progress/Runtime 接近 `384px`；Artifact 可更宽，但也要沿用同一 shell |
| 右侧栏背景 | 使用接近主对话区的浅灰/白底，不使用强色块，不出现深色侧边面板感 |
| 卡片宽度 | 卡片应占满右侧栏内容宽度，左右留白稳定 |
| 卡片圆角 | 小到中等，优先 `rounded-xl` 以下的克制尺度；禁止大气泡 |
| 卡片边框 | 1px 极淡边框，不能成为第一视觉焦点 |
| 卡片阴影 | 极轻，可无；禁止 dashboard 式浮起阴影 |
| 卡片间距 | 用垂直 gap 分隔，避免横线堆叠 |
| 标题 | 14-16px、semibold、短标题；不要 hero 标题 |
| 正文 | 13-14px，行高适合扫读 |
| 说明文字 | 12-13px，浅灰，短句 |
| 图标 | 16px 左右，小而克制；只在标题动作、文件类型、状态、折叠中使用 |
| 按钮 | 低调，边框或透明背景；主按钮只在 Approval 等阻塞场景出现 |
| 颜色 | 默认黑白灰；蓝色只给当前选择/链接，琥珀/红只给风险 |
| 空态 | 简短一句说明 + 轻插图/小图标即可，不做大面积营销说明 |

特别禁止：

- 渐变背景、彩色光斑、装饰性图标列。
- 多层卡片套卡片。
- 大量 uppercase 标签污染正文。
- 大圆角 pill 堆满右侧。
- 强阴影、强边框、强 hover 动画。
- 为每个设置项配一个图标。
- 把右侧做成 shadcn admin dashboard。

### 14.5 Claude Code 布局规则

右侧栏在空间关系上必须像 Claude Code：

- 左侧会话栏、中央时间线、右侧工作区三者并列。
- 右侧工作区不覆盖中央内容，除非在窄屏进入 overlay/sheet。
- 右侧工作区顶部和主内容顶部对齐，不做浏览器边缘抽屉感。
- 右侧工作区不需要整栏边框包围；边界主要通过背景差异、留白和卡片轮廓表达。
- 中央时间线宽度被右侧影响时，输入框和消息宽度仍保持可读，不发生挤压错乱。
- 右侧关闭后，中央自然扩展；重新打开后不改变中央内容顺序和 scroll 状态。

### 14.6 Claude Code 卡片规则

Progress 卡：

- 标题为 `Progress`。
- 用阶段点、check 状态、简短说明展示任务进度。
- 不做百分比大仪表盘。
- 不展示完整日志流。

项目/文件卡：

- 标题可为当前项目名或 `Lecquy`。
- 列表项包含小文件图标、文件名、必要后缀信息。
- 文件名截断优雅，行高稳定。
- 右上可有 folder / chevron 等少量动作。

Context 卡：

- 标题为 `Context`。
- 内容是当前任务的引用对象和上下文来源摘要。
- 可有轻量插图或状态图，但不大面积占位。
- 说明文字短，例如“Track tools and referenced files used in this task.” 这种一句话密度。

Approval 卡：

- 可以比其他卡更突出，但仍属于右侧栏卡片，不变成全屏 modal。
- 命令/diff preview 应使用轻量代码块。
- 操作按钮明确、少量、靠近风险说明。

Runtime 卡：

- 视觉上仍应像 Claude Code 的右侧卡片，不像传统设置页。
- 分区可以多，但每个分区应是“任务运行时检查项”，不是 admin form。
- 控件密度要高，避免大表单留白。

### 14.7 验收红线

后续任何实现如果出现以下情况，视为没有遵守本规范：

- 右侧仍由 `SettingsDrawer` 独立抽屉主导。
- 右侧仍由 `openDocument` 和 `isSettingsOpen` 两套状态互斥。
- 新 UI 明显继承了当前 Lecquy 设置栏的卡片堆叠、图标污染或抽屉感。
- Claude Code 截图中右侧栏的“轻、白、伴随、低噪声、卡片化”没有成为主视觉。
- 为了少改代码牺牲了状态清晰度和组件边界。
- 代码里出现大量补丁式 className 条件来兼容旧 UI。
- 视觉验收只看功能可用，不截图对比 Claude Code 右侧栏。

后续编码任务开始前，必须先读本节，再读现有源文件。读源文件时只继承业务逻辑，不继承旧视觉方向。

## 15. 实现边界

本规范定义交互逻辑，不要求一次性实现所有后端能力。

第一阶段可落地：

- RightRail 状态机。
- Runtime 归位。
- Artifact 归位。
- Approval 归位。
- Progress 从现有 messages/todo/tool state 派生只读摘要。
- Context 展示静态 session / route / context file 摘要。

后续阶段再做：

- Memory recall 明细通过 WS/API 暴露。
- Memory 修正动作。
- Context Stack 的 token/层级可视化。
- Artifact 历史版本。
- 复杂 diff 审批。

## 16. 验收场景

### 16.1 长任务不抢用户文件

步骤：

1. 打开一个 artifact。
2. 让 agent 执行长任务。
3. 等待 progress 更新。

通过标准：

- 右侧仍显示用户打开的 artifact。
- Progress 只以轻提示或中央时间线出现，不抢占。

### 16.2 draft 弱自动打开

步骤：

1. 关闭右侧。
2. 让 agent `write_file` 生成文档。

通过标准：

- 右侧自动打开 Artifact。
- 中央时间线保留 `正在写入 filename`。
- draft 流式更新。

### 16.3 Runtime 不被普通事件覆盖

步骤：

1. 打开 Runtime。
2. agent 同时开始 progress 或生成 draft。

通过标准：

- Runtime 保持打开。
- 只有 Approval 可以覆盖。

### 16.4 Approval 置顶并恢复

步骤：

1. 打开 artifact。
2. 触发权限审批。
3. 批准或拒绝。

通过标准：

- 右侧切到 Approval。
- 审批完成后回到 artifact。

### 16.5 切换 session 不泄漏右侧对象

步骤：

1. 在 A 会话打开 artifact。
2. 切到 B 会话。

通过标准：

- B 会话不显示 A 的 artifact。
- B 默认 Context 或 Progress。

### 16.6 Claude Code 视觉对齐

步骤：

1. 打开宽屏页面。
2. 让右侧显示 Progress、项目文件、Context 或 Runtime。
3. 对照 Claude Code 截图检查。

通过标准：

- 右侧是伴随式卡片栏，不是设置抽屉。
- 卡片轻、白、低阴影、低边框、小圆角。
- 无多层气泡、无装饰图标污染、无强 dashboard 感。
- 代码结构是新 RightRail 原语，不是在旧 UI 上堆补丁。

## 17. 最终判断

以 Claude Code 为主参考时，Lecquy 不应追求“右侧功能更多”，而应追求“右侧更懂当前任务”。

一句话规范：

> 用户在中央推进任务，右侧只在需要检查、确认、追溯或调整时出现；系统可以提示，但不能随便抢焦点；只有审批能打断一切。
