## 1. 这个圆圈是什么

输入框旁边的圆环进度条（如截图红框所示），显示当前 session 已消耗的上下文窗口百分比。

```
已消耗 token 总量 / model.limit.context * 100 = 百分比
```

hover 时展示：
- N tokens（总消耗量）
- N% used（占比）
- $0.00X spent（累计费用）

---

## 2. 数据链路（OpenCode 实现）

```
AI SDK finish-step 事件
  → session.ts getUsage()
      从 LanguageModelUsage 拆出：
        input / output / reasoning / cache.read / cache.write
      total = 五者相加（不是只算 input）
  → assistant 消息写入 tokens 字段，推送前端
  → session-context-metrics.ts getSessionContextMetrics()
      取最后一条 tokens.output > 0 的 assistant 消息
      total = msg.tokens.input + output + reasoning + cache.read + cache.write
      limit = provider.models[modelID].limit.context
      usage = limit ? Math.round(total / limit * 100) : null
  → session-context-usage.tsx SessionContextUsage
      把 usage % 传给 ProgressCircle
  → progress-circle.tsx
      SVG stroke-dashoffset 渲染圆弧
```

---

## 3. 圆圈 SVG 实现原理

用 `stroke-dasharray` + `stroke-dashoffset` 实现环形进度：

```ts
const viewBoxSize = 16
const center = 8              // viewBox 中心
const radius = center - strokeWidth / 2
const circumference = 2 * Math.PI * radius  // 圆周长

// dashoffset 越小 → 显示的弧越长
const offset = circumference * (1 - percentage / 100)
```

```svg
<!-- 背景圈（CSS 控制颜色，通常是灰色） -->
<circle cx="8" cy="8" r="6.5" stroke-width="2" />

<!-- 进度圈 -->
<circle cx="8" cy="8" r="6.5"
  stroke-dasharray="40.84"       <!-- 圆周长，固定值 -->
  stroke-dashoffset="24.5"       <!-- percentage=40% 时的值 -->
  stroke-width="2" />
```

CSS 加 `transform: rotate(-90deg)` 让起点从 12 点钟方向开始（SVG 默认从 3 点方向）。

组件接口：

```ts
interface ProgressCircleProps {
  percentage: number   // 0-100，自动 clamp
  size?: number        // 默认 16px
  strokeWidth?: number // 默认 3
}
```

## 4. token 计算的有效性分析

这是这套方案最容易踩坑的地方，需要区分三种情况。

### 4.1 token 总量（total）—— 始终准确

token 总量来自 **API 响应本身**的 `usage` 字段，不是客户端估算。

只要 provider 在响应里返回了 `usage`（几乎所有主流 API 都支持），total 就是准确的。无论：
- 官方 API（Anthropic、OpenAI、Qwen 等）
- 自部署 OpenAI-compatible API（vLLM、Ollama 等）
- 模型切换（每次请求返回的是当前模型的 token 用量）

**这个不会因为模型切换而出问题。**

### 4.2 context limit —— 有效的前提是"limit 已知"

`usage = total / model.limit.context * 100`

`model.limit.context` 来源有三个，优先级从高到低：

| 来源 | 说明 |
| --- | --- |
| 用户在 config 里手动配置 | 最高优先级，用户覆盖 |
| models.dev 数据库 | OpenCode 内置，覆盖主流商业模型 |
| fallback 默认值 | **= 0**，即"未知" |

当 `limit.context = 0` 时：
- `usage` 计算结果为 `null`，UI 不显示百分比（或显示 0%）
- `isOverflow()` 直接返回 `false`（compaction 永不触发）
- 圆圈停在空状态

**结论：圆圈是否有意义，取决于当前 model 的 context limit 是否已知。**

### 4.3 不同 tokenizer 会互相影响吗

不会。原因如下：

每个模型用自己的 tokenizer 计算 token，API 响应里返回的 `usage.inputTokens` 就是该模型 tokenizer 的计数结果，context limit 也是同一个 tokenizer 空间里的数字。

所以：

```
qwen3-6b:   total_A tokens / limit_A = 合法的 %
qwen3-30b:  total_B tokens / limit_B = 合法的 %
```

两边各自在自己的 token 空间里算，**不会互相干扰**。

切换 model 后，UI 读的是"最新一条 assistant 消息"的 token，自然就是新 model 的数据，自动切换，无需额外处理。

## 5. Qwen 模型切换的具体分析

以 `qwen3-6b-27b`（密集模型）切换到 `qwen3-30b-a3b`（MoE 模型）为例：

### token 总量：没问题

两个模型虽然架构不同（MoE vs 密集），但都是 Qwen3 家族，使用同一套 tokenizer（tiktoken BPE，约 150k 词表）。同样的输入文本，两个模型的 token 计数几乎相同（误差 < 1%）。

即使 tokenizer 有差异，也不影响计算正确性，因为每个模型返回的 `usage` 是自己的计数，和自己的 `limit` 对齐。

### context limit：可能有问题

| 模型 | context window（参考） |
| --- | --- |
| Qwen3-6B / 14B / 27B | 128k tokens |
| Qwen3-30B-A3B（MoE） | 128k tokens |
| Qwen3-235B-A22B（MoE） | 128k tokens |

Qwen3 全系列 context window 相同（128k），所以这次切换 limit 不变，不会有问题。

**潜在问题出现在跨系列切换时：**

| 场景 | 问题 |
| --- | --- |
| Qwen3（128k）→ 某个 32k 模型 | limit 从 131072 变成 32768，% 会跳变（正确行为） |
| 切换到 models.dev 数据库里没有的自部署模型 | limit = 0，圆圈消失，compaction 失效 |
| 切换到 Ollama 本地模型 | token 计数正常，但 limit 通常 = 0（Ollama 不在 models.dev） |

## 6. 自部署/本地模型的已知问题

### Ollama / vLLM / LM Studio 等

token 总量：通常正常（OpenAI-compatible API 返回 `usage`，AI SDK 能读到）。

context limit：**几乎必然是 0**，因为这些本地模型不在 models.dev 数据库里。

后果：
- 圆圈显示 0%（不随对话增长）
- `isOverflow()` 永远返回 false
- 无论 context 实际有多满，compaction 都不会触发

**解决方式（OpenCode 的做法）：** 用户在配置文件里手动指定 `limit.context`：

```yaml
# opencode.yaml 或等效配置
models:
  - id: "your-local-model"
    limit:
      context: 32768
      output: 4096
```

### 通过 OpenAI-compatible API 调用 Qwen（如阿里云 DashScope）

token 总量：准确（API 返回标准 `usage`）。

context limit：取决于该 model ID 是否在 models.dev 里。Qwen 主流模型（如 `qwen-plus`、`qwen-turbo`）通常已收录，limit 正确。小版本或最新发布的模型可能尚未收录，limit 为 0。

## 7. Lecquy 目前的状态

Lecquy 当前**没有**对应的上下文占比指示器。

缺少的东西：

| 缺失项 | 说明 |
| --- | --- |
| token 数据回路 | `finish-step` 等价事件需要把 token 用量推送到前端（可能已有但未暴露） |
| model context window 数据 | 前端需要能查到当前 model 的 `contextWindow`，目前不明确是否已有 |
| `ProgressCircle` 组件 | SVG 实现本身约 30 行，可直接参考 OpenCode 实现 |
| 数据聚合层 | 类似 `session-context-metrics.ts` 的前端计算层 |

如果要做这个功能，核心链路：

```
backend finish-step → WS 推送 token 用量 → 前端状态更新
→ 计算 total / model.contextWindow * 100
→ ProgressCircle 渲染
```

## 8. ProgressCircle 实现参考（可直接用于 Lecquy）

```tsx
// components/ui/ProgressCircle.tsx
interface ProgressCircleProps {
  percentage: number   // 0-100
  size?: number        // px，默认 16
  strokeWidth?: number // 默认 2
  className?: string
}

export function ProgressCircle({
  percentage,
  size = 16,
  strokeWidth = 2,
  className,
}: ProgressCircleProps) {
  const viewBoxSize = 16
  const center = viewBoxSize / 2
  const radius = center - strokeWidth / 2
  const circumference = 2 * Math.PI * radius
  const clamped = Math.max(0, Math.min(100, percentage))
  const offset = circumference * (1 - clamped / 100)

  return (
    <svg
      width={size}
      height={size}
      viewBox={`0 0 ${viewBoxSize} ${viewBoxSize}`}
      fill="none"
      className={className}
      style={{ transform: 'rotate(-90deg)' }}  // 12点钟起点
    >
      {/* 背景圈 */}
      <circle
        cx={center} cy={center} r={radius}
        strokeWidth={strokeWidth}
        className="stroke-muted"
      />
      {/* 进度圈 */}
      <circle
        cx={center} cy={center} r={radius}
        strokeWidth={strokeWidth}
        strokeDasharray={circumference}
        strokeDashoffset={offset}
        className="stroke-primary transition-all duration-300"
      />
    </svg>
  )
}
```

颜色建议：
- 0-70%：主色（`stroke-primary`）
- 70-90%：警告色（`stroke-warning`）
- 90-100%：危险色（`stroke-destructive`）

## 9. 核心结论

| 问题 | 结论 |
| --- | --- |
| token 总量计算是否准确 | 准确，来自 API 响应，不是客户端估算 |
| 切换模型会不会出问题 | 不会，每次请求自动使用当前 model 的 token 数和 limit |
| 不同 tokenizer 会相互影响吗 | 不会，各模型在自己的 token 空间内计算 |
| Qwen3 密集 → MoE 切换有问题吗 | 没问题，context window 相同（128k），tokenizer 同族 |
| 本地部署模型能用吗 | token 计数通常正常，但 limit 默认为 0，圆圈失效 |
| limit = 0 时的后果 | 圆圈不显示 %，compaction 永不触发 |
| 解决 limit = 0 的方法 | 在 config 里手动指定该模型的 context window 大小 |
