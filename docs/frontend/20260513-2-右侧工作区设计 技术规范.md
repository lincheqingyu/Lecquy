# 右侧工作区设计 技术规范

> 更新日期：2026-05-13
> 类型：技术规范
> 关联：[个人强 Agent 路线](../项目级/20260408-2-个人强%20Agent%20路线%20开发规划.md)、[system prompt 上下文工程最终取舍](../项目级/20260512-2-系统提示词上下文工程最终取舍%20技术规范.md)、[设置栏极简重设计](./20260513-1-设置栏极简重设计%20技术规范.md)

## 1. 结论

Lecquy 的右侧不应该长期叫“设置栏”，也不应该继续把 `DocumentPanel` / `ArtifactPanel` / `SettingsDrawer` 分成两套布局概念。

更契合项目定位的设计是统一的 **RightRail 右侧工作区**：

```text
左侧会话栏 | 主工作台
              ├─ TopBar
              └─ 中央任务时间线 | RightRail
```

RightRail 的职责不是“放更多面板”，而是让 kira 在当前任务中随时看清三件事：

1. Lecquy 正在做什么。
2. Lecquy 依据了什么上下文和记忆。
3. 当前有什么产物、风险、确认或运行时配置需要处理。

这比单独复刻 Claude Code 的右栏更贴合 Lecquy。Claude Code 的右侧主要是任务上下文和文件预览；Lecquy 还必须承载长期记忆、本地 prompt 文件、运行时配置、工具权限、任务连续性这些个人强 Agent 特有内容。

## 2. 设计依据

### 2.1 项目身份

`CLAUDE.md` / `AGENTS.md` 已经把 Lecquy 定义为 kira 的个人 JARVIS：唯一入口、长期记忆、自我进化、本地优先。右侧工作区必须服务这个方向，而不是服务通用 AI Web 聊天产品。

因此右侧不应该优先展示“模型参数表”，而应该优先展示当前任务需要的上下文、产物、记忆和确认。

### 2.2 当前阶段

近期形态是单 agent + 自建 harness，第一批能力是 agent loop、文件工具、bash、git、Plan 模式、subagent 调度、沙箱、diff 渲染。

RightRail 应该提前给这些能力留下稳定位置：

- 文件和 artifact 预览。
- Plan / worker 进度。
- sandbox / 权限审批。
- diff 和命令结果。
- runtime settings。
- memory recall 与 context stack。

### 2.3 上下文工程

20260512-2 已经锁定 7 层 prompt 架构：

```text
System → Mode → StartupContext → UserPreference → SkillRuntime → MemoryRecall → LiveTurn
```

RightRail 应该把这套结构在 UI 上“可观察但不喧宾夺主”地呈现出来。用户不需要每轮都看见完整 prompt，但应该能在需要时知道当前回答受哪些稳定画像、项目规则、技能、记忆召回和 live context 影响。

### 2.4 前端时间线

20260422-1 与 20260423-1 已经把对话区定义为事件时间线：展示“发生了什么”，而不是展示底层工具名。

RightRail 应与时间线形成互补：

- 中央时间线展示发生顺序。
- RightRail 展示被点击对象、当前上下文、运行状态和后续可操作入口。

## 3. 不采用的方向

### 3.1 不做独立 Settings Drawer

设置栏不应继续是和文档区同级但概念分裂的抽屉。它应变成 RightRail 的一个 mode：`Runtime`。

原因：

- 模型参数、采样、reasoning、tools、kernel、memory runtime 都是 agent runtime 的一部分。
- 打开设置不应该破坏文档预览和任务上下文的空间模型。
- 长期看会出现更多右侧内容，如果每种内容都做一个独立抽屉，布局会变成互斥逻辑堆叠。

### 3.2 不做 Claude Code 复刻

Claude Code 的右侧卡片栏值得借鉴，但不能直接照搬。

Claude Code 的右栏偏向：

- Progress
- 项目文件
- Context
- 文件预览

Lecquy 需要额外覆盖：

- 长期记忆召回。
- `.lecquy/` prompt 文件。
- 本地运行时状态。
- 权限审批。
- 自我进化入口。

### 3.3 不做后台管理面板

RightRail 不是 admin panel。不要把所有可配置项、所有 memory 表、所有工具 inventory 都展示出来。

规则：

- 默认看任务态，不看配置态。
- 配置只在 `Runtime` mode 里出现。
- 高风险和异常状态优先浮出。
- 成功状态尽量静默。

## 4. RightRail 信息架构

RightRail 建议收敛为 6 个 mode。

| Mode | 主要用途 | 触发方式 | 默认可见性 |
|---|---|---|---|
| `Context` | 当前任务上下文栈：项目、规则、记忆、附件、route | 任务开始或用户点上下文按钮 | 默认 mode |
| `Progress` | plan、todo、worker/subagent、长任务进度 | plan/manager/worker 运行时自动打开 | 运行时优先 |
| `Artifact` | 文件、文档、生成产物、diff、图片/PDF 预览 | 点击 artifact / attachment / draft 自动打开 | 用户点击优先 |
| `Memory` | 召回的记忆、决策、错误、承诺、可修正入口 | 点击 memory chip 或 context stack | 按需 |
| `Runtime` | 当前 agent 的模型、采样、tools、reasoning、kernel、memory runtime | 顶栏设置按钮 | 按需 |
| `Approval` | 权限确认、危险命令、外发数据、沙箱升级 | 系统请求确认时自动置顶 | 强制优先 |

## 5. 优先级与互斥规则

RightRail 不应只是 `isSettingsOpen` / `openDocument` 两个布尔值互相打架。需要一个显式状态模型。

建议状态：

```ts
type RightRailMode =
  | 'context'
  | 'progress'
  | 'artifact'
  | 'memory'
  | 'runtime'
  | 'approval'

interface RightRailState {
  isOpen: boolean
  mode: RightRailMode
  pinnedMode: RightRailMode | null
  artifactRef: ArtifactRef | null
  memoryRef: MemoryRef | null
  approvalRef: ApprovalRef | null
  lastNonBlockingMode: RightRailMode
}
```

优先级：

1. `Approval`：最高优先级。权限确认出现时自动切到 `approval`，不能被普通 artifact/settings 覆盖。
2. 用户显式点击：用户点文件、设置、memory 时，应尊重用户选择，并记录 `pinnedMode`。
3. `Artifact draft`：当右侧未被用户 pin 住时，`write_file` draft 可弱自动打开。
4. `Progress`：长任务运行时，如果没有显式打开 artifact/runtime，则自动显示 progress。
5. `Context`：默认 fallback。
6. 窄屏：移动端和小宽度下 RightRail 变成覆盖层或底部 sheet，不常驻挤压主内容。

关闭规则：

| 当前 mode | Esc 行为 | 关闭按钮行为 | 自动切换 |
|---|---|---|---|
| `approval` | 不关闭，除非审批完成/取消 | 不关闭或只折叠详情 | 审批完成后回到 `lastNonBlockingMode` |
| `artifact` | 回到 `context` 或关闭 rail | 关闭 artifact，回到 context | 新 artifact 不抢 pinned runtime |
| `runtime` | 关闭 rail | 关闭 rail | 不被普通 progress 抢占 |
| `progress` | 关闭 rail 或回 context | 关闭 rail | 任务结束后回 context |
| `memory` | 回 context | 回 context | 无 |
| `context` | 关闭 rail | 关闭 rail | 无 |

## 6. 各 mode 的内容设计

### 6.1 Context

Context 是默认态，不是设置态。

建议卡片顺序：

1. 当前任务：会话标题、mode、工作区、route。
2. Context Stack：`StartupContext` / `UserPreference` / `MemoryRecall` / `LiveTurn` 的摘要。
3. 相关文件：本轮读过、改过、引用过的文件。
4. 当前约束：来自 `CLAUDE.md`、`AGENTS.md`、skills 的关键约束。
5. 风险提示：上下文过大、记忆低置信、工具失败等。

Context Stack 只展示摘要，不展示全文。点击后可进入 `Memory` 或 `Runtime > Kernel`。

### 6.2 Progress

Progress 承载 plan / manager / worker 状态。

内容：

- 当前 goal。
- todo 列表。
- in-progress step。
- worker/subagent 状态。
- 最近工具动作摘要。
- blocked / waiting for user 的原因。

Progress 不应该做成花哨进度条。Lecquy 的任务通常不是百分比任务，更适合“阶段 + 状态 + 下一步”的工程表达。

### 6.3 Artifact

Artifact 是 `DocumentPanel` / `ArtifactPanel` 的统一归宿。

内容类型：

- attachment preview。
- markdown / code artifact。
- diff preview。
- PDF / 图片预览。
- generated document。

交互：

- 点击中央时间线里的文件事件切到 `artifact`。
- draft artifact 可在 rail 未 pinned 时弱自动打开。
- draft 流式写入时保持锁底；用户手动上滚后解除锁底。
- artifact ready 后提供 `Show in Folder` / copy path / open external 等动作。

### 6.4 Memory

Memory mode 不等于 memory 数据库管理后台。它只展示当前任务相关的记忆。

内容：

- recalled decisions。
- recalled mistakes。
- recalled preferences。
- recalled project facts。
- commitments。
- confidence / source / status。

必要动作：

- “这条过时了”。
- “这条记错了”。
- “不要再用这条”。
- “提升为稳定偏好”。

这些动作可以先作为设计占位，不必立即落地。它们对应 `CLAUDE.md §8.1` 的长期记忆治理问题。

### 6.5 Runtime

Runtime 是当前设置栏的归宿。

内容沿用 20260513-1 七分区：

```text
AGENT
SAMPLING
TOOLS
REASONING
ROLE CONTEXT
KERNEL
MEMORY RUNTIME
```

但命名上建议从 `Settings` 转为 `Runtime` 或 `Agent Runtime`。它不是产品设置页，而是当前 agent 运行时检查与编辑区。

当前阶段可以保留 384px 宽度、小圆角白底模块、模块之间露出对话区底色。后续统一 RightRail 后，Runtime 作为一个 mode 存在，不再作为独立 drawer。

### 6.6 Approval

Approval 是当前右侧设计最容易遗漏的一块。

未来自建 harness 会涉及：

- bash 执行确认。
- 文件删除 / overwrite。
- git reset / checkout。
- 网络访问。
- secret / 外发数据。
- 沙箱升级。

这些不适合塞进聊天时间线底部，也不适合弹浏览器 alert。RightRail 的 approval mode 应显示：

- 请求动作。
- 风险说明。
- 影响范围。
- 原始命令或 diff。
- 允许 / 拒绝 / 修改后允许。
- 是否记住本次规则。

审批完成后自动回到之前的非阻塞 mode。

## 7. 视觉语言

### 7.1 总体气质

RightRail 应该是“安静的本地工作台”，不是装饰型侧边栏。

关键词：

- 本地。
- 可审计。
- 长期。
- 工程密度。
- 小圆角。
- 低阴影。
- 少图标。
- 状态明确。

### 7.2 背景与卡片

建议：

- RightRail 背景使用对话区底色：`bg-surface-alt`。
- 内容模块用 `bg-surface`。
- 圆角控制在 `rounded-lg` 或更小。
- 边线/阴影弱到只负责分层，不成为视觉主角。
- Runtime settings 可以采用“模块占满 rail 宽度”的密集表达。
- Context / Progress 这类信息卡可视实际密度保留轻微内边距，但不要出现卡片套卡片。

### 7.3 字体与信息层级

建议：

- section label：uppercase、小号、muted。
- 值：右对齐，必要时 monospace。
- 文件路径、命令、模型名：monospace 或 code chip，但 chip 不应泛滥。
- 状态色只给 warning/error/active，不做彩色装饰。

### 7.4 图标策略

图标只用于结构和动作：

- 展开/折叠。
- 文件。
- 命令。
- 警告。
- 确认。
- 复制/打开/关闭。

不要给每个配置项都配图标。

## 8. 响应式与窗口宽度

桌面默认：

| 场景 | 推荐宽度 |
|---|---|
| Context / Progress / Runtime | 384px |
| Memory | 384px 或 420px |
| Artifact 文档预览 | 520px 起，可拖拽到 45vw |
| Diff / 宽代码 | 640px 起，必要时进入主区 split |

小屏策略：

- 小于 900px：RightRail 作为覆盖层，不挤压主内容。
- 小于 640px：RightRail 变为全屏 sheet，保留返回按钮。
- 输入区聚焦时，非 approval rail 可自动收起。

## 9. 状态持久化

RightRail 状态应按 session 维度保存，而不是全局保存。

建议：

- `mode`：每个 session 记住最后显式 mode。
- `artifactRef`：切换 session 后清空或按 session 恢复。
- `runtime`：不随 session 强制打开。
- `pinnedMode`：只在当前 session 生命周期有效，避免冷启动恢复到奇怪面板。

## 10. 与现有实现的关系

当前实现大致是：

```text
ConversationSidebar | MainShell
                       ├─ TopBar
                       └─ ConversationArea + DocumentPanel
                    + SettingsDrawer
```

目标实现应收敛为：

```text
ConversationSidebar | MainShell
                       ├─ TopBar
                       └─ ConversationTimeline | RightRail
```

重构与迁移建议：

本阶段不是把旧右侧 UI “搬家”，而是重构 RightRail。这里的“迁移”只指迁移仍然有价值的业务逻辑和数据流，例如 preset 保存、context file 保存、memory runtime 配置、artifact 预览数据绑定；旧视觉、旧抽屉结构、旧卡片层级不迁移。

1. 新建 `RightRailState`，先只托管 `runtime`。
2. 重构新的 `RightRailShell` / `RightRailCard` / mode content 组件。
3. 从 `SettingsDrawer` 迁移 model preset、context file、memory runtime 的保存逻辑，重写为 `Runtime` mode 内容。
4. 从 `DocumentPanel` / `ArtifactPanel` 迁移预览逻辑和数据绑定，重写为 `Artifact` mode 内容。
5. 增加 `context` mode，只放当前会话、相关文件、引用规则摘要。
6. 增加 `progress` mode，对接 todo / plan / worker 状态。
7. 增加 `approval` mode，对接权限协议。
8. 增加 `memory` mode，只展示当前任务召回的 memory，不做数据库后台。
9. 实现阶段允许从头重构右侧组件，禁止在当前 `SettingsDrawer` / `DocumentPanel` 外继续叠补丁；旧文件只继承业务逻辑，不继承视觉方向。

## 11. 需要避免的坑

1. 不要再新增第三种右侧面板。
2. 不要让 settings 和 artifact 用不同层级的布局。
3. 不要把 memory mode 做成全量记忆管理器。
4. 不要让自动打开覆盖用户显式打开的内容。
5. 不要把 approval 当普通 toast。
6. 不要把 Context Stack 展示成 prompt 全文。
7. 不要为了像 Claude Code 而牺牲 Lecquy 的 memory/runtime 特性。
8. 不要在右侧堆插件市场、MCP 列表、多用户配置。
9. 不要把当前 Lecquy 旧 UI 当作样式基准继续打补丁。后续编码必须严格以 Claude Code 右侧栏的轻量卡片、低噪声、小圆角、淡边框和伴随式布局为视觉基准。
10. 不要为了少改代码保留旧抽屉结构。RightRail 是主布局重构点，必要时允许重写组件边界，追求长期代码质量。

## 12. 验收口径

后续落地时，右侧工作区应满足：

| 检查项 | 通过标准 |
|---|---|
| 单一右侧模型 | 代码中有统一 `RightRail` 或等价状态模型，不再由 `openDocument` 与 `isSettingsOpen` 两套互斥逻辑主导 |
| mode 完整 | 至少支持 `context` / `artifact` / `runtime`，并预留 `progress` / `memory` / `approval` |
| 设置归位 | `SettingsDrawer` 不再作为独立抽屉概念，而是 Runtime 内容 |
| 文档归位 | `DocumentPanel` / `ArtifactPanel` 进入 Artifact mode |
| 用户显式优先 | 用户打开 runtime/artifact 后，普通自动事件不得抢占 |
| approval 优先 | 权限确认能置顶显示，完成后回到之前 mode |
| 视觉一致 | RightRail 背景、模块、圆角、阴影、标题、控件风格统一 |
| Claude Code 对齐 | 右侧栏截图对比 Claude Code：轻量白底卡片、极淡边框、低阴影、小圆角、少图标、非抽屉感；不得呈现旧 Lecquy 设置栏气泡堆叠 |
| 代码质量 | 有新的 RightRail 原语和单一状态模型；不得在旧 `SettingsDrawer` 上通过大量条件 className 叠补丁 |
| 移动端可退化 | 小宽度下不会挤坏主时间线和输入区 |
| 无泛化污染 | 不出现多用户、插件市场、MCP inventory 之类与当前阶段无关的 UI |

## 13. 最终判断

经过补充检查，右侧设计思路已经完整到可以作为后续前端重构的方向备份。

一句话版本：

> Lecquy 的右侧应该是个人 Agent 的上下文工作区，不是设置抽屉；设置、文档、进度、记忆、审批都应统一进入 RightRail，并按当前任务优先级切换。

## 14. 工作台命名规范

后续代码、文档和规则统一使用下面的命名，避免继续混用“设置栏”“抽屉”“文档面板”等旧概念。

| 区域 | 代码命名 | 中文名称 | 说明 |
|---|---|---|---|
| 整体三栏布局 | `WorkbenchShell` | 工作台壳层 | 承载左侧任务侧栏、主工作区和右侧工作区 |
| 左侧栏 | `TaskSidebar` | 任务侧栏 | New task / Projects / Recents / status |
| 中间主区 | `MainWorkbench` | 主工作区 | 当前任务的主要阅读和操作区域 |
| 顶部栏 | `WorkbenchHeader` | 工作台标题栏 | 当前任务标题、breadcrumb、少量动作 |
| 对话 / 事件流 | `TaskTimeline` | 任务时间线 | 消息、工具动作、产物事件 |
| 底部输入区 | `InputDock` | 输入停靠栏 | 固定底部的输入框和快捷动作 |
| 右侧整体 | `RightRail` | 右侧工作区 | 伴随式上下文 / 检查栏 |
| 右侧运行时内容 | `Runtime` / `AgentRuntime` | 运行时 | 原“设置栏”的新名称 |
| 右侧卡片 | `RightRailCard` | 右侧卡片 | Progress / Context / Runtime 等一级卡片 |
| 右侧卡片头 | `RightRailCardHeader` | 卡片头 | 可折叠触发区 |
| 右侧卡片内容 | `RightRailCardBody` | 卡片内容 | 折叠时隐藏 |

命名结论：

- 不再使用 `SettingsDrawer` / “设置栏”作为产品概念。
- 右侧整体统一称为 `RightRail`，中文名“右侧工作区”。
- 模型、采样、工具、推理、内核、记忆运行时统一归入 `Runtime` mode，中文名“运行时”。
- 顶栏齿轮按钮语义为 `Runtime toggle`，中文名“运行时入口”。
