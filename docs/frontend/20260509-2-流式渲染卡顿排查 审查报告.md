# 流式渲染卡顿排查 审查报告

> 更新日期：2026-05-09
> 类型：审查报告
> 关联：[20260417-1-Markdown 渲染排障 技术规范](./20260417-1-Markdown%20渲染排障%20技术规范.md)

## 问题

Web 端调用 API 或本地模型时，消息流式渲染不够流畅，经常出现卡顿 / 瞬间加载一大段的情况。

## 数据链路

```
LLM Provider → pi-agent-core (SDK) → Backend notifier → WebSocket
→ useChat (rAF 批处理) → React setState → MessageList
→ MessageItem × N → StreamdownMarkdown → DOM
```

## 诊断结论

**前端渲染是主因**，网络 / 后端处理 / API 响应不是瓶颈。

### 排除项

| 维度 | 结论 |
|------|------|
| 网络 | WebSocket 本地回环（或同机部署），延迟可忽略 |
| 后端处理 | `notifier` 直接写 WS，无阻塞队列，不引入缓冲延迟 |
| API 响应 | LLM 流式返回，后端透传不积攒 |
| CSS mask-image | 有轻微 GPU 合成开销但不是主因 |

---

## 瓶颈点（按严重程度排序）

### 1. MessageItem 缺少 React.memo — 每次 token 到达全局重渲染

**位置** `frontend/src/components/chat/MessageItem.tsx:238`

每个 `step_delta` token 到达后，`useChat` 通过 `updateMessage` 更新某一条消息，创建全新的 `messages` 数组。`MessageList` 中直接 `.map()` 渲染所有 `MessageItem`。由于 `MessageItem` 没有 `React.memo` 包裹，**所有历史消息都会重新渲染**，即使它们的数据完全没变。

每次渲染 `MessageItem` 都要执行：

- `groupMessageBlocks()` 块分组计算（`message-blocks.ts`）
- `blocksToText()` / `blocksToThinkingText()` 文本提取
- `StreamdownMarkdown` 组件树重建（含 `streamdown` 内部 markdown AST 解析）

如果对话有 50 条消息，每个 token 到达（rAF 帧 ≈ 最高 60fps）就要重算 50 条消息的块分组 + 文本提取 + markdown 渲染。

**影响**：严重。是卡顿的最大来源。

### 2. StreamdownMarkdown 在流式模式下每 token 重解析 markdown

**位置** `frontend/src/components/chat/StreamdownMarkdown.tsx:189`

流式输出时 `isAnimating=true`，`Streamdown` 以 `mode="streaming"` 渲染。每收到一个 token，`content` prop 就变了，`Streamdown` 内部重新解析整个 markdown 文本生成 AST。虽然 streamdown 是为流式设计的，但高频 token 下（本地模型可能每秒数百 token）这个开销不可忽略。

**影响**：严重。配合问题 1 放大——每次重渲染都附带 markdown 解析。

### 3. thinking 计时器每 100ms 触发 MessageItem 重渲染

**位置** `frontend/src/components/chat/MessageItem.tsx:299-308`

```ts
const THOUGHT_TIMER_INTERVAL_MS = 100
// 当 hasRunningThinkingBlocks 为 true 时：
// setInterval(() => setCurrentTimeMs(Date.now()), 100)
```

思考期间每秒 10 次 `setState`，由于没有 memo，会连带触发全部消息重渲染。

**影响**：中等。思考期间持续制造额外渲染，但在文本流式场景不触发。

### 4. rAF 批处理在本地模型高吞吐场景下不够

**位置** `frontend/src/hooks/useChat.ts:569-589`

`enqueueStepDelta` 用 `requestAnimationFrame` 做帧级批处理，把相邻的同 step delta 合并。在远程 API（几十 token/秒）下没问题，但本地模型可能一帧内积压大量 token，合并后的 content 仍然很长，导致"瞬间加载一大段"的视觉。

当前合并逻辑（`useChat.ts:575`）只看 `stepId`、`kind`、`stream` 是否匹配，合并的是 content 字符串拼接。但一帧内可能收到 20-30 个 delta，合并后一次 `setState` 的 payload 仍然很重。

**影响**：中等。是"瞬间加载"而非"逐字输出"的直接原因。

### 5. 没有虚拟化 — 长对话 DOM 节点过多

**位置** `frontend/src/components/chat/MessageList.tsx:220`

```tsx
{messages.map((message) => (
  <MessageItem key={message.id} message={message} ... />
))}
```

直接渲染所有消息，没有用虚拟列表。对话越长，DOM 节点越多，layout/reflow 开销越大。配合同一帧内所有节点重渲染（问题 1），规模效应明显。

**影响**：低到中等。在短对话中不明显，对话超过 30 条后逐渐显著。

## 修复建议（按投入产出比排序）

| 优先级 | 改动 | 预期收益 | 风险 |
|--------|------|----------|------|
| P0 | `React.memo` 包裹 `MessageItem` + 自定义比较函数 | 消除 90% 无意义重渲染 | 极低 |
| P0 | `useMemo` 包裹 `groupMessageBlocks` / `blocksToText` / `blocksToThinkingText` | 减少每次重渲染的计算量 | 极低 |
| P1 | `StreamdownMarkdown` 外层加 `React.memo`，content 相同则跳过 | 避免流式模式下 AST 重建 | 低 |
| P1 | 在 `enqueueStepDelta` 或 `flushPendingStepDeltas` 中加 throttle（例如最少 50ms 间隔、或者一帧内合并后仍过长的做截断） | 平滑本地模型的高吞吐输出 | 低 |
| P2 | 移除 100ms thinking 计时器的全局影响（将 `currentTimeMs` 改为 ref + CSS animation 驱动而非 setState） | 消除思考期间的额外 10fps 渲染 | 低 |
| P2 | 引入虚拟列表（`@tanstack/react-virtual`） | 长对话场景大幅减少 DOM 节点 | 中 |
