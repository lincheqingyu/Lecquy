## 0. 执行原则（不可违反的红线）

读这份指令的人（kira / codex / 其它 LLM）必须先承诺三件事，否则不要动键盘：

1. **先实测，再动手**。两份审查报告都是 code reading + 推理，没用 React Profiler / Chrome Performance trace 实测过。**第 1 节没做完，不要进第 2 节及以后。**
2. **删代码优先于加代码**。本指令第 2 节四件事至少减 200 行 JS。如果你做完发现代码量没减反增，停下来重读 CLAUDE.md 第 4 节。
3. **治本不治标**。明确拒绝：`requestAnimationFrame` 50 ms 节流（用降帧率换流畅感，骗眼睛）。明确不做：任何只对短对话起效、长对话仍会复发的修复。

## 1. 必做：实测定位（5 分钟）

**目的**：用真实数据决定第 2 节做完之后还需不需要进第 3 节。

**操作**：

1. Chrome 打开 dev server，开 React DevTools Profiler。
2. 起一段流式对话——优先用本地模型，因为它最容易触发"瞬间加载"现象。
3. Record，让模型输出 5 秒文本（一段中等长度的回答），Stop。
4. 看：
   - **commit duration 分布**——p50 / p95 / max 各是多少。一帧预算 16.6 ms，p95 > 8 ms 就值得修。
   - **哪个组件占 commit time 最多**——`MessageList` / `MessageItem` / `StreamdownMarkdown` / 别的？
   - **commit 频率**——是不是每帧都在 commit？
5. 同时开 Network 面板，看 `/api/v1/chat/ws` 的 frames 时间线：
   - 本地模型 frame 间隔是不是均匀？还是本身就 burst？
   - 如果后端本身就 burst，"瞬间加载"的根因部分在后端，前端再优化也只能消极防御。

**输出**：把 Profiler 截图 + Network frame 时间线截图存到 `docs/frontend/历史归档/20260509-profiler-baseline/`，作为后续验收的对照基线。

**这一步不做不要进第 2 节**。否则你不知道在治什么。

### 2.3 MessageList 手写贴底改 CSS overflow-anchor

**这是阶段一里收益最大的一件**。当前 `MessageList.tsx` 有 ~70 行手写"贴底跟随 + 用户主动滚开后释放"的逻辑（`ResizeObserver` + 三个 `useEffect` + `userInteractingRef` + `isPinnedRef` + `scheduleUserInteractionRelease` + `markUserInteraction` + 两个阈值常量 + `scrollToBottom` 函数）。

**这套逻辑在每个 token 都会触发"读 scrollHeight + 写 scrollTop"的强制同步布局**。`overflow-anchor` 是浏览器原生能力（Chrome/Edge/Safari/Firefox 全支持），完全替代这套 JS。

**做法**：

1. 在 `MessageList.tsx` 顶层容器添加底部 sentinel：

   ```tsx
   <div ref={containerRef} className="chat-scroll-mask chat-scrollbar ...">
     <div ref={contentRef} className="...">
       {messages.map(...)}
     </div>
     <div className="chat-scroll-anchor" aria-hidden="true" />
   </div>
   ```

2. 在 `frontend/src/index.css` 加（或追加到现有 chat-scroll 段）：

   ```css
   .chat-scroll-mask {
     overflow-anchor: none;  /* 默认所有元素退出锚点候选 */
   }
   .chat-scroll-anchor {
     overflow-anchor: auto;  /* 只让底部哨兵当锚点 */
     height: 1px;
     width: 100%;
   }
   ```

   行为：内容增长时浏览器自动让锚点保持在视口位置——锚点是底部哨兵→ 自动贴底。用户主动向上滚后锚点退出视口，浏览器自然停止贴底，无需 JS 干预。

3. **删除清单**（`MessageList.tsx`）：
   - 删 `BOTTOM_THRESHOLD_PX` / `STRICT_BOTTOM_THRESHOLD_PX` / `USER_SCROLL_COOLDOWN_MS` 三个常量。
   - 删 `userInteractingRef` / `userScrollCooldownRef` / `isPinnedRef` 三个 ref。
   - 删 `isPinnedToBottom` state 与同步它的 useEffect。
   - 删 `isNearBottom` / `syncPinnedState` / `scheduleUserInteractionRelease` / `markUserInteraction` 四个函数。
   - 删 `scrollToBottom` 函数。
   - 删 Effect 1 / Effect 2（`ResizeObserver`）/ Effect 3。
   - 删 `onScroll={syncPinnedState}` / `onWheel={markUserInteraction}` / `onTouchMove={markUserInteraction}` 三个事件绑定。
   - 删 `bottomAnchorRef` 与对应的 `<div ref={bottomAnchorRef}>`（被新的 `.chat-scroll-anchor` 取代，但用 className 不用 ref）。
4. **保留**：`scrollRequestVersion` 用 props 接收时仍要做"显式跳底"——会话切换、用户主动跳底场景。但实现方式简化为：

   ```tsx
   useEffect(() => {
     if (messages.length === 0 || !containerRef.current) return
     const el = containerRef.current
     el.scrollTop = el.scrollHeight
   }, [scrollRequestVersion])
   ```

   只这一个 effect，依赖只 `scrollRequestVersion`。

**风险与回退**：

- `overflow-anchor` 在用户主动滚动后行为会变得"粘"——某些浏览器可能在用户向上滚一下又恢复贴底前有不一致。**做完后必须人工测试**：
  - 流式输出中向上滚 → 应停止贴底，停在该位置。
  - 流式输出中滚到最底 → 应自动恢复贴底。
  - Safari / Chrome / Firefox 各跑一遍。
- 如果某浏览器行为异常，**不要**回滚到原版手写——保留 CSS 方案 + 加最小化 fallback：仅在最底部时给 `<html data-chat-pinned>` 加一个属性，由它驱动行为。具体细节这份指令不展开，发现再写。

**预期减少**：约 70 行 JS + 加 8 行 CSS = 净减 62 行。

## 3. 阶段二：消除冗余渲染（实测后决定）

**前提**：阶段一做完，实测仍卡。常见症状：commit duration p95 仍 > 8ms，且火焰图显示大头是"50 条 MessageItem 全部 re-render"。

**这一节里所有改动都是治本，但都是"消除冗余"层面的——不是架构改动**。

### 3.1 MessageItem 上 `React.memo` + 自定义比较函数

**核心思路**：每个 token 只让"props 真的变了的"那条消息重渲染。其它 49 条历史消息引用稳定 → memo 跳过。

**比较函数要点**：

```tsx
export const MessageItem = memo(MessageItemImpl, (prev, next) => {
  if (prev.message !== next.message) return false  // 引用变了肯定要渲染
  if (prev.isLastAssistant !== next.isLastAssistant) return false  // [此 prop 已在阶段一删除，本行可不要]
  if (prev.activeAttachmentKey !== next.activeAttachmentKey) return false
  // 回调函数 props 由 useChat 用 useCallback 稳定，不需要比较
  return true
})
```

**前置条件**：`useChat.ts` 里所有 `setMessages(prev => prev.map(m => m.id === id ? updater(m) : m))` 必须保证**不变的消息引用真的不变**——目前看是这样（只 patch 命中那一条），但要扫一遍确认。

### 3.2 StreamdownMarkdown 上 `React.memo`

```tsx
export const StreamdownMarkdown = memo(StreamdownMarkdownImpl)
```

默认浅比较即可——`content` 相同就跳过 streamdown 内部的 markdown AST 重建。

### 3.3 MessageItem 内重计算用 `useMemo`

`MessageItem.tsx:255-289` 每次渲染都跑：

- `groupMessageBlocks(message.blocks ?? [])`
- `blocksToText(message.blocks)`
- `blocksToThinkingText(message.blocks)`
- 各种 `.some(...)` 遍历

把这些用 `useMemo([message.blocks])` 包住。即使 memo 在阶段一/二某条没拦住，至少这条消息内部不重算。

### 3.4 dark-mode 提到全局 atom

**做法**：

1. 新增 `frontend/src/lib/theme.ts`，导出一个全局 `useIsDarkMode()` hook。内部只挂**一次** `MutationObserver` 监听 `<html>` 的 class，用一个简单的 ref-counting 或全局 store（避免引入 zustand 等大库——直接 module-level state + `useSyncExternalStore` 即可）。
2. 删 `StreamdownMarkdown.tsx` 里的 `useDocumentDarkMode` 与 `detectDarkMode` 函数。
3. 替换 `useDocumentDarkMode()` 调用为 `useIsDarkMode()`。
4. 同样地，`MermaidBlockRenderer` 内部也用同一个 hook。

**预期效果**：N 个观察者 → 1 个观察者；切换主题时只 broadcast 一次。

### 阶段二验收

- Profiler 火焰图里历史消息的 `MessageItem` 不应再出现在 commit 上（已被 memo 跳过）。
- 只有最新那一条 assistant 消息每 token 在 commit。
- commit duration p95 应进一步下降（参考线：< 4ms）。

**如果体感够好** → 停。**长对话场景**（>100 条历史消息）下要补一次实测：滚到顶 + 起一段流式输出，确认 DOM 节点数对帧率影响。

## 4. 阶段三：架构层（仅当阶段二不够）

**绝对前提**：阶段一 + 阶段二都做完，且**长对话**实测仍不达标。短对话不要做这一节。

这一节的改动会动 `useChat` 的核心数据契约——属于"结构性重构"，**需要 kira 二次确认才动**，不要看到这份指令直接做。

### 4.1 历史消息虚拟化

引入 `@tanstack/react-virtual`。MessageList 改为虚拟列表：

- mount 数量与对话长度脱钩。
- 流式那一条必须强制 mount——不能被滚出视口就虚拟掉。需要在 virtualizer 里把它的 index 加到 `overscan` 或单独保留。
- 与阶段一的 `overflow-anchor` 协同：虚拟列表通常自己控制 scroll 行为，可能要重新评估贴底策略。

风险：与现有的 `mask-image` / `chat-scroll-mask` 视觉效果可能冲突，需测试。

### 4.2 流式消息脱离 React 数据流

**只对当前正在流的那一条消息做特殊处理**，其它消息照常走 React。

数据结构二分：

```ts
interface UseChatState {
  frozenMessages: ChatMessage[]      // 已结束的消息，引用稳定
  streamingMessage: {                // 当前流的那一条，单独维护
    id: string
    elementRef: React.RefObject<HTMLDivElement>
    blocks: MessageBlock[]           // 用 ref 持有，不进 React state
  } | null
}
```

WS `step_delta` 路径变成：

- text stream delta：直接 `streamingMessage.elementRef.current!.appendChild(document.createTextNode(delta.content))`，**不**经 React。
- thinking stream / tool_call / step_state 等：仍走 React（频率低，不在性能关键路径上）。
- `step_state` 完成时：把 `streamingMessage.blocks` 序列化成完整消息塞进 `frozenMessages`，由 streamdown 接管 markdown 渲染。

**取舍点（kira 必须明确表态）**：

- **选项 A**：流式期间显示 plain text（无 markdown 渲染），结束后才完整 streamdown。优点是最简单、最快。缺点是流式期间看到的是裸字符。
- **选项 B**：流式期间用一个轻量增量 markdown 渲染器（仅识别段落 / 列表 / 代码块的开关，不做 syntax highlighting / mermaid），结束后才走 streamdown。优点是流式期间有基本格式感。缺点是要写或选一个轻量渲染器。

**预期效果**：commit duration 不再随对话长度 / 最新消息长度变化——O(1)。

### 阶段三验收

- 500 条历史消息 + 持续流式输出 5 秒：FPS 应稳定 >50。
- 单条 5 万 token 的回答流式输出：FPS 应稳定 >50。
