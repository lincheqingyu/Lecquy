# 消息时间线与工具动作呈现 技术规范

## 背景

Lecquy 当前的对话消息不是单纯的“文本气泡”，而是由以下几类内容共同组成的一条时间线：

- 思考片段
- 助手正文
- tool 动作
- 文件产物动作

此前前端存在两个问题：

- 文件动作卡片被固定渲染在整条 assistant 消息顶部，导致 `Created a file` 提前出现，不在真实执行位置
- 原始 tool 名和动作语义混用，出现 `已调用 write_file`、`已调用 skill` 这类偏实现细节的文案

本规范定义 Lecquy 前端对“思考 / tool / 文件动作”的统一呈现口径。

## 目标

- 用户看到的是“发生了什么”，不是“底层调用了哪个 API 名”
- 文件动作必须出现在对应执行位置，而不是消息顶部或底部的独立区域
- 实时流式渲染与历史回放保持同一套时间线语义
- 内置 tool 和未来配置型 tool 共享同一套展示框架

## 核心原则

### 1. 时间线优先

assistant 消息按实际发生顺序渲染。

顺序基于 `message.blocks` 中的内容块序列，并允许在 tool 位置插入对应的动作节点。

正确顺序示例：

1. 第一段思考
2. 正文
3. `Loaded fin-bookkeeper`
4. 第二段思考
5. 正文
6. `Created a file`
7. 最终汇报正文

### 2. 动作语义优先于原始 tool 名

对用户展示时，不直接使用原始 `toolName`。

展示层统一映射到动作语义，例如：

- `skill` -> `Loaded {skillName}`
- `read_file` -> `Viewed {fileName}`
- `bash` -> `Ran {commandSummary}`
- `write_file` 成功 -> 不显示原始 tool 卡，改为文件动作节点
- 未知 tool -> `Used a tool`

原始 `toolName` 只保留给协议、调试和错误排查使用。

### 3. 文件动作是时间线节点，不是独立顶部区域

`write_file` 成功后，用户应看到的是 `Created a file` / `Updated a file` 这样的动作节点，并且该节点必须插入在对应 `write_file` 调用发生的位置。

禁止以下做法：

- 把所有文件动作统一挂到消息顶部
- 把文件动作单独做成与正文完全脱离的“前置摘要”
- 同时展示 `write_file` 成功卡和 `Created a file` 卡，造成重复

### 4. 思考按“段”管理，不按“整条消息”管理

每个 thinking 段独立计时、独立折叠。

规则：

- 第一段思考结束后，时长冻结
- 后续新思考段重新开始单独计时
- 不使用“继续思考”“继续思考中”这种口径
- 统一显示为每段自己的 `思考了 x.xs`，无时长时显示 `思考`

## 前端实现约束

## 数据层

- `MessageThinkingBlock` 持有自己的 `startedAt`、`endedAt`、`durationMs`、`status`
- `MessageToolCallBlock` 仍表示原始 tool 调用
- 文件动作不再通过“消息顶部独立区域”表达，而是在渲染时按 tool 顺序插入

## 渲染层

关键文件：

- `frontend/src/components/chat/MessageItem.tsx`
- `frontend/src/components/chat/ToolCallCard.tsx`
- `frontend/src/components/chat/ToolGroupCard.tsx`
- `frontend/src/components/artifacts/ArtifactTrace.tsx`

关键规则：

- `MessageItem` 负责按消息时间线组装最终渲染顺序
- `ToolCallCard` 负责把原始 tool 映射为动作语义文案
- `write_file` 成功时，隐藏原始成功 tool 卡
- 匹配到的文件动作节点插入到对应 tool 的位置
- 位置匹配优先使用文件名；若 tool block 缺少稳定文件名，则按当前消息内尚未消费的 `write_file` 动作顺序兜底匹配
- 若极端情况下无法匹配到位置，允许作为末尾 fallback 展示，但不得回退到顶部

## 实时流式与历史回放

无论是 WebSocket 实时消息还是历史会话恢复，展示口径必须一致：

- 思考段独立计时
- tool 文案使用动作语义
- 文件动作出现在执行处

如果后续要继续增强，优先方向不是再加顶部摘要，而是把“文件动作节点”进一步下沉为显式的 timeline block。

## 未来扩展

针对未来配置型 tool，统一按两层处理：

- 协议层保留原始 `toolName`
- 展示层使用标准化动作语义

建议后续为配置型 tool 增加可选展示元数据：

- `intent`
- `subject`
- `presentation`

如果未提供展示元数据，则回退到通用文案 `Used a tool`，但仍遵守时间线排序。

## 验收标准

- `Created a file` 出现在对应 `write_file` 执行位置
- 消息顶部不再有独立前置文件动作区域
- 成功的 `write_file` 不再显示 raw tool 成功卡
- 思考段各自独立展开与计时
- 不再出现 `已调用 skill`、`已调用 write_file`、`继续思考` 这类口径
