## 0. 适用范围

本规范只描述 RightRail **状态机本身**的契约：状态形状、action 集合、transition 表、不变量、Esc / focus 行为、与外部 ref 的边界、单元测试口径。

本规范**不覆盖**：

- Context / Progress / Artifact / Memory / Approval 各 mode 的内容实现——属于后续四份独立规范。
- DocumentPanel / ArtifactPanel 业务逻辑迁移到 Artifact mode——属于 Artifact mode 规范覆盖。
- 中央时间线 ↔ 右侧的点击路径接通——属于联动验收阶段。

按 [20260513-3 §14.2](./20260513-3-右侧工作区人机交互逻辑%20技术规范.md) 的术语口径：本任务是 **结构重构**，不是视觉迁移，也不是补丁路线。

## 1. 为什么先做这件事

按 20260513-3 §14.7 的验收红线："右侧仍由 `openDocument` 和 `isSettingsOpen` 两套状态互斥"是不通过项。当前实现里 RightRail 已经有一个 reducer，但：

- `openDocument`（HomePageLayout 本地 useState）和 `rightRailState`（reducer）是**两套并行状态**，通过手动 `dispatchRightRail({ type: 'close' })` 配合 `setOpenDocument(null)` 维持一致——这是补丁，不是单一状态模型。
- 没有 `request-approval` / `resolve-approval` action，approval 阻塞 + 恢复 `lastNonBlockingMode` 的关键流转不存在。
- `close` 行为对所有 mode 一致（关闭 rail 并清 pinned），但 20260513-3 §6.3 / §11.1 要求按 mode 分级：artifact 应回 context，runtime 应关闭 rail，approval 不允许普通关闭。
- 没有 `set-memory-ref` / `set-approval-ref` / `set-progress-ref` action，三个 ref 字段虽然在 state shape 里，但无人写入。
- 没有单元测试覆盖 transition 表，回归靠肉眼。

只要 RightRail 里只有 Runtime 一个 mode，上面问题都隐身。加 Context（第二个 mode）的那一刻起，每条都会立即触发补丁式 `useEffect` 抢状态——这正是本次状态机规范要先做的原因。

## 3. 状态机契约

### 3.1 State 形状（**不变**）

保留现有 `RightRailState`。本规范不修改 state 形状，只补 action 与 transition。

```ts
interface RightRailState {
  isOpen: boolean
  mode: RightRailMode               // 'context' | 'progress' | 'artifact' | 'memory' | 'runtime' | 'approval'
  openReason: RightRailOpenReason   // 'user' | 'system' | 'restore' | 'approval'
  pinnedMode: RightRailMode | null
  lastNonBlockingMode: RightRailMode
  artifactRef: RightRailArtifactRef | null
  memoryRef:   RightRailMemoryRef   | null
  approvalRef: RightRailApprovalRef | null
  progressRef: RightRailProgressRef | null
}
```

### 3.2 Action 集合（**补齐**）

补齐后的 action 列表共 11 项，按职责分组：

```ts
type RightRailAction =
  // 用户显式操作
  | { type: 'open-mode'; mode: RightRailMode; reason?: RightRailOpenReason; pin?: boolean }
  | { type: 'toggle-mode'; mode: RightRailMode; reason?: RightRailOpenReason; pin?: boolean }
  | { type: 'close'; trigger?: 'esc' | 'button' }
  // 系统弱自动
  | { type: 'progress-event'; ref: RightRailProgressRef }
  | { type: 'artifact-draft-detected'; ref: RightRailArtifactRef }
  // 阻塞
  | { type: 'request-approval'; ref: RightRailApprovalRef }
  | { type: 'resolve-approval' }
  // 引用同步
  | { type: 'set-artifact-ref'; ref: RightRailArtifactRef | null }
  | { type: 'set-memory-ref';   ref: RightRailMemoryRef   | null }
  | { type: 'set-progress-ref'; ref: RightRailProgressRef | null }
  // 生命周期
  | { type: 'reset-session' }
```

不再有 `set-approval-ref`：approval ref 与 approval mode 进入是一次性原子动作，由 `request-approval` 同步写入；外部对 approval 的所有意图都通过 `request-approval` / `resolve-approval` 表达，避免出现"ref 已写但 mode 没切"或反之的不一致中间态。

### 3.3 Transition 表（**强约束**）

下表是 reducer 实现的唯一参考，落地时必须有对应单元测试一对一覆盖（§5）。

| # | 当前 state | action | 新 state | 不变量 |
|---|---|---|---|---|
| 1 | 任意（mode ≠ approval） | `open-mode` reason=user | `isOpen=true, mode=action.mode, pinnedMode=action.mode（若 isUserInspectMode 或 pin=true）, lastNonBlockingMode=action.mode` | pinnedMode 只在 user inspect 类（artifact/memory/runtime）或显式 pin=true 时写入 |
| 2 | mode=approval | `open-mode`（任意非 approval） | 不变 | approval 不可被普通 open 覆盖 |
| 3 | pinnedMode≠null，pinnedMode≠action.mode | `open-mode` reason=system | 不变 | system 必须尊重 user pinned |
| 4 | isOpen=true, mode=action.mode | `toggle-mode` | 等价于 `close`（按 §3.4 close 子表分级） | 同 mode 再按 toggle 即关闭 |
| 5 | isOpen=false 或 mode≠action.mode | `toggle-mode` | 等价于 `open-mode` | 不同 mode toggle 即打开 |
| 6 | 任意非 approval（含 pinnedMode≠null） | `progress-event` | 若 pinnedMode=null 且 mode∈{closed, context, progress}：切到 progress，progressRef=ref，openReason=system；否则只写 progressRef，不切 mode | 用户检查态（artifact/memory/runtime pinned）不被 progress 抢占 |
| 7 | 任意非 approval | `artifact-draft-detected` | 若 pinnedMode=null 且 mode∈{closed, context, progress}：切到 artifact，artifactRef=ref，openReason=system；否则只写 artifactRef，不切 mode | 同上，user pinned 不被弱自动打开抢 |
| 8 | mode=approval | `progress-event` / `artifact-draft-detected` | 只写对应 ref，不切 mode | approval 期间 ref 仍可累积，恢复后回 last mode 时可见 |
| 9 | 任意 | `request-approval` | `isOpen=true, mode=approval, approvalRef=ref, openReason=approval, lastNonBlockingMode=（保留当前非 approval mode）` | approval 总是置顶；保存恢复点 |
| 10 | mode=approval | `resolve-approval` | `mode=lastNonBlockingMode, approvalRef=null, openReason='restore', isOpen=（保留原值或 true）` | 完成后回到非阻塞 mode；isOpen 不被自动关闭 |
| 11 | mode≠approval | `resolve-approval` | 不变 | approval 不在时 resolve 是 noop（避免误触） |
| 12 | 任意非 approval | `close` trigger='esc' | 按 mode 分级（见 §3.4） | Esc 行为按 §11.1 Esc 表 |
| 13 | 任意非 approval | `close` trigger='button' / 缺省 | `isOpen=false, pinnedMode=null` | 显式关按钮总是直接关闭 rail |
| 14 | mode=approval | `close` 任意 trigger | 不变 | approval 不允许普通关闭 |
| 15 | 任意 | `set-artifact-ref` / `set-memory-ref` / `set-progress-ref` | 仅写对应 ref，不切 mode、不改 isOpen、不改 pinnedMode | ref 写入与 mode 切换严格解耦 |
| 16 | 任意 | `reset-session` | 完全等价于 `createInitialRightRailState()` | 切换 session 必须清空所有 ref + pinnedMode |

### 3.4 close trigger='esc' 的 mode 分级子表

来自 20260513-3 §11.1：

| 当前 mode | Esc 结果 |
|---|---|
| `runtime` / `context` / `progress` | `isOpen=false, pinnedMode=null`（关闭 rail） |
| `artifact` | 若 lastNonBlockingMode='context'：切到 context，pinnedMode=null；否则关闭 rail |
| `memory` | 切到 context，pinnedMode=null |
| `approval` | 不变（Esc 在 approval 期间被吞掉，由 mode 内部组件处理焦点） |

实现时把这张子表放在 reducer 的 `close` case 内分支处理；不要把 Esc 行为放到 React 组件的 keydown 监听里手写。

### 3.5 不变量（**reducer 内部断言**）

每次 reducer 返回新 state 前必须满足：

1. `mode === 'approval'` ⟺ `approvalRef !== null`。
2. `mode === 'approval'` 时，`lastNonBlockingMode` 必须是上一次非 approval 的 mode（不允许是 approval 本身）。
3. `pinnedMode` 只能是 `null` 或属于 user inspect 类（`artifact` / `memory` / `runtime`）；不允许 pin 到 `context` / `progress` / `approval`。
4. `isOpen === false` ⟹ `mode === lastNonBlockingMode`（关闭后不残留奇怪 mode，下次打开恢复到关闭前的 mode 而不是闪一下 approval）。
5. `reset-session` 后必须等于 `createInitialRightRailState()`，没有任何 ref 残留。

建议在开发期通过 `assertInvariants(state)` 在每个 case 返回前调用一次；生产构建可通过 `import.meta.env.DEV` 关闭。

## 4. 与外部状态的边界

### 4.1 `openDocument` 必须收敛到 `artifactRef`

20260513-3 §14.7 红线之一是"两套互斥状态"。本规范要求：

- **删除** `HomePageLayout.tsx:288` 的 `const [openDocument, setOpenDocument] = useState<OpenDocument | null>(null)`。
- artifact / attachment 的"打开"统一通过 `dispatch({ type: 'open-mode', mode: 'artifact', reason: 'user', pin: true })` + `dispatch({ type: 'set-artifact-ref', ref })`。
- 当前 `OpenDocument` 联合类型（`attachment` / `artifact` 两 kind）迁入 `RightRailArtifactRef`：

  ```ts
  interface RightRailArtifactRef {
    kind: 'attachment' | 'artifact'
    key: string
    sessionKey?: string | null
    // 必须能反查回 attachment 或 artifact 的具体对象——由 messageId + index 派生，由 Artifact mode 在渲染时从 ConversationArea 提供的 currentArtifacts / messages 中查找。
    messageId: string
    indexInMessage: number
  }
  ```

- `currentArtifacts` 镜像数据（HomePageLayout.tsx:290）保留，作为 Artifact mode 渲染时的数据源；它不是状态，是订阅。
- `showDocumentWorkspace` / `documentPanelWidth` / DocumentPanel 渲染分支（HomePageLayout.tsx:618, 885-934）属于 Artifact mode 规范覆盖范围，本规范**不要求在本阶段删除**，但要求把 `openDocument` 字段名替换成由 reducer 派生的 `artifactRef`，渲染分支照旧从派生值读取。删除分支由后续 Artifact mode 规范统一执行。

### 4.2 弱自动打开必须用 action，不用 useEffect 双写

当前 `HomePageLayout.tsx:690-702` 的 draft auto-open useEffect 必须重构为：

```ts
useEffect(() => {
  const nextDraft = currentArtifacts.find(/* ...findFirstUnseenDraft... */)
  if (!nextDraft) return
  dispatchRightRail({
    type: 'artifact-draft-detected',
    ref: toArtifactRef(nextDraft),
  })
}, [currentArtifacts])
```

新增的 `artifact-draft-detected` 由 reducer 内部按 transition #7 决定"切 mode + 写 ref"还是"只写 ref"，外部不需要判断 `rightRailState.pinnedMode`。这是把弱自动打开的决策从 useEffect 收敛到 reducer 的关键。

### 4.3 顶栏齿轮 / Esc 监听层

- 顶栏齿轮（HomePageLayout.tsx:822-837 的 PanelRight 按钮）：`onClick` 改为 `dispatchRightRail({ type: 'toggle-mode', mode: 'runtime', reason: 'user', pin: true })`，删除手动 `setOpenDocument(null)`（由 transition 表自然处理）。
- 全局 Esc 监听：建议在 `HomePageLayout` 用一个 `useEffect` 监听 `keydown`，命中 Esc 且未在输入框 / textarea 内时 dispatch `{ type: 'close', trigger: 'esc' }`。**禁止**在 RightRail 子组件各自加 keydown 监听，避免抢键。

## 6. 持久化（沿用 §13）

本规范不强制实现持久化；如果实现，建议如下边界：

- 持久化键：`lecquy.rightRailState.{sessionKey}`。
- 持久化字段：`{ mode, lastNonBlockingMode, isOpen, artifactRef }`。
- 不持久化：`approvalRef`、`pinnedMode`、`progressRef`、`memoryRef`、`openReason`。
- 持久化时机：reducer 派生 selector，外部 effect 防抖写入；不在 reducer 内部 `localStorage.setItem`。

持久化不是本阶段验收项；若推迟到后续 Context mode 规范一起做，本规范不阻塞。

## 7. 不做什么

- **不引入** Context / Progress / Artifact / Memory / Approval 任何 mode 的内容渲染（保留现有 RuntimeMode + EmptyState 占位）。
- **不引入** 持久化（§6 仅建议）。
- **不引入** WS 协议层用于 progress / approval / memory 事件（这是后端先长能力的事）。
- **不引入** 中央时间线 → 右侧的点击路径（属于联动验收阶段）。
- **不引入** 响应式断点逻辑（§12 / §8 留给视觉规范阶段）。
- **不改动** RuntimeMode 组件本身（功能已完成，状态机重构与 Runtime mode 内容解耦）。

## 8. 验收口径

代码层落地后必须满足：

| 检查项 | 通过标准 |
|---|---|
| Action 完整 | `RightRailAction` 联合类型包含 §3.2 列出的 11 项 |
| Transition 表覆盖 | `rightRailState.test.ts` 至少有 §5 列出的全部 case 通过 |
| 不变量断言 | 开发模式下 reducer 每个 case 返回前调用 `assertInvariants`，违反时抛出 |
| 单一状态模型 | `grep -n 'openDocument' HomePageLayout.tsx` 返回 0 行 |
| draft 弱自动打开收敛 | `HomePageLayout.tsx` 中 draft 检测 effect 内部只调用 `dispatchRightRail({ type: 'artifact-draft-detected', ref })`，不再读取 `pinnedMode` / `openDocument` 判断 |
| Esc 行为 | RuntimeMode 打开时按 Esc 关闭 rail；artifact mode 打开时按 Esc 回 context；approval mode 打开时按 Esc 不动作（本阶段 approval 不可达，靠手写 `request-approval` dispatch 在测试里验证） |
| approval 恢复 | dispatch `request-approval` → dispatch `resolve-approval` 后 mode 等于 request 之前的 mode |
| 顶栏齿轮一致 | 齿轮按钮触发 `toggle-mode runtime user`，再次点击关闭 rail；不残留 `openDocument` 状态 |
| 文件预览基线 | 当前 DocumentPanel / ArtifactPanel 渲染分支仍能从 `artifactRef` 派生显示（视觉与现状一致，不退化），后续 Artifact mode 规范再统一替换 |
| 没有 console error | 用户操作路径（runtime 切换 / artifact 点开 / 切 session / 关闭 rail）无 console 报错 |

## 10. 红线

- 不允许在 `HomePageLayout` 用 useEffect 直接判断 `pinnedMode` / `mode` 来决定是否 dispatch；判断必须在 reducer 内部。
- 不允许保留 `openDocument` 作为与 reducer 并行的状态——这是 20260513-3 §14.7 验收红线之一。
- 不允许在 RightRail 子组件内部各自加 keydown 监听抢 Esc。
- 不允许跳过单元测试直接补 reducer——本规范的契约本体是 §3.3 transition 表，测试是契约的可执行表达。
- 不允许把"Approval / Progress / Memory ref 字段"删掉换更"简洁"的设计——这些字段是后续 mode 落地的接口，删了后面要返工。
