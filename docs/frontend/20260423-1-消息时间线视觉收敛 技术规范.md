# 消息时间线视觉收敛 技术规范

## 文档关系

- 本文档**替代** [20260422-2-消息时间线 UI 问题修复 技术规范](./历史归档/20260422-2-消息时间线%20UI%20问题修复%20技术规范.md)（已归档）
- 本文档**继承** [20260422-1-消息时间线与工具动作呈现 技术规范](./20260422-1-消息时间线与工具动作呈现%20技术规范.md) 定义的语义映射与时间线顺序，不修改任何业务语义，仅做视觉与交互收敛

## 背景

20260422-2 曾针对三处视觉缺陷（流式草稿默认折叠、思考模块图标错位、文件卡片视觉层级过重）给出局部修复方案。落地评估时发现：三处症状的共同根因是**时间线上存在 4 种并行的视觉语言**（`OperationRow` / `ReadyOperationRow` / `ReadyArtifactLink` / `renderThinkingGroup`），每种都自带字号、对齐规则、展开策略，局部修补只能缓解单点症状，不能消除不一致的结构性来源。

同时在确认 20260422-2 问题三的"右侧面板可用性"时发现：

- 真正承担 `write_file` 产物渲染的是 `ArtifactPanel`（而非 20260422-2 错误指向的 `DocumentPanel`）。`HomePageLayout` 在 `openDocument.kind === 'attachment'` 时渲染 `DocumentPanel`，`'artifact'` 时才渲染 `ArtifactPanel`
- `HomePageLayout.openDocument` 目前是通过 `useState` 保存的打开瞬间快照，**即使消息侧的 draft 内容流式更新，`openDocument.artifact` 也不会重新绑定**。换言之，数据压根没有流到右侧面板
- `ArtifactPanel` 的 `StreamdownMarkdown` 调用未传 `isAnimating`，默认 `false`，没有流式光标与动画
- `ArtifactPanel` 的预览滚动容器没有任何滚动锁定逻辑

因此本次收敛不是单纯的视觉重构，而需要**视觉层收敛 + 数据流打通**一并实施。

## 目标

- 把时间线上"文件事件（draft / ready / 失败）"与"思考块"统一为**同一个视觉原语** `TimelineEvent`
- 把 `write_file` 流式内容的承载点从"时间线内联预览"迁移到"右侧 `ArtifactPanel`"，彻底移除 21rem 内联预览区
- 思考模块的图标与展开竖线的对齐、字号不一致问题通过"共享坐标系"天然解决，无需像素级微调
- 不破坏 20260422-1 定义的任何语义映射与时间线顺序

## 本轮范围说明

**纳入本轮**：文件事件（`write_file` draft/ready、失败）、思考块（`renderThinkingGroup`）、`ArtifactPanel` 流式跟随。

**不纳入本轮**：Tool 调用（`viewed` / `ran` / `loaded`）的 `ToolCallCard` / `ToolGroupCard` 视觉收敛。这两者保留现有卡片样式，由后续独立规范处理。本轮结束后，时间线会在**过渡期**内存在两种视觉语言——文件/思考走 `TimelineEvent` 单行，tool 调用仍走现有卡片——这是已知并接受的取舍，换取本轮改动面可控、风险集中。

## 设计方案

### 视觉层：时间线事件原语 `TimelineEvent`

新建组件 `frontend/src/components/chat/TimelineEvent.tsx`，替代现有 `OperationRow` / `ReadyOperationRow` / `ReadyArtifactLink` 三种卡片形态，并承担 `renderThinkingGroup` 的触发按钮骨架。

```tsx
interface TimelineEventProps {
  icon: ReactNode            // 状态图标 (FileText / Sparkles / Terminal / LoaderCircle / AlertCircle)
  verb: ReactNode            // 动词短语 ("Created" / "Updated" / "正在写入" / "思考了" / "写入失败")
  target?: ReactNode         // 目标 (文件名 / 思考耗时 / tool 参数)
  status?: 'pending' | 'streaming' | 'ready' | 'error'
  onTargetClick?: () => void // 目标可点击时的处理（仅文件名这类"链接"生效，动词保持纯文本）
  expandable?: boolean       // 是否可展开（仅思考块使用）
  expanded?: boolean
  onToggleExpanded?: () => void
  children?: ReactNode       // 展开内容（仅思考块使用）
  actions?: ReactNode        // 展开态右上角的操作区（如思考块的 Copy 按钮）
}
```

**共享视觉规范（消除当前不一致）：**

- 单行基础高度 `h-6`，布局 `flex items-center gap-2`
- 图标统一 `size-3.5`；`status='streaming'` 时图标带 `animate-spin`（LoaderCircle）或 `animate-pulse`；`status='error'` 时图标色 `text-red-500`
- 字号统一 `text-[13px] leading-[1.55]`；**思考块触发按钮与展开内容共享同一字号**（彻底消除 20260422-2 问题二中 `[13px]` / `[14px]` 不一致）
- 动词 `text-text-secondary`，目标 `text-text-muted`；`onTargetClick` 存在时目标部分 hover 下划线 + `text-text-primary`
- 无 chevron、无 badge、无嵌套 pill；`expandable=true` 的事件（仅思考块）才显示 `ChevronDown/Up`
- 事件条之间垂直间距由上层容器用 `space-y-1` 控制（当前 `space-y-2` 偏松，单行化后需收紧）

**展开内容的竖线实现（天然对齐）：**

展开态用 `relative` 定位容器，竖线通过 `absolute` 定位在图标正下方：

```tsx
<div className="relative">
  <button className="flex h-6 items-center gap-2 ...">
    <Sparkles className="size-3.5" /> {/* 图标中心 x = 7px（14px/2） */}
    ...
  </button>
  {expanded && (
    <div className="relative pl-[18px] pt-1.5">
      <span
        aria-hidden
        className="absolute left-[7px] top-1.5 bottom-0 w-px bg-border"
      />
      <div className="text-[13px] leading-[1.55] text-text-secondary">
        {children}
      </div>
    </div>
  )}
</div>
```

竖线的 `left-[7px]` 与图标 `size-3.5`（14px）的中心轴**共享父容器坐标系**，不再需要像 20260422-2 那样对 `ml-[7px]` 做像素级实测。字号在触发按钮与展开容器上都是 `text-[13px]`，不一致问题消失。

### 事件类型映射（本轮范围）

| 事件 | icon | verb | target | status | 展开 |
|------|------|------|--------|--------|------|
| 思考 | `Sparkles` | `思考了` | `2.3s` | `ready` | 是（思考正文 + 可选 Copy） |
| 文件 draft | `LoaderCircle`（spin） | `正在写入` | 文件名 | `streaming` | 否 |
| 文件 ready（created） | `FileText` | `Created` | 文件名（可点击） | `ready` | 否 |
| 文件 ready（updated） | `FileText` | `Updated` | 文件名（可点击） | `ready` | 否 |
| 文件失败 | `AlertCircle`（红） | `写入失败` | 文件名 + 错误摘要 | `error` | 否 |

文件 ready 的目标点击行为：调用 `onOpenArtifact(artifact)` 打开右侧 `ArtifactPanel`。动词本身保持纯文本、非按钮语义。

Tool 调用（`viewed` / `ran` / `loaded`）不在本轮范围，保留现有 `ToolCallCard` / `ToolGroupCard` 实现，后续规范再处理。

### 数据层：`ArtifactPanel` 流式跟随

视觉收敛前必须先把数据流打通，否则"移除 21rem 内联预览"会导致流式过程不可见。

**① artifacts 状态从 `ConversationArea` 镜像到 `HomePageLayout`（通过 `onArtifactsChange` 回调）**

当前 `useChat` 在 `ConversationArea` 内部持有 `messages` 与派生的 `artifacts`，而 `openDocument` state 在 `HomePageLayout` 第 173 行。`HomePageLayout` 无法直接看到 artifacts 的变化，`openDocument.artifact` 保持为打开瞬间快照。

解决方案：`ConversationArea` 新增 prop `onArtifactsChange: (artifacts: ChatArtifact[]) => void`，`HomePageLayout` 持有镜像 state，驱动 `openDocument.artifact` 的同步与自动打开。

```tsx
// ConversationArea.tsx 新增
interface ConversationAreaProps {
  // ...现有 props
  onArtifactsChange?: (artifacts: ChatArtifact[]) => void
}

// 在 ConversationArea 内部，读取 useChat 派生的 artifacts（具体字段取决于 useChat 实现，
// 若 useChat 当前未直接暴露 artifacts，需要从 messages 中派生；可参考 ArtifactTrace 的做法）
const artifacts = useMemo(() => extractArtifactsFromMessages(messages), [messages])

useEffect(() => {
  onArtifactsChange?.(artifacts)
}, [artifacts, onArtifactsChange])
```

```tsx
// HomePageLayout.tsx 新增
const [currentArtifacts, setCurrentArtifacts] = useState<ChatArtifact[]>([])

// 把回调传给 ConversationArea
<ConversationArea
  // ...现有 props
  onArtifactsChange={setCurrentArtifacts}
/>

// 同步 effect：openDocument.artifact 跟随 currentArtifacts 中最新版本
useEffect(() => {
  if (!openDocument || openDocument.kind !== 'artifact') return
  const latest = currentArtifacts.find(
    (a) => a.artifactId === openDocument.artifact.artifactId,
  )
  if (!latest || latest === openDocument.artifact) return
  setOpenDocument({ ...openDocument, artifact: latest })
}, [currentArtifacts, openDocument])
```

`draft → ready` 的状态切换与 `content` 流式增长都会自动触发 `ArtifactPanel` 内部 `useEffect` 重新计算 `detail`。

实现注意事项：`onArtifactsChange` 回调需要引用稳定（`HomePageLayout` 侧直接传 `setCurrentArtifacts` 即可，setter 由 React 保证稳定）；`useMemo` 依赖 `messages` 即可，不需要过度优化。单用户场景下 artifacts 变化频次低（每次 tool 调用或 artifact content chunk 到达时），性能无风险。

**② 弱自动打开策略（单用户场景简化）**

`write_file` 首次进入 `draft` 时：

- 若右侧面板当前**未打开**（`openDocument === null`）：自动打开并绑定到该 draft artifact
- 若右侧面板**已打开**：不抢焦点，仅在时间线上呈现 `正在写入` 事件条，用户手动点击才切换

```tsx
// HomePageLayout.tsx
useEffect(() => {
  if (openDocument) return // 已打开时不抢焦
  const firstDraft = currentArtifacts.find((a) => a.status === 'draft')
  if (!firstDraft) return
  // 调用现有 handleOpenArtifact，保持打开逻辑与手动点击一致
  // 注意：handleOpenArtifact 需要 messageId 与 artifactIndex，需要从 artifacts 回推定位
  const messageId = firstDraft.messageId // 假设 ChatArtifact 已带 messageId 字段
  const artifactIndex = currentArtifacts.indexOf(firstDraft)
  handleOpenArtifact(messageId, artifactIndex, firstDraft)
}, [currentArtifacts, openDocument, handleOpenArtifact])
```

若 `ChatArtifact` 未带 `messageId`，在 `onArtifactsChange` 回调侧补充索引信息一并传出（例如改成 `onArtifactsChange: (items: Array<{ artifact: ChatArtifact, messageId: string, artifactIndex: number }>) => void`）。具体形态在实现第 1 步时确定。

单用户场景下不需要考虑多 tab / 多用户并发，逻辑保持极简。

**③ `ArtifactPanel` 滚动锁定**

Draft 期间滚动容器保持锁底；用户手动上滚时解除锁定，避免干扰阅读。

```tsx
// ArtifactPanel
const [isScrollLocked, setIsScrollLocked] = useState(true)

useEffect(() => {
  if (!isDraft || !isScrollLocked || !previewRef.current) return
  previewRef.current.scrollTop = previewRef.current.scrollHeight
}, [isDraft, isScrollLocked, content])

const handleScroll = (e: UIEvent<HTMLDivElement>) => {
  if (!isDraft) return
  const el = e.currentTarget
  const atBottom = el.scrollHeight - el.scrollTop - el.clientHeight < 8
  setIsScrollLocked(atBottom) // 用户滚到底部→重新锁定；否则解锁
}

// draft → ready 切换时重置锁定状态，避免影响后续查看
useEffect(() => {
  if (!isDraft) setIsScrollLocked(true)
}, [isDraft])
```

**④ `StreamdownMarkdown` 流式动画**

`StreamdownMarkdown`（`frontend/src/components/chat/StreamdownMarkdown.tsx`）已经完整实现 `isAnimating` prop——第 165 行声明、第 176 行根据值切换 `mode` 为 `'streaming'` / `'static'`、第 191 行直接透传给底层 `<Streamdown>`。流式光标、淡入动画由 streamdown 库自身处理。

本规范只需要在 `ArtifactPanel.tsx` 第 235 行把默认值改掉：

```tsx
// 修改前
<StreamdownMarkdown content={content} />
// 修改后
<StreamdownMarkdown content={content} isAnimating={isDraft} />
```

### 思考块展开风格（决策 2：保持推荐）

展开内容仅用竖线分隔，无背景色、无缩进缩减、无额外描边。保持"安静"阅读风格，与 Codex / Claude.ai 的 `Thought for Xs` 展开区一致。Copy 按钮沿用现有 hover 淡入逻辑，定位到展开容器右上角（`absolute top-1.5 right-0`），不再挤在触发按钮旁边。

### 多次编辑同一文件的呈现（决策 E）

同一 artifact 在一次对话中被连续 `write_file → edit → write_file`：**时间线上呈现三条独立事件条**，保留"发生了什么、什么时候发生"的审计感。但所有事件条的目标点击都指向该 artifact 的**最新版本**（与 `ArtifactPanel` 当前显示内容一致）。历史版本的 diff 查看不在本次收敛范围，由后续 artifact 版本管理方案覆盖。

### 失败态规范（决策 D）

`write_file` / `edit` 失败时：

- 事件条：`icon=<AlertCircle className="text-red-500" />`、`verb="写入失败"`、`target=${filename} · ${errorSummary}`
- `errorSummary` 取后端返回的 `error.message` 前 60 字符，超出截断加省略号
- 事件条不可点击展开；完整错误信息由 tool 原始卡在异常路径下正常渲染（20260422-1 规范已定义 `write_file` 失败时回落到原始 tool 卡）
- 因此失败事件与成功事件在时间线上**不是并存关系**：成功走 `TimelineEvent`，失败走原始 tool 卡。这条规则从 20260422-1 继承，本次不修改

## 实施步骤

按"风险由低到高 + 数据流先通"排序：

1. **`ArtifactPanel` 接通流式动画**：第 235 行 `<StreamdownMarkdown content={content} isAnimating={isDraft} />`；同时加滚动锁定逻辑。此步不依赖任何上层改动，可独立验证 streamdown 动画是否正常
2. **`ConversationArea` + `HomePageLayout` artifacts 镜像**：`ConversationArea` 新增 `onArtifactsChange` prop、从 useChat 派生 artifacts；`HomePageLayout` 新增 `currentArtifacts` state 与 `openDocument.artifact` 同步 effect。此步完成后，已打开的 draft artifact 能看到内容流式出现，构成新方案的最小可用状态
3. **Draft 弱自动打开 effect**：在数据已通的前提下加自动打开 effect，注意 `openDocument === null` 时才触发
4. **新建 `TimelineEvent`**，先在 `renderThinkingGroup` 中落地，验证对齐与字号一致（用共享坐标系实现竖线）
5. **用 `TimelineEvent` 重写 `ArtifactTrace` 的 ready 形态**（Created / Updated），移除嵌套 pill 与 chevron
6. **用 `TimelineEvent` 重写 draft 形态**（`正在写入`），移除 21rem 内联预览区与 `PreviewContent` 组件
7. **补齐失败态**（`写入失败` 事件条），回归测试 write_file 异常路径

每一步都能独立验证，不依赖后续步骤稳定。

## 验收标准

**视觉**

- 时间线上本轮范围内的事件（思考、文件 draft/ready、失败）外观一致：单行、`text-[13px]`、图标 `size-3.5`、间距 `space-y-1`；Tool 卡暂保持现有样式，过渡期不一致是预期行为
- 思考展开内容的竖线与 `Sparkles` 图标中心对齐（通过共享坐标系实现，不依赖 `ml-[7px]` 实测）
- 思考触发按钮与展开内容字号均为 `text-[13px]`
- 文件事件条无 chevron、无 badge、无嵌套 pill
- 失败事件条图标为红色 `AlertCircle`，文案 `写入失败 filename · 错误摘要`

**交互**

- `write_file` 首次进入 draft 时，若右侧面板未打开，自动打开并流式显示内容；已打开时不抢焦
- Draft 期间 `ArtifactPanel` 滚动锁底、带流式光标；用户上滚后解锁、滚回底部重新锁定
- Draft → Ready 切换后滚动锁定状态重置
- 文件事件条的文件名可点击，打开右侧 `ArtifactPanel`；动词部分不可点击
- 时间线上 21rem 内联预览区不再出现

**语义**

- 20260422-1 定义的时间线顺序与语义映射完全保留
- 同一文件多次编辑呈现为多条独立事件条，目标点击始终打开最新版本
- `write_file` 失败时回落到原始 tool 卡（沿用 20260422-1 规则），不使用 `TimelineEvent` 失败态（仅 `edit` 等其他失败路径使用）
