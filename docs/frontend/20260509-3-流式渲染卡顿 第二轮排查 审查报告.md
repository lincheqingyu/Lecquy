## 第一轮结论的复核

第一轮的 5 条结论（`MessageItem` 缺 memo / `StreamdownMarkdown` 流式重解析 / 思考计时器 / rAF 合并不够 / 没虚拟化）**全部成立**，源码定位与第一轮一致，不再赘述。

但在第一轮指出的"组件层渲染量"之外，还有**渲染管线层 / 副作用层**的结构性问题没被点到——这一轮主要补这一块。卡顿不只是"组件多了"，而是每个 token 顺手触发了一连串"强制同步布局 / 反复构建观察者 / 反复字符串化"的操作。

## 第二轮新发现（按对帧率影响排序）

### 1. `MessageList` 流式期间存在双重布局回弹（**最大新增瓶颈**）

**位置** `frontend/src/components/chat/MessageList.tsx:152-179`

每次 token 到达：

1. `useChat` 通过 `setMessages` 产生新的 `messages` 数组引用
2. `MessageList` 对 `contentRef` 装的 `ResizeObserver`（行 152-170）会因为 DOM 高度变化触发 `maybeStickToBottom()`，里面同步执行 `el.scrollTop = el.scrollHeight`
3. 与此同时 `useEffect`（行 174-179）以 `[messages, isStreaming, isWaiting, isPinnedToBottom]` 为依赖再调一次 `scrollToBottom('auto')`

也就是**每个 token 产生 2 次"读 scrollHeight + 写 scrollTop"**。读写交替本身就会让浏览器同步跑布局（forced reflow）；把这件事放到 60fps 节奏里执行就是典型的"layout thrashing"。

雪上加霜：`maskImage`（行 205-206）让滚动容器进入 GPU 合成层 + 软合成路径，每次滚动位置变化都要重做 mask 合成；`chat-scroll-mask` 的渐变 mask 也会让浏览器倾向于重画整个容器。

**为什么在第一轮没被点到**：第一轮把矛头都指向"组件重渲染"，但 React commit 之后的"布局/合成回弹"才是用户感知到"瞬间加载/抖一下"的直接原因。即便把 `MessageItem` `memo` 全部加上，只要 `messages` 引用每帧变化，这两条 effect 仍然每帧触发布局同步。

**影响**：严重。**先解决这个，再讨论 memo / 虚拟化**——否则 memo 的收益会被 reflow 吞掉。

**建议方向**（不在本文档拍板）：

- 方案 A：放弃 Effect 3 兜底（行 174-179），只保留 `ResizeObserver`，并把 `scrollTop = scrollHeight` 包进 `requestAnimationFrame`，把"读 scrollHeight"和"写 scrollTop"分到不同帧避开 forced reflow。
- 方案 B：用 CSS `overflow-anchor`（现代浏览器都支持）+ 一个底部哨兵元素，让浏览器自己处理"内容增长时贴底"，前端不再手动 set scrollTop。能彻底退出滚动写回的热路径。
- 方案 C：保留双重监听但**节流**——`ResizeObserver` 内部用 `rAF` 去重，且只有"距底 < 阈值"才执行。

### 2. `createBlocksSignature` 每次 token 触发 `JSON.stringify` 全量序列化

**位置** `frontend/src/components/chat/MessageItem.tsx:310-326`、`frontend/src/lib/chat-stream-debug.ts:89-94`

```ts
useEffect(() => {
  if (!isAssistant || (message.blocks?.length ?? 0) === 0) return
  const signature = createBlocksSignature(message.blocks)  // ← 每帧全量 stringify
  if (renderDebugSignatureRef.current === signature) return
  renderDebugSignatureRef.current = signature
  logChatStream(...)
}, [isAssistant, message.blocks, message.content, message.id, message.stepId, message.stepStatus, message.thinkingContent])
```

`createBlocksSignature` 内部依次：

1. `summarizeBlocks(blocks)` —— 全量 `map` blocks
2. `summarizeGroups(blocks)` —— 调用 `groupMessageBlocks` 全量分组（第一轮 P0 收益里也有它）
3. `JSON.stringify(...)` —— 序列化整棵结构

**而且早退判断在序列化之后**——即使最终签名相同也已经付完代价。再叠加"`MessageItem` 没 memo → 每条历史消息都过一遍这条 effect"，长对话下每个 token 在调试日志这条死路里至少是 50 次 `JSON.stringify`。

**影响**：严重，且**就算 `localStorage` 里没开 `lecquy.debug.chatStream` 也照样跑**——因为 `logChatStream` 内部的开关只决定"是否 console.debug"，不决定"是否计算 signature"。

**建议方向**：

- 把 `createBlocksSignature` 调用包进 `if (isChatStreamDebugEnabled()) { ... }`，调试关掉时这条 effect 完全不跑。
- 或者把签名换成 O(blocks.length) 的轻量统计（块数 + 末块长度 + 末块 status）即可识别变化。

### 3. `useChat` 的 `logAssistantSnapshot` 在每个 step_delta 里同样做无谓计算

**位置** `frontend/src/hooks/useChat.ts:387-402`、`520-553`

```ts
const logAssistantSnapshot = useCallback((scope, meta, message) => {
  logChatStream(scope, {
    ...meta,
    blocks: summarizeBlocks(message.blocks),   // ← 立即求值，非懒
    groups: summarizeGroups(message.blocks),   // ← 立即求值，非懒
  })
}, [])
```

JS 参数是急切求值，`summarizeBlocks` / `summarizeGroups` **在 debug 关闭时也照算**。每 token 至少调用 1 次，与上一条叠加即"前端每个 token 至少把 blocks 数组扫 4 遍只为打日志"。

**建议方向**：调用点先检查 `isChatStreamDebugEnabled()`，或者把 `logAssistantSnapshot` 改成接受 `() => payload` 的懒求值闭包。

### 4. `StreamdownMarkdown` 每个实例独立挂 `MutationObserver` 监听根元素

**位置** `frontend/src/components/chat/StreamdownMarkdown.tsx:138-159`

```ts
function useDocumentDarkMode() {
  const [isDark, setIsDark] = useState(detectDarkMode)
  useEffect(() => {
    ...
    const observer = new MutationObserver(syncDarkMode)
    observer.observe(document.documentElement, { attributes: true, attributeFilter: ['class'] })
    return () => observer.disconnect()
  }, [])
  return isDark
}
```

每条 assistant 消息可以包含**多个** `StreamdownMarkdown`（外层正文 + thinking 内层 + 每个 plan task），整个对话很容易达到几十个观察者**全部监听同一个 `<html>` 节点**。在切换主题之外，这本身不会持续触发回调，但：

- 观察者注册/取消有恒定开销
- 一旦根 class 真的变化，N 个观察者依次回调 N 次，每个回调都 `setState` → N 个 `StreamdownMarkdown` 同时重渲染

**建议方向**：把 dark-mode 提到 `App.tsx` 一个全局 context / Zustand atom，所有 `StreamdownMarkdown` 订阅同一个值。**这条本身不是"流式卡顿主因"，但一旦做了 P0 / P1 修复，它会变成相对的次大头**，顺手清掉。

### 5. `StreamdownMarkdown` 的 hooks 调用顺序违反 React Rules of Hooks（潜在 bug）

**位置** `frontend/src/components/chat/StreamdownMarkdown.tsx:168-176`

```ts
export function StreamdownMarkdown({ content, className, isAnimating = false }) {
  if (!content.trim()) return null   // ← 先 early return
  const isDark = useDocumentDarkMode()  // ← 后调用 hook
  ...
}
```

`useDocumentDarkMode`、内层的 `useState/useEffect` 都在 early return **之后**。同一个组件实例如果经历"内容空 → 内容非空"的切换（流式开始时常发生），React 会因"渲染了比上次更多/更少的 hooks"抛错。

**目前看起来没炸**的原因：所有调用点（`MessageItem.tsx:651-653`、`746-749` 等）外层都在拿到非空内容时才挂载 `StreamdownMarkdown`，即同一个实例在生命周期内 `content.trim()` 一直非空。但这是依赖调用方的偶然性约束，不是组件自身保证。

**建议方向**：把 early return 移到 hooks 之后，或在外层用 `<>{content.trim() ? <StreamdownMarkdown .../> : null}</>` 收紧契约。优先级低于性能问题但容易顺手改掉。

### 6. `MessageList` 的 `lastAssistantId` 每个 token 全量回扫

**位置** `frontend/src/components/chat/MessageList.tsx:190-195`

```ts
const lastAssistantId = useMemo(() => {
  for (let i = messages.length - 1; i >= 0; i--) {
    if (messages[i].role === 'assistant') return messages[i].id
  }
  return null
}, [messages])
```

每个 token 都让 `messages` 引用变化 → useMemo 重算 → 从后往前线性扫描。运气好遇到末尾就是 assistant 还行，**但如果 plan 模式下末尾是 plan event，且后面 assistant 在中间**，就要扫到中间。

更严重的问题：当前代码里这个 ID 只用来注释里说的"品牌标识"，而那个 UI 已经被注释掉了（`MessageItem.tsx:786-797`）。**功能根本没启用，但每帧仍在算**。

**建议方向**：直接删掉 `lastAssistantId` 计算（连同 `isLastAssistant` prop），等品牌标识真的复活再加回来。一行删除收益不大但白白省。

### 7. Streamdown `lineNumbers={true}` 的"逐行块级类名"在流式下被反复重建

**位置** `frontend/src/components/chat/StreamdownMarkdown.tsx:188-205`

```tsx
<Streamdown
  mode={mode}
  isAnimating={isAnimating}
  ...
  lineNumbers   // ← 每次 content 变化要给每一行重新挂 block 类名
  ...
>
```

注释（行 195-196）说明 `lineNumbers=true` 是为了"给每一行挂 block 级类名"以便 `index.css` 隐藏行号 + 命中样式。代价是 streamdown 每次 content 变更要给整段输出每行都生成 className。本地模型每秒几百 token 时，这个量级会显著。

**建议方向**：

- 选项 A：去掉 `lineNumbers` 依赖，改用更轻的样式钩子（例如 `data-streamdown="paragraph"` 之类原生 selector）。需要先复测当前 `index.css` 里依赖 lineNumbers 的样式还能不能命中。
- 选项 B：保留 `lineNumbers`，但流式期间（`isAnimating === true`）跳过——等 step 完成后再开。trade-off 是流式期间样式钩子失效，得评估视觉影响。

### 8. `enqueueStepDelta` 合并策略只看相邻一条

**位置** `frontend/src/hooks/useChat.ts:569-589`

```ts
const lastIndex = pendingStepDeltasRef.current.length - 1
const last = pendingStepDeltasRef.current[lastIndex]
if (last && last.stepId === delta.stepId && last.kind === delta.kind && (last.stream ?? 'text') === stream) {
  pendingStepDeltasRef.current[lastIndex] = { ...delta, content: last.content + delta.content }
} else {
  pendingStepDeltasRef.current.push(delta)
}
```

只跟队尾一条比较。在"thinking 与 text 交替输出"或"多 step 并发"场景下，**队列里会保留多条独立 delta**，一帧 flush 时仍然多次 `setMessages`（每条 delta 一次 `applyStepDelta`）。React 18 在 rAF 内**会**自动批处理 setState，所以渲染是合并的；但每个 `updateMessage` 内部的 `messages.map(...)` 仍然会跑 N 次。

**建议方向**：在 `flushPendingStepDeltas` 里按 `(stepId, kind, stream)` 做最终聚合，每个 stream key 只调一次 `updateMessage`。或者在 `setMessages` 的 reducer 里一次性处理整个 deltas 数组。

第一轮的 P1 建议（throttle / 截断）是从"输出节奏"角度切；这里是从"内部合并粒度"切，两者不冲突，互补。

## 渲染热路径全景（第二轮整理）

把第一轮 + 第二轮的现象按"一次 token 到达"展开：

```
WS step_delta 到达
  ↓
enqueueStepDelta (合并仅队尾)                          ← 第二轮第 8 条
  ↓ rAF 触发
flushPendingStepDeltas
  ↓ 队列里有 N 条 → setMessages × N
applyStepDelta
  └→ updateMessage: messages.map(50 entries)            ← 每条 delta 一次
  └→ summarizeBlocks/Groups 立即求值（即便 debug 关）   ← 第二轮第 3 条
  ↓
React commit
  ↓ 50 个 MessageItem 重渲染（无 memo）                 ← 第一轮第 1 条
  ├→ groupMessageBlocks × 50                            ← 第一轮第 1 条
  ├→ blocksToText / blocksToThinkingText × 50           ← 第一轮第 1 条
  ├→ createBlocksSignature × 50（含 JSON.stringify）    ← 第二轮第 2 条
  ├→ StreamdownMarkdown 重解析 markdown                  ← 第一轮第 2 条
  │   ├→ useDocumentDarkMode 各自挂 MutationObserver     ← 第二轮第 4 条
  │   └→ lineNumbers 重算每行类名                        ← 第二轮第 7 条
  ↓
DOM 提交，scrollHeight 改变
  ↓
ResizeObserver 触发 scrollTop = scrollHeight (强制 reflow) ← 第二轮第 1 条
+ Effect 3 触发 scrollTop = scrollHeight (再次)            ← 第二轮第 1 条
  ↓
mask-image 合成层重绘
```

第一轮主要在中段（"组件渲染"那一段）开刀；第二轮指出**头尾两端**还有持续耗能。

## 关键判断（写给作者，留待决策）

1. **如果只能动 1 处**——动第二轮 §1（`MessageList` 的双重布局回弹）。它最便宜（删一个 effect + rAF 包一下），并且解释了"卡顿/瞬间加载"里"卡顿"那一面（持续 layout thrashing 让 frame budget 漏出去）。
2. **如果还能动 1 处**——上 `React.memo`（第一轮 §1）。它解释了"瞬间加载一大段"那一面（commit 阶段 cost 高 → 帧间集中渲染）。
3. **debug 日志的清理（第二轮 §2、§3）属于"零风险净收益"**，不需要专门排期，做上面任何 P0 时顺手收掉即可。
4. **虚拟化 / `lineNumbers` / dark-mode hoisting** 这三件影响都不是当前主因。**先做 P0，复测，再决定要不要继续**——很可能做完 P0 后体感已经够好，不必再投。
5. CLAUDE.md 第 2 节明确"frontend 不主动维护——只修阻塞性 bug"。所以**这份报告里只有 P0 / P1 是"阻塞性"层级**（直接影响"agent 用着不流畅"），P2 / P3 不应自发动手。
