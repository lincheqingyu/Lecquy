# 代码文件头双语摘要补齐

> 更新日期：2026-05-13
> 类型：执行指令
> 关联：[CLAUDE.md](../../CLAUDE.md) / [AGENTS.md](../../AGENTS.md)

## 1. 背景

`CLAUDE.md` / `AGENTS.md` 已新增规则：项目中所有代码文件开头必须有简洁的中英双语文件级注释，说明该文件负责什么、位于什么链路、主要被谁调用或影响谁，便于 Codex / Claude Code 快速定位代码。

本指令用于后续交给 GPT-5.3 执行全仓代码文件头注释补齐。执行者只做注释补齐，不做业务逻辑、架构、格式化或命名调整。

## 2. 执行目标

为仓库内所有项目自有代码文件补齐文件头双语摘要：

```ts
// 中文：本文件负责 ...
// English: This file ...
```

要求：

- 每个摘要 1-3 行，短、准、可定位。
- 中文和英文表达同一件事，不需要逐字翻译，但语义必须一致。
- 说明文件职责、所在链路、关键调用关系；不要泛泛写"工具函数"、"组件文件"。
- 已有文件头注释时，补齐或改写为中英双语，不重复堆叠。
- 不解释实现细节，不写历史，不写需求背景，不写 TODO。

## 3. 范围

必须处理：

- `frontend/`、`backend/`、`shared/` 下的项目自有源码。
- 仓库根部和子目录中的项目自有脚本、配置代码、测试代码。
- 常见扩展名：`.ts`、`.tsx`、`.js`、`.jsx`、`.mjs`、`.cjs`、`.css`、`.scss`、`.html`、`.sh`、`.sql`。
- TypeScript declaration 文件 `.d.ts` 如果是项目手写文件，也要补。

必须跳过：

- `node_modules/`、`dist/`、`build/`、`coverage/`、`.turbo/`、`.vite/`、`.cache/`、`.lecquy/pg/` 等生成或依赖目录。
- lockfile：`pnpm-lock.yaml`、`package-lock.json`、`yarn.lock`。
- 纯数据文件：`.json`、`.jsonl`、`.yaml`、`.yml`、`.toml`，除非该文件本身明确承载可执行脚本逻辑。
- Markdown 文档、图片、字体、二进制文件、第三方 vendored 代码、自动生成文件。
- 文件顶部明确写着 generated / auto-generated / do not edit 的文件。

## 4. 注释格式

按文件语言使用合法注释：

```ts
// 中文：负责首页对话区布局，连接消息列表、输入框和右侧工作区。
// English: Owns the home conversation layout, wiring the message list, composer, and right rail.
```

```css
/* 中文：定义前端全局样式入口，接入 Tailwind 层和基础主题变量。
 * English: Defines the frontend global style entry, wiring Tailwind layers and base theme variables.
 */
```

```sh
#!/usr/bin/env bash
# 中文：启动本地后端开发服务，并传入仓库约定的环境变量。
# English: Starts the local backend dev server with the repository's expected environment variables.
```

```sql
-- 中文：创建记忆检索所需的 SQLite 表和 FTS 索引。
-- English: Creates the SQLite tables and FTS indexes required for memory retrieval.
```

特殊位置规则：

- `#!` shebang 必须保留第一行，摘要放在 shebang 后。
- 文件有 license / copyright banner 时，保留 banner 第一，摘要放在 banner 后。
- `"use client"`、`"use strict"` 等运行指令不得被破坏；如果不确定，保留现有指令位置，只在其后添加摘要。
- TSX / JSX 文件顶部仍使用 `//` 文件级注释，不要塞进 JSX 节点。

## 5. 执行步骤

1. 先读 `CLAUDE.md` 和 `AGENTS.md`，确认当前规则。
2. 用 `git status --short` 记录已有脏改，后续不要覆盖用户改动。
3. 用 `git ls-files` 生成候选文件清单，再按第 3 节过滤。
4. 分批阅读候选文件，每次根据 imports / exports / 主要函数或组件判断职责。
5. 为每个文件写入或更新文件头双语摘要。
6. 每批修改后用 `git diff --check` 检查空白错误。
7. 最后输出处理统计：已补齐数量、跳过数量、存在疑问需要人工确认的文件。

## 6. 强约束

- 不改业务逻辑。
- 不改 import / export 顺序。
- 不做格式化全仓跑动。
- 不重命名文件、目录、变量、函数、类型。
- 不修顺手看到的 bug。
- 不新增依赖。
- 不为了注释而改变运行语义。
- 不给生成文件、第三方代码和纯数据文件强行加注释。
- 遇到职责判断不清的文件，先写保守摘要；仍不确定则列入"需要人工确认"，不要臆造。

## 7. 验收标准

- 所有项目自有代码文件都有文件头中英双语摘要。
- 注释格式符合对应语言语法。
- `git diff --check` 通过。
- `pnpm build` 不因注释补齐产生新错误；如果本地环境无法跑 build，必须说明原因。
- 最终 diff 除文件头摘要外没有其他逻辑修改。
