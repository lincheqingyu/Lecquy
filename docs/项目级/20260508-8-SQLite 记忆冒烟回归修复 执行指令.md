**已确认成功的事**（不要破坏）：

- 写入触发正常
- `project_id` 推导正确（git remote 路径）
- LLM 提取在工作（importance=7 出现，非关键词降级）
- `summary` 抓住对话语义

**已确认的 bug**：

最后两条 `理解docx样式，告诉我有哪些问题` 完全重复——根因是每次 turn 完成时把最近 8 条消息整段送 LLM 提取，没有"水位机制"避免重复处理同一条消息。

---

## 二、根因分析（先看清问题再动手）

### 当前 `extractAndPersistOnTurnComplete`（在 `coordinator.ts`）的逻辑：

```ts
async function extractAndPersistOnTurnComplete(projection, manager, cwd) {
  const messageCount = countDurableCandidateMessages(manager, 0)   // ← 从 0 开始数，每次都是全量
  if (messageCount < EVENT_EXTRACTION_MESSAGE_THRESHOLD) return

  const input = buildEventExtractionInput(projection, manager)     // ← 取最近 8 条（含已处理过的）
  const items = (await extractEventMemoryItems(input)).map(...)
  insertSqliteMemoryItems(items)
}
```

每次 turn 完成都重跑一次"取最近 8 条丢给 LLM"——上一轮处理过的消息会被反复处理。

### PG 那一支怎么处理的：

`coordinator.ts` 第 70 行用 `getLatestTriggerEventSeq(pool, sessionId, 'extract_event')` 查"这个 session 上次提取到第几条 seq"，然后 `countDurableCandidateMessages(manager, latestTriggerEventSeq)` 只看新增的。

迁移到 SQLite 时**这个水位机制丢了**。

## 三、不要做什么

- **不要**改 `extraction-runner.ts` 内部的 LLM 提取逻辑——它是好的
- **不要**改 5 类 event_type 的分类——这是产品体验问题，留到第二周末单独调研
- **不要**触碰 PG 那一支代码（`db/memory-repository.ts` 的 `getLatestTriggerEventSeq` 等）—— 第三件才删
- **不要**改前端
- **不要**因为修复需要新增表就重设计 schema——只加最小必要的水位表

## 四、修复方案

### 4.1 在 `sqlite-store.ts` 新增水位表 + API

**新表**：

```sql
CREATE TABLE IF NOT EXISTS memory_extraction_watermark (
  session_id TEXT PRIMARY KEY,
  last_extracted_seq INTEGER NOT NULL,
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

放在 `initializeSchema` 函数里，与 `memory_items` 同批建表。

**新 API**（在 `sqlite-store.ts` 导出）：

```ts
export function getLastExtractedSeq(sessionId: string): number
export function setLastExtractedSeq(sessionId: string, seq: number): void
```

实现要点：

- `getLastExtractedSeq`：查不到 session_id 时返回 `0`（首次提取）
- `setLastExtractedSeq`：使用 `INSERT ... ON CONFLICT DO UPDATE` 一次写入
- 都用 prepared statement 缓存

### 4.2 改造 `extractAndPersistOnTurnComplete`

**目标行为**：

```
1. 读 lastSeq = getLastExtractedSeq(sessionId)
2. 数从 lastSeq 起的新增消息数
3. 不足阈值 → 不提取（不更新水位）
4. 够阈值 → 取从 lastSeq 起的最近 N 条（不要回头看 lastSeq 以前）
5. 调 LLM 提取
6. 写入提取结果到 SQLite（用现有 transaction）
7. 在 **同一个事务**里更新 lastSeq = 当前 manager.getEntries().length
```

**关键变更**：

- `countDurableCandidateMessages(manager, 0)` → `countDurableCandidateMessages(manager, lastSeq)`
- `buildEventExtractionInput` 增加参数 `fromSeq`，只取 seq > fromSeq 的消息
- `insertSqliteMemoryItems(items)` 包成事务，事务内 + `setLastExtractedSeq(sessionId, currentSeq)`

**伪代码**：

```ts
export async function extractAndPersistOnTurnComplete(projection, manager, cwd) {
  const sessionId = projection.sessionId
  const lastSeq = getLastExtractedSeq(sessionId)
  const currentSeq = manager.getEntries().length

  const newMessageCount = countDurableCandidateMessages(manager, lastSeq)
  if (newMessageCount < EVENT_EXTRACTION_MESSAGE_THRESHOLD) return

  const input = buildEventExtractionInput(projection, manager, lastSeq)
  if (input.messages.length < EVENT_EXTRACTION_MESSAGE_THRESHOLD) return

  const projectId = deriveProjectId(cwd)
  const items = (await extractEventMemoryItems(input)).map((item) => ({
    ...item, projectId,
  }))

  // ▼ 这里很重要：即使 items.length === 0（LLM 提取出 0 条），也要更新水位
  //   否则下次会再次重跑同样的对话窗口
  insertItemsAndAdvanceWatermark(items, sessionId, currentSeq)

  logger.info('SQLite event memory 已落库', {
    sessionKey: projection.key,
    sessionId,
    projectId,
    count: items.length,
    fromSeq: lastSeq,
    toSeq: currentSeq,
  })
}
```

**`insertItemsAndAdvanceWatermark` 是新的事务包装函数**，放在 `sqlite-store.ts`：

```ts
export function insertItemsAndAdvanceWatermark(
  items: readonly SQLiteMemoryItemInsert[],
  sessionId: string,
  newSeq: number,
): void {
  const database = getDb()
  const tx = database.transaction(() => {
    if (items.length > 0) {
      // 复用既有 insert 逻辑（不要再写一遍）
      // 提示：把当前 insertMemoryItems 的内部循环抽成 private 函数
      _insertItemsImpl(items)
    }
    _setWatermarkImpl(sessionId, newSeq)
  })
  tx()
}
```

注意：要重构现有 `insertMemoryItems` 把它的核心循环抽出来，避免事务嵌套（better-sqlite3 不支持嵌套事务）。

### 4.3 改造 `buildEventExtractionInput`

加 `fromSeq` 参数，让消息过滤时跳过已处理部分：

```ts
function buildEventExtractionInput(
  projection: SessionProjection,
  manager: SessionManager,
  fromSeq: number = 0,
): EventExtractionInput {
  const messages = manager.getEntries()
    .map((entry, index) => ({ entry, seq: index + 1 }))
    .filter(({ seq }) => seq > fromSeq)              // ← 新增过滤
    .filter(({ entry }) => /* 现有过滤逻辑保留 */)
    .slice(-EVENT_EXTRACTION_MAX_MESSAGES)
    .map(/* 现有 mapping 保留 */)

  return { sessionContext: ..., messages }
}
```

### 4.4 清理已经存在的重复脏数据

冒烟跑过的库里有重复条目，本次也要清掉。给 kira 一个**可手动执行**的 SQL（不要自动执行——让 kira 看清楚再决定）：

新建文件 `backend/src/memory/dedupe.sql`（或放在测试 fixture 里），内容：

```sql
-- 清理 memory_items 中按 (kind, event_type, project_id, summary, content) 完全重复的条目
-- 保留每组中 rowid 最小的那一条
-- 执行前先备份：cp .lecquy/memory/memory.db .lecquy/memory/memory.db.bak

BEGIN;

DELETE FROM memory_items
WHERE rowid NOT IN (
  SELECT MIN(rowid)
  FROM memory_items
  GROUP BY
    kind,
    COALESCE(event_type, ''),
    COALESCE(project_id, ''),
    summary,
    content
);

-- 重建 FTS（保险起见）
INSERT INTO memory_items_fts(memory_items_fts) VALUES('rebuild');

COMMIT;
```

在执行报告里告诉 kira：执行前先 `cp memory.db memory.db.bak`，再 `sqlite3 .lecquy/memory/memory.db < backend/src/memory/dedupe.sql`。

### 4.5 测试

新增 / 扩展测试覆盖：

1. **`getLastExtractedSeq` 默认值**：未写入过的 session 返回 0
2. **`setLastExtractedSeq` 幂等**：同 session 多次写入只保留最后一次
3. **重复提取的回归测试**：模拟 manager 有 10 条消息，第一次 extract 后水位 = 10；第二次新增 2 条，buildInput 只看到 2 条新的
4. **事务原子性**：mock LLM 提取抛错 → 水位**不应该**被更新

FROM memory_items
GROUP BY kind, COALESCE(event_type, ''), COALESCE(project_id, ''), summary, content
