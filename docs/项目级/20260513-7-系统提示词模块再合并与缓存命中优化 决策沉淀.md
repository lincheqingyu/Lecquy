## 1. 修订口径与背景

### 1.1 为什么再写一份

`20260512-2` 在「梳理出 system prompt 应该关心的所有维度」这件事上是合格的——它把 SOUL / IDENTITY / USER / MEMORY / AGENTS / TOOLS / mode / safety / 工具纪律 / capability / 用户画像切片 / 偏好预算 / skill / memory recall / live turn 等概念全部列了出来，并为每个概念定了职责、预算和拆分条件。

但当这套结构准备落地到 prompt builder 代码时，暴露出两个工程上必须修正的问题：

第一，**缓存命中被破坏**。原文档把 7 层结构里 1-5 层全部放进 system 字段，其中混入了 `time.md`（每天变）、`capability block`（运行时生成）、`preference 解析诊断`（每次重算）等动态片段，导致整个 system 前缀的稳定性从「天级」掉到「会话级」甚至「单次启动级」。在 API 调用模式下（Lecquy 长期形态），prompt cache 命中率是边际成本的最大变量，比这件事更重要的工程指标几乎没有。

第二，**认知负荷过载**。1-5 层加起来 26 个子项，1-7 层共 36 个子项。每个子项都被赋予了名字、职责、可缓存性、预算和拆分条件，但作为一份给单人开发者使用的设计文档，**子项数量超过短期记忆容量（~7±2）就会反向变成负担**：开发者既记不住每项在做什么，也无法判断每项是真的在贡献信息还是凑数补完。

对标 Claude Code / Codex / opencode 三家在生产环境跑了大半年的 system prompt 工程实践，它们的共同选择不是「更细的模块拆分」，而是「**一份大模板 + 动态环境隔离 + cache breakpoint**」。本文把 Lecquy 校准到这个方向，并进一步采用"会话级 FrozenSystemSnapshot + `<system_prompt_update>` 增量覆盖"来保证单个会话内 system 字段不变。

### 1.2 取代关系

- `20260512-2` 标记为废弃，正文保留，文档顶部加 banner 指向本文。结论性的"6 层 / 7 层"叙述、26 子项的拼接表、cache boundary 在 `UserPreference` 之后的判断，全部以本文为准。
- `20260512-1` 审查指令本身仍有效，它对开源项目的分析结论是事实素材；但据此推导出的"26 子项最终架构"已被本文收敛。
- `20260512-3` 执行指令的 Phase 0（现状勘察）和 Phase 1（`.lecquy/` 文件骨架）仍可继续执行——这两个 Phase 涉及的文件本身没被本次决策削减。Phase 2（prompt builder 7 层重构）和 Phase 3（memory schema）必须基于本文重新出执行指令，旧指令不要直接跑。
- `CLAUDE.md §1.3` 与根 `AGENTS.md` 必须同步引用本文。Codex 入口读取 `AGENTS.md`，如果 `AGENTS.md` 仍停在 `20260512-2`，后续实现会被旧架构带偏。
- 2026-05-18 补丁口径：结合当前路线，Lecquy 第一优先级适配 **OpenAI Chat Completions-compatible** API（vLLM / llama.cpp / SGLang / Qwen Cloud / DeepSeek / Kimi / GLM 等国产或本地模型的最大公约数）。Anthropic Messages API 与 `cache_control` 是 provider adapter 专属能力，不作为当前主链路的通用假设。

## 2. 触发本次修订的两个工程问题

### 2.1 缓存命中被破坏的具体位置

在 `20260512-2` §6 的 36 个子项里，有 7 个子项的字节稳定性不达标，分布如下：

| 子项 | 类型 | 失效频率 | 影响范围 |
|------|------|----------|----------|
| 1.2 tooling + tool schema summary | 运行时生成 | 每次启动可能不同（序列化抖动） | system 前缀 |
| 1.6 workspace | 半静态 + 半动态 | session 临时目录每会话不同 | system 前缀 |
| 1.8 `time.md` | 实例值 | 每天 | system 前缀 |
| 3.1 capability block | 运行时生成 | 每次启动可能不同 | system 前缀 |
| 4.3 preference 解析诊断摘要 | 实例值 | 每次重算 | system 前缀 |
| 6.x MemoryRecall | 检索结果 | 每轮 | system 前缀（按原设计） |
| 7.x LiveTurn | 当轮输入 | 每轮 | system 前缀（按原设计） |

其中 1.8、4.3、6.x、7.x 是结构性问题：内容本质动态，不应该放在 system 前缀的位置。1.2、1.6、3.1 是工程性问题：内容本可稳定，但实现里没有做确定性序列化，导致字节抖动。

按最严格的显式 prompt cache 机制（以 Anthropic prefix caching 为代表），**只要 cache breakpoint 之前任意位置出现字节差异，从该位置起所有 token 都需要重新计算**。当前结构下，1.8 `time.md` 这一个子项就足以让整段稳定前缀每天清空缓存一次；6.x / 7.x 进入 system 字段则让前缀每轮都失效。即使在 OpenAI-compatible / vLLM / 国产模型 API 这类没有 Anthropic `cache_control` 的路径里，稳定前缀也仍然有价值：它让本地日志、payload diff、未来 provider adapter 和自动前缀缓存都能建立在同一套字节边界上。

### 2.2 认知负荷的具体表现

26 个子项对一个单人项目而言意味着：

- 每加一条规则要先在脑里走一遍"它属于 SOUL / IDENTITY / USER / AGENTS / TOOLS / safety / runtime / extra-instructions 哪一个"，分类成本高于内容本身的写作成本
- 1.6 `workspace.md`、1.7 `documentation.md`、1.10 `extra-instructions.md`、1.11 `AGENTS.md` 的语义高度重叠（"项目稳定规则"），但拆成 4 个文件后实际维护时不知道改哪个
- 3.2 `SOUL.md` 和 3.3 `IDENTITY.md` 在「人格 vs 使命」边界上存在持续争议，每次编辑都需要重新判断归属
- 4.1 `preference slice`、4.2 截断结果、4.3 诊断摘要本质是同一份内容的三个处理阶段，拆成三个子项后让阅读者误以为是三种不同信息

这些问题不是「文档写得不好」，而是「拆分粒度本身和单人项目的承载力错配」。同样的拆分粒度在多人团队 + 平台化产品里是合理的（每个模块有专人维护），在单人 + 个人 Agent 场景里就是负担。

## 3. 与对标项目的实际差距

### 3.1 Claude Code

Claude Code 的 system prompt 在概念上是**一份连续长文档**，按英文段落小节区分 identity / tone / refusal / tools / artifacts / skills / env，整体约 4000–6000 tokens。运行时只动态生成两块：

- 工具 schema 摘要（在 system 字段固定位置）
- 末位 `<env>` 块（在 system 字段末尾）

`<env>` 块的形式相对固定：

```text
<env>
Working directory: /Users/.../project
Platform: darwin
Today's date: 2026-05-13
Model: claude-opus-4-7
</env>
```

cache breakpoint 卡在 `<env>` 块之前，前面所有内容在支持显式缓存的 provider 上可以拿更长 TTL；`<env>` 块本身约 100-200 tokens，每天 / 每会话重算可接受。项目级 `CLAUDE.md` / `AGENTS.md` 内容追加在 system 末尾但在 `<env>` 之前，归入稳定前缀。对 OpenAI-compatible 主路径而言，这一边界先作为 provider-agnostic 的逻辑切分，不要求立刻发出 Anthropic 风格的 `cache_control`。

### 3.2 Codex CLI

Codex CLI 的 system prompt 比 Claude Code 更简短，整体约 2000–3000 tokens：identity + 工具使用纪律 + 输出风格 + AGENTS.md + env-like 块。文件层面就 2–3 个模板，没有 SOUL / IDENTITY / USER 这种人格化拆分——OpenAI 的产品定位决定它不做"个人化"，但模板结构对 Lecquy 仍有参考价值。

### 3.3 opencode / Open Claw

`sst/opencode` 走 base prompt + provider-specific patches + project AGENTS.md + env 路线，文件 2–3 个。

`Open Claw` 按 markdown 文件顺序拼接，简化到极致，没有 cache breakpoint 设计，也没有 env 块隔离——这种做法在小项目里能跑，规模上去后会同时撞上 cache 失效和模板冲突两个问题。

### 3.4 三家共同模式

去除产品差异后，三家在结构上的共同选择是：

1. **物理文件 ≤ 5 个**——多了维护不动
2. **一份"基底大模板"承载 60% 内容**——identity / style / tools-discipline / safety / runtime 全在里面
3. **项目级文件 1–2 个**——AGENTS.md（+ 可选 TOOLS.md）作为可被用户编辑的项目层
4. **末位 `<env>` 块**——所有实例化值集中在这里
5. **单 cache breakpoint**——卡在 `<env>` 之前

本文保留"动态环境隔离 + 单稳定断点"这个工程目标,但不照搬"每轮 `<env>` 进 system 字段"的物理形态。Lecquy 进一步要求单个会话内 `system` 字段不变,所以当前日期、核心文件变更摘要等实例化值改由 `<system_prompt_update>` 追加到 messages 尾部,并在 compact / resnapshot 时吸收进新的 FrozenSystemSnapshot。

Lecquy 当前结构和这个共同模式的偏差点：

- 文件数：Lecquy 计划 10 个常驻 + system-prompt 子目录 ≥ 14 个模板 vs 共同模式 ≤ 5
- 基底模板：Lecquy 拆成 1.3 / 1.4 / 1.7 / 1.9 / 1.10 五个独立子项 vs 共同模式 1 个大模板
- 实例化值：Lecquy 散落在 1.2 / 1.6 / 1.8 / 3.1 / 4.3 五处 vs 共同模式集中在 1 个末位 `<env>` 块
- cache 策略：Lecquy 把 boundary 放在 `UserPreference` 之后（4.3 之后）但前面已经被 1.8 污染 vs 共同模式 boundary 放在 `<env>` 块之前并清场

## 4. 最终结论

### 4.1 概念维度从 26 收敛到 6

Lecquy system prompt 的真正概念维度只有：

| 维度 | 回答的问题 | 当前散落在哪些子项 |
|------|-----------|--------------------|
| Agent 基底 | 你是谁、怎么说话、不能做什么、什么场合怎么做 | 1.1 / 1.3 / 1.4 / 1.7 / 1.9 / 1.10 |
| 工具能力 | 你能用什么工具、怎么用、出错怎么办 | 1.2 / 1.12 / 2.1 / 2.3 |
| 项目约束 | 当前项目里硬规则是什么 | 1.11 |
| 用户画像 | 用户是谁、长期偏好是什么、长期记忆里有什么 | 3.2 / 3.3 / 3.4 / 3.5 / 4.1 |
| 技能上下文 | 当前任务命中什么 skill | 5.1 / 5.2 / 5.3 |
| 实例化值 | 当前是什么时间 / 哪个目录 / 哪个会话 / 哪个模型 | 1.6 / 1.8 / 1.5 部分 / 3.1 |

mode（2.x）、capability block（3.1）、preference 截断 / 诊断（4.2 / 4.3）、memory recall（6.x）、live turn（7.x）不构成独立维度：mode 是「Agent 基底」的一个 if 分支；capability 是「实例化值」的一种；4.2 / 4.3 是处理阶段不是内容；memory recall 和 live turn 是 messages 字段内容，不属于 system 字段维度。

### 4.2 物理文件从 26 项收敛到 ~8 个

```text
.lecquy/
├── BASE.md                    # 维度①Agent 基底（identity + style + safety + docs + runtime + extra）
├── TOOLS.md                   # 维度②工具能力（工具纪律 + 调用风格，不含 inventory）
├── AGENTS.md                  # 维度③项目约束
├── SOUL.md                    # 维度④用户画像 - 人格底色
├── IDENTITY.md                # 维度④用户画像 - 使命与边界
├── USER.md                    # 维度④用户画像 - 稳定画像与长期偏好（不切片）
├── MEMORY.md                  # 人工可读 memory 入口和 fallback
├── MEMORY.summary.md          # 维度④用户画像 - 长期记忆 frozen snapshot
├── memory/
│   └── memory.db              # 长期记忆事实来源
└── skills/
    └── <skill-name>/
        └── SKILL.md           # 维度⑤技能上下文，按需加载
```

工具 schema、skills index、`<system_prompt_update>` 块在运行时生成，不落文件。

相比 `20260512-2` 的取舍变化：

- 删除 `.lecquy/system-prompt/` 整个目录——15 个模板文件合并为 `BASE.md` 一份
- `SOUL.md` 与 `IDENTITY.md` 保留双文件——这是少数从 `20260512-2` 沿用的拆分，因为它们的人格 vs 使命边界长期审计意义大于合并节省的认知成本
- `USER.md` 不强制做 profile / preference 切片——切片由 prompt builder 在注入时按 heading 解析，不要求用户在写作时区分
- `MEMORY.md` 与 `MEMORY.summary.md` 双文件保留——人工入口 vs 系统维护的 snapshot 职责区分清晰

### 4.3 system 字段与 messages 字段的物理边界

这是本文最关键的工程边界，必须明确写进 prompt builder 代码注释和测试断言。

**核心约束**：每个会话创建时生成一份 `FrozenSystemSnapshot`，之后在下一次 compact / resnapshot 之前，`system` 字段字节**绝不改变**。SOUL / IDENTITY / USER.md / MEMORY.summary.md 等可变文件的会话内更新，不回写当前 `system` 字段，而是作为增量块追加到下一次用户问题之前。这样可以让 `system snapshot + 历史 API replay transcript` 最大限度保持 prefix cache 命中，同时让用户体感上“修改后下一轮立即生效”。

```text
system 字段（FrozenSystemSnapshot；会话创建 / compact resnapshot 时生成，两次 resnapshot 之间字节不可变）：
  [BASE.md]
  [TOOLS.md]
  [AGENTS.md]
  [SOUL.md]
  [IDENTITY.md]
  [USER.md]
  [MEMORY.summary.md]
  [skills index]                ← 运行时生成，字典序
  [matched SKILL.md] (可选)     ← 仅限 snapshot 生成时已激活的 skill
  <cache breakpoint>            ← 主断点

messages 字段（API replay transcript，不等同于用户可见 transcript）：
  [...历史 API replay 轮次...]
  [<system_prompt_update> 块]（可选；上轮之后的核心文件变化 / 当前日期 / 会话内覆盖）
  [当前轮 API user message]
    └─ <retrieved_memory> 块（本轮召回结果）
    └─ 用户实际输入
```

历史 API replay 轮次中,**每一轮当时注入的 `<retrieved_memory>` 块必须原封保留在该轮的 user message 内**——这是 prompt cache prefix-match 的硬约束。如果在 API replay 记录里剥掉 memory 注入,下一轮发送时 messages 字段在该轮位置和上次发送字面不同,cache 立刻失效。用户可见 transcript 仍然保存干净的 kira 实际输入,两者不能混用。

`<system_prompt_update>` 是 Lecquy 托管的 synthetic user/context block,不是用户真实发言,也不是 tool result。它只能覆盖 frozen snapshot 中明确授权的可变片段（SOUL / IDENTITY / USER.md / MEMORY.summary.md / 当前日期等实例化事实）,不能覆盖 BASE / TOOLS / AGENTS / safety / 工具权限 / 用户当前明确请求。底层 provider 如果不接受连续 user message,adapter 可以在发送前把它与当前 user message 合并为同一条 user role 内容,但 API replay transcript 必须保存当次发送给模型的完整字节。

### 4.4 cache 策略

本文里的 cache breakpoint 首先是 **Lecquy 内部的稳定字节边界**，不是要求所有 provider 都支持同一种显式缓存字段。当前主路线是 OpenAI Chat Completions-compatible API：vLLM / llama.cpp / SGLang / Qwen Cloud / DeepSeek / Kimi / GLM 等都优先走这一类 adapter。该路径默认不发 Anthropic `cache_control`，只保证 prompt builder 产出的稳定前缀字节不抖动。

- **OpenAI-compatible 主路径**：输出会话级稳定 system 字符串；当前日期、模型、核心文件增量等变化进入 messages 尾部的 `<system_prompt_update>`；不注入 `cache_control`；如 provider 有自动 prefix cache 或自定义缓存参数，由 provider adapter 单独处理。
- **Anthropic adapter 路径**：把 FrozenSystemSnapshot 拆成 content blocks,在稳定 block 末尾设置 `cache_control: { type: "ephemeral", ttl: "1h" }`。`cache_control` 不得被写进普通 prompt 字符串。
- **次 cache breakpoint** 暂不作为 P0。未来如果给 skill 段加第二断点，必须遵守 Anthropic TTL 顺序约束：长 TTL 断点在前，短 TTL 断点在后，不能出现“5 分钟断点在 1 小时断点之前”的结构。
- **messages 字段 cache** 只在确认 provider 支持、且历史 replay 字节稳定后再开启。它依赖 §6 的 MemoryRecall 持久化边界先落稳。

因此当前落地只强制一个逻辑主断点：FrozenSystemSnapshot 结束处。是否把这个逻辑断点翻译成 provider 真实字段，由各 provider adapter 决定。

## 5. FrozenSystemSnapshot 最终拼接顺序

| 顺序 | 内容 | 来源 | 缓存性 | 失效频率 |
|------|------|------|--------|----------|
| 1 | Agent 基底 | `.lecquy/BASE.md`(含 mode 子段) | 静态 | 极低,主动 commit 后 |
| 2 | 工具纪律 | `.lecquy/TOOLS.md` | 静态 | 极低 |
| 3 | 工具 inventory(schema 摘要) | 运行时生成 | 半静态 | 工具集变化时,需确定性序列化 |
| 4 | 项目约束 | `.lecquy/AGENTS.md` | 静态 | kira 编辑时 |
| 5 | 人格底色 | `.lecquy/SOUL.md` | 静态 | kira 编辑时 |
| 6 | 使命与边界 | `.lecquy/IDENTITY.md` | 静态 | kira 编辑时 |
| 7 | 稳定画像与偏好 | `.lecquy/USER.md` | 静态 | kira 编辑时 |
| 8 | 长期记忆 snapshot | `.lecquy/MEMORY.summary.md` | 静态 | 晋升流程发生时 |
| 9 | skills index | 运行时生成,字典序 | 半静态 | skill 安装 / 卸载时 |
| 10 | 命中的 SKILL.md(可选) | snapshot 生成时已激活的 skill | 半静态 | 切换 skill 时走 `<system_prompt_update>` 或新会话 |
| `<bp1>` | 逻辑主断点；Anthropic adapter 才翻译为 `cache_control` | | | |

以上拼接结果在会话创建时冻结。后续如果 kira 修改 SOUL / IDENTITY / USER.md / MEMORY.summary.md,或当前日期、模型、附件等实例化事实变化,不重建当前 system 字段,而是在下一次用户问题前追加 `<system_prompt_update>`。只有 compact / resnapshot 或新会话才重新生成新的 FrozenSystemSnapshot。

**`<system_prompt_update>` 块结构(固定):**

```text
<system_prompt_update priority="high" source="lecquy">
Current date: <YYYY-MM-DD>
Time zone: <IANA tz>
Changed files since snapshot:
- SOUL.md: <cumulative stable diff summary>
- USER.md: <cumulative stable diff summary>
Active skill update: <skill name / version / summary>
</system_prompt_update>
```

注入内容仅限"会话内可变 snapshot 的累积变化"和"当前调用实例化事实"。所有字段均为可选:只有相对 FrozenSystemSnapshot 已经发生变化、且本轮需要模型感知的字段才输出;未变字段不写 `none` / `null` / 空行。它不放工具权限、不放安全规则、不放完整文件正文；文件变化先做稳定 diff 摘要,必要时才追加短片段。块名固定为 `<system_prompt_update>`，闭合标签同名,不要再使用 `<runtime_context_update>`。

**Mode 处理:**

`20260512-2` 把 mode 当成一个独立层(2.x),本文把它降级为 `BASE.md` 内部的 if 分支 + `<system_prompt_update>` 里的当前 mode 标志位。理由:

- simple / manager / worker 的差异 ≤ 300 tokens
- 三个模板 90% 内容相同,拆三个文件后维护时容易漂移
- worker 的 minimal prompt 策略可以通过 prompt builder 的 `mode=worker` 参数关掉一部分注入,不需要独立文件

未来真的出现 ≥ 3 个稳定可路由 agent、每个 agent 的 prompt 差异 ≥ 500 tokens 时,再考虑拆 `BASE.<mode>.md`。

## 6. messages 字段约定

### 6.1 MemoryRecall 的注入位置与形态

MemoryRecall 必须区分三份数据,不能混成一份:

- **用户可见消息(user-visible transcript)**:只保存 kira 实际输入,用于 UI 展示、搜索、用户导出和人工审查。
- **API replay 消息(api transcript)**:发送给模型和后续重放 cache prefix 时使用,可在同一条 user role 内容前部附加 `<retrieved_memory>`。
- **augmentation 记录**:保存本轮 memory recall 的来源、排序、hash 和是否注入,用于调试与审计。

发送给模型时,每一轮 user role 内容由两段拼成,**在同一条 user 消息内**,不要拆成两条连续 user:

```text
<retrieved_memory priority="low">
[memory item 1 - kind=decision, project=Lecquy, importance=0.8]
...
[memory item N - kind=fact, project=global, importance=0.6]
</retrieved_memory>

{用户实际输入}
```

注意事项:

- Anthropic API 在 messages 数组里只允许 user / assistant 两种 role,memory recall 必须挂在 user role 内；OpenAI-compatible API 虽然允许 `system` / `tool` 等 role,但为了跨 provider 一致,也按同一条 user role 内容处理。
- 包裹 tag 固定为 `<retrieved_memory priority="low">`(英文小写,属性顺序固定),让模型能稳定识别为"低权限检索结果"
- 内容部分按项目维度 + 重要性 + 时间衰减排序后再 topK 截断,不要每次输出顺序抖动
- 不允许把 memory 块塞进 system 字段——这一条是本次决策的红线
- 不允许把 `<retrieved_memory>` 直接写进用户可见 transcript；如果当前存储 schema 只有一份 message,先补 hidden api transcript / augmentation entry,不要用污染用户历史的方式换 cache。

### 6.2 历史轮次的保留规则

prompt cache 命中要求"本次发送给模型的 API transcript 是上次发送 transcript 的 prefix extension"。这意味着:

- **写入 API replay 历史的 user message 必须是当时发给 API 的完整字节**,包含 `<retrieved_memory>` 块
- 用户可见 transcript 仍然只保存 kira 实际输入；前端展示、会话标题、用户导出、memory extraction 默认读取干净 transcript,避免把系统注入误认为 kira 发言
- 不允许在 API replay 持久化时剥离 memory 块求"历史干净"——这是看似合理但会摧毁所有历史轮次 cache 的反优化
- 历史 API replay message 的 memory 块**只读**,不再被新一轮 memory 召回的结果覆盖

如果 session 持续到 memory 块累积导致 token 压力,处理方式是触发 compact(见 6.5),不是修改历史 message。

### 6.3 SystemPromptUpdate 的注入位置与形态

`<system_prompt_update>` 用来解决"当前会话 system snapshot 不变"和"用户修改核心文件后希望下一轮立即生效"之间的冲突。它是会话内增量覆盖块,不是新的 system prompt,也不是用户真实发言。

本文采用 **cumulative since snapshot** 路线。每个 `<system_prompt_update>` 块的内容是"相对 FrozenSystemSnapshot 创建时点的累积变化",不是"相对上一次 update 的增量"。最新一个 update 块 + FrozenSystemSnapshot 就必须能还原当前等价状态,不要求模型回看历史 update 并在上下文里 fold 多条 delta。本文明确不采用 delta-since-previous 路线,理由见 §8.4。

触发条件:

- SOUL.md / IDENTITY.md / USER.md / MEMORY.summary.md 在当前 FrozenSystemSnapshot 生成后发生变化
- 当前日期 / 时区 / 模型 / mode / 附件状态需要以当轮事实覆盖 snapshot 创建时的旧事实
- 当前会话内激活或切换 skill,但不希望重建 system 字段

无变化不 emit。一次普通用户提问如果没有跨日、核心文件变化、mode / model / attachment 等实例化事实变化,就不追加 `<system_prompt_update>`。

发送给模型时,`<system_prompt_update>` 放在当前 user message 之前。默认实现为独立 synthetic user context message;如果某个 provider 不接受连续 user role,adapter 可以在发送前把它并入当前 user message 的最前部。无论采用哪种物理形态,API replay transcript 必须记录**最终发送给模型的完整字节**,用户可见 transcript 仍然不显示该块。

```text
<system_prompt_update priority="high" source="lecquy">
Current date: 2026-05-18
Time zone: Asia/Shanghai
Changed files since snapshot:
- USER.md: 新增偏好"文档日期必须使用当前真实日期,不得沿用会话创建日期"。
</system_prompt_update>
```

优先级规则:

- 它只覆盖 frozen snapshot 中明确授权为可变的内容:SOUL / IDENTITY / USER.md / MEMORY.summary.md / 当前日期等实例化事实。
- 它不能覆盖 BASE / TOOLS / AGENTS / safety / 工具权限 / 用户当前明确请求。
- FrozenSystemSnapshot 的 BASE.md 必须预先声明:如果后续 messages 中出现 `<system_prompt_update>`,模型应把它视为 Lecquy 托管的高优先级会话内更新,但作用域仅限可变 snapshot。
- 更新摘要必须稳定、短小、按固定顺序排序。不要把完整文件正文反复塞进该块;大改动等 compact / resnapshot 吸收。
- 文件 diff 必须是相对 FrozenSystemSnapshot 的累积差异。同一个文件在第 5 轮新增 X、第 10 轮新增 Y,第 10 轮的 update 应显示"新增 X 和 Y",而不是只显示"新增 Y"。

### 6.4 LiveTurn 内容的去层级化

`20260512-2` 7.x 列出的 6 个 LiveTurn 子项(current user message / route / session / attachments / options / tool results / todo state / worker outputs / 临时 extra instruction)在工程上不是 prompt 内容,是**调用上下文**:

- current user message → 直接是 user message 内容
- tool results / worker outputs → 通过 tool_use / tool_result 块自然落在 messages 数组里,不需要单独拼接
- route / session / attachments / options / mode → 作为当轮实例化事实进入 `<system_prompt_update>` 或请求 metadata
- todo state → 由 todo 工具结果落在 messages 数组里
- 临时 extra instruction → 由用户当轮在 user message 里说明,不进 system

去层级化的好处:这些内容回到它们本来的物理位置,prompt builder 不需要为它们做"拼接到 LiveTurn 层"的特殊处理,代码量直接减少。

### 6.5 compact 与 memory / system_prompt_update 的关系

compact 触发时,历史轮次会被摘要重写,并允许重新生成新的 FrozenSystemSnapshot。摘要规则约束:

- 摘要 prompt 内部必须识别 `<retrieved_memory>` tag,把它视为"系统注入,非 kira 发言",不要把召回内容写成"kira 说"
- 摘要 prompt 内部必须识别 `<system_prompt_update>` tag,把它视为"Lecquy 会话内增量覆盖",不要把它写成用户真实发言
- 历史轮次的 memory 块在摘要中**可以丢弃**——这些事实本来就在 memory.db 里,历史副本只是为了 cache prefix,摘要后无需保留
- 历史轮次的 system_prompt_update 块在摘要中**可以吸收进新的 FrozenSystemSnapshot 或 compact summary**,吸收后旧 update 块无需继续注入
- 这条规则等于"compact 同时也是历史 memory / update bloat 的清理机会",不是额外成本
- compact 产出的用户可见摘要不得包含 `<retrieved_memory>` 原文；API replay 摘要可以保留"曾注入过记忆"这一元信息,但不能把 recall 内容当成用户陈述。

## 7. `<system_prompt_update>` 块设计要点

### 7.1 内容清单

允许进 `<system_prompt_update>` 的内容,必须满足"会话内可变 snapshot 增量"或"调用实例化事实"定义——一旦改变,只影响当前轮和后续轮,不应该改写已冻结的 system 字段:

- 工作目录、session id、mode 选择、当前模型 id、附件计数
- 当前日期、时区
- SOUL.md / IDENTITY.md / USER.md / MEMORY.summary.md 相对 FrozenSystemSnapshot 的累积差异摘要
- 当前会话内激活 skill 的名称、版本或短摘要

不允许进 `<system_prompt_update>` 的内容:

- BASE / TOOLS / AGENTS / safety / 工具权限等不可变规则
- 完整 SOUL / IDENTITY / USER.md / MEMORY.summary.md 文件正文
- 记忆召回(进 messages 字段)
- 用户真实输入(保持在当前 user message 主体)

### 7.2 序列化要求

`<system_prompt_update>` 块位于 messages 尾部,本身不属于稳定 system prefix,但它会进入 API replay transcript。为了让下一轮仍然是 prefix extension,它必须字节稳定:

- 字段顺序固定(按本文 §5 顺序)
- 未变字段不输出占位;不要写 `none` / `null` / 空行
- 日期格式 ISO 8601(`YYYY-MM-DD`),时区 IANA
- 数字字段不带千分位
- changed files 按文件名固定顺序输出:`SOUL.md` → `IDENTITY.md` → `USER.md` → `MEMORY.summary.md`
- diff 摘要使用稳定句式;同一文件内容和同一旧 snapshot 输入两次,输出必须字节相同
- 文件 diff 是 cumulative since snapshot,不是 delta-since-previous。同一 snapshot 下相同最终状态必须产出相同 block 字节,不依赖中间修改顺序

### 7.3 与用户体感的对齐

用户修改核心文件后,当前会话不需要重开即可在下一轮体感生效:

- UI 可以提示"当前会话已通过增量上下文生效;compact 或新会话后会进入新的 system snapshot"
- 普通偏好、称呼方式、日期规则等低风险更新走 `<system_prompt_update>` 立即生效
- safety、工具权限、AGENTS/TOOLS 等高风险更新不走增量覆盖;必须新会话或显式 resnapshot 后生效
- compact 时把增量块吸收到新的 FrozenSystemSnapshot,之后历史中的旧 update 块可以被摘要清理

## 8. 缓存策略与确定性要求

### 8.1 确定性序列化清单

下列内容必须由 prompt builder 用 stable serializer 输出,不依赖语言默认序列化:

| 内容 | 确定性要求 |
|------|-----------|
| 工具 schema 摘要 | 按工具名字典序,字段按字典序,JSON 缩进固定为 2 空格 |
| skills index | 按 skill 名字典序,description 取原始字节,不做 markdown 二次渲染 |
| MEMORY.summary.md | 系统维护时按 heading 顺序固定输出,人工编辑时不打散结构 |
| `<retrieved_memory>` 内容 | 按 (importance desc, recency desc, id asc) 三键排序后 topK 截断 |
| `<system_prompt_update>` 块 | 字段顺序固定(§7.2),changed files 按固定文件顺序输出,文件 diff 为 cumulative since snapshot |

必要时在 prompt builder 里加单元测试,断言"同样输入两次调用,字节相等"。

### 8.2 失效场景与代价

| 场景 | 失效范围 | 频率 | 处理 |
|------|---------|------|------|
| kira 编辑 USER.md / SOUL.md / IDENTITY.md | 当前会话仅新增 `<system_prompt_update>`；新会话 / compact 后重建 system snapshot | 周级 | 当前会话即时体感生效,不破坏历史前缀 cache |
| 安装 / 卸载 skill | 当前会话仅新增 `<system_prompt_update>`；新会话 / compact 后重建 skills index | 月级 | 可接受 |
| 切换命中的 SKILL.md | 当前会话仅新增 `<system_prompt_update>` 或要求新会话 | 任务级 | 不直接改当前 system 字段 |
| 跨日 | 仅新增 `<system_prompt_update>` | 天级 | FrozenSystemSnapshot 不变 |
| 切换 session / mode / 模型 | 当前会话仅新增 `<system_prompt_update>` 或请求 metadata | 调用级 | FrozenSystemSnapshot 不变 |
| memory.db 召回结果不同 | 仅当轮 user message | 每轮 | 由 messages 字段缓存策略保护历史轮次 |

### 8.3 实例化值进 system 还是 messages

本文最终选择:会话创建时的少量 snapshot metadata 可以进入 FrozenSystemSnapshot,但当前日期、模型、mode、附件计数、核心文件改动摘要等会话内变化进入 messages 末尾的 `<system_prompt_update>`,不再作为每轮动态 `<env>` 写进 system 字段。理由:

- 每个会话的 system 字段在生成后不改变,能最大化 `system snapshot + 历史 API replay transcript` 的 prefix cache 命中。
- 用户修改 SOUL / IDENTITY / USER.md 后,下一轮通过 `<system_prompt_update>` 立即生效,不需要重开会话。
- FrozenSystemSnapshot 的 BASE.md 预先授权 `<system_prompt_update>` 覆盖可变 snapshot,因此它不是普通低权限用户文本,但作用域仍被限制在可变内容。
- tool role 不适合承载这类更新:OpenAI-compatible 和 Anthropic 都要求 tool result 对应前序 tool call,synthetic user/context block 跨 provider 更稳。

### 8.4 为什么选 cumulative 不选 delta

`<system_prompt_update>` 有两条可能路线:

- **cumulative since snapshot**:每个 update 块展示相对 FrozenSystemSnapshot 创建时点的当前完整累积差异。最新 update 块单独可读,不依赖历史 update。
- **delta-since-previous**:每个 update 块只展示相对上一个 update 的变化。模型必须顺序读取多条历史 update,在上下文里 fold 出当前状态。

本文选择 cumulative since snapshot,明确不采用 delta-since-previous。理由:

- 用户体感上需要的是"当前状态",不是"修改历史"。最新 update 块必须能让模型直接看到当前 USER / SOUL / IDENTITY 等价状态。
- 长上下文存在中间段注意力衰减。对 cumulative 路线,旧 update 进入中间后被淡化是好事,因为末尾最新 update 已覆盖它;对 delta 路线,任一中间 delta 被弱化都会造成当前状态重建错误。
- Lecquy 主路径偏 OpenAI-compatible / vLLM / Qwen / DeepSeek / GLM 等模型,不应把可靠性押在"模型正确折叠多条历史 delta"这种 reconstructive reasoning 上。
- token 代价可控。`<system_prompt_update>` 只在真实变化时 emit,且未变字段省略;相比 memory recall、工具结果和长对话历史,它不是主要上下文消耗来源。

## 9. 模块消融方法论

### 9.1 判断一个模块是否真的有效的两个手段

无先验方法,只有以下两个事后手段:

**手段 1:消融实验**

把某个模块从 system 字段拿掉,跑 10 个典型任务(覆盖 simple / manager / worker、覆盖 frontend / backend / 文档场景),对比 baseline 输出。如果 agent 行为无明显降级(工具调用顺序合理 / 不出红线行为 / 不漏关键规则),该模块永久删除或合并到上位模块。

**手段 2:token 占比 + 触发频率**

- token 占比 < 2%:候选删除
- 日常对话里规则被触发频率 < 1%:候选删除
- 同时满足两条:必须删除或合并

### 9.2 第一批建议消融的候选

按本文新结构,下列原 `20260512-2` 子项建议在落地时直接消融或合并,不需要单独工程支持:

3. **FrozenSystemSnapshot 工程化**:会话创建时生成 system snapshot,之后直到 compact / resnapshot 前不得修改 system 字段。当前日期、模型、核心文件变化不再进动态 system。
4. **`<system_prompt_update>` 工程化**:在当前用户问题前追加会话内 update 块,路线固定为 cumulative since snapshot,未变字段不输出占位,并明确它只覆盖可变 snapshot,不能覆盖 BASE / TOOLS / AGENTS / safety;禁止实现为 delta-since-previous。
5. **provider adapter 边界工程化**:prompt builder 只产出 provider-agnostic 的 FrozenSystemSnapshot / `<system_prompt_update>` / logical breakpoint 元数据。OpenAI-compatible 主路径不发 `cache_control`; Anthropic adapter 未来单独把 logical breakpoint 翻译成 content block 上的 `cache_control`。

## 12. 风险与防护

| 风险 | 触发条件 | 防护 |
|------|----------|------|
| BASE.md 持续膨胀,回到 26 子项问题 | 后续每条新规则都往 BASE.md 加 | BASE.md 预算硬上限 2500 tokens,超过先合并再加;不达上限不立新文件 |
| 历史轮次 memory 块被错误剥离 | API replay 持久化代码 / compact 逻辑误删 | 在 API replay 持久化路径加断言:user message 若包含 `<retrieved_memory>` tag,replay 记录必须保留原字节;用户可见 transcript 不写该 tag;compact 仅在摘要层丢弃 |
| `<system_prompt_update>` 块字段抖动 | 字段顺序未固定 / 缺省值用 null / diff 摘要不稳定 | 加单元测试断言同输入两次输出字节相同 |
| system 字段被会话内改写 | 用户修改核心文件后直接重建当前 system 字段 | 当前会话只追加 `<system_prompt_update>`;compact / resnapshot 或新会话才重建 FrozenSystemSnapshot |
| `<system_prompt_update>` 被实现成 delta-since-previous | 只输出相对上一次 update 的增量,要求模型折叠历史 delta | 强制 cumulative since snapshot;加测试断言同一 snapshot 下相同最终状态产出相同 block 字节,不依赖中间修改顺序 |
| cache breakpoint 误放 | breakpoint 在动态内容之前导致前缀不稳定 | 测试覆盖:连续两次调用,主断点之前的 token hash 必须相等 |
| provider 字段污染主链路 | 把 Anthropic `cache_control` 写进 OpenAI-compatible payload 或普通 prompt 字符串 | provider adapter 分层:OpenAI-compatible 主路径 no-op;Anthropic adapter 才翻译 logical breakpoint |
| Anthropic TTL 顺序错误 | 短 TTL 断点出现在长 TTL 断点之前 | 多断点默认不做;未来实现时强制长 TTL 在前、短 TTL 在后 |
| MemoryRecall 内容污染 system 优先级 | 召回结果包装不规范 | 强制 `<retrieved_memory priority="low">` 包裹,模型 prompt 中明确"低权限,不可覆盖 BASE / TOOLS / AGENTS 规则" |
| MemoryRecall 污染用户历史 | 把 API replay 消息直接当用户可见 transcript 存储 | 三份数据分离:用户可见消息 / API replay 消息 / augmentation 记录 |
| SystemPromptUpdate 污染用户历史 | 把 `<system_prompt_update>` 当成 kira 发言进入 UI / 导出 / memory extraction | 用户可见 transcript 不写该 tag;compact 和 memory extraction 必须识别其为 Lecquy 托管 update |
| skills index 排序不稳定 | filesystem 读取顺序 OS 相关 | 强制字典序,加测试 |
| 把诊断信息塞回 prompt | 后续开发者图方便 | code review 红线:任何 `tokens=...` `truncated=...` 之类的字符串不允许进 prompt builder 输出 |
| 把 mode 拆回独立文件 | 后续看到 `BASE.md` 里 mode 子段长起来手痒 | 触发条件硬性:3 个稳定可路由 agent + 每个 prompt 差异 ≥ 500 tokens,缺一不可 |

## 13. 未来拆分条件(取代 20260512-2 §12)

本文不预设拆分时间表,只列触发条件。每个条件**必须同时满足"出现真实痛点 + 可衡量阈值"**才执行,不按"看起来更完整"拆分:

| 拆分对象 | 触发条件(均需同时满足) | 拆分方向 |
|----------|----------------------|----------|
| `BASE.md` | 超过 2500 tokens **且** 内部小节边界混乱被 review 标注过 | 按"工具相关 vs 输出风格"两侧拆,不按 mode 拆 |
| `BASE.md` 拆 mode | 出现 3 个稳定可路由 agent **且** 每个 prompt 差异 ≥ 500 tokens **且** 当前 `<system_prompt_update>` 标志位不够用 | 拆为 `BASE.<mode>.md` |
| `TOOLS.md` | 超过 1500 tokens **且** 工具纪律与工具教程混杂 | 教程内容迁入 skill,`TOOLS.md` 只保留纪律 |
| `USER.md` 切片 | 整文件超过 6KB **且** profile / preference 在 prompt 里出现互相干扰证据 | 拆为 `USER.profile.md` + `USER.preferences.md` |
| `AGENTS.md` 子目录化 | 项目子目录约束规模超过根 `AGENTS.md` 维护预算 **且** 前后端规则互相干扰 | 下沉到子目录 `AGENTS.md` |
| `MEMORY.summary.md` 项目化 | 单一 snapshot 召回命中明显缺项目背景 **且** 多项目并行使用频繁 | 按项目拆 `MEMORY.<project>.summary.md` |
| 引入 agent registry | 出现 ≥ 3 个稳定 agent **且** 不同 agent 工具权限不同 **且** 自动路由需求出现 | 引入 `agents/<id>/role.md` + 轻量 registry |

---

最终原则不变:**Lecquy 的 system prompt 工程只服务一个目标——让单人 Agent 的长期执行力越用越懂 kira**。文件数要少、职责要硬、动态内容交给 messages 字段、复杂流程交给 skill、项目约束交给 `.lecquy/AGENTS.md` 和子目录 `AGENTS.md`。在这个目标之外的"完整性""规整性""可扩展性",**默认不做**。
