## 1. 修订口径与背景

### 1.1 为什么再写一份

`20260512-2` 在「梳理出 system prompt 应该关心的所有维度」这件事上是合格的——它把 SOUL / IDENTITY / USER / MEMORY / AGENTS / TOOLS / mode / safety / 工具纪律 / capability / 用户画像切片 / 偏好预算 / skill / memory recall / live turn 等概念全部列了出来，并为每个概念定了职责、预算和拆分条件。

但当这套结构准备落地到 prompt builder 代码时，暴露出两个工程上必须修正的问题：

第一，**缓存命中被破坏**。原文档把 7 层结构里 1-5 层全部放进 system 字段，其中混入了 `time.md`（每天变）、`capability block`（运行时生成）、`preference 解析诊断`（每次重算）等动态片段，导致整个 system 前缀的稳定性从「天级」掉到「会话级」甚至「单次启动级」。在 API 调用模式下（Lecquy 长期形态），prompt cache 命中率是边际成本的最大变量，比这件事更重要的工程指标几乎没有。

第二，**认知负荷过载**。1-5 层加起来 26 个子项，1-7 层共 36 个子项。每个子项都被赋予了名字、职责、可缓存性、预算和拆分条件，但作为一份给单人开发者使用的设计文档，**子项数量超过短期记忆容量（~7±2）就会反向变成负担**：开发者既记不住每项在做什么，也无法判断每项是真的在贡献信息还是凑数补完。

对标 Claude Code / Codex / opencode 三家在生产环境跑了大半年的 system prompt 工程实践，它们的共同选择不是「更细的模块拆分」，而是「**一份大模板 + 末位 `<env>` 块 + cache breakpoint**」。这是本文要把 Lecquy 校准到的方向。

### 1.2 取代关系

- `20260512-2` 标记为废弃，正文保留，文档顶部加 banner 指向本文。结论性的"6 层 / 7 层"叙述、26 子项的拼接表、cache boundary 在 `UserPreference` 之后的判断，全部以本文为准。
- `20260512-1` 审查指令本身仍有效，它对开源项目的分析结论是事实素材；但据此推导出的"26 子项最终架构"已被本文收敛。
- `20260512-3` 执行指令的 Phase 0（现状勘察）和 Phase 1（`.lecquy/` 文件骨架）仍可继续执行——这两个 Phase 涉及的文件本身没被本次决策削减。Phase 2（prompt builder 7 层重构）和 Phase 3（memory schema）必须基于本文重新出执行指令，旧指令不要直接跑。
- `CLAUDE.md §1.3` 与根 `AGENTS.md` 必须同步引用本文。Codex 入口读取 `AGENTS.md`，如果 `AGENTS.md` 仍停在 `20260512-2`，后续实现会被旧架构带偏。
- 2026-05-18 补丁口径：结合当前路线，Lecquy 第一优先级适配 **OpenAI Chat Completions-compatible** API（vLLM / llama.cpp / SGLang / Qwen Cloud / DeepSeek / Kimi / GLM 等国产或本地模型的最大公约数）。Anthropic Messages API 与 `cache_control` 是 provider adapter 专属能力，不作为当前主链路的通用假设。

## 2. 触发本次修订的两个工程问题

### 2.1 缓存命中被破坏的具体位置

在 `20260512-2` §6 的 36 个子项里，有 7 个子项的字节稳定性不达标，分布如下：

| 子项 | 类型 | 失效频率 | 影响范围 |
|------|------|----------|----------|
| 1.2 tooling + tool schema summary | 运行时生成 | 每次启动可能不同（序列化抖动） | system 前缀 |
| 1.6 workspace | 半静态 + 半动态 | session 临时目录每会话不同 | system 前缀 |
| 1.8 `time.md` | 实例值 | 每天 | system 前缀 |
| 3.1 capability block | 运行时生成 | 每次启动可能不同 | system 前缀 |
| 4.3 preference 解析诊断摘要 | 实例值 | 每次重算 | system 前缀 |
| 6.x MemoryRecall | 检索结果 | 每轮 | system 前缀（按原设计） |
| 7.x LiveTurn | 当轮输入 | 每轮 | system 前缀（按原设计） |

其中 1.8、4.3、6.x、7.x 是结构性问题：内容本质动态，不应该放在 system 前缀的位置。1.2、1.6、3.1 是工程性问题：内容本可稳定，但实现里没有做确定性序列化，导致字节抖动。

按最严格的显式 prompt cache 机制（以 Anthropic prefix caching 为代表），**只要 cache breakpoint 之前任意位置出现字节差异，从该位置起所有 token 都需要重新计算**。当前结构下，1.8 `time.md` 这一个子项就足以让整段稳定前缀每天清空缓存一次；6.x / 7.x 进入 system 字段则让前缀每轮都失效。即使在 OpenAI-compatible / vLLM / 国产模型 API 这类没有 Anthropic `cache_control` 的路径里，稳定前缀也仍然有价值：它让本地日志、payload diff、未来 provider adapter 和自动前缀缓存都能建立在同一套字节边界上。

### 2.2 认知负荷的具体表现

26 个子项对一个单人项目而言意味着：

- 每加一条规则要先在脑里走一遍"它属于 SOUL / IDENTITY / USER / AGENTS / TOOLS / safety / runtime / extra-instructions 哪一个"，分类成本高于内容本身的写作成本
- 1.6 `workspace.md`、1.7 `documentation.md`、1.10 `extra-instructions.md`、1.11 `AGENTS.md` 的语义高度重叠（"项目稳定规则"），但拆成 4 个文件后实际维护时不知道改哪个
- 3.2 `SOUL.md` 和 3.3 `IDENTITY.md` 在「人格 vs 使命」边界上存在持续争议，每次编辑都需要重新判断归属
- 4.1 `preference slice`、4.2 截断结果、4.3 诊断摘要本质是同一份内容的三个处理阶段，拆成三个子项后让阅读者误以为是三种不同信息

这些问题不是「文档写得不好」，而是「拆分粒度本身和单人项目的承载力错配」。同样的拆分粒度在多人团队 + 平台化产品里是合理的（每个模块有专人维护），在单人 + 个人 Agent 场景里就是负担。

## 3. 与对标项目的实际差距

### 3.1 Claude Code

Claude Code 的 system prompt 在概念上是**一份连续长文档**，按英文段落小节区分 identity / tone / refusal / tools / artifacts / skills / env，整体约 4000–6000 tokens。运行时只动态生成两块：

- 工具 schema 摘要（在 system 字段固定位置）
- 末位 `<env>` 块（在 system 字段末尾）

`<env>` 块的形式相对固定：

```text
<env>
Working directory: /Users/.../project
Platform: darwin
Today's date: 2026-05-13
Model: claude-opus-4-7
</env>
```

cache breakpoint 卡在 `<env>` 块之前，前面所有内容在支持显式缓存的 provider 上可以拿更长 TTL；`<env>` 块本身约 100-200 tokens，每天 / 每会话重算可接受。项目级 `CLAUDE.md` / `AGENTS.md` 内容追加在 system 末尾但在 `<env>` 之前，归入稳定前缀。对 OpenAI-compatible 主路径而言，这一边界先作为 provider-agnostic 的逻辑切分，不要求立刻发出 Anthropic 风格的 `cache_control`。

### 3.2 Codex CLI

Codex CLI 的 system prompt 比 Claude Code 更简短，整体约 2000–3000 tokens：identity + 工具使用纪律 + 输出风格 + AGENTS.md + env-like 块。文件层面就 2–3 个模板，没有 SOUL / IDENTITY / USER 这种人格化拆分——OpenAI 的产品定位决定它不做"个人化"，但模板结构对 Lecquy 仍有参考价值。

### 3.3 opencode / Open Claw

`sst/opencode` 走 base prompt + provider-specific patches + project AGENTS.md + env 路线，文件 2–3 个。

`Open Claw` 按 markdown 文件顺序拼接，简化到极致，没有 cache breakpoint 设计，也没有 env 块隔离——这种做法在小项目里能跑，规模上去后会同时撞上 cache 失效和模板冲突两个问题。

### 3.4 三家共同模式

去除产品差异后，三家在结构上的共同选择是：

1. **物理文件 ≤ 5 个**——多了维护不动
2. **一份"基底大模板"承载 60% 内容**——identity / style / tools-discipline / safety / runtime 全在里面
3. **项目级文件 1–2 个**——AGENTS.md（+ 可选 TOOLS.md）作为可被用户编辑的项目层
4. **末位 `<env>` 块**——所有实例化值集中在这里
5. **单 cache breakpoint**——卡在 `<env>` 之前

Lecquy 当前结构和这个共同模式的偏差点：

- 文件数：Lecquy 计划 10 个常驻 + system-prompt 子目录 ≥ 14 个模板 vs 共同模式 ≤ 5
- 基底模板：Lecquy 拆成 1.3 / 1.4 / 1.7 / 1.9 / 1.10 五个独立子项 vs 共同模式 1 个大模板
- 实例化值：Lecquy 散落在 1.2 / 1.6 / 1.8 / 3.1 / 4.3 五处 vs 共同模式集中在 1 个末位 `<env>` 块
- cache 策略：Lecquy 把 boundary 放在 `UserPreference` 之后（4.3 之后）但前面已经被 1.8 污染 vs 共同模式 boundary 放在 `<env>` 块之前并清场

## 4. 最终结论

### 4.1 概念维度从 26 收敛到 6

Lecquy system prompt 的真正概念维度只有：

| 维度 | 回答的问题 | 当前散落在哪些子项 |
|------|-----------|--------------------|
| Agent 基底 | 你是谁、怎么说话、不能做什么、什么场合怎么做 | 1.1 / 1.3 / 1.4 / 1.7 / 1.9 / 1.10 |
| 工具能力 | 你能用什么工具、怎么用、出错怎么办 | 1.2 / 1.12 / 2.1 / 2.3 |
| 项目约束 | 当前项目里硬规则是什么 | 1.11 |
| 用户画像 | 用户是谁、长期偏好是什么、长期记忆里有什么 | 3.2 / 3.3 / 3.4 / 3.5 / 4.1 |
| 技能上下文 | 当前任务命中什么 skill | 5.1 / 5.2 / 5.3 |
| 实例化值 | 当前是什么时间 / 哪个目录 / 哪个会话 / 哪个模型 | 1.6 / 1.8 / 1.5 部分 / 3.1 |

mode（2.x）、capability block（3.1）、preference 截断 / 诊断（4.2 / 4.3）、memory recall（6.x）、live turn（7.x）不构成独立维度：mode 是「Agent 基底」的一个 if 分支；capability 是「实例化值」的一种；4.2 / 4.3 是处理阶段不是内容；memory recall 和 live turn 是 messages 字段内容，不属于 system 字段维度。

### 4.2 物理文件从 26 项收敛到 ~8 个

```text
.lecquy/
├── BASE.md                    # 维度①Agent 基底（identity + style + safety + docs + runtime + extra）
├── TOOLS.md                   # 维度②工具能力（工具纪律 + 调用风格，不含 inventory）
├── AGENTS.md                  # 维度③项目约束
├── SOUL.md                    # 维度④用户画像 - 人格底色
├── IDENTITY.md                # 维度④用户画像 - 使命与边界
├── USER.md                    # 维度④用户画像 - 稳定画像与长期偏好（不切片）
├── MEMORY.md                  # 人工可读 memory 入口和 fallback
├── MEMORY.summary.md          # 维度④用户画像 - 长期记忆 frozen snapshot
├── memory/
│   └── memory.db              # 长期记忆事实来源
└── skills/
    └── <skill-name>/
        └── SKILL.md           # 维度⑤技能上下文，按需加载
```

工具 schema、skills index、`<env>` 块在运行时生成，不落文件。

相比 `20260512-2` 的取舍变化：

- 删除 `.lecquy/system-prompt/` 整个目录——15 个模板文件合并为 `BASE.md` 一份
- `SOUL.md` 与 `IDENTITY.md` 保留双文件——这是少数从 `20260512-2` 沿用的拆分，因为它们的人格 vs 使命边界长期审计意义大于合并节省的认知成本
- `USER.md` 不强制做 profile / preference 切片——切片由 prompt builder 在注入时按 heading 解析，不要求用户在写作时区分
- `MEMORY.md` 与 `MEMORY.summary.md` 双文件保留——人工入口 vs 系统维护的 snapshot 职责区分清晰

### 4.3 system 字段与 messages 字段的物理边界

这是本文最关键的工程边界，必须明确写进 prompt builder 代码注释和测试断言：

```text
system 字段（一次调用内字节恒定 + 跨调用尽量稳定）：
  [BASE.md]
  [TOOLS.md]
  [AGENTS.md]
  [SOUL.md]
  [IDENTITY.md]
  [USER.md]
  [MEMORY.summary.md]
  [skills index]                ← 运行时生成，字典序
  [matched SKILL.md] (可选)     ← 命中时加载
  <cache breakpoint>            ← 主断点
  [<env> 块]                    ← 运行时生成，末位

messages 字段（API replay transcript，不等同于用户可见 transcript）：
  [...历史 API replay 轮次...]
  [当前轮 API user message]
    └─ <retrieved_memory> 块（本轮召回结果）
    └─ 用户实际输入
```

历史 API replay 轮次中,**每一轮当时注入的 `<retrieved_memory>` 块必须原封保留在该轮的 user message 内**——这是 prompt cache prefix-match 的硬约束。如果在 API replay 记录里剥掉 memory 注入,下一轮发送时 messages 字段在该轮位置和上次发送字面不同,cache 立刻失效。用户可见 transcript 仍然保存干净的 kira 实际输入,两者不能混用。

### 4.4 cache 策略

本文里的 cache breakpoint 首先是 **Lecquy 内部的稳定字节边界**，不是要求所有 provider 都支持同一种显式缓存字段。当前主路线是 OpenAI Chat Completions-compatible API：vLLM / llama.cpp / SGLang / Qwen Cloud / DeepSeek / Kimi / GLM 等都优先走这一类 adapter。该路径默认不发 Anthropic `cache_control`，只保证 prompt builder 产出的稳定前缀字节不抖动。

- **OpenAI-compatible 主路径**：输出单条稳定 system 字符串 + 末位 `<env>` 字符串；不注入 `cache_control`；如 provider 有自动 prefix cache 或自定义缓存参数，由 provider adapter 单独处理。
- **Anthropic adapter 路径**：把 system 拆成 content blocks，在 `<env>` 之前的稳定 block 上设置 `cache_control: { type: "ephemeral", ttl: "1h" }`。`cache_control` 不得被写进普通 prompt 字符串。
- **次 cache breakpoint** 暂不作为 P0。未来如果给 skill 段加第二断点，必须遵守 Anthropic TTL 顺序约束：长 TTL 断点在前，短 TTL 断点在后，不能出现“5 分钟断点在 1 小时断点之前”的结构。
- **messages 字段 cache** 只在确认 provider 支持、且历史 replay 字节稳定后再开启。它依赖 §6 的 MemoryRecall 持久化边界先落稳。

因此当前落地只强制一个逻辑主断点：`BASE.md` → `MEMORY.summary.md` → skills index → (可选 SKILL.md) 结束，`<env>` 之前。是否把这个逻辑断点翻译成 provider 真实字段，由各 provider adapter 决定。

## 5. system 字段最终拼接顺序

| 顺序 | 内容 | 来源 | 缓存性 | 失效频率 |
|------|------|------|--------|----------|
| 1 | Agent 基底 | `.lecquy/BASE.md`(含 mode 子段) | 静态 | 极低,主动 commit 后 |
| 2 | 工具纪律 | `.lecquy/TOOLS.md` | 静态 | 极低 |
| 3 | 工具 inventory(schema 摘要) | 运行时生成 | 半静态 | 工具集变化时,需确定性序列化 |
| 4 | 项目约束 | `.lecquy/AGENTS.md` | 静态 | kira 编辑时 |
| 5 | 人格底色 | `.lecquy/SOUL.md` | 静态 | kira 编辑时 |
| 6 | 使命与边界 | `.lecquy/IDENTITY.md` | 静态 | kira 编辑时 |
| 7 | 稳定画像与偏好 | `.lecquy/USER.md` | 静态 | kira 编辑时 |
| 8 | 长期记忆 snapshot | `.lecquy/MEMORY.summary.md` | 静态 | 晋升流程发生时 |
| 9 | skills index | 运行时生成,字典序 | 半静态 | skill 安装 / 卸载时 |
| 10 | 命中的 SKILL.md(可选) | 运行时按需加载 | 半静态 | 切换 skill 时 |
| `<bp1>` | 逻辑主断点；Anthropic adapter 才翻译为 `cache_control` | | | |
| 11 | `<env>` 块 | 运行时生成 | 动态 | 每次调用可能不同 |

**`<env>` 块结构(固定):**

```text
<env>
Working directory: <cwd>
Session ID: <session-id>
Mode: <simple|manager|worker>
Today's date: <YYYY-MM-DD>
Time zone: <IANA tz>
Model: <model-id>
Attachments: <count or none>
</env>
```

注入内容仅限"当前调用的实例化事实",不放规则,不放偏好,不放检索结果。

**Mode 处理:**

`20260512-2` 把 mode 当成一个独立层(2.x),本文把它降级为 `BASE.md` 内部的 if 分支 + `<env>` 块里一个标志位。理由:

- simple / manager / worker 的差异 ≤ 300 tokens
- 三个模板 90% 内容相同,拆三个文件后维护时容易漂移
- worker 的 minimal prompt 策略可以通过 prompt builder 的 `mode=worker` 参数关掉一部分注入,不需要独立文件

未来真的出现 ≥ 3 个稳定可路由 agent、每个 agent 的 prompt 差异 ≥ 500 tokens 时,再考虑拆 `BASE.<mode>.md`。

## 6. messages 字段约定

### 6.1 MemoryRecall 的注入位置与形态

MemoryRecall 必须区分三份数据,不能混成一份:

- **用户可见消息(user-visible transcript)**:只保存 kira 实际输入,用于 UI 展示、搜索、用户导出和人工审查。
- **API replay 消息(api transcript)**:发送给模型和后续重放 cache prefix 时使用,可在同一条 user role 内容前部附加 `<retrieved_memory>`。
- **augmentation 记录**:保存本轮 memory recall 的来源、排序、hash 和是否注入,用于调试与审计。

发送给模型时,每一轮 user role 内容由两段拼成,**在同一条 user 消息内**,不要拆成两条连续 user:

```text
<retrieved_memory priority="low">
[memory item 1 - kind=decision, project=Lecquy, importance=0.8]
...
[memory item N - kind=fact, project=global, importance=0.6]
</retrieved_memory>

{用户实际输入}
```

注意事项:

- Anthropic API 在 messages 数组里只允许 user / assistant 两种 role,memory recall 必须挂在 user role 内；OpenAI-compatible API 虽然允许 `system` / `tool` 等 role,但为了跨 provider 一致,也按同一条 user role 内容处理。
- 包裹 tag 固定为 `<retrieved_memory priority="low">`(英文小写,属性顺序固定),让模型能稳定识别为"低权限检索结果"
- 内容部分按项目维度 + 重要性 + 时间衰减排序后再 topK 截断,不要每次输出顺序抖动
- 不允许把 memory 块塞进 system 字段——这一条是本次决策的红线
- 不允许把 `<retrieved_memory>` 直接写进用户可见 transcript；如果当前存储 schema 只有一份 message,先补 hidden api transcript / augmentation entry,不要用污染用户历史的方式换 cache。

### 6.2 历史轮次的保留规则

prompt cache 命中要求"本次发送给模型的 API transcript 是上次发送 transcript 的 prefix extension"。这意味着:

- **写入 API replay 历史的 user message 必须是当时发给 API 的完整字节**,包含 `<retrieved_memory>` 块
- 用户可见 transcript 仍然只保存 kira 实际输入；前端展示、会话标题、用户导出、memory extraction 默认读取干净 transcript,避免把系统注入误认为 kira 发言
- 不允许在 API replay 持久化时剥离 memory 块求"历史干净"——这是看似合理但会摧毁所有历史轮次 cache 的反优化
- 历史 API replay message 的 memory 块**只读**,不再被新一轮 memory 召回的结果覆盖

如果 session 持续到 memory 块累积导致 token 压力,处理方式是触发 compact(见 6.4),不是修改历史 message。

### 6.3 LiveTurn 内容的去层级化

`20260512-2` 7.x 列出的 6 个 LiveTurn 子项(current user message / route / session / attachments / options / tool results / todo state / worker outputs / 临时 extra instruction)在工程上不是 prompt 内容,是**调用上下文**:

- current user message → 直接是 user message 内容
- tool results / worker outputs → 通过 tool_use / tool_result 块自然落在 messages 数组里,不需要单独拼接
- route / session / attachments / options / mode → 进 `<env>` 块
- todo state → 由 todo 工具结果落在 messages 数组里
- 临时 extra instruction → 由用户当轮在 user message 里说明,不进 system

去层级化的好处:这些内容回到它们本来的物理位置,prompt builder 不需要为它们做"拼接到 LiveTurn 层"的特殊处理,代码量直接减少。

### 6.4 compact 与 memory 块的关系

compact 触发时,历史轮次会被摘要重写。摘要规则约束:

- 摘要 prompt 内部必须识别 `<retrieved_memory>` tag,把它视为"系统注入,非 kira 发言",不要把召回内容写成"kira 说"
- 历史轮次的 memory 块在摘要中**可以丢弃**——这些事实本来就在 memory.db 里,历史副本只是为了 cache prefix,摘要后无需保留
- 这条规则等于"compact 同时也是历史 memory bloat 的清理机会",不是额外成本
- compact 产出的用户可见摘要不得包含 `<retrieved_memory>` 原文；API replay 摘要可以保留"曾注入过记忆"这一元信息,但不能把 recall 内容当成用户陈述。

## 7. 末位 `<env>` 块设计要点

### 7.1 内容清单

允许进 `<env>` 的内容,必须满足"调用实例化值"定义——一旦改变,只影响当前调用,不应该影响 Lecquy 的规则:

- 工作目录、session id、mode 选择
- 当前日期、时区
- 当前模型 id、附件计数

不允许进 `<env>` 的内容:

- 用户偏好(进 USER.md)
- 工具纪律(进 TOOLS.md)
- 记忆召回(进 messages 字段)
- 任何"规则性"语句

### 7.2 序列化要求

`<env>` 块虽然不参与 cache 命中(它每次都是新的),但要求**形式稳定**,因为下一轮 user message 进入历史时,本轮的 `<env>` 不再被发送——只有 system 字段会重发,而 system 字段不包含历史 `<env>`。所以 `<env>` 不会污染历史,但**模型识别 env 的能力依赖形式稳定**,因此:

- 字段顺序固定(按本文 §5 顺序)
- 缺省值用 `none` 字面量,不要 `null` 或空行
- 日期格式 ISO 8601(`YYYY-MM-DD`),时区 IANA
- 数字字段不带千分位

### 7.3 与 Claude Code 的对齐

Lecquy `<env>` 块的形式刻意贴近 Claude Code,目的:

- 让 kira 在 Claude Code / Codex / Lecquy 之间切换时感受一致
- 便于将来把 Lecquy 的会话从其它 agent 迁过来时直接对齐结构
- 让 prompt builder 的实现可以参考 Claude Code 的已知工程经验

## 8. 缓存策略与确定性要求

### 8.1 确定性序列化清单

下列内容必须由 prompt builder 用 stable serializer 输出,不依赖语言默认序列化:

| 内容 | 确定性要求 |
|------|-----------|
| 工具 schema 摘要 | 按工具名字典序,字段按字典序,JSON 缩进固定为 2 空格 |
| skills index | 按 skill 名字典序,description 取原始字节,不做 markdown 二次渲染 |
| MEMORY.summary.md | 系统维护时按 heading 顺序固定输出,人工编辑时不打散结构 |
| `<retrieved_memory>` 内容 | 按 (importance desc, recency desc, id asc) 三键排序后 topK 截断 |
| `<env>` 块 | 字段顺序固定(§7.2) |

必要时在 prompt builder 里加单元测试,断言"同样输入两次调用,字节相等"。

### 8.2 失效场景与代价

| 场景 | 失效范围 | 频率 | 处理 |
|------|---------|------|------|
| kira 编辑 USER.md / SOUL.md / IDENTITY.md | system 字段从该文件起后续全部 | 周级 | 可接受,文件变更天然要重建 cache |
| 安装 / 卸载 skill | skills index 起后续 + `<env>` | 月级 | 可接受 |
| 切换命中的 SKILL.md | 该 SKILL 起后续 + `<env>` | 任务级 | 由次断点保护前缀 |
| 跨日 | 仅 `<env>` | 天级 | 主断点之前不受影响 |
| 切换 session / mode / 模型 | 仅 `<env>` | 调用级 | 主断点之前不受影响 |
| memory.db 召回结果不同 | 仅当轮 user message | 每轮 | 由 messages 字段缓存策略保护历史轮次 |

### 8.3 实例化值进 system 还是 messages

本文选择 `<env>` 块进 system 字段(对齐 Claude Code),不是放在 messages 字段第一条 user 之前。理由:

- system 字段长度限制宽松,`<env>` 约 100–200 tokens 不构成压力
- messages 字段保持纯净,只装对话和 memory 注入,工程心智简单
- 模型对 system 字段的服从优先级天然高于 messages,`<env>` 里的事实(如"今天是 X 月 Y 日")作为环境前提比作为"用户说的话"更合适

## 9. 模块消融方法论

### 9.1 判断一个模块是否真的有效的两个手段

无先验方法,只有以下两个事后手段:

**手段 1:消融实验**

把某个模块从 system 字段拿掉,跑 10 个典型任务(覆盖 simple / manager / worker、覆盖 frontend / backend / 文档场景),对比 baseline 输出。如果 agent 行为无明显降级(工具调用顺序合理 / 不出红线行为 / 不漏关键规则),该模块永久删除或合并到上位模块。

**手段 2:token 占比 + 触发频率**

- token 占比 < 2%:候选删除
- 日常对话里规则被触发频率 < 1%:候选删除
- 同时满足两条:必须删除或合并

### 9.2 第一批建议消融的候选

按本文新结构,下列原 `20260512-2` 子项建议在落地时直接消融或合并,不需要单独工程支持:

## 13. 未来拆分条件(取代 20260512-2 §12)

本文不预设拆分时间表,只列触发条件。每个条件**必须同时满足"出现真实痛点 + 可衡量阈值"**才执行,不按"看起来更完整"拆分:

| 拆分对象 | 触发条件(均需同时满足) | 拆分方向 |
|----------|----------------------|----------|
| `BASE.md` | 超过 2500 tokens **且** 内部小节边界混乱被 review 标注过 | 按"工具相关 vs 输出风格"两侧拆,不按 mode 拆 |
| `BASE.md` 拆 mode | 出现 3 个稳定可路由 agent **且** 每个 prompt 差异 ≥ 500 tokens **且** 当前 `<env>` 标志位不够用 | 拆为 `BASE.<mode>.md` |
| `TOOLS.md` | 超过 1500 tokens **且** 工具纪律与工具教程混杂 | 教程内容迁入 skill,`TOOLS.md` 只保留纪律 |
| `USER.md` 切片 | 整文件超过 6KB **且** profile / preference 在 prompt 里出现互相干扰证据 | 拆为 `USER.profile.md` + `USER.preferences.md` |
| `AGENTS.md` 子目录化 | 项目子目录约束规模超过根 `AGENTS.md` 维护预算 **且** 前后端规则互相干扰 | 下沉到子目录 `AGENTS.md` |
| `MEMORY.summary.md` 项目化 | 单一 snapshot 召回命中明显缺项目背景 **且** 多项目并行使用频繁 | 按项目拆 `MEMORY.<project>.summary.md` |
| 引入 agent registry | 出现 ≥ 3 个稳定 agent **且** 不同 agent 工具权限不同 **且** 自动路由需求出现 | 引入 `agents/<id>/role.md` + 轻量 registry |

最终原则不变:**Lecquy 的 system prompt 工程只服务一个目标——让单人 Agent 的长期执行力越用越懂 kira**。文件数要少、职责要硬、动态内容交给 messages 字段、复杂流程交给 skill、项目约束交给 `.lecquy/AGENTS.md` 和子目录 `AGENTS.md`。在这个目标之外的"完整性""规整性""可扩展性",**默认不做**。
