## 问题 1：记忆系统是不是真的？

### 事实

**写入内容与格式**：

记忆系统有两条路径，一条始终在跑，一条受 PG_ENABLED 开关控制（默认 false）。

**路径 A（始终在跑）— 轮次刷新写文件**：
- `flush.ts:44-70` 的 `recordMemoryTurnAndMaybeFlush` 在每轮 agent 调用结束后触发
- `flush.ts:21-42` 的 `buildFlushEntry` 只提取最后一条 user 消息 + 最后一条 assistant 消息的纯文本，拼成固定模板（Reason / User Summary / Assistant Summary / Candidate Durable Facts）
- `flush.ts:52-56` 的阈值由 `getMemoryConfig().flushTurns` 控制，达到阈值后计数器归零并写入
- `store.ts:67-73` 的 `appendDailyMemoryEntry` 以 ISO 时间戳为 section 头，追加写入 `.lecquy/memory/memory-YYYY-MM-DD.md`

**路径 B（PG_ENABLED=true 才生效）— 事件提取 + 向量召回**：
- `coordinator.ts:33-140` 的 `MemoryCoordinator` 在 turn 完成后检查新消息数，≥4 条就入队 `extract_event` job 到 PG
- `extraction-runner.ts:360-400` 先尝试 LLM 提取（最多 3 次重试），失败则降级到关键词规则启发式提取
- `prompt-injector.ts:89-127` 的 `buildMemoryRecallMessages` 优先走 PG 向量搜索（`searchEventMemories`），失败/空结果则回退到文件系统读取

**flush 触发条件**：
- `flush.ts:54-58`：每轮 `turnState.counter` 自增，达到 `cfg.flushTurns` 阈值后 flush 并重置计数器。这是**纯粹的轮次计数**，没有事件驱动、没有语义触发。

**记忆注入位置**：
- `prompt-injector.ts:81-87`：记忆被包装为 role='user' 的 AgentMessage，tag 为 `<LAYER:memory_recall>`，注入到 `contextMessages` 数组
- `agent-runner.ts:126`：`contextMessages` 通过 `agentLoop` 的第二个参数 `{ messages: contextMessages }` 注入，作为 steering messages

**长期检索能力判断**：
- 文件系统路径：只有按日期命名的 daily markdown 文件（`store.ts:27-29`），**没有索引、没有全文搜索、没有按项目维度分片**
- PG 路径：`types.ts:89-96` 定义了 `MemoryRecallQuery`（含 sessionId / sessionKey / userQuery / mode / route / limit），`prompt-injector.ts:95-106` 调用 PG 向量搜索，**有索引、有时间戳**（`occurredAt`），但**没有项目维度字段**
- MEMORY.md 摘要文件：`compact.ts:397-401` 在压缩完成后写入 `MEMORY.summary.md`，`store.ts:76-84` 读取时截断到 400 token 预算

### 判断

**半实现**。文件系统 flush（路径 A）在跑但产出的是原始对话 dump，不是结构化记忆。PG 事件提取+召回（路径 B）有完整代码但默认关闭，且实际写入的是 PG 表而非文件系统可读内容。两条路径之间没有桥接。

## 问题 2：一次请求的上下文是怎么拼出来的？

### 事实

**拼装顺序（完整跟踪）**：

```
[用户输入] → [7层系统提示组装] → [contextMessages（含记忆召回）] → [agentLoop 内部拼接] → [调用 LLM]
```

**第 1 步：系统提示组装**（`system-prompts.ts:262-291` + `prompt-layer-types.ts:7-14`）

7 层结构，按优先级从上到下：

| 层 | 内容来源 | 截断策略 |
|---|---|---|
| 1. System | `identity-{role}` 模板 + 工具列表 + skill 索引 + 工作区路径 + 文档路径 + 时区 | 无截断（由模板文件控制长度） |
| 2. Mode | `role-{role}` 模板（角色差异化指令） | 无截断 |
| 3. StartupContext | SOUL.md + IDENTITY.md + USER.md(profile段) + MEMORY.summary.md + Capability | `STARTUP_BUDGETS.startupTotal = 1500 tokens`，`context-files.ts:207-229` 从后往前逐段裁切 |
| 4. SkillRuntime | 命中 skill 的 SKILL.md 正文 | 无截断 |
| 5. UserPreference | USER.md(preference段) | `STARTUP_BUDGETS.userPreference = 200 tokens` |
| 6. MemoryRecall | PG 向量召回 或 MEMORY.md + 今日/昨日 daily memory | 无明确上限 |
| 7. LiveTurn | 当前轮用户消息 + 工具调用结果 | 无截断 |

**第 2 步：contextMessages**（记忆召回注入点）

- `agent-runner.ts:75` 接收 `contextMessages` 参数
- `agent-runner.ts:126` 传给 `agentLoop(messages, { systemPrompt, messages: contextMessages, tools })`
- 记忆召回通过 `prompt-injector.ts:89-127` 生成 `AgentMessage[]`，包装为 `role: 'user'` + `<LAYER:memory_recall>` 标签

**第 3 步：项目级上下文注入**

- `system-prompts.ts:179-196` 的 `buildProjectContextSection` 读取 `.lecquy/` 下的 SOUL.md / IDENTITY.md / USER.md / AGENTS.md / TOOLS.md / MEMORY.md
- `system-prompts.ts:109-133` 的 `buildDocumentationSection` 检查 `docs/README.md`、`docs/backend/`、`backend/AGENTS.md` 是否存在，仅注入路径提示，**不注入内容**
- `system-prompts.ts:152-170` 的 `buildTimeSection` 从前端传入的 `route.userTimezone` 格式化当前日期时间注入
- `system-prompts.ts:105-108` 的 `buildWorkspaceSection` 注入当前工作区路径

**截断策略（系统层）**：
- `context-files.ts:149-160` 的 `truncateToTokenBudget`：按 `tokenBudget * 3.5` 估算字符上限，从尾部逐字符裁切
- `context-files.ts:207-229` 的 `fitStartupContentWithinBudget`：启动内容超标时从最后一个 section 往前裁，而不是保头或保尾策略
- `agent/types.ts:19-20`：`TOOL_OUTPUT_LIMIT = 50_000` 字符（工具输出截断）
- LLM API 级别的 `max_tokens` 由 `LLM_MAX_TOKENS` 环境变量控制，默认 8192（`env.ts:36`）

**个人化注入点判断**：

当前有个人化注入：SOUL.md（人格气质）、IDENTITY.md（角色边界）、USER.md（用户偏好/profile），均从 `.lecquy/` 目录读取。

但**缺失的是"历史决策摘要"注入点**——没有任何代码把 past session 的关键决策摘要注入到当前上下文中（除非 PG 向量召回命中）。

最自然的新增注入点在 `agent-runner.ts:105-113` 的 `buildSimpleSystemPrompt` 调用处或 `agent-runner.ts:126` 的 `contextMessages` 数组——可以在 `contextMessages` 中追加一条 role='user' 的决策历史摘要消息。

### 判断

**实现**。7 层系统提示拼装齐全，token 预算控制到位。但个人化目前只来自静态 .md 文件，缺乏动态的"基于历史行为学习"的注入点。

## 问题 3：主循环和子 Agent 现在到底在跑什么？

### 事实

**主 Agent 循环**：
- `agent-runner.ts:66-213` 的 `runSimpleAgent` 是唯一实际运行的 agent 入口
- `agent-runner.ts:125-170`：调用 `agentLoop`（来自 `@mariozechner/pi-agent-core`）
- 终止条件（`agent-runner.ts:138-146` + `agent/types.ts:7-8`）：
  - `MAX_ITERATIONS = 10`：达到最大迭代次数 → 注入强制停止 steering message
  - `MAX_TOOL_FAILURES = 3`：工具连续失败次数过多 → 同上的强制停止
  - 无 tool_call 时由 `agentLoop` 内部自然终止
  - **没有显式 done 信号**，依赖 `agentLoop` 的底层实现

**子 Agent / Worker**：
- **`backend/src/agent/sub-agent-runner.ts` 不存在**
- `agent/tools/index.ts:55-57` 的 `createManagerTools` 只含有 read/skill/todo_write/request_user_input/sessions_send/sessions_spawn，这是一个"编排者"角色，但没有对应的 Manager 运行循环
- `agent/tools/index.ts:60-62` 的 `createWorkerTools` 等于 `createSimpleTools` 减去 `todo_write` 和 `sessions_spawn`
- `agent/tools/session-tools/sessions-spawn.ts` 存在（用于派生子会话），但没有任何代码把它接入实际的 Agent 循环
- `prompt-layer-types.ts:103-111` 定义了 `WorkerReceipt` 类型（result / validation / nextHint），但**没有任何函数返回这个类型**

**Plan 模式**：
- `todo-manager.ts:26-142` 是纯内存数据结构（校验/渲染/状态转换），不持久化、不驱动执行
- `todo-write.ts` 工具存在，允许 agent 写 todo
- `context-files.ts:252` 的 AGENTS.md 模板描述了 plan 模式工作流（"plan 模式先规划 todo，再串行执行"），但这是 **prompt 层面的指令**，不是代码层面的执行引擎
- `agent/tool-permission.ts:45-56` 有 Manager/Worker 白名单/黑名单，但 Manager 角色在 `agent-runner.ts` 中从未被实例化——`runSimpleAgent` 固定传 `role: 'simple'`

### 判断

**半实现**。Simple 模式的主循环在跑。Plan/Manager-Worker 模式是只存在于 prompt 指令和 helper 类型里的"幽灵代码"——工具白名单、WorkerReceipt 类型、sessions_spawn 工具都写了，但没有把它们串起来的执行器。

## 问题 4：压缩做到哪儿了？

### 事实

**全仓关键词搜索结果**：

| 位置 | 状态 | 说明 |
|---|---|---|
| `memory/compact.ts:276-321` `buildCompactSummary` | **在跑** | LLM 调用 `completeSimple` 生成结构化摘要，失败降级到模板 |

| `memory/compact.ts:21` `COMPACT_TRIGGER_MESSAGE_EVENTS = 50` | **在跑** | 触发阈值：50 条 user/assistant 消息 |
| `memory/compact.ts:22` `COMPACT_RECENT_TAIL = 10` | **在跑** | 保留最近 10 条，其余被压缩 |
| `memory/compact.ts:137-175` `formatHistoryForCompaction` | **在跑** | 把 session entries 序列化为可读文本供 LLM 压缩 |
| `memory/compact.ts:397-401` `writeMemorySummary` | **在跑** | 压缩结果同时写入 `MEMORY.summary.md` |
| `core/prompts/context-files.ts:149-160` `truncateToTokenBudget` | **在跑** | startup context 的 token 预算截断 |
| `core/prompts/context-files.ts:207-229` `fitStartupContentWithinBudget` | **在跑** | startup 总预算超限时的逐段裁切 |
| `core/prompts/user-md-parser.ts:31-51` `truncateToBudget` | **在跑** | USER.md profile/preference 的独立截断 |
| `memory/store.ts:43-55` `truncateToBudget` | **在跑** | MEMORY.summary.md 读入时的截断 |
| `agent/types.ts:19-20` `TOOL_OUTPUT_LIMIT = 50_000` | **在跑** | 工具输出截断上限（字符数） |
| `system-prompts.ts:388-391` `buildSummarizePrompt` | **半跑** | 引用了"用于 nodes.ts summarize"，但 `nodes.ts` 文件不存在于仓库中 |
| `session-v2/session-pruner.ts` | **半跑** | 环境变量 `SESSION_PRUNING_MODE` 默认 `'cache-ttl'`，但这是 session entry 淘汰而非上下文语义压缩 |

**当前唯一在生效的"上下文瘦身"核心机制**：

`applyCompactionIfNeeded`（`memory/compact.ts:419-457`）→ 触发条件：50 条消息事件 → 保留最近 10 条 → 其余消息序列化为文本，调 LLM 生成结构化摘要 → 摘要同时写入 session entries（compaction 类型）和 `MEMORY.summary.md`

但**这个摘要没有被注入回 agent 的系统提示**——`system-prompts.ts` 的 `buildStartupLayerSlice` 只读取 `MEMORY.summary.md`（经过 400 token 预算截断），而压缩摘要是作为 session entry 插入的（通过 `SessionManager.appendCompaction`），依赖 session 持久化和回放机制来进入上下文。两者是两条不同路径。

### 判断

**实现**。LLM 压缩摘要是真实在跑的，触发阈值和降级策略都到位。但压缩摘要的回注路径有两个：session entries（依赖持久化回放）和 MEMORY.summary.md（启动时按预算截断后注入），两者不一致。此外没有 token counting（粗估用 1 token ≈ 3.5 字符），没有基于实际 context window 的动态压缩。

## 问题 5：自我进化的"接口"现在能不能开？

### 事实

**写文件工具能否写入自身配置路径**：

- `runtime-paths.ts:110-116` 的 `resolvePathWithinRoot`：所有文件工具（read/write/edit/bash）的路径解析都过 `isWithinRoot` 检查，只防止逃逸工作区根目录，**不阻止写入 `.lecquy/` 子目录**
- `dangerous-paths.ts:68-78`：`.claude` 和 `.lecquy` 被列为 `DANGEROUS_DIRECTORIES`
- `tool-permission.ts:345-461` 的 `createPermissionAwareTools`：当 `LAYERED_PROMPT=true` 时，写 `.lecquy/` 下的文件会触发 permission check（`classifyToolPermission` → `isInDangerousDirectory`）
- 但 `agent-runner.ts:92`：`layeredPermissionEnabled = process.env.LAYERED_PROMPT === 'true'` —— 默认不启用
- **结论**：默认情况下 write_file / edit_file 可以写入 `.lecquy/` 下的 SOUL.md / IDENTITY.md / USER.md / MEMORY.md，无显式拦截
- `.claude/` 目录在 `DANGEROUS_DIRECTORIES` 中，但如果 `LAYERED_PROMPT` 不是 `'true'`，这个检查也不会触发

**Skill 系统加载时机**：

- `skill-loader.ts:147-392`：`SkillLoader` 是一个普通 class，每次调用 `getSkillContent()` / `listSkills()` / `scanSkills()` 都会重新 `scanSkills()`
- `scanSkills()`（`skill-loader.ts:315-325`）每次都同步读取文件系统：bundled skills（编译内置）→ `backend/skills/`（开发期覆盖）→ `.lecquy/skills/`（运行时扩展）
- `system-prompts.ts:15`：`import { SKILLS } from '../core/skills/skill-loader.js'` —— 模块级单例
- **结论**：**热加载**。运行时修改 `.lecquy/skills/` 下的 SKILL.md 文件，下一次 skill 工具调用时会读到新内容。但 `context-files.ts:303-305` 的 TOOLS.md 模板说"磁盘 SKILL.md 热更新不影响当前会话的冻结版本"——这是 prompt 层面的约束，不是代码层面的。

**系统提示能否运行时替换**：

- `system-prompts.ts:45-53` 的 `buildIdentityLine` 调用 `renderPromptModuleTemplate` 从模板文件渲染
- `system-prompts.ts:179-196` 的 `buildProjectContextSection` 从 `.lecquy/` 的 SOUL.md / IDENTITY.md 等文件读取
- `system-prompts.ts:376-378` 的 `buildSimpleSystemPrompt` 是每次请求都重新调用的
- **结论**：系统提示**每次请求都动态构建**，修改 `.lecquy/` 下的人格文件 + 模板文件会立即生效。但内置模板文件在编译产物中（`backend/src/core/prompts/` 下的 .md 模板），如果代码被打包/编译，这些模板就固化了。
- `agent-runner.ts:105` 有 `systemPromptOverride` 参数，允许调用方完全覆盖系统提示。

**Agent 自我反思痕迹**：

- **没有**。搜索全仓未发现 agent 读取自己上一轮输出并写回的代码。
- `memory/extraction-runner.ts` 是记忆提取（从对话中提取事实），不是自我反思
- `memory/compact.ts` 是上下文压缩（摘要对话），不是自我反思

### 判断

**半实现**。写自身的"能力"（文件工具可达 `.lecquy/`）是存在的，但没有任何机制驱动 agent 去做这件事。Skill 是天然热加载的。系统提示每次动态构建，改文件即生效。但缺乏"触发自我修改"的 prompt 指令、反思循环或评估反馈回路。

## 问题 6：沙箱实际能不能用？

### 事实

**bash 工具当前隔离方式**：

- `agent/tools/bash.ts:23`：使用 `execSync`，**不是 `spawn`**，**不是 `ChildProcessSandbox`**
- `agent/tools/bash.ts:24`：`cwd: PROJECT_ROOT` —— cwd 锁定
- `agent/tools/bash.ts:25`：`timeout: 120_000` —— 2 分钟超时
- `agent/tools/bash.ts:27`：`stdio: ['pipe', 'pipe', 'pipe']` —— stdin 为 pipe（不可交互）

**sandbox-adapter.ts 的状态**：

- `sandbox-adapter.ts:240-253` 有 `NullSandbox`（假沙箱，仅用于测试）

**permissions 目录的补充**：

- `dangerous-paths.ts:21-172`：定义了危险文件/目录/系统路径的静态黑名单，但只在 `tool-permission.ts` 的 `createPermissionAwareTools` 中被引用，而该函数仅当 `LAYERED_PROMPT=true` 时启用

## 补充观察

1. **`.ZxhClaw/` 目录是遗产**：`runtime-paths.ts:38-39` 将其列为 `LEGACY_ROOT_RUNTIME_DIR_NAMES`，`.claude/worktrees/zen-golick/.ZxhClaw/` 下的人格文件（IDENTITY/SOUL/MEMORY/USER.md）是旧格式——内容描述的是"档案领域助手"，与当前 Lecquy 定位无关。当前生效的配置路径是 `.lecquy/`。

2. **sub-agent-runner.ts 缺失是整个 Plan 模式的瓶颈**：Manager/Worker 的工具白名单、WorkerReceipt 类型、sessions_spawn 工具都写了，但没有执行器把它们串起来。写这个文件就是 Plan 模式从"幽灵"变"落地"的关键一步。

3. **PG 依赖是最大的复杂度来源**：`db/` + `memory/coordinator.ts` + `extraction-runner.ts` + `foresight-sync.ts` 合计约 2,000+ 行，全受 `PG_ENABLED=false` 屏蔽。如果承认单人场景不需要 PG，这些代码可以物理删除而不影响任何命脉功能。
