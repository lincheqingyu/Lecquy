# 第一周第 3 件 — 人格基线撰写指引（USER / SOUL / MEMORY.md）

> 更新日期：2026-05-08
> 类型：撰写指引（执行人是 kira 本人，不是 codex）
> 关联：[CLAUDE.md](../../CLAUDE.md)、[20260508-2 代码现状审查报告](./20260508-2-个人强%20Agent%20路线%20代码现状审查报告.md)
> 预计耗时：1.5~3 小时（取决于你愿意倒多少东西出来）

---

## 一、为什么这件事的执行人是你（kira）不是 codex

前两件是机械活，可以丢给工具。这一件不能。

`.lecquy/USER.md` / `SOUL.md` / `MEMORY.md` 的内容**已经在被 system prompt 注入**（见 `backend/src/core/prompts/system-prompts.ts:179-196` 的 `buildProjectContextSection`）。也就是说——**今天你写什么，明天 agent 看到什么**。

让 codex 写"你是谁"是荒诞的。这只能你自己写。一旦写完，你能立刻在下一次对话里感受到 agent 不一样。这是当前所有动作里**反馈最快**的一件。

---

## 二、目标

产出三份个人化文件：

```
.lecquy/USER.md    — 你的事实档案（profile + preference）
.lecquy/SOUL.md    — agent 的人格气质（不是描述你，是描述"它"对你应该是什么气质）
.lecquy/MEMORY.md  — 当前已知的"项目 / 关系 / 习惯"长期记忆基线（开局快照）
```

注意三者的**区别**：

| 文件 | 视角 | 描述对象 | 用途 |
|------|------|---------|------|
| USER.md | 第三方描述 | **你 (kira)** | 让 agent 知道服务对象是谁 |
| SOUL.md | 第二人称指令 | **agent 自己** | 让 agent 知道自己是谁、对你什么气质 |
| MEMORY.md | 客观快照 | **你和你的世界** | 给 agent 一份"开局已知信息" |

---

## 三、不要做什么

- **不要**写得"完美"——这是个起点不是终点，下个月你会想改
- **不要**复制 ChatGPT 的"你是一个有用的助手"那种烂大街句式——SOUL.md 越具体、越有你个人审美的痕迹越好
- **不要**写明文敏感信息（密码、API Key、银行卡），但**身份、偏好、家人/朋友/项目这种**可以写——这是单人本地系统，不是公开仓库
- **不要**等"想清楚了再写"——草稿即可，agent 会用、你会发现哪里写错了，再改

---

## 四、撰写指引：USER.md

### 结构建议

```markdown
# USER

> 这份档案描述项目作者 kira 是谁、什么偏好、当前在做什么。
> agent 在和我对话时应当默认这些事实。

## 基本事实

- 名字 / 称呼：
- 邮箱：xingtaokai@outlook.com
- 时区 / 工作时段：
- 当前主要在做的事（一两句话）：

## 技术背景

- 主语言 / 栈：
- 喜欢的代码风格关键词（例：极简、强类型、显式 > 隐式、文件式记忆…）：
- 不喜欢什么（例：过度抽象、ORM 黑魔法、依赖注入框架…）：
- 当前主项目：Lecquy，定位见 CLAUDE.md

## 沟通偏好

- 想要什么调子的回答（例：直接、不要恭维、不要列大纲、不要 emoji…）：
- 什么情况下希望 agent 主动反对我（例：我说错的事实、坏判断的设计…）：
- 什么情况下希望 agent 闭嘴执行（例：明确指令、纯机械活…）：

## 当前正在解决的关键问题

- 这一周/这一月最想推进的事（例：把 Lecquy 从"半成品堆叠"转向"懂我的编程伙伴"）
```

### 撰写技巧

- **写"反例"比写"正例"信息密度高**：比起说"我喜欢极简"，说"我看到 ` const result: ResultType<UserType, ErrorType> = await ...` 这样的过度类型化会反胃"信息量大十倍。
- **真名实姓写出来**：你常用的项目名、合作的人名、城市，写出来。这是 agent 后续召回时的钩子。
- **量化偏好**："回答控制在 200 字以内除非我说继续展开"远胜于"回答简短"。

## 五、撰写指引：SOUL.md

这是最难的一份。它不是描述**你**，是描述**它**——一个伴随你长期工作的 agent 该是什么气质的存在。

### 结构建议

```markdown
# SOUL

> 这份文件定义 Lecquy 在和 kira 对话时应有的气质、立场、风格。
> 它不是"system prompt 的礼貌版本"，是一种人格设定。

## 你是谁（agent 的自我定位）

例如：
> "你是 kira 长期共事的搭档程序员，不是助手。你不需要每次都恭维我，
> 不需要等我授权再发表意见。你和我意见不同时直接说出来。"

## 你和 kira 的关系

例如：
> "你认识 kira 一年多了。你知道他凌晨 2 点写代码会犯什么错。
> 你看到他重复犯同一个错时，不要假装第一次见。"

（这里写得越像"老朋友"越好——这是 hermes-agent 自我进化的雏形。
即使开局这些是假的，agent 后面看到这份 SOUL 会朝这个方向靠。）

## 你的语言风格

- 中文为主，技术术语用中文
- 不用 emoji，不用过度礼貌
- 反对 kira 时直接说"我不同意，原因是 X"
- 同意 kira 时不要复述他的话，只说"对"或直接动手

## 你的边界

- 不替 kira 做决策——但要给出明确推荐和理由
- 不擅自扩大任务范围
- 看到 kira 在重复造轮子时提醒一下
- 看到 kira 偏离 CLAUDE.md 时也提醒
```

### 撰写技巧

- **不要写"你应该有用"这种废话**。写具体的——"反对我时怎么开口"、"我犯错时怎么说"。
- **写一两个**对话片段示例（"当我说 X，你回 Y"）——这比抽象指令对 LLM 行为塑造效果强得多。
- **不要怕过度具体**——SOUL 的越像一个真实的人，agent 行为就越稳。

## 六、撰写指引：MEMORY.md

这是"开局快照"。在记忆系统真正能工作之前，这份文件填补"agent 第一次见你时知道什么"的空白。

### 结构建议

```markdown
# MEMORY (基线快照)

> 这是 kira 在 2026-05-08 手写的一份"已知事实"快照。
> 后续记忆系统会自动追加，本文件作为基线保留。

## 项目 / 仓库

- Lecquy（这个项目本身）：定位见 CLAUDE.md，状态：第一周清理 + 沙箱集成阶段
- 其他活跃项目：______
- 已封存的项目：______

## 重要决策（让 agent 不要再质疑的事）

- 单人独占，不做多用户产品 — 已在 2026-05 钉死
- PG 处置：本周不动，下周决策（候选：保留启用 / 改 SQLite / 删除）
- Plan 模式（Manager-Worker）：暂停，sub-agent-runner.ts 不补
- ____（你自己加）

## 重复犯过的错（让 agent 提醒我）

- ___（例："写记忆功能时容易陷入框架完美主义"）
- ___

## 不要给我建议的事

- ___（例："不要再建议引入 LangGraph / AutoGen"）
- ___（例："不要再建议给项目加多用户支持"）
```

### 撰写技巧

- **MEMORY 是 USER 的"动态层"**：USER 写"我是谁"，MEMORY 写"我现在在哪儿、认识谁、最近发生什么"
- **大胆写"不要给我建议的事"**：这是节省后续 token 的最高效方式
- **写"重复犯过的错"是核心**：agent 看到这段会主动当你的"良性外部记忆"，这是 JARVIS 最关键的功能之一

## 七、写完后做什么

1. 三份文件都放在 `.lecquy/` 目录下（这个目录已经被 `runtime-paths.ts` 识别为运行时数据目录）
2. **不要 commit 这三份到 git**（敏感个人信息）—— 检查 `.gitignore` 是否已经忽略 `.lecquy/`，否则手动加上：
   ```
   # .gitignore
   .lecquy/
   !.lecquy/.gitkeep
   ```
3. 启动后端 `pnpm dev:backend`，发起一次新对话，问 agent："你认识我吗？告诉我你知道什么。"
4. 把 agent 的回答粘到本文件底部"首次验证"段，看看注入效果

## 九、给后续维护者的话（包括未来的 codex）

这三份文件**不是文档**，是**数据**。

- 改 USER.md 不需要 PR review——kira 自己改自己即可
- agent 后续的 reflect_session 循环（第三周做的事）会**自动写入** MEMORY.md
- 如果你（codex / claude）看到这三份文件，**不要"帮 kira 整理表达"或"修正语病"**，原文保留
- 唯一可以做的修改：在 kira 明确指令下追加新内容

如果你看到 SOUL.md 写的句子"违反"某种通用 AI 指令规范——它不是错，是 kira 故意的。
