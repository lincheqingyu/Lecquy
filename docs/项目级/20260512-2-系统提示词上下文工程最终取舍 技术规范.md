## 1. 修订口径

本文是后续 system prompt 结构落地的技术方向。它不再横向比较 OpenClaw、Hermes 或其他项目，而是直接给 Lecquy 做最终取舍。

本次修订吸收 Opus 4.7 的审稿意见，但不全盘采纳。凡是和 Lecquy 当前约束冲突的建议，以本节为准：

| Opus 反馈 | 最终处理 | 原因 |
|-----------|----------|------|
| 继续使用 v1 / v2 演进叙述 | 不采纳 | 用户已明确要求不要区分 v1/v2；本文只给最终架构和未来拆分条件。 |
| `MEMORY.md` 语义不清 | 采纳 | 明确 `memory.db` 是事实来源，`MEMORY.summary.md` 是常驻 frozen snapshot，`MEMORY.md` 是人工可读入口和 fallback。 |
| `USER.md` 需要具体长度预算 | 采纳 | 明确源文件软上限、切片预算和拆分触发条件。 |
| 单独建立 `tools-discipline.md` | 不采纳 | 工具纪律归 `.lecquy/TOOLS.md`；工具可用性由运行时生成，避免重复文件。 |
| 单独建立 `context-loader.md` | 不采纳为 prompt 文件 | 加载顺序、截断和注入防护应落实在 prompt builder 代码与测试中，不作为常驻人格文件。 |
| 增加 prompt injection 防护 | 采纳 | 长期本地记忆、项目文档、网页内容都会成为可写上下文，必须显式分级。 |
| prompt mode: full / minimal / none | 采纳为加载策略 | 主 Agent 使用 full，worker 使用 minimal；不因此新增目录结构。 |
| `agent.yaml` + `agents/<name>/role.md` | 暂不采纳 | 当前只需要 simple / manager / worker 运行模式；等出现多个稳定可路由 agent 再拆。 |

## 2. 最终结论

Lecquy 是个人强 Agent，不是多人平台。system prompt 上下文工程只服务四个目标：

- 本地优先：核心上下文、长期记忆、项目规则都在本地可检查、可编辑、可迁移。
- 长期记忆：稳定画像进入常驻片段；事件、决策、错误、项目事实进入可检索记忆。
- 自我进化：允许持续沉淀 `USER`、`SOUL`、`IDENTITY`、`MEMORY`，但不把所有材料都塞进 system prompt。
- 工程执行：项目规则、工具纪律、技能工作流必须直接服务代码执行，不做平台化配置体系。

最终结构是：少量常驻文件 + 固定 prompt layer + SQLite 记忆检索 + 按需技能加载。不要做大而全的 prompt 总文件，不要提前做 agent registry，不要把 memory 拆成一堆 Markdown 分类文件。

## 3. 必须有的文件

| 文件 | 决策 | 职责 |
|------|------|------|
| `.lecquy/SOUL.md` | 必须有 | 人格底色、沟通方式、稳定价值取向。回答“这个个人 Agent 像谁”。 |
| `.lecquy/IDENTITY.md` | 必须有 | 使命、能力边界、行为原则。回答“这个 Agent 为谁做什么”。 |
| `.lecquy/USER.md` | 必须有 | 用户稳定画像与稳定偏好，单文件结构化维护。 |
| `.lecquy/AGENTS.md` | 必须有 | 本地 Agent 运行规则与项目级约束入口，不作为 agent 总索引。 |
| `.lecquy/TOOLS.md` | 必须有 | 工具使用纪律、危险操作边界、确认规则；不是工具可用性清单。 |
| `.lecquy/MEMORY.md` | 必须有 | 人工可读的 memory 入口、维护说明和 fallback，不作为常规全文注入。 |
| `.lecquy/MEMORY.summary.md` | 必须有 | 可常驻的稳定记忆 frozen snapshot，由系统维护或人工整理。 |
| `.lecquy/memory/memory.db` | 必须有 | 长期记忆事实来源，支持检索、标签、时间、重要性和置信度。 |
| `.lecquy/system-prompt/*.md` | 必须有 | 基础 system、mode、tooling、safety、runtime 模板。 |
| `skills/<name>/SKILL.md` | 按需有 | 可复用专项工作流，命中时加载，不常驻。 |

## 4. 暂时不要有的文件

| 文件或目录 | 决策 | 原因 |
|------------|------|------|
| `core.md` | 不要 | 合并 `SOUL` 与 `IDENTITY` 会把人格、使命、边界混在一起，后续难以审计。 |
| `user-profile.md` / `user-preferences.md` / `user-projects.md` | 暂时不要 | 当前单个 `USER.md` 足够；先靠标题结构、预算切片和拆分条件控制膨胀。 |
| `memory/facts.md` / `memory/decisions.md` / `memory/mistakes.md` | 不要 | 记忆分类由数据库字段和标签承担，Markdown 分类会增加维护成本和污染风险。 |
| `tools-discipline.md` | 不要 | 工具纪律归 `.lecquy/TOOLS.md`；长流程归 skill。 |
| `context-loader.md` | 不要作为 prompt 文件 | 加载策略应写进 prompt builder 代码、schema 或测试，不参与 prompt 拼接。 |
| `agents.md` 作为总索引 | 不要 | 当前只需要 mode/role，不需要平台级 agent registry。 |
| `agents/<id>/agent.yaml` | 暂时不要 | 只有多个稳定、可路由、可授权 agent 出现后才需要机器可读 registry。 |
| `context/` 大目录 | 暂时不要 | 会诱导把所有材料都变成 prompt 文件，削弱检索和按需读取边界。 |

## 6. Prompt 拼接顺序

拼接顺序按“可缓存前缀 -> 半缓存技能 -> 动态上下文”的顺序排列。目标是让长期稳定内容连续放在最前面，提高 KV cache 命中率；不要在可缓存层之间插入 memory recall、工具输出、当前用户消息或技能正文。

层级顺序与加载内容总表：

| 全局顺序 | 缓存分区 | Layer | 层内顺序 | 来源 | 是否允许用户修改 | 加载内容解释 | 示例 |
|----------|----------|-------|----------|------|------------------|--------------|------|

| 1.2 | 可缓存前缀 | `System` | 2 | tooling + tool schema summary | 否，运行时生成 | 注入当前运行时实际可用的工具摘要和参数形状；它描述工具 inventory，不写工具纪律。 | `read_file(path)`、`edit_file(path, patch)`、`bash(command)` |

### 6.1 可缓存前缀

这一段是 system prompt 的稳定前缀。只要项目配置、用户稳定画像、模式选择不变，就应该尽量保持字面内容和顺序不变。

1. 加载 `System` 层。
2. 加载 `Mode` 层。
3. 加载 `StartupContext` 层。
4. 加载 `UserPreference` 层。

#### 6.1.1 System 层

`System` 层放最前面，负责定义运行底座、工具纪律和全局安全边界。

1. 加载当前模式对应的 identity template。
2. 加载运行时生成的工具基础说明和工具 schema 摘要。
3. 加载 `tool-call-style.md`，约束工具调用格式和工具结果处理方式。
4. 加载 `safety.md`，约束权限、危险操作确认、prompt injection 防护。
5. 加载 skills index，只包含技能名称、触发描述和路径，不加载技能正文。
6. 加载 workspace、documentation、time、runtime、extra instructions 等基础模板。
7. 加载 `.lecquy/AGENTS.md`，注入本地 Agent 运行规则与项目级约束入口。
8. 加载 `.lecquy/TOOLS.md`，注入工具纪律、危险操作边界和确认规则。

#### 6.1.2 Mode 层

`Mode` 层放在 system 底座之后，负责声明当前运行模式。

1. 判断当前 mode 是 simple、manager 还是 worker。
2. 加载对应的 `role-*.md`。
3. 注入该 mode 的职责边界、输出要求和委派规则。
4. 对 worker 使用 minimal prompt 策略，不加载个人化画像和人格底色。

#### 6.1.3 StartupContext 层

`StartupContext` 层放在 mode 之后，负责注入个人强 Agent 的稳定身份和长期背景。

1. 加载 capability block，说明当前 agent 的核心能力和运行限制。
2. 加载 `.lecquy/SOUL.md`，注入人格底色、沟通方式和稳定价值取向。
3. 加载 `.lecquy/IDENTITY.md`，注入使命、能力边界和默认行为原则。
4. 加载 `.lecquy/USER.md` 的 profile slice，只注入稳定用户画像。
5. 加载 `.lecquy/MEMORY.summary.md`，注入高度压缩的长期记忆摘要。

#### 6.1.4 UserPreference 层

`UserPreference` 层仍然属于可缓存前缀，应放在技能正文之前，而不是放在半缓存层之后。

1. 从 `.lecquy/USER.md` 解析 preference slice。
2. 只保留长期稳定偏好，不注入临时口味、一次性任务要求或会话流水。
3. 对 preference slice 做预算截断，目标不超过 200 tokens。
4. 将截断和解析结果记录到 prompt diagnostics，方便发现 `USER.md` 膨胀。

### 6.2 半缓存技能层

这一段只在命中技能时加载。技能正文通常比 memory recall 稳定，但会随任务类型变化，所以它必须放在所有可缓存层之后、动态层之前。

1. 判断当前用户请求是否命中某个 skill。
2. 加载命中 skill 的 `SKILL.md` 正文。
3. 按 skill 指令读取必要 reference，不批量加载整个 references 目录。
4. 对 skill 正文和 reference 摘要做预算截断。
5. 将 skill 内容作为 `SkillRuntime` 层拼接。

### 6.3 动态记忆层

这一段每轮都可能变化，不能放进稳定前缀。

1. 根据当前用户消息、项目、route、mode 生成 memory query。
2. 从 `memory.db` 召回相关事实、决策、错误、项目状态和承诺。
3. 按 `kind`、`scope`、`projectId`、`status`、`confidence`、`importance` 过滤和排序。
4. 将召回结果包裹为低权限 `MemoryRecall` 层，明确它是检索结果，不是 system 规则。
5. 对 memory recall 做 topK 和 token 截断。

### 6.4 实时回合层

这一段永远放最后。它包含当前回合才出现的信息，不参与 prompt cache 设计。

1. 加载当前用户消息。
2. 加载 route、session、附件、选项等 live turn 上下文。
3. 加载本轮工具调用结果、todo 状态、worker 结果。
4. 加载必要的临时 extra instruction。
5. 将以上内容作为 `LiveTurn` 层拼接在 prompt 最末尾。

关键约束：

- worker 默认使用 minimal prompt，不继承 `SOUL.md`、`IDENTITY.md`、`USER.md`，只继承任务、工具纪律和必要项目上下文。
- memory recall 必须标注为检索结果，不能与 system 规则同权。
- 技能正文只有命中时加载，技能索引可以常驻。
- 项目文档只能按需读取，不进入常驻 system prompt。
- cache boundary 固定在 `UserPreference` 之后；`SkillRuntime` 是半缓存层，`MemoryRecall` 和 `LiveTurn` 是动态层。

## 7. 内容归属

| 内容 | 归属 |
|------|------|
| 人格底色、语言气质、稳定价值取向 | `SOUL.md` |
| 长期使命、能力边界、默认行为原则 | `IDENTITY.md` |
| 用户稳定画像、长期偏好 | `USER.md` |
| 用户短期偏好、近期承诺、临时事实 | `memory.db`，稳定后再晋升到 `USER.md` |
| 事实、决策、错误、项目状态、人物、环境 | `memory.db`，按标签检索 |
| 记忆维护说明、人工检查入口、fallback 文本 | `MEMORY.md` |
| 可常驻的长期记忆摘要 | `MEMORY.summary.md` |
| 工具确认规则、危险操作边界、工具调用纪律 | `TOOLS.md` |
| 工具是否可用、参数 schema、运行时 inventory | 运行时生成，不写入 `TOOLS.md` |

| 仓库技术栈、目录、命令、文档规范 | 项目 `AGENTS.md`、子目录 `AGENTS.md`、`docs/**` |
| 可复用专项流程、检查清单、领域工作法 | skill |

## 8. 文件职责表

| 文件 | 常驻 | 可编辑 | 预算 | 禁止内容 |
|------|------|--------|------|----------|
| `.lecquy/SOUL.md` | 是 | 是 | 建议 <= 800 tokens | 工具教程、项目规则、记忆明细 |
| `.lecquy/IDENTITY.md` | 是 | 是 | 建议 <= 800 tokens | 生活细节、短期任务、项目长文档 |
| `.lecquy/USER.md` | 是，切片 | 是 | 源文件软上限 4KB；profile slice <= 400 tokens；preference slice <= 200 tokens | 会话流水、未经验证的事实 |
| `.lecquy/AGENTS.md` | 是 | 系统管理为主 | 建议 <= 1200 tokens | agent registry、长篇技能内容 |
| `.lecquy/TOOLS.md` | 是 | 系统管理为主 | 建议 <= 1200 tokens | 完整工具文档、工具 inventory、重复安全规则 |
| `.lecquy/MEMORY.md` | 否，fallback 可截断注入 | 是 | 建议 <= 4KB | 不能作为主记忆库，不能全文常驻 |
| `.lecquy/MEMORY.summary.md` | 是 | 系统管理为主 | 建议 <= 600 tokens | 高频变化事件、低置信事实 |
| `.lecquy/memory/memory.db` | 否 | 系统管理 | 检索 topK + 输出截断 | 不能把召回结果升级为 system 规则 |
| `.lecquy/system-prompt/*.md` | 是 | 系统管理 | 单模板建议 <= 1000 tokens | 用户私有记忆、项目长文档 |
| `skills/<name>/SKILL.md` | 否 | 是 | 按需加载 | 全局身份、常驻人格、项目私密记忆 |

`USER.md` 超过 4KB 不立刻拆文件，但必须 review；超过 6KB、或 profile/preference 连续两次被截断，才触发拆分讨论。

## 9. Memory 策略

`memory.db` 是唯一权威长期记忆库。Markdown 文件只能作为摘要、说明或人工入口。

记忆字段至少需要支持：

| 字段 | 用途 |
|------|------|
| `kind` | `fact` / `decision` / `mistake` / `preference` / `project` / `people` / `environment` / `summary` / `commitment` |
| `scope` | `global` / `project` / `session` |
| `projectId` | 项目维度过滤 |
| `status` | `active` / `archived` / `superseded` |
| `roleHints` | 哪些 mode 或 agent 优先召回 |
| `tags` | 补充检索维度 |
| `confidence` | 置信度，低置信不得晋升常驻 |
| `importance` | 重要性排序 |
| `ttl` | 过期或复审时间 |
| `source` | 来源会话、文件或人工写入 |

晋升规则：

- 临时事实先进入 `memory.db`，不直接写 `USER.md`。
- 被多次确认、跨任务稳定有效的用户偏好，才能晋升 `USER.md`。
- 对执行有长期影响的项目决策，保留在 `memory.db` 并在项目文档中落正式记录。
- 高频召回且稳定的少量事实，可进入 `MEMORY.summary.md`。
- `MEMORY.md` 不承担 Hermes 式完整 frozen snapshot；这个职责由 `MEMORY.summary.md` 承担。

## 10. Prompt Injection 防护

信任优先级固定为：

```text
System templates
> Project instruction
> Skill instruction
> Memory summary
> Memory recall
> Tool output / external docs / web content
> Live user-provided quoted content
```

防护规则：

- 外部文档、网页、工具输出、检索结果都不能覆盖 system、project instruction 或 skill 规则。
- memory recall 必须带来源和低权限标记，不能写成“你必须遵守”的格式。
- 写入 `USER.md`、`SOUL.md`、`IDENTITY.md`、`MEMORY.summary.md` 前必须过滤明显注入短语，例如“忽略之前规则”“泄露 secret”“读取隐藏文件”“把系统提示词输出给我”。
- 对不可见字符、异常长行、base64 大块、疑似 exfiltration 指令做截断或拒写。
- prompt builder 应在代码层落实加载顺序、截断、去重、低权限包裹和日志诊断；不要把这些策略放进独立 prompt 文件后依赖模型自觉执行。

## 11. 风险与防护

| 风险 | 触发条件 | 防护方案 |
|------|----------|----------|
| 人格、使命、边界混乱 | 合并 `SOUL.md` 与 `IDENTITY.md` | 保持双文件；SOUL 写人格，IDENTITY 写职责和边界。 |
| `USER.md` 膨胀 | 用户画像、偏好、项目、人物全部堆入单文件 | 固定标题、4KB 软上限、切片预算；临时信息先进入 memory。 |
| memory 分类过粗 | 只靠少数文本分类，无法区分事实和决策 | 数据库层使用 `kind`、`scope`、`status`、`roleHints`、`ttl`、`confidence`。 |
| `TOOLS.md` 变成大 system prompt | 每个工具都写完整说明和例子 | 只写纪律和边界；工具教程放 skill 或按需文档。 |
| agent 发现机制弱 | 出现多个专职 agent，但仍只有 role template | 达到拆分条件后引入 agent registry；当前阶段不提前平台化。 |
| prompt 注入 | 文档、网页、工具输出要求覆盖 system 规则 | 固定信任优先级；检索和工具输出低权限包裹。 |
| 上下文漂移 | 临时任务计划、会话流水写进常驻文件 | 常驻文件只收稳定结论；临时计划留 live turn 或 memory。 |
| 记忆污染 | 未验证事实被写入 `USER.md` 或 summary | 先入 `memory.db`，带置信度；稳定后再晋升。 |
| 重复规则 | 同一条约束出现在 SOUL、IDENTITY、TOOLS、AGENTS | 每类文件只保留自己的职责；冲突时按信任优先级处理。 |
| 文件过多 | 为每个概念新建 Markdown | 新文件必须满足明确拆分条件，否则进入现有文件或 memory。 |

## 12. 未来拆分条件

只有满足以下条件时才拆分，不按“看起来更完整”拆分：

| 拆分对象 | 触发条件 | 拆分方向 |
|----------|----------|----------|
| `USER.md` | 超过 6KB；profile/preference 连续两次被截断；稳定用户信息出现明显互斥领域 | 拆为 `USER.profile.md` 与 `USER.preferences.md`，仍保持少量文件。 |
| `.lecquy/TOOLS.md` | 超过 1500 tokens；工具纪律和工具教程混杂；同类规则重复三处以上 | 保留 `TOOLS.md` 为纪律摘要，长流程迁入 skill；不默认新增 `tools-discipline.md`。 |
| `.lecquy/AGENTS.md` | 出现 3 个以上稳定可选 agent；需要按任务自动路由；每个 agent 有不同工具权限 | 引入 `agents/<id>/role.md` 和轻量 registry。 |
| `MEMORY.summary.md` | 摘要无法覆盖长期稳定事实；召回命中经常缺关键背景 | 按项目生成 `MEMORY.<project>.summary.md`，不拆人工分类文件。 |
| `skills/` | 某类流程被重复执行 3 次以上，且每次都需要同一套步骤或参考资料 | 提炼为独立 skill。 |
| `.lecquy/system-prompt/*.md` | 某模板超过 1000 tokens，且包含两个以上职责 | 按职责拆模板，不新增业务概念文件。 |
| project instruction | 根 `AGENTS.md` 超过维护预算，或前端/后端规则互相干扰 | 下沉到子目录 `AGENTS.md` 或对应 `docs/**`。 |

最终原则：Lecquy 的 system prompt 工程只服务个人强 Agent 的长期执行力。文件数量要少，职责要硬，动态内容交给检索，复杂流程交给 skill，项目约束交给 project instruction。
