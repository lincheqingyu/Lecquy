## 二、要修的 bug（一句话）

**关闭浏览器再打开 localhost:8000，UI 显示"新会话空白页"，但用户发出的消息却被路由进了上一次的会话。**

复现路径：

1. 首次打开 → localStorage 写入 `lecquy.peerId = peer_A`
2. 在"会话1"中问答几轮（后端生成 session key `agent:...:dm:peer_A`）
3. 关闭浏览器窗口
4. 重开 localhost:8000 → UI 渲染空白态（`selectedSessionKey === null`）
5. 此时输入消息 → `useChat` 用当前 `peerId = peer_A` 发 `run_start` → 后端拼出**同样的 session key** → 消息追加到"会话1"
6. 用户刷新或点开"会话1" → 看到刚刚的新消息出现在旧会话里

---

## 三、根因（结构层）

代码里有**两个相关状态**，生命周期不一致：

| 状态 | 来源 | 持久化 | 关闭浏览器后 |
|---|---|---|---|
| `peerId`（身份/路由键） | `localStorage['lecquy.peerId']` | ✅ 持久 | 保留旧值 |
| `selectedSessionKey`（UI 当前会话引用） | `useState(null)` in `HomePageLayout` | ❌ 不持久 | 复位为 `null` |

后端 dm 路由（`backend/src/session-v2/session-key.ts:13-29`）只看 `peerId`：

```ts
key: `agent:${agentId}:${channel}:${accountId}:dm:${peer}`
```

所以 peerId 不变 → 后端永远路由到同一个 session。但前端 UI 因为 `selectedSessionKey` 未持久化而表现成"新会话"，**两者撕裂**。

## 四、决策（关键，先看再动）

### 4.1 不走的方向

**不采纳"每次进首页 resetPeerId"**（DeepSeek 的方案 A）。

理由：
- 该方案让"关闭浏览器再打开 = 新会话"成为默认行为，**直接破坏长期记忆产品方向**（CLAUDE.md 第 1 节核心关键词之一）。
- Lecquy 的产品参考是 JARVIS——重启后仍然是同一个"它"，不是 CLI agent 那种"启动即新进程"。
- Web 用户的预期也是"关掉再开继续上次"（ChatGPT、Claude.ai、微信网页版都这么做）。

### 4.2 不在本次做的方向

**不在本次执行里把后端 peerId 重命名为 sessionId**。

理由：
- 后端 `runtime/session-key.ts` 与 `session-v2/session-key.ts` 是两条并行的 session 路径（CLAUDE.md 第 4 节"遇到混杂目录不要硬拆"红线提到的混杂模式之一）。peerId 同时被两条路径消费，强行重命名会触发跨模块连锁修改。
- 这次的目标是**修 bug，不是重构**。peerId 概念清理留待单独"结构性重构任务"。

### 4.3 本次采用的方向

**前端持久化"最近活动会话引用"，进首页时按它反查后端，自动恢复 UI 与 peerId 的绑定。**

新增前端 localStorage 键：`lecquy.lastActiveSessionKey`（string，存当前/最近一次活动的 session key）。

行为约定：

| 场景 | 行为 |
|---|---|
| 进首页（冷启动） | 读 `lastActiveSessionKey` → 若 `fetchSessions()` 中能找到，自动 `handleSelectConversation`（这会顺带把 peerId 切到该 session 绑定的 peerId）；找不到则进入空白新会话视图 |
| 用户点"新建会话" | `resetPeerId()` + 清除 `lastActiveSessionKey` + UI 复位（沿用现有 `handleStartNewConversation`，**新增清除 lastActiveSessionKey 一行**） |
| 用户从侧栏选会话 | `handleSelectConversation` 末尾**新增写入 `lastActiveSessionKey`** |
| 后端首次解析新 session（`onSessionResolved`） | 现有逻辑保留，**新增写入 `lastActiveSessionKey`** |
| 用户删除当前会话 | `handleDeleteConversation` 已 fallback 到 `handleStartNewConversation`，由 4.3 的"新建会话"分支自动清除 |

这样：
- peerId 仍然是 dm 路由键（后端零改动）
- UI 当前会话引用变成显式持久化的派生态
- 关闭浏览器再开 → 自动恢复到上次会话（UI 状态和 peerId 重新对齐）
- bug 消失，且符合"长期记忆 / 越用越懂"的产品方向

## 六、执行步骤（按顺序）

### 6.1 在 `frontend/src/lib/session.ts` 新增 lastActiveSessionKey 工具函数

**文件**：`frontend/src/lib/session.ts`

**动作**：在文件末尾追加三个新函数（不要改动已有的 `getPeerId` / `resetPeerId` / `setPeerId` / `getSessionId` / `resetSessionId`）：

```ts
const LAST_ACTIVE_SESSION_KEY = 'lecquy.lastActiveSessionKey'

export function getLastActiveSessionKey(): string | null {
  try {
    const value = localStorage.getItem(LAST_ACTIVE_SESSION_KEY)
    return value && value.trim().length > 0 ? value : null
  } catch {
    return null
  }
}

export function setLastActiveSessionKey(sessionKey: string): void {
  try {
    localStorage.setItem(LAST_ACTIVE_SESSION_KEY, sessionKey)
  } catch {
    // 静默失败：localStorage 不可用时退化成"每次冷启动都是空白态"
  }
}

export function clearLastActiveSessionKey(): void {
  try {
    localStorage.removeItem(LAST_ACTIVE_SESSION_KEY)
  } catch {
    // 静默失败
  }
}
```

**说明**：
- 不复用现有的 `memoryPeerId` 内存 fallback——`lastActiveSessionKey` 在 localStorage 不可用时直接退化成"无持久化"（这种情况下用户也享受不到"关掉再开继续"的体验，但不会出 bug）
- 不暴露给任何其它模块——只在 `HomePageLayout.tsx` 调用

**验收**：`pnpm -F @lecquy/frontend typecheck` 通过。

### 6.3 在 `HomePageLayout.tsx` 增加冷启动恢复逻辑

**位置**：找到 `useEffect(() => { void refreshSessions() }, [])`（约 line 339）。

**现有逻辑**：仅刷新会话列表。

**改造目标**：列表加载完成后，若 `lastActiveSessionKey` 存在且能在列表中找到，自动 `handleSelectConversation`。

**实现方式**：新增一个独立的"已尝试冷启动恢复"标志位，避免后续 `refreshSessions` 重复触发恢复。

#### 6.3.1 在组件顶部 state 区新增 ref（紧跟现有 `seenDraftArtifactKeysRef` 之后）

```ts
const hasAttemptedRestoreRef = useRef<boolean>(false)
```

#### 6.3.2 在 `useEffect(() => { void refreshSessions() }, [])` 之后新增一个新的 useEffect

```ts
useEffect(() => {
  if (hasAttemptedRestoreRef.current) return
  if (isSessionListLoading) return
  if (sessionItems.length === 0) {
    hasAttemptedRestoreRef.current = true
    return
  }

  hasAttemptedRestoreRef.current = true

  const targetKey = getLastActiveSessionKey()
  if (!targetKey) return

  const target = sessionItems.find((item) => item.id === targetKey)
  if (!target) {
    // 上次活动的会话已被删除或不在列表里 — 退化为新会话视图
    clearLastActiveSessionKey()
    return
  }

  void handleSelectConversation(targetKey)
}, [isSessionListLoading, sessionItems])
```

**关键点**：
- `hasAttemptedRestoreRef` 保证**整个组件生命周期内只恢复一次**——后续手动刷新会话列表（如新建会话后 `refreshSessions`）不会触发恢复
- 依赖数组只有 `isSessionListLoading` 和 `sessionItems`——不要把 `handleSelectConversation` 加进依赖（它每次渲染都是新引用，会触发循环）。如果 lint 警告，加 `// eslint-disable-next-line react-hooks/exhaustive-deps` 并注释说明
- 找不到 target 的 fallback 分支必须 `clearLastActiveSessionKey`，否则下次冷启动还会再失败一次

**验收**：
- 手工冷启动（关浏览器 → 重开 localhost:8000）回到上次会话
- 主动点"新建会话" → 进入空白态 → 关浏览器 → 重开 → 仍然是空白态（不会被旧的 `lastActiveSessionKey` 拉回去）
- 删除当前会话 → 自动进入空白态 → 关浏览器 → 重开 → 仍然是空白态

### 6.4 处理一个 corner case — 残留的 lecquy.peerId

**背景**：现有用户的 localStorage 里已经有 `lecquy.peerId = peer_xxx`，对应一个真实的后端 session。新逻辑下：
- 第一次冷启动 → `lastActiveSessionKey` 不存在 → 走"空白新会话"分支 → 但 `activePeerId` 仍是 `peer_xxx` → 用户发第一条消息 → 后端路由到旧 session → 用户看到"新会话被合并到旧会话"——还是会撞上原 bug 的尾巴一次

**对策**：组件初始化时，**只有在 `lastActiveSessionKey` 存在时才信任 localStorage 里的 peerId**；否则在进入"空白新会话"分支前主动 reset 一次。

**实现位置**：在 6.3.2 新增的 useEffect 中，把"找不到 target"的 fallback 改写为：

```ts
const target = sessionItems.find((item) => item.id === targetKey)
if (!target) {
  clearLastActiveSessionKey()
  const nextPeerId = resetPeerId()
  setActivePeerId(nextPeerId)
  return
}
```

并在另一处分支处理"`lastActiveSessionKey` 直接为空"的情况——在 useEffect 体的开头新增：

```ts
const targetKey = getLastActiveSessionKey()
if (!targetKey) {
  // 没有"最近活动会话"记录，但 localStorage 可能残留了 peerId
  // 主动 reset 一次，确保进入真正的"空白新会话"
  const currentPeer = getPeerId()
  // 仅当存在残留 peerId 且没有任何 session 引用它时才 reset
  const orphan = !sessionItems.some((item) => item.peerId === currentPeer)
  if (orphan) {
    const nextPeerId = resetPeerId()
    setActivePeerId(nextPeerId)
  }
  return
}
```

**说明**：`SessionListItemVm` 已经在 `frontend/src/lib/session-management.ts` 中暴露 `peerId` 字段（取自后端 `entry.origin.peerId`），可以直接用作判断依据。

**完整片段示意**（6.3.2 + 6.4 合并后）：

```ts
useEffect(() => {
  if (hasAttemptedRestoreRef.current) return
  if (isSessionListLoading) return
  if (sessionItems.length === 0) {
    hasAttemptedRestoreRef.current = true
    return
  }

  hasAttemptedRestoreRef.current = true

  const targetKey = getLastActiveSessionKey()

  if (!targetKey) {
    const currentPeer = getPeerId()
    const orphan = !sessionItems.some((item) => item.peerId === currentPeer)
    if (orphan) {
      const nextPeerId = resetPeerId()
      setActivePeerId(nextPeerId)
    }
    return
  }

  const target = sessionItems.find((item) => item.id === targetKey)
  if (!target) {
    clearLastActiveSessionKey()
    const nextPeerId = resetPeerId()
    setActivePeerId(nextPeerId)
    return
  }

  void handleSelectConversation(targetKey)
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [isSessionListLoading, sessionItems])
```
