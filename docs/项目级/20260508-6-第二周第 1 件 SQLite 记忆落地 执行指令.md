## 二、目标（一句话）

新建 SQLite memory store，把 `extraction-runner.ts` 的"LLM 提取 + 关键词降级"逻辑接到 SQLite，让 agent 每次会话结束都能把对话提取成结构化事件并持久化到本地 `.lecquy/memory/memory.db`。

**这是"先建"，不是"切换"也不是"拆 PG"**。本任务结束时 PG 路径仍保留可用，但默认不再走它。

## 四、技术决策（已拍板，不要改）

| 项 | 决策 | 理由 |
|---|---|---|
| SQLite 库 | `better-sqlite3` | 同步 API + 性能好 + 社区主流 |
| 数据库文件 | `.lecquy/memory/memory.db` | `.lecquy/` 已是 runtime 数据目录 |
| 模式 | WAL（Write-Ahead Logging） | 并发读 + 防写阻塞 |
| 全文搜索 | 内置 FTS5 | 无外部依赖、性能足够 |
| 异步 polling | **不要** | 单人场景同步调用即可，简化代码 |
| 项目维度 tag | `project_id` 字段（cwd 的 git remote 优先，fallback 到 cwd 路径） | 单一字段、查询简单 |

```sql
  id            TEXT     PRIMARY KEY,
  kind          TEXT     NOT NULL,           -- 'event' | 'compaction'
  event_type    TEXT,                        -- 'user_fact'|'assistant_commitment'|'tool_action'|'decision'|'observation'
  project_id    TEXT,                        -- 新增：cwd 的 git remote 或 cwd 路径

  session_key   TEXT,
  summary       TEXT     NOT NULL,
  content       TEXT     NOT NULL,
  tags_json     TEXT     NOT NULL DEFAULT '[]',
  importance    REAL     NOT NULL DEFAULT 5,
  confidence    REAL     NOT NULL DEFAULT 0.5,
  status        TEXT     NOT NULL DEFAULT 'active',
  source_event_ids_json TEXT NOT NULL DEFAULT '[]',
  source_session_id     TEXT,
  occurred_at   TEXT     NOT NULL,
  created_at    TEXT     NOT NULL DEFAULT (datetime('now')),
  updated_at    TEXT     NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_project_occurred
  ON memory_items(project_id, occurred_at DESC)
  WHERE status = 'active';

CREATE INDEX IF NOT EXISTS idx_event_type_created
  ON memory_items(event_type, created_at DESC)
  WHERE status = 'active';

CREATE INDEX IF NOT EXISTS idx_session_created
  ON memory_items(session_id, created_at DESC);

-- FTS5 虚拟表 + content table 联动
CREATE VIRTUAL TABLE IF NOT EXISTS memory_items_fts USING fts5(
  summary, content, tags_text,
  content='memory_items',
  content_rowid='rowid',
  tokenize='unicode61 remove_diacritics 2'
);

-- 触发器：保持 FTS 与主表同步
CREATE TRIGGER IF NOT EXISTS memory_items_ai AFTER INSERT ON memory_items BEGIN
  INSERT INTO memory_items_fts(rowid, summary, content, tags_text)
  VALUES (new.rowid, new.summary, new.content, replace(replace(new.tags_json, '[', ''), ']', ''));
END;

CREATE TRIGGER IF NOT EXISTS memory_items_ad AFTER DELETE ON memory_items BEGIN
  INSERT INTO memory_items_fts(memory_items_fts, rowid, summary, content, tags_text)
  VALUES ('delete', old.rowid, old.summary, old.content, replace(replace(old.tags_json, '[', ''), ']', ''));
END;

CREATE TRIGGER IF NOT EXISTS memory_items_au AFTER UPDATE ON memory_items BEGIN
  INSERT INTO memory_items_fts(memory_items_fts, rowid, summary, content, tags_text)
  VALUES ('delete', old.rowid, old.summary, old.content, replace(replace(old.tags_json, '[', ''), ']', ''));
  INSERT INTO memory_items_fts(rowid, summary, content, tags_text)
  VALUES (new.rowid, new.summary, new.content, replace(replace(new.tags_json, '[', ''), ']', ''));
END;
```

### 5.3 新建辅助 `backend/src/memory/project-id.ts`

只做一件事：根据 cwd 推导出稳定的 `projectId` 字符串。

逻辑：

```ts
export function deriveProjectId(cwd: string): string {
  // 1. 尝试读取 cwd 下 .git/config 里的 remote.origin.url
  //    如果有就返回（标准化：去掉 .git 后缀、去掉协议前缀）
  // 2. 否则，回退到 cwd 的绝对路径作为 projectId
  // 3. 如果 cwd 也无法读取，返回 'unknown'
}
```

要点：
- **稳定**：同一仓库不论 clone 在哪都返回相同值（用 git remote URL）
- **可读**：直接用作 tag 字段，不要 hash
- **无副作用**：不改 git 状态、只读

参考实现：

```ts
import { execSync } from 'node:child_process'
import { resolve } from 'node:path'

export function deriveProjectId(cwd: string): string {
  try {
    const url = execSync('git config --get remote.origin.url', {
      cwd, encoding: 'utf-8', timeout: 1000, stdio: ['ignore', 'pipe', 'ignore'],
    }).trim()
    if (url) {
      return url.replace(/\.git$/, '').replace(/^https?:\/\//, '').replace(/^git@([^:]+):/, '$1/')
    }
  } catch {}
  return resolve(cwd)
}
```

### 5.4 改造 `extraction-runner.ts`

当前 `extractEventMemoryItems(input: EventExtractionInput)` 函数本身（LLM 调用 + zod 校验 + 关键词降级）**不要动**——它是这套设计的精华。

把这个新入口在 `session-runtime-service.ts` 调用 `compact` 的同位置接上（每次 turn 完成后）。

## 九、把握不准时怎么办

1. **schema 设计觉得有改进空间** → 先按指令的来，第二件做召回时再调整。schema 改动是廉价的（migration 可以用 `ALTER TABLE` 或重建库——单人无历史数据）
2. **deriveProjectId 在边缘情况怎么办**（无 .git / 无 remote / 多 remote） → 按指令的回退顺序来，复杂逻辑留到后续
3. **better-sqlite3 在 monorepo 编译失败** → SQLite 是 native 模块，可能需要 `pnpm rebuild`。如果还失败，停下来报告
4. **既有 PG 测试因为引入 SQLite 而互相干扰** → 把 SQLite 测试用 `:memory:` 模式，不应该影响 PG 测试。如果还冲突，报告
