## 一、事故现象（kira 复现的真实数据）

kira 在跑 [20260508-7 第二件] 的产品级冒烟时，第 2 个新会话第 3 个问题（"ConversationArea 那个 bug 我说要什么时候修？"）触发以下错误：

```
400 This model's maximum context length is 1048576 tokens.
However, you requested 1283839 tokens (1251839 in the messages, 32000 in the completion).
Please reduce the length of the messages or completion.

最近一次工具错误：会话不存在: current。
如果这是查询类任务，请补充更准确的表名、字段名或筛选条件，或者先让我探查可用 schema。
```

**关键事实**：

- 模型上下文窗口 **1,048,576 tokens**（1M）
- 请求大小 **1,283,839 tokens**（1.25M），超 22.5%
- messages 部分 **1,251,839 tokens**（1.19M）—— 几乎全部都在 messages 里
- 工具错误 `会话不存在: current` 同时出现 —— 可能是同一根因
- 这次会话只跑了 8 轮左右，正常应在 5K~50K tokens 量级 —— **超出预期 100 倍**

## 三、首要任务：精确定位炸 token 的来源

不要立刻改代码——先用一份 **诊断脚本** 找出 1.25M tokens 是怎么积起来的。

### 3.1 加临时 token-by-section 日志

在 `backend/src/agent/agent-runner.ts` 中调用 `agentLoop` 之前（参考 `:126` 附近），加一段**临时**诊断日志（不要 commit 到主分支，但用一个分支 `debug/context-bloat-investigation` 提交）：

```ts
import { logger } from '../utils/logger.js'

// 临时诊断：dump 每段 prompt 的字符数和粗估 token 数
function debugDumpContext(systemPrompt: string, contextMessages: AgentMessage[]) {
  const estimateTokens = (s: string) => Math.ceil(s.length / 3.5)

  logger.warn('[DEBUG_CONTEXT_BLOAT] system prompt', {
    chars: systemPrompt.length,
    tokens_est: estimateTokens(systemPrompt),
  })

  contextMessages.forEach((msg, idx) => {
    const content = typeof msg.content === 'string'
      ? msg.content
      : JSON.stringify(msg.content)
    logger.warn('[DEBUG_CONTEXT_BLOAT] context msg', {
      idx,
      role: msg.role,
      chars: content.length,
      tokens_est: estimateTokens(content),
      preview: content.slice(0, 100),
    })
  })

  const totalChars = systemPrompt.length
    + contextMessages.reduce((sum, m) => {
      const c = typeof m.content === 'string' ? m.content : JSON.stringify(m.content)
      return sum + c.length
    }, 0)
  logger.warn('[DEBUG_CONTEXT_BLOAT] total', {
    chars: totalChars,
    tokens_est: estimateTokens(totalChars),
  })
}

// 调用 agentLoop 之前
debugDumpContext(systemPrompt, contextMessages)
```

### 3.3 期望诊断结果可能指向的根因（候选清单）

根据 [DEBUG_CONTEXT_BLOAT] 输出，最大的那一段就是根因。下面是可能性排序：

| 可能根因 | 诊断特征 | 修复方向 |
|---|---|---|
| A. memory_recall 注入失控 | system prompt 单段 >100K tokens，且其中包含 `<LAYER:memory_recall>` | 给召回加 token budget |

## 四、紧急已知 bug：`会话不存在: current` 工具错误

这是一个**独立 bug**，但可能跟 token 爆炸**同源**。

### 4.1 排查路径

```bash
grep -rn "会话不存在" backend/src --include="*.ts"
grep -rn "'current'" backend/src/agent/tools --include="*.ts"
```

定位到工具实现，看 `current` 是怎么被传进来的。可能性：

- agent 自主决定用 `current` 作为 sessionKey 占位（看 system prompt 里是否提到这种用法）
- 某个工具描述里写了 `current` 作为示例值，agent 照着用

### 4.2 修复

如果工具支持 `current` 关键字，那应该把它解析成实际的 session：

```ts
// 在 sessions_history / sessions_send 等工具里
if (sessionKey === 'current' || sessionKey === '__current__') {
  sessionKey = currentRuntimeSessionKey  // 从注入的 runtime 里取
}
```

如果工具不支持 `current`，那应该在工具 description 里**明确禁止**这个值，让 agent 永远不传它。

但**注意**：这个工具错误本身可能不是 1.25M token 的根因——错误提示只是在最后追加显示。token 爆炸更可能是被工具**返回的内容**触发，而不是错误信息本身。

## 五、修复策略（按诊断结果分支）

### 5.1 如果诊断指向"召回注入失控"（A）

在 `searchForRecall` 调用方（`prompt-injector.ts`）加一个**总 token 预算** 上限：

```ts
const RECALL_TOKEN_BUDGET = 1500   // 整个 memory_recall 块最多 1500 tokens
```

实现：
1. `searchForRecall` 返回 top K 条（K=5）
2. 拼装时按 score 顺序累计字符数，超过 `RECALL_TOKEN_BUDGET * 3.5` 就停
3. 至少保证最高分的那 1 条能进去（即使它本身就超预算，也只截断它）

### 5.2 如果诊断指向"工具返回过大"（B/C）

定位具体工具，给它加输出截断或分页：

- `sessions_history`：默认只返回最近 N 条（N=10），超过提示用户用游标
- `sessions_list`：默认只返回最近活跃的 N 个 session
- `read_file` / `bash`：检查现有 50K 截断是否真的生效

### 5.3 如果诊断指向"file system fallback"（D）

`store.ts:loadMemoryInjectionText` 加 token budget 截断：

- 当前 `truncateToBudget` 是按字符数截断的吗？budget 设了多少？
- 加一个总上限：整个文件 fallback 不超过 2000 tokens

### 5.4 如果诊断指向"compact 失效"（E）

检查：
- compact 阈值 `COMPACT_TRIGGER_MESSAGE_EVENTS = 50` 是否在你这次冒烟里被触发了
- 如果未到 50 条，但单条消息巨大，应该有 token-aware 触发（按 token 数而非消息数）—— 看 `architecture-evolution.md` 里 Phase 2 token-aware 触发策略是否落地

### 5.5 如果诊断指向"自递归"（F）

这是最 nasty 的——找到拼装环节的 bug。常见情况：

- system prompt 包含 `MEMORY.summary.md`，而 compact 写入 `MEMORY.summary.md` 时把当前的 summary 也带上了
- 形成"summary of summary of summary..." 递归

## 六、不要做什么

- **不要**盲目改 token 上限（如把模型 max_tokens 调成 4M）—— 那是治标不治本
- **不要**在没有诊断输出前盲目改任何代码 —— 你不知道根因在哪
- **不要**禁用 SQLite 路径就完事 —— kira 已经做了 `LECQUY_MEMORY_DISABLED=true` 紧急回滚，但产品体验降级到没有"懂我"，必须找到真正根因
- **不要**触碰 `extraction-runner.ts` —— 写入路径在第 1 件已稳定
- **不要**触碰前端
