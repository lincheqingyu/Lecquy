### 2.1 生产链路（已审查确认）

```

  → buildContextMessages(bound, mode, input)

```

`session-v2/SessionService` / `ws/simple-handler.ts` / `ws/plan-handler.ts` / `agent/message-utils.ts` **是死代码**，不在生产链路上。本次不动它们。

### 2.2 `transformContext` 钩子未被使用

`agent-runner.ts:128-168` / `manager-runner.ts:153-192` / `worker-runner.ts:209-269` 三处 `agentLoop(...)` 调用，config 都**只设了** `convertToLlm`（role 过滤），**没设** `transformContext`。

`grep -rn "transformContext" backend/src` 全工程零命中。

### 2.3 单 turn 内累积无任何降压

`pi-agent-core/dist/agent-loop.js:99-105` 每个 tool 执行完 `currentContext.messages.push(result)`。
`agent-loop.js:134-141` 每 turn 把 `currentContext.messages` 全量过 `convertToLlm` 直接送 LLM。
`MAX_ITERATIONS = 10`、`TOOL_OUTPUT_LIMIT = 50_000` → 单 turn 最坏累积 ~500k 字符。
glm-4.7 中文场景 ~1.5 char/token → ~333k tokens，远超 128k window。

### 2.4 `applyCompactionIfNeeded` 在 finally 跑

`session-runtime-service.ts:1531-1539` 在 `executeRun` 的 `finally` 块调用。
**只能事后清理，无法预防 turn 内爆炸。**
触发条件 `compact.ts:21 COMPACT_TRIGGER_MESSAGE_EVENTS = 50` 是消息数，对工具型短对话不灵敏。

### 2.5 上传层无体积上限

`session-runtime-service.ts:430-470 createUserContent` 直接吃 `attachment.data`（image base64）和 `attachment.text`（file 文本），**无大小校验、无截断**。

`pi-session-core/session-manager.ts:115-117` 在 LLM 投影时也保留完整 image base64 进 LLM。

### 2.6 跨 turn 的"工具失忆"是有意设计

`pi-session-core/session-manager.ts:142` `createContextAssistantMessage` `.filter(part => part.type === 'text')` 把所有 toolCall 块从 LLM 投影里砍掉。**这是设计选择，不是 bug。**本次不动。

## 3. 不要做什么

- **不要**改 `pi-agent-core` 上游代码——它给我们留了 `transformContext` 钩子，照用就行。
- **不要**在 `transformContext` 里调 LLM——in-loop 必须零 LLM 调用，纯规则。
- **不要**在 in-loop 压缩里对图片 base64 做"重新编码 / 缩放"——直接整块替换为占位文字。
- **不要**让 in-loop 压缩 mutate 入参 `messages`——必须返回新数组。
- **不要**把"截断后的 LLM 视图"写回事件树——事件树永远存原始数据，截断只发生在 LLM 视图。
- **不要**碰 `memory/coordinator.ts` / `memory/compact.ts` 的 LLM 摘要逻辑——本次只改触发条件（T2），不改摘要本身。
- **不要**修改 `agent/types.ts:TOOL_OUTPUT_LIMIT`——50k 单条上限保留。
- **不要**删 `session-v2/` 任何文件——T4 后续做。
- **不要**改 frontend——上传上限只在后端 WS 入口校验，前端体验问题留给单独任务。
- **不要**为了"以后扩展"在 in-loop compactor 里加策略接口、插件机制、配置 DSL——能写 inline 就写 inline。

## 4. 动手清单

### T1（P0）— 实现 in-loop compactor 并接入三个 runner

#### 4.1 新建文件 `backend/src/runtime/context/in-loop-compactor.ts`

**契约**：

```ts
import type { AgentMessage } from '@mariozechner/pi-agent-core'

export interface InLoopCompactorOptions {
  /** 总字符数超过此阈值才启动压缩 */
  readonly maxTotalChars: number
  /** 末尾 K 个 assistant turn 内的消息保持原样不动 */
  readonly keepRecentTurns: number
  /** 单条 toolResult 超过此字符数才被替换；小于的不动 */
  readonly minToolResultChars: number
}

export const DEFAULT_IN_LOOP_OPTIONS: InLoopCompactorOptions = {
  maxTotalChars: 80_000,        // ≈ 20k tokens 留给历史
  keepRecentTurns: 2,
  minToolResultChars: 2_000,
}

/**
 * In-loop 上下文压缩
 *
 * - 纯函数，不调 LLM，不修改入参
 * - 总字符数低于阈值则返回原数组（同引用）
 * - 超过阈值时，把"末尾 keepRecentTurns 个 assistant turn 之前"的
 *   大 toolResult 替换为占位文字，把同区间的 image 块替换为占位文字
 * - 末尾 K 个 turn 一律不动
 *
 * 返回值：要么是入参原数组（无操作），要么是新数组（有替换）
 */
export function compactInLoop(
  messages: AgentMessage[],
  options: InLoopCompactorOptions = DEFAULT_IN_LOOP_OPTIONS,
): AgentMessage[] { /* ... */ }
```

**实现要点**（按顺序，不要颠倒）：

1. **估算总字符数**：照搬 `session-v2/session-pruner.ts:4-11 estimateChars` 的写法（`typeof content === 'string' ? .length : JSON.stringify(content).length`）。**不要** import session-v2，直接 inline 这个 helper。
2. **早退**：总字符数 < `maxTotalChars` 直接 `return messages`（同引用）。
3. **找 cutoff 索引**：从尾向头扫，每遇到一个 `role === 'assistant'` 就计数 +1，到 `keepRecentTurns` 时记录 cutoff 索引。`cutoff` 之后（含）的消息一律保留原样。
4. **构造新数组**：
   - `cutoff` 之后：原引用
   - `cutoff` 之前：
     - `role === 'toolResult'` 且 `estimateChars(m) >= minToolResultChars` → 替换 `content` 为 `[{ type: 'text', text: '[tool_result truncated: tool=<toolName> id=<toolCallId> originalChars=<n>]' }]`
     - `role === 'user'` 且 content 数组里有 `type === 'image'` 块 → 替换 image 块为 `{ type: 'text', text: '[image removed from in-loop context: <name?>, <bytes>B]' }`，文本块原样
     - `role === 'assistant'` 且 content 数组里有 `type === 'toolCall'` 块 → 不动 toolCall 块本身（agent-loop 还要靠 id 配对），但如果 `output` 字段超长，把 `output` 字段截断到 200 字符 + `... (truncated)`
     - 其他情况 → 原引用

5. **保护**：`keepRecentTurns + 1` 个 assistant 不存在时（即历史不足），直接 `return messages` 不压缩。

**测试**（同目录 `in-loop-compactor.test.ts`）：

最少 4 个用例：
- a) 总字符 < 阈值 → 返回原引用（用 `assert.strictEqual` 验证引用相等）
- b) 10 个 toolResult × 50k → 返回新数组，前 8 个被替换，末 2 turn 原样
- c) 中间夹一个 image base64 user → image 被替换为占位
- d) 边界：`keepRecentTurns + 1` 个 assistant 不存在 → 返回原引用

### T3（P1）— 上传体积上限（双层）

#### 3.1 入口校验（拒绝过大）

**`backend/src/types/api.ts`**（zod schema）：在 attachment schema 里加 max 校验。常量：

```ts
export const MAX_IMAGE_BYTES = 5 * 1024 * 1024            // 5 MB base64
export const MAX_FILE_TEXT_CHARS = 200_000                 // 200k chars
export const MAX_TOTAL_ATTACHMENT_BYTES = 8 * 1024 * 1024  // 8 MB per message
```

校验位置：runStartSchema → modelOptions.attachments → 单条 attachment 的 `data?.length` / `text?.length`，超出直接 zod refine 失败。

**`backend/src/ws/chat-ws.ts:166-196 run_start`** 的 `runStartSchema.safeParse` 会自动用上面的 schema，**不需要**额外代码——只要 schema 正确就行。

错误返回 `code: 'ATTACHMENT_TOO_LARGE'`，message 写明哪个超 / 上限多少 / 实际多少。

#### 3.2 内部静默截断（兜底）

**`backend/src/runtime/session-runtime-service.ts:430-470 createUserContent`**：

- 对每个 image attachment：如果 `attachment.data.length > MAX_IMAGE_BYTES`（不应该走到这里，但兜底），不进 LLM 内容，替换为 `{ type: 'text', text: '[image dropped: oversized <bytes>B > <MAX>B]' }`。
- 对每个 file attachment：如果 `attachment.text.length > MAX_FILE_TEXT_CHARS`，截断到 `MAX_FILE_TEXT_CHARS` 并在末尾加 `\n... (truncated, original <n> chars)`，并 `truncated: true`。

**不要**抛异常——这一层是 last resort，应该静默且记录 `logger.warn`。

#### 3.3 测试

`session-runtime-service` 暂无单元测试基础设施，可以在 `backend/src/types/api.test.ts`（如果不存在则新建）里加 schema 校验测试：

- 单图 6 MB → reject
- 单文件 250k chars → reject
- 总和 9 MB → reject
- 边界值通过

`createUserContent` 单测如果搭起来代价大，跳过——schema 层防住就够。
