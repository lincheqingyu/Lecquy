## 四、技术决策

### 4.1 召回排序公式（这是产品核心）

最终排序分数（高到低）：

```
score = 0.40 * bm25_norm
      + 0.25 * importance_norm
      + 0.25 * recency_decay
      + 0.10 * project_match_bonus
```

- `bm25_norm` = SQLite FTS5 的 BM25 分（已是越大越相关）；如果当前轮没有 query 文本（冷启动），此项为 0
- `importance_norm` = `importance / 10`（事件 schema 里 1-10）
- `recency_decay` = `0.5 ^ (days_ago / 30)` —— 30 天半衰期，30 天前的事件分数砍半，60 天再砍半
- `project_match_bonus` = `project_id == current_project_id ? 1.0 : 0.0`

实现位置：在 `sqlite-store.ts` 新增 `searchForRecall(opts: RecallOptions): MemoryItemRow[]` 函数，**不要**改既有的 `searchMemoryItems`。

### 4.2 项目维度的"软优先"语义

不是"只看本项目"，是"本项目优先 + 全局兜底"。具体策略：

1. 先按 `project_id == current` 跑一次召回，取 top K 条
2. 如果不足 K 条，剩余名额从全局（不限项目）补
3. 最后按合成分数重排

这样既保证项目维度的强相关性，又不至于在跨项目场景下完全失忆。

### 4.3 冷启动 query 处理

新会话第一轮用户消息可能很短或不在 FTS5 词典里。当 `userQuery.trim().length < 2` 时：

- **不要**走 BM25 路径（FTS MATCH 会报错）
- 改为按 `recency + importance + project_match` 的元数据召回（无 query）

### 4.4 关键参数（供调试时调）

```ts
const MEMORY_RECALL_TOP_K = 5             // 已有，保持
const RECALL_RECENCY_HALF_LIFE_DAYS = 30  // 时间衰减半衰期
const SCORE_WEIGHTS = {
  bm25: 0.40,
  importance: 0.25,
  recency: 0.25,
  projectMatch: 0.10,
}
```

把这些放在 `sqlite-store.ts` 顶部作为可调常量。

## 五、动手清单

### 5.1 在 `sqlite-store.ts` 新增 `searchForRecall` 函数

API 契约：

```ts
export interface RecallOptions {
  readonly currentProjectId?: string  // cwd 推导出的项目 id
  readonly userQuery?: string          // 当前轮用户消息文本（可能为空）
  readonly limit?: number              // 默认 MEMORY_RECALL_TOP_K
  readonly sinceDays?: number          // 可选：只看最近 N 天
}

export function searchForRecall(opts: RecallOptions): MemoryItemRow[]
```

**实现要点**：

1. 用一个 SQL 一次性算出合成分数（用 SQLite 内置的 `(julianday('now') - julianday(occurred_at))` 算 days_ago）
2. **不**用 application 层做合成——SQLite 算更快，且能直接 ORDER BY
3. 没有 query 时跳过 FTS JOIN，直接走 metadata 排序
4. 有 query 时 LEFT JOIN FTS（确保即使 FTS miss，metadata 高分项仍被召回）

**SQL 草稿**（仅参考，落地按更优写法调整）：

```sql
SELECT
  memory_items.*,
  (
    0.40 * COALESCE(NULLIF(-bm25(fts), 0), 0) / NULLIF(:max_bm25, 0)
    + 0.25 * (memory_items.importance / 10.0)
    + 0.25 * power(0.5, (julianday('now') - julianday(memory_items.occurred_at)) / 30.0)
    + 0.10 * CASE WHEN memory_items.project_id = :current_project_id THEN 1.0 ELSE 0.0 END
  ) AS composite_score
FROM memory_items
LEFT JOIN memory_items_fts fts ON ... (when query)
WHERE memory_items.status = 'active'
ORDER BY composite_score DESC
LIMIT :limit
```

**注意**：SQLite 没有内置 `power` 函数——better-sqlite3 可以注册 user-defined functions，或者在 application 层为每行后处理。建议**在 better-sqlite3 注册一个 JS function `pow_decay(days, half_life)`** 让 SQL 直接调用，避免 application 层后处理排序。

### 5.2 改造 `prompt-injector.ts`

**当前代码**（`buildMemoryRecallMessages` 函数）：

```ts
if (args.pgEnabled && normalizeWhitespace(args.userQuery).length >= 2) {
  // ... PG 召回 ...
}
if (!recallText.trim()) {
  recallText = await loadMemoryInjectionText(args.workspaceDir)
}
```

**改造后**：

```ts
// 优先：SQLite 召回（默认开启，除非环境变量 LECQUY_MEMORY_DISABLED=true）
if (process.env.LECQUY_MEMORY_DISABLED !== 'true') {
  try {
    const projectId = deriveProjectId(args.workspaceDir)
    const items = searchForRecall({
      currentProjectId: projectId,
      userQuery: args.userQuery,
      limit: MEMORY_RECALL_TOP_K,
    })
    recallText = formatMemoryRecallBlock(items)
  } catch (error) {
    logger.warn('SQLite memory recall 失败，已回退为文件系统 recall', {
      sessionId: args.sessionId,
      error: error instanceof Error ? error.message : String(error),
    })
  }
}

// 降级 1：PG legacy（仅当 MEMORY_PG_LEGACY=true，第三件会删）
if (!recallText.trim() && process.env.MEMORY_PG_LEGACY === 'true') {
  // ... 既有 PG 召回逻辑保留 ...
}

// 降级 2：文件系统 fallback（保留作为最后防线）
if (!recallText.trim()) {
  recallText = await loadMemoryInjectionText(args.workspaceDir)
}
```

注意：

- `pgEnabled` 字段从对外接口里**保留但忽略**——不要改 `BuildMemoryRecallMessagesArgs` 类型签名（外面有调用方），但在函数内不再用它判断 SQLite 路径
- 加一个新的内部 helper 函数包装 SQLite 召回，让函数本体保持可读

当前模板格式（猜测，按实际改）：

```

```

- 项目名取 `projectId` 的最后一段（例：`xingtaokai/lecquy` → `lecquy`，`/Users/hqy/Documents/zxh/projects/Lecquy` → `Lecquy`）
